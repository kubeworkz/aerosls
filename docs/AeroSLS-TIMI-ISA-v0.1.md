# AeroSLS TIMI: Rudimentary Instruction Set — Design Plan v0.2

**Status:** Draft for review — encoding, comparison model, ABI, and numeric semantics now resolved (see §3, §4, §8 changelog); floats confirmed deferred.
**Scope:** Core execution instruction set only (arithmetic, control flow, memory/pointer ops, calls). Object-typing, capability/authority instructions, and template-based translation hooks are explicitly deferred to later phases (see §7).

---

## 1. Where this fits

AeroSLS is building toward a System/38-style stack: Single Level Storage (SLS) as the addressing foundation, TIMI as the technology-independent instruction layer that sits between compiled application objects and the native hardware, and SLIC as the licensed-internal-code layer that implements TIMI and manages the machine underneath it. LPAR-style partitioning is a longer-term goal that depends on both being solid.

This document only covers step one: what a TIMI *instruction* looks like. Nothing here about object types, authorities, or partitioning — those come after there's a working bytecode to build on.

### 1.1 What we're borrowing, and from where

Two IBM lineages matter here, and they pull in different directions:

- **System/38 MI / TIMI** — the *architectural role*: a virtual ISA that is never interpreted. It is generated by compilers, embedded in the program object alongside (or instead of) native code, and translated to the host machine's real instructions before first execution. If the object moves to a machine with a different native ISA, it gets re-translated from the embedded MI/TIMI, not the native code. That's the "technology independence" property we want.
- **PL.8's intermediate language ("W-code" in the prompt's sense)** — the *instruction shape*: PL.8 compiles to "an intermediate machine-independent language with symbolic registers and machine-like operations" before its backend maps it to a real register file and ISA (801, S/370, POWER, etc.). That's a three-address, symbolic-register RISC-like IR — closer to LLVM IR or a load/store RISC ISA than to a stack machine.

These are not the same thing architecturally. Actual System/38 MI instructions are *not* simple fixed-format RISC ops — they're variable-length, opcode-plus-up-to-four-operand instructions where each operand is an index into an Object Definition Table (ODT) that supplies type, size, and location out of band. That's what makes MI opcodes generic (one `ADD NUMERIC` handles every numeric type — the ODT tells the machine how). It's powerful but heavyweight for a first cut.

The plan below takes System/38's *role* (embedded, translated-not-interpreted, technology-independent) but gives TIMI v0.1 PL.8-style *instruction shape* (fixed-format, symbolic-register, three-address, RISC-like) instead of the full ODT/template machinery. That's what "rudimentary" and "resembling W-code" both point at. Full ODT-style generic typing is a §7 phase, not v0.1.

## 2. Design principles for v0.1

1. **Never interpreted, always translated.** TIMI code is a compile-time artifact. SLIC translates it to native machine code before the object runs — at install time, first invocation, or on move to new hardware. There is no TIMI interpreter loop in the execution path (a *reference interpreter* is still useful for testing — see §6 — but it's a development tool, not part of the production architecture).
2. **Symbolic registers, not a stack.** Instructions name an unbounded set of symbolic (virtual) registers, not stack slots. SLIC's translator is responsible for allocating these to whatever real registers the host provides. This matches PL.8's IL and is what makes the format "W-code-like" rather than "p-code/JVM-like."
3. **Fixed-width, three-address encoding.** Every instruction is the same size and shape: opcode, up to three register operands, optional immediate. No variable-length operand streams, no out-of-band descriptor tables in v0.1. This is the single biggest simplification relative to real MI, and it's deliberate — it makes the assembler, translator, and object format all trivial to bring up first.
4. **Minimal but real typing.** Each register operand carries a small type tag (integer width, pointer, float) inline in the instruction rather than in a separate ODT. Enough to make load/store and arithmetic well-defined; not enough to be "generic opcodes" yet. That richer, MI-style out-of-band typing is exactly what §7 phases in.
5. **Embedded in the OS object, alongside a translation record.** Every program object carries its TIMI instruction stream plus enough metadata (entry points, register count, symbol/relocation info) for SLIC to regenerate native code without the original compiler. This is what gives you the System/38 "re-translate on new hardware" property.
6. **Addressing is SLS-native.** TIMI has no notion of physical addresses or segments-and-offsets in the traditional sense — all memory operands are single-level-store pointers. Pointer arithmetic and load/store instructions operate on that pointer model from day one, even before capability/authority checking exists.

## 3. Instruction encoding

Fixed 64-bit instruction word. v0.2 tightens the register fields from v0.1's placeholder 16-bit width down to 10 bits, and spends the reclaimed bits on a much wider inline immediate (see §8 changelog for the reasoning).

```
 63        56 55   52 51        42 41        32 31                          4 3   0
+-----------+-------+-------------+-------------+------------------------------+----+
|  opcode   | type  |    rD       |     rA       |          rB / imm           |flg |
|  (8 bits) |(4 bit)|  (10 bits)  |  (10 bits)   |          (28 bits)          |(4) |
+-----------+-------+-------------+-------------+------------------------------+----+
```

- **opcode (8 bits):** up to 256 instructions; §4 defines the v0.1/v0.2 subset (~34).
- **type (4 bits):** operand type tag — `i8 i16 i32 i64 u8 u16 u32 u64 f32 f64 ptr bool` (12 used, 4 reserved for later phases). Applies to the instruction's operation, not each register independently. Signed vs. unsigned integer variants of an opcode (e.g. `DIV`, `MOD`, arithmetic vs. logical shift eligibility) are selected by this tag, not by separate opcodes — see §4.7.
- **rD, rA (10 bits each):** symbolic virtual register numbers, 0–1023, **scoped per procedure** (renumbered from 0 at each `ENTER`, not a global SSA id). See §8 Q1 for why 1024 is enough and what happens when it isn't.
- **rB / imm (28 bits):** dual-purpose, selected by a flag bit:
  - *Register mode:* low 10 bits are `rB` (0–1023); upper 18 bits reserved, must be zero.
  - *Immediate mode:* full 28 bits, sign-extended, giving a range of roughly ±134 million — enough for the overwhelming majority of constants and load/store displacements without touching the literal pool. Values that don't fit still go through `LOADI64`'s two-word literal-pool form.
- **flags (4 bits):** per-opcode-class modifiers — register/immediate select, relation code on `CMP`, branch-invert bit on `BC`, signed/unsigned or volatile/atomic markers elsewhere. See §4 for per-opcode assignment.

This is intentionally closer to a MIPS/RISC-V-style fixed encoding than to MI's variable-length ODT-indexed stream — that's the "lower-level, W-code-like" trade-off the plan calls for. It is *not* meant to be the final encoding; §7 covers the migration path once generic/typed opcodes are needed.

## 4. Core opcode classes (v0.1)

### 4.1 Arithmetic / logical
`ADD SUB MUL DIV MOD` (int + float variants via type tag), `AND OR XOR NOT`, `SHL SHR SAR`, `NEG`.

### 4.2 Data movement
`MOV` (register-register), `LOADI` (immediate/literal), `LOADI64` (wide literal, two-word form).

### 4.3 Comparison — resolved: separate, register-based, no condition-code register
`CMP rD, rA, rB` — **no hidden flags/condition-code register.** TIMI has no implicit machine state (consistent with design principle #2 — everything is a named symbolic register), so `CMP` writes an ordinary `bool`-typed result into `rD`. The 4-bit flag field selects the relation being tested: `EQ NE LT LE GT GE LTU LEU GTU GEU` (10 codes used, 6 reserved). Signed vs. unsigned is therefore selected by the relation code for `LT/LE/GT/GE` vs. `LTU/LEU/GEU/GTU`, independent of the operand type tag.

Keeping `CMP` and `BC` as two instructions (rather than a fused compare-and-branch) was the deliberate choice: the result register can be reused by multiple downstream branches, and it gives SLIC's translator freedom to schedule the comparison away from the branch, or to lower it onto a host's hardware condition codes, register-compare, or predicate-register model as appropriate for that target — the IR itself stays target-agnostic either way.

### 4.4 Control flow
`BR` (unconditional, PC-relative via imm), `BC rA, imm` (conditional — branches if `rA` is nonzero, or if `rA` is zero when the invert flag bit is set; `rA` is normally a `CMP` result but any nonzero register works, matching common `BEQZ`/`BNEZ`-style RISC idioms), `CALL` (symbolic target — resolved by SLIC at translation time against the object's entry-point table), `RET`.

### 4.5 Memory / pointer
`LOAD`, `STORE` (both take an SLS pointer register + signed displacement via rB/imm — now up to 28 bits, see §3), `LEA` (compute pointer = base + offset without dereferencing), `PTRADD` (pointer + integer, typed by element size — placeholder until §7's real pointer/type model lands).

Pointers are a fixed **64-bit** width (matches the SLS single-level virtual address precedent). Instruction words, the literal pool, and all in-object data are encoded **little-endian** — an arbitrary but now-fixed choice, stated here so the object format and any cross-tool (dis)assembler agree.

### 4.6 Procedure frame
`ENTER` / `LEAVE` — bracket a procedure body, tell SLIC how many symbolic registers (≤1024, see §8 Q1) and how much local storage the translator needs to provision. Not "real" instructions in the ALU sense; more like directives, but keeping them in-stream (rather than in a side table) keeps the "everything is a flat instruction stream" property intact.

### 4.7 Numeric semantics — resolved
- **Overflow:** `ADD/SUB/MUL/NEG` wrap silently in two's-complement, matching plain hardware ALU behavior and most RISC ISAs. No trap. A checked/trapping-arithmetic variant is deferred to a later phase (flag-bit variant or new opcode) if AeroSLS ever needs it.
- **Division:** `DIV`/`MOD` by zero, and signed `MIN_INT / -1`, are **trap conditions**, not wraparound — division can't produce a meaningful wrapped result. The exact trap/exception delivery mechanism depends on SLIC's activation and exception infrastructure, which doesn't exist yet (Phase 4); this section fixes the *contract* now so Phase 1's reference interpreter and Phase 3's translator implement the same behavior from day one instead of drifting apart.
- **Shifts:** `SHL` (left, fills zero) is unambiguous either way. Logical vs. arithmetic right-shift are kept as **separate opcodes** rather than derived from the type tag: `SHR` (logical, zero-fill — valid on any type) and `SAR` (arithmetic, sign-extending — valid only on signed integer types; the assembler should reject `SAR` on a `u*` type tag rather than silently reinterpreting it). Shift amount comes from `rB`/imm, masked to 6 bits (0–63) regardless of operand width.

### 4.8 Calling convention (ABI) — resolved for v0.1/v0.2
Simplest workable option, chosen over adding dedicated argument-marshaling opcodes: **`r0`–`r7` are a reserved, caller-saved argument/return convention by documentation, not by hardware.** Up to 8 integer/pointer arguments go in `r0`–`r7` before `CALL`; the return value comes back in `r0` after `RET`. Anything beyond 8 arguments is passed via frame-local memory (`STORE` into the callee's incoming stack area, mirroring what every real ABI does once it runs out of registers) — exact frame-argument layout is a Phase 1 assembler/ABI-doc detail, not an ISA encoding concern.

This was chosen over first-class `SETARG`/`GETPARAM`/`SETRET`/`GETRET` opcodes to keep the opcode count down for a "rudimentary" v0.1 — it costs SLIC's register allocator a documented convention it must respect around call sites (values live across a call into `r0`–`r7` need to be treated as caller-saved), but adds zero new instructions. If that convention proves awkward once real procedures are being compiled (e.g. it collides badly with the per-procedure renumbering in §8 Q1), promoting to explicit marshaling opcodes is a candidate for Phase 6, not a v0.2 blocker.

### 4.9 Fixed-point math (interim substitute for floats)
Since floats are deferred (§8 Q3), fixed-point is available now as a **pure software convention on top of existing integer opcodes — no new instructions required**:
- Represent fractional values as `Q16.16` in an `i32` register or `Q32.32` in an `i64` register (scale factor is a frontend/compiler convention, not something TIMI encodes).
- `ADD`/`SUB` work unmodified — fixed-point addition is just integer addition at a common scale.
- Multiply: `MUL` the two scaled values, then `SAR` by the fractional bit count (16 or 32) to bring the result back to the right scale.
- Divide: `SHL` the dividend by the fractional bit count first, then `DIV`, to avoid losing fractional precision.
No `MULFX`/`DIVFX` opcodes are being added in v0.1 — the two-instruction idiom above is enough to unblock fractional math for now. If profiling later shows that idiom is a hot-path cost, a fused fixed-point multiply/divide pair is a small, self-contained Phase 6 addition.

That's roughly 32–34 opcodes — enough to translate straight-line arithmetic, conditionals, loops, and non-virtual procedure calls with a real (if minimal) ABI. No object creation, no capability/pointer-authority instructions, no floating point, no generic/ODT-typed ops — all explicitly out of scope for v0.1/v0.2 per §7.

## 5. Object embedding format (sketch)

A TIMI-bearing OS object needs, at minimum:

1. **Header** — object type, TIMI version, entry point count, symbolic register count (or "high water mark"), literal pool size.
2. **TIMI instruction stream** — the fixed-width words from §3, back to back.
3. **Literal pool** — wide constants (`LOADI64` targets, float constants).
4. **Entry point table** — symbolic name → instruction offset, for `CALL` resolution both within the object and from other objects.
5. **Relocation/reference table** — external symbols the translator must bind at translate time (calls out to other objects, referenced by name/pointer, not by resolved address — consistent with SLS single-level addressing).
6. **(Optional, post-translation) Native code cache** — the last-translated native instructions for the current host, plus a fingerprint of that host, so SLIC can skip re-translation when nothing has changed. This is what System/38-style hardware migration invalidates and regenerates.

This is deliberately thinner than System/38's Program Template (instructions + ODT + user data) — there's no ODT because v0.1 has no out-of-band operand descriptors. Adding one is the natural extension point in §7.

## 6. Toolchain needed to validate this

A spec is only real once something can run against it:

1. **Assembler** — hand-writable text form of §3/§4, emits the §5 object format. Needed before any compiler backend exists, for hand-testing.
2. **Reference interpreter** — a *development-only* tool that directly executes TIMI for correctness testing. This deliberately violates principle #1 ("never interpreted in production") but is standard practice for bringing up a new ISA — you need a ground truth before you trust a translator.
3. **Disassembler** — for debugging translator output and object dumps.
4. **Native translator (one target first)** — SLIC-side component that walks the instruction stream and emits real machine code for a single initial host architecture. Register allocation here can start as simple as linear-scan; it doesn't need to be good, it needs to exist.
5. **Test suite** — small TIMI programs (arithmetic, loops, calls) with known outputs, run through interpreter and translator, results compared.

## 8. Open questions — resolutions (v0.2)

**Q1. Register count ceiling per procedure — resolved: 1024 (10-bit field), scoped per procedure, not global.**
The 16-bit field in v0.1 was not free — every bit spent on register width is a bit not spent on immediate range, and it bloats every single instruction word regardless of whether that instruction needs a big register number. Bit budget for the 64-bit word: 8 (opcode) + 4 (type) + 4 (flags) = 16 bits fixed overhead, leaving 48 bits for operands. v0.1 spent that as 16/16/16 (three register-sized fields, immediates capped at 16 bits). v0.2 spends it as 10/10/28: two 10-bit register fields plus a 28-bit register-or-immediate field — register ceiling drops from 65,536 to 1024, and the payoff is a ±134M-range inline immediate instead of ±32K, which cuts a lot of literal-pool traffic for everyday constants and displacements.

The reason 1024 is enough, not just "smaller": register numbers are **scoped per procedure** and reset to 0 at each `ENTER` (§4.6) — this was already implicit in v0.1 but is now stated explicitly because it's load-bearing for this answer. That means the ceiling isn't "1024 registers total across the program," it's "1024 distinct symbolic temporaries live within one procedure's numbering," which is a different and much easier bar to clear. The remaining risk is a frontend that does naive SSA numbering (fresh id per assignment, no coalescing) on a very large generated procedure. Rather than widen the encoding for that edge case, the contract is: frontends must renumber/coalesce densely before emission (standard compiler practice — real backends do this regardless of encoding width), and any procedure that still can't fit under 1024 live temporaries spills the excess to frame-local memory via ordinary `LOAD`/`STORE`, exactly like a physical register allocator does under real register pressure. No ISA change needed for that case.

**Q2. CMP/BC — resolved: kept separate, register-based (no hardware condition-code register).**
See §4.3 for the encoding. The deciding factor beyond "SSA-friendliness": TIMI's own design principle #2 already commits to no implicit machine state (symbolic registers, not a stack, not hidden flags), so a condition-code register would have been an inconsistency with the rest of the ISA, not just a stylistic choice. No code-size or hardware constraint is currently known that would push toward fusion — if one shows up once a real target's calling/branch costs are measured in Phase 3, fusing is a localized encoding change, not a redesign.

**Q3. Floats — resolved: deferred; integer/pointer-only for v0.1/v0.2.**
Three items identified as the actual remaining gaps for the integer/pointer core (now resolved above rather than left open):
1. Integer overflow/wraparound and division-trap contract — §4.7.
2. Calling convention for `CALL`/`RET`/`ENTER`/`LEAVE` — §4.8.
3. Pointer width and endianness — §4.5 (64-bit pointers, little-endian).

Fixed-point as an interim substitute for fractional math is specified in §4.9 as a pure software convention on existing opcodes — no new instructions needed for v0.1.

**Q4. Object format reuse vs. divergence from the existing SLS object header — resolved by Phase 2 (see §9).**
Looked at the real `kernel/object_catalog.h` and `kernel/loader.c` in the AeroSLS repo. Short answer: reuse the §5 layout byte-for-byte as the *payload*, don't touch the catalog entry struct itself. Details in §9.

## 9. Phase 2 findings — wiring §5 into the real AeroSLS repo

Connected the actual `aerosls2` codebase and read `kernel/object_catalog.h`, `kernel/loader.c/.h`, `kernel/process.c`, and `kernel/persist.c` rather than guessing. Findings:

**The catalog entry (`struct SLSObjectEntry`) is metadata-only** — object_id, name, type, `base_vaddr`, `size_pages`, owner/tier/role/perm, active flag. It does not carry program bytes. `OBJ_TYPE_PROGRAM` (type 7) already exists, documented as "the SLS-native replacement for a filesystem exec." Good news: no new catalog object type was needed for TIMI — it slots in as a third payload format under the type that already exists.

**Program bytes actually live in a separate staging array**, not in SLS-mapped memory: `struct ServiceBinary service_binaries[MAX_SERVICE_BINARIES]` in `loader.c` — 16 slots, 16 KiB each, uploaded in chunks via `SYS_SLS_UPLOAD_BINARY`, then mapped into a process's page table at spawn time by `loader_load_into_process()`. That function already switches on a detected format (`is_elf` magic-sniffed at upload time) between an ELF64 loader and a flat-binary loader, both of which map the bytes as **directly executable** native pages. This is the exact seam TIMI needed: a third format, detected the same way, that does *not* get mapped executable (design principle #1 — never interpreted, never run as native code, always translated first).

**What Phase 2 actually changed** (in the real repo, not just this doc):
- `kernel/loader.h` — added `TIMI_MAGIC`, `struct TimiObjectHeader` and `struct TimiEntryRec` (packed, byte-identical to the Phase 1 host tool's `.tmo` layout — verified, see below), an `is_timi` flag on `ServiceBinary`, and a `SYS_SLS_TIMI_INFO` (173) syscall number.
- `kernel/loader.c` — `timi_validate()` (magic + bounds-checked header, rejecting truncated/corrupt uploads before anything walks the arrays), TIMI detection alongside the existing ELF magic-sniff in `sys_sls_upload_binary()`, a `loader_timi_info()` introspection function, and a new branch in `loader_load_into_process()` that validates a TIMI object and then **refuses to spawn it** with a clear message — instead of falling through to the flat-binary path, which would have mapped bytecode words in as fake x86-64 instructions and crashed (or worse).
- `kernel/syscall_dispatch.c` — wired `SYS_SLS_TIMI_INFO`.
- `net/http.c` — three sites that reported `is_elf ? "ELF64" : "flat"` (JSON object listing, upload response, binary-download header) now go through a shared `binary_format_name()` helper so a TIMI upload doesn't get silently mislabeled "flat" over the HTTP API.
- `kernel/object_catalog.h` — one-line comment update on `OBJ_TYPE_PROGRAM` noting the TIMI sub-format.

**Verification:** no `x86_64-elf-gcc` cross-compiler is available in this environment, so I couldn't build the actual kernel binary. Two things I could and did check: (1) the modified/touched files (`loader.c`, `loader.h`, `http.c`, `syscall_dispatch.c`, `object_catalog.c`) all compile clean with host `gcc -ffreestanding` — not a perfect stand-in for the real cross-compiler, but it does catch type errors, undeclared identifiers, and struct mismatches; (2) I extracted `timi_validate()`'s exact logic into a standalone host program and ran it against all 6 real `.tmo` files produced by the Phase 1 assembler — all 6 validated with the correct instruction/literal/entry counts and correct entry-point names, and two deliberately corrupted files (truncated, bad magic) were correctly rejected. That's a real, tested proof that a `.tmo` from `timi-asm` is byte-compatible with the kernel's parser, not just an assertion that it should be.

**A pre-existing gap this surfaced, not caused by TIMI:** `OBJ_TYPE_PROGRAM` objects don't actually live in single-level storage today — they're staged in that bounded 16×16 KiB side array and only touch the SLS-mapped `base_vaddr` range at spawn time (as native page-table mappings, not as a persistent object). That's a real divergence from "everything is a persistent SLS object," but it predates TIMI and applies equally to ELF/flat programs — not something to silently paper over inside the TIMI work. Worth a decision later: whether Phase 3 (native translator) should also be the point where PROGRAM objects become properly SLS-resident (bytecode *and* translated-native-cache both persisted as real objects, per §5/§7 phase 4's original intent), or whether that's tracked as a separate, non-TIMI cleanup.

## 10. Phase 3 findings — the native x86-64 translator, and proof it actually works

**Design chosen:** naive load-operate-store codegen — every symbolic register is a memory slot in an rbp-relative stack frame (`TX_MAX_REGS = 64` slots, a documented v1 narrowing from the ISA's 1024 ceiling), not a real allocated host register. Every TIMI instruction becomes "load operand(s) into RAX/RCX, compute, store result back." Correct over fast, deliberately — the goal was proving the translate-and-map-and-run pipeline end to end, not a competitive JIT. Real register allocation is future work, not a blocker.

**Calling convention:** `CALL`/`RET` use real x86-64 `call`/`ret` (so nested/recursive TIMI calls get a real hardware stack for free), with r0–r7 marshalled through a small outgoing-argument area on the stack at each call site, and the callee's `ENTER`-emitted prologue reading them back into its own frame slots. This is the concrete mechanism behind §8 Q1's "register numbers reset per procedure" and §4.8's "r0–r7 argument convention" — those two answers only work together if something physically copies values across the frame boundary, and this is that something.

**Two hard problems, both resolved:**
1. *How does a translated TIMI program get a legitimate memory address to `LOAD`/`STORE` against?* TIMI has no real object/pointer allocation model yet (that's §7 Phase 6/7). Resolved with a stopgap: register **r7 is reserved by Phase 3 trampoline convention** as a pointer to a small scratch page the trampoline provides — a real mapped address, not part of the ISA spec itself. `mem_ops.timi` (Phase 1, uses address 0 — valid only in the interpreter's simulated memory) was left untouched, and `mem_ops_native.timi` was added alongside it using the r7 convention for real hardware.
2. *Where does that scratch page's address come from, when the trampoline needs to bake it in as an immediate before the page is mapped, and the mapping address depends on the translated code's size?* Resolved by translating twice — once with a placeholder to learn the byte length (the scratch pointer's numeric value never changes how many bytes get emitted, only what immediate gets baked in), then once for real with the now-known scratch address. Cheap, deterministic, no patch-after-the-fact fragility.

**Verification — this is the part worth being precise about, because "I wrote a translator" and "I proved a translator correct" are different claims:**
- No `x86_64-elf-gcc` cross-compiler or QEMU is available in this environment, so the actual kernel binary was not built or booted.
- What *was* done: the translator core (`timi_x86.c`) was written as a portable, no-libc file — the same file living at both `outputs/timi_x86.c` (host toolchain) and `kernel/timi_x86.c` (kernel), byte-identical. A host test harness (`timi-jit-test`) translates a real `.tmo` from the Phase 1 assembler, `mmap`s the output `PROT_EXEC`, and **calls it as a real function on the host CPU** — genuine execution, not a simulation or a static read of the bytes.
- First run: 5 of 6 tests passed, one crashed. The crash was real and useful — it caught a genuine ModRM byte-encoding bug (`0x81` used where `0x88` was correct, a reg/rm-field arithmetic mistake affecting every `LOAD`/`STORE` variant) that static review had missed. Fixed, re-verified: **9/9 programs pass under the Phase 1 interpreter, 8/8 applicable programs pass under real native execution** (one skipped by design — see below), covering the full opcode set: arithmetic, `DIV`/`MOD`, shifts, signed and unsigned `CMP` relations, `BR`/`BC`/`CALL`/`RET`, typed `LOAD`/`STORE`, `LEA`/`PTRADD`, `LOADI`/`LOADI64`. Two new test programs (`extra_ops.timi`, `ptr_ops.timi`) were added specifically to exercise opcodes the original six didn't touch.
- The kernel-side files (`kernel/timi_x86.c`, `kernel/timi_translate.c`, and the modified `kernel/loader.c`) all compile clean under host `gcc -ffreestanding` — not the real cross-compiler, but real evidence against type errors, undefined references, and struct mismatches, which is what actually broke last time (a `struct TxEntryRec` name collision between `loader.h` and `timi_x86.h`, caught immediately by this same check and fixed by renaming).

**What's still unverified, stated plainly:** whether this actually boots and runs correctly inside real AeroSLS — page-table mapping, the `SYS_SLS_EXIT` handoff, and the kernel's own scheduler/paging code have not been exercised by anything in this session. That requires the real toolchain and QEMU, neither available here.

**Kernel wiring:** `loader_load_into_process()`'s TIMI branch (Phase 2's "refuse to spawn" stub) now calls `timi_translate_and_map()`, which runs the translator into a static 192 KiB output buffer (no general-purpose allocator exists in this kernel — matches the codebase's existing static-array discipline), maps the resulting code executable page by page via the same `allocate_physical_ram_frame()`/`user_map_page()` calls the ELF64/flat paths already use, maps one additional R+W scratch page for r7, and appends a small kernel-specific stub (kept out of the portable translator core) that turns the translated program's return value into a `SYS_SLS_EXIT` syscall. **v1 limitation:** only a TIMI object's `.entry main` is spawnable — no plumbing exists yet to choose among multiple entry points through `program_spawn()`'s call chain.

**Known, explicitly flagged gap:** `DIV`/`MOD` by zero and signed `MIN_INT / -1` are left to real x86-64 hardware `#DE` (divide error) rather than a software-checked trap. The ISA spec's §4.7 calls these trap conditions, and hardware `#DE` is literally that — but AeroSLS doesn't yet wire an ISR0 handler for user processes, so today an unhandled `#DE` likely hangs or resets rather than delivering a clean fault. Not solved here; it's process-fault-handling infrastructure, not specifically a TIMI problem, similar in kind to the `OBJ_TYPE_PROGRAM` staging gap flagged in §9.

## 11. Phase 4 findings — activation cache (translate-on-first-use)

**What "activation" means here, concretely:** on System/38, `CREATE PROGRAM` compiles source to MI and stores it; the separate *activation* step — first call — translates MI to native machine code and caches that translated form, so a program is translated once and invoked many times, not re-translated per call. Phase 3 didn't have that split: `loader_load_into_process()` called `timi_translate_and_map()` on every single spawn, re-running the full two-pass encoder from scratch each time. Phase 4 closes that gap.

**The key architectural fact that made this simple rather than requiring a redesign:** `process.c`'s `program_spawn()` (the `OBJ_TYPE_PROGRAM` path TIMI objects use) always maps a program's code at the same fixed constant, `USER_PROC_CODE_BASE`, in every process's page table — not a per-spawn or per-process-chosen address. That's true single-level-store semantics: an object's effective address doesn't depend on who's looking at it. It also means the translated code's internal offsets and the r7 scratch-pointer `movabs` immediate baked into the trampoline (§10) are identical across every activation of the same object — nothing about the cached bytes needs to change per process. That's what makes sharing physical code frames across processes safe without any general capability for shared-memory mapping elsewhere in the kernel: this one case happens to satisfy the precondition for free.

**Design:** a bounded `struct TimiActivation g_activations[MAX_SERVICE_BINARIES]` in `kernel/timi_translate.c` (one slot per possible TIMI object, mirroring `ServiceBinary`'s own capacity — consistent with the codebase's static-array-everywhere discipline, no allocator involved). Each slot records the object name, an FNV-1a hash + byte size of the `.tmo` content it was translated from, and the physical frame addresses of the translated code (code + appended exit stub — both immutable at runtime, so both are safely shared).

- **Cache hit** (`timi_translate_and_map()` finds a slot whose name, hash, and size all match the current upload): skips the translator entirely. Loops over the cached frame array and `user_map_page()`s each one — same physical frames — into the new process's page table as R+X. Allocates one **fresh** physical frame for the scratch page (never shared: TIMI code writes through r7, so two concurrent processes sharing one scratch frame would corrupt each other's transient state) and maps it R+W+NX at the same scratch vaddr the cached code already expects.
- **Cache miss** (no slot yet, or the object was re-uploaded with different content — hash/size mismatch): runs the existing Phase 3 two-pass translate-and-stub-append pipeline unchanged, then additionally stashes the resulting frame pointers into the object's activation slot before returning, so the *next* spawn takes the hit path.
- **Invalidation:** keyed on an FNV-1a hash of the uploaded bytes (not just the object name), so re-uploading the same-named object with different bytes is correctly detected as a cache miss rather than serving stale native code. The hash is not cryptographic and doesn't need to be — the only question it answers is "did this object's bytes change since I last translated it," and an object's own owner already controls its own bytes.
- **Known, explicitly flagged gap:** a re-upload that invalidates a cache entry does not reclaim the old translation's physical frames — this kernel has no frame-free path anywhere yet (not for `ServiceBinary` slots, not for the process table, nothing), so the old frames are simply overwritten in the activation struct and orphaned, exactly like every other static-array resource here today. Not a regression Phase 4 introduces so much as the same pre-existing gap now applying to one more resource type. A real fix needs kernel-wide frame refcounting/freeing, out of scope for TIMI specifically.
- **Diagnostics:** `timi_activation_info()` reports cache state (page count, entry offset, content hash, or "not yet translated") and is now called from `loader_timi_info()`, so the existing `timi-info` report shows activation status alongside the header/entry-point dump it already printed.

**Verification:** no cross-compiler is available in this sandbox (same constraint as Phases 2–3), so the kernel binary wasn't rebuilt here. What was checked: (1) the FNV-1a hash function was copy-pasted into a standalone host test and run against all 9 real `.tmo`-source `.timi` files in `tools/timi/tests/` — deterministic (same bytes hash the same twice), zero pairwise collisions across all 9, and a single-byte mutation of one file changes its hash, confirming the invalidation trigger actually fires on content changes rather than being a no-op; (2) both modified files, `kernel/timi_translate.c` and `kernel/loader.c`, compile clean with zero warnings under host `gcc -fsyntax-only -ffreestanding -Wall -Wextra -std=c11` — the same catch-type-errors-and-undeclared-identifiers check used in Phases 2–3, which is what caught real bugs (the `TxEntryRec` collision, the `0x81`/`0x88` ModRM bug) before this session.

**What's still unverified, stated plainly:** whether a cache hit's re-mapping of shared frames into a second process's freshly-cloned page table actually works correctly on real hardware — i.e., whether spawning the same TIMI object twice concurrently produces two working processes that don't corrupt each other — has not been exercised by anything in this session. That requires the real toolchain, a running kernel, and at least two concurrent spawns of the same object, none of which are available here. The architectural argument above (fixed `base_vaddr`, private scratch frame) is sound on paper; it hasn't been run.

**Addendum — this was subsequently verified for real, and it surfaced a genuine pre-existing kernel bug unrelated to TIMI.** With the user's own toolchain and QEMU (this session still has neither), the full pipeline was exercised end to end via the real HTTP API: upload a `.tmo`, spawn it (cache MISS, translate), spawn the same object again (cache HIT, reuse frames). The first spawn worked; the second triple-faulted the VM. The serial log showed a clean exit for PID 100 (`exited (code=5)`) followed by a `program_spawn` line for PID 101 with no matching exit — QEMU's exception trace (`-d int -no-reboot`) pinned it to a General Protection Fault at a kernel address with a garbage `RSP`, cascading through a double fault into a triple fault.

Root cause, and it has nothing to do with the activation cache: `process_exit()` (`kernel/process.c`, reached via `SYS_SLS_EXIT`) restores the kernel's `CR3`/`RSP` and `ret`s straight back into `program_spawn()`'s C code, bypassing `syscall_entry_stub`'s (`arch/x86/syscall.asm`) matching `swapgs` in `.syscall_return`. `syscall_entry_stub` does `swapgs` on entry so `[gs:0]`/`[gs:8]` address this core's `PerCPUData` (`user_rsp`/`kernel_rsp`, `arch/x86/user_paging.h`); skipping the exit-side `swapgs` leaves `GS_BASE`/`KERNEL_GS_BASE` permanently swapped after the *first* process ever exits this way. The *next* process's own syscall entry then swaps them the wrong direction — `GS_BASE` becomes `0` instead of `&per_cpu_data[0]` — so `mov rsp,[gs:8]` reads from physical address `0x8` instead of the real per-CPU struct. That's exactly the garbage `RSP` in the crash dump. This is why it never showed up before: nothing in this project had ever spawned a process, had it exit via a syscall, and then spawned a second one in the same boot session until this activation-cache verification actually exercised that sequence.

Fix: one extra `swapgs` added to `process_exit()`'s inline asm, immediately before the `CR3`/`RSP` restore, undoing the entry-side `swapgs` exactly as `.syscall_return` would have. Verified afterward with two full MISS-then-HIT cycles back to back against two different objects (PIDs 100-103), all four exiting cleanly with the correct result (`code=5`). This closes the "what's still unverified" gap above for real: the activation cache's shared-frame re-mapping across independently-cloned page tables is now confirmed correct on (emulated) real hardware, not just architecturally sound on paper.

## 12. Phase 5 findings — second native target (RV64), and what "prove re-translation-on-migration works" actually means

**Scoping this against what actually exists.** The roadmap's Phase 5 line reads "second native target, to prove re-translation-on-migration actually works." Two things had to be checked before writing any code: whether this sandbox can build/run RV64 code at all, and whether the RISC-V side of the AeroSLS kernel has anywhere to wire a translator into. Neither was available: no `riscv64-unknown-elf-gcc`, no `qemu-riscv64`/`qemu-system-riscv64`, and no root to install either (same hard constraint Phase 2/3 hit for the x86 cross-compiler). More importantly, `kernel/kernel_riscv.c` is a 10-line boot stub — `process.c`, `loader.c`, and `object_catalog.c` are x86-only in the root `Makefile`'s `X86_C_SRC`; the RISC-V kernel has no process/loader/object-catalog subsystem at all yet. So Phase 5's real, honest deliverable is the host toolchain: a second translator that consumes the exact same `.tmo` bytecode format and produces correct results, proving the retargeting claim at the level where it's actually checkable right now. Kernel-side wiring is deferred, blocked on RISC-V kernel infrastructure that's a separate, non-TIMI undertaking — not something to fake by wiring into a subsystem that doesn't exist.

**Design, mirrored deliberately against `timi_x86.c`:** `tools/timi/timi_riscv.c` uses the identical naive load-operate-store model (every symbolic register is a stack slot, not a real allocated register) and the identical real-call-instruction calling convention (`jal ra,target` / `jalr x0,0(ra)` in place of x86's `call`/`ret`, still giving nested/recursive TIMI calls a real call chain), so the two translators are easy to audit side by side. A few things came out differently, forced by real ISA differences rather than by choice:
- **No movabs equivalent.** RV64 has no single instruction that loads an arbitrary 64-bit constant. Every 64-bit value the translator needs — `LOADI64` literals, the trampoline's `scratch_ptr` — goes through a literal pool appended after all code, loaded via the standard `auipc rd,%pcrel_hi; ld rd,%pcrel_lo(rd)` pattern real RISC-V toolchains use for symbol loads, resolved in a second patch pass once the pool's address is known (the same two-pass shape as the branch/call fixups, just a second independent fixup list).
- **12-bit immediates.** RV64 load/store/addi immediates are 12 bits (±2048), far short of TIMI's 28-bit displacement field. Every `LOAD`/`STORE`/`LEA`/`PTRADD` displacement is materialized into a register first (a plain `ADDI` if it fits, the literal-pool path otherwise) rather than relying on the instruction's own immediate field the way x86's disp32 could.
- **`ra` isn't pushed automatically.** x86's `call` pushes a return address for free; RV64's `jal` just clobbers `ra`. Every procedure's prologue/epilogue explicitly saves/restores `ra` around its own body, and the trampoline does the same around its one call into the entry procedure — x86's trampoline needed no equivalent bookkeeping.
- **CMP and DIV/MOD are simpler, not harder.** RV64's M-extension `div`/`divu`/`rem`/`remu` are independent single-result instructions — no `cqo`/paired-register dance like x86's `idiv`. All 10 CMP relations reduce to 1-2 instructions each via `slt`/`sltu`/`xori`/`sub` (see `emit_cmp()`), and RV64 has no flags register at all, so — unlike x86 (§8 Q2) — there was never a "should CMP set implicit machine state" question to weigh; register-based comparison is just how the ISA works.

**A real bug, caught the same way the x86 ModRM bug was — by execution, not review.** The first version of `reg_disp()` placed symbolic register slot 0 at `[s0-8]` and slot 1 at `[s0-16]` — exactly where the prologue also saves the caller's `ra` and `s0`. Static review missed it (it's a one-line arithmetic choice that looks fine in isolation). Running `add.tmo` through the verifier below produced a wrong return value that traced back to `RET`'s `ld ra,8(sp)` reloading `5` (a value the test program had stored into register slot 0) instead of the real return address — the two were the same memory cell. Fixed by shifting every slot down by 16 bytes so the register-slot region sits below, not overlapping, the saved `ra`/`s0` pair (see the comment on `reg_disp()` in `timi_riscv.c`). A second, smaller bug — a sign-extension cast in the verifier's own `sext()` helper landing an intermediate result in `uint32_t` before narrowing to `int32_t`, so negative offsets silently became huge positive ones — was caught the same way, one level down: it's a bug in the verification tool, not the translator, and would never have been visible without single-stepping actual execution and noticing `sp` jump by `2^32`.

**Verification — and an honest comparison to Phase 3's standard.** No riscv64 cross-compiler and no QEMU means the encoder's output was never run on real or emulated RISC-V hardware. What was built instead, in the same spirit as Phase 1's reference interpreter existing to give something independent to check against: `tools/timi/rv64_exec.c`, a small purpose-built RV64 decoder+executor covering exactly the instruction encodings `timi_riscv.c` emits (no CSRs, no FP, no atomics, no compressed instructions, no traps — not a general RISC-V simulator). `tools/timi/timi_riscv_verify.c` translates a real `.tmo`, feeds the bytes into this executor with a sentinel return address standing in for "the call has returned," and checks the result register. This is real execution of the actual encoded bytes — it caught two genuine bugs static review didn't — but it is **weaker evidence than Phase 3's real-CPU execution**: a matching bug in both the encoder and this independently-written decoder wouldn't be caught, whereas x86's `mmap`+`PROT_EXEC`+call ran the translator's output on silicon that has no opinion about what the encoder intended. Stated plainly, not glossed over. All 8 applicable programs from the same test suite Phase 3 uses (`mem_ops.timi` skipped for the same address-0 reason it's skipped in the x86 native suite) pass with results identical to the x86 target — `make test-native` and `make test-riscv` from `tools/timi/` run the same `.tmo` files through both targets and agree on every result, which is the concrete, checkable form of "the same bytecode retargets to a second ISA."

**What's still unverified, stated plainly:** whether this encoder's output runs correctly on real or QEMU-emulated RV64 hardware — the executor above is an independent implementation, not a reference, and independent-implementation agreement is meaningfully weaker evidence than real hardware execution (see above). Also unverified: everything about kernel integration — there is no RISC-V process/loader subsystem to wire into yet, so unlike x86's Phase 3→4 progression, there's no "port into the kernel" step to even attempt here. Both are natural follow-ups once a riscv64 toolchain/QEMU is available and the RISC-V kernel grows a process/loader subsystem, respectively — the second of which is a substantially larger, non-TIMI undertaking.

## 13. Phase 6 findings — object-typed operations (ODT-equivalent), and the first opcodes that make real calls out of translated code

**What "MI-style generic/typed operations" meant in practice.** The roadmap line for Phase 6 pointed at System/38's ODT (Object Definition Table) — the mechanism by which MI instructions reference other system objects by name/handle rather than by raw address. TIMI's answer, kept deliberately small: one new type tag (`T_OBJREF`) and three new opcodes — `RESOLVE rD, "name"` (look up a live SLS object by name, return its `base_vaddr` as a `T_OBJREF`, or 0 if not found), `OBJSIZE rD, rA, TYPE` (byte size of the object a `T_OBJREF` points at), and `OBJTYPE rD, rA, TYPE` (its `SLSObjectType` tag). This is the first point where translated TIMI code needs to ask the *host environment* a question it cannot answer from its own bytecode alone — every prior opcode was self-contained arithmetic/control-flow/memory-access over values already in registers or the object's own bytes.

**Object format v0.2 → v0.3: a fourth pool, a breaking header change.** `RESOLVE`'s name operand is resolved at assemble time to an index into a new name pool appended after the entry table — fixed 32-byte slots (`TimiName`/`TxEntryRec`-shaped), deduplicated by exact string match (`tools/timi/timi_asm.c`'s `find_or_add_name()`), populated either explicitly via an optional `.name "..."` directive or implicitly on `RESOLVE`'s first use of a given string. The `.tmo` header grew a fifth `uint32_t` field (`num_names`), which shifts every fixed-offset calculation after it — the same "clean bump, no back-compat shim" call made for every prior format change in this project (nothing external consumes `.tmo` files; everything is regenerated from source by this same toolchain). `timi_isa.h`, `timi_obj.c`, `timi_asm.c`, `timi_interp.c`, `timi_disasm.c`, `timi_x86.h/.c`, `timi_riscv.h/.c`, and `kernel/loader.h/.c` all needed their header-parsing/size-accounting updated in lockstep; a compile-time or first-test-run mismatch would have caught a missed one immediately (`timi_validate()`'s 64-bit arithmetic size check is exactly the belt-and-suspenders that exists for this).

**The runtime-call ABI: `r6` joins `r7`, and translated code makes real C-ABI calls.** `RESOLVE`/`OBJSIZE`/`OBJTYPE` can't be pure codegen the way `ADD` or `LOAD` are — answering them means querying a live, environment-specific object catalog. The design: every native translator gains three new parameters — `rt_resolve_fn`, `rt_objsize_fn`, `rt_objtype_fn` — addresses of real C functions (`uint64_t timi_rt_resolve(const char*)`, `uint64_t timi_rt_objsize(uint64_t)`, `uint64_t timi_rt_objtype(uint64_t)`), baked into the generated code as translate-time constants exactly the way `scratch_ptr` already was for `r7`. A new reserved register, **`r6` = `namepool_ptr`**, mirrors `r7`'s existing scratch-pointer convention: the translator computes it *itself* from `obj_data` (a raw pointer straight into the TIMI object's own name-pool bytes, never copied) and bakes it into every trampoline, so `RESOLVE` can compute `namepool_ptr + idx*32` and hand that string pointer to `rt_resolve_fn` with zero marshaling. On x86 this is `movabs r11,fn_addr` / `call r11` (real indirect call, wrapped in an explicit `mov r10,rsp` / `and rsp,-16` / … / `mov rsp,r10` realignment — unlike this translator's own intra-object `call`s, the callee here is a normal compiler-generated C function that may assume strict SysV 16-byte stack alignment for its own stack-local spills, an assumption the rest of this naive codegen never had to satisfy before). On RV64 it's the direct analog: the function address goes through the same `auipc+ld` literal-pool path as any other 64-bit constant, called via a real `jalr ra,0(reg)`, with the single argument passed in `a0` (the real RV64 ABI arg0/return register — every other opcode in this translator uses only `t0`–`t2`, so `a0` is reserved exclusively for this).

**Each environment binds those three function pointers to something different — that's the whole point.** The Phase 1 reference interpreter, the x86 host JIT harness, and the RV64 verify harness all bind them to small identical mock catalogs (`"timi_add_test2"` → size 303/type PROGRAM, `"timi_verify3"` → size 100/type DATA), so `tests/obj_ops.timi` — `RESOLVE` a known name, `OBJSIZE`/`OBJTYPE` it, `RESOLVE` an unknown name and confirm it comes back 0 — produces the identical expected result (305) across every implementation, the same cross-implementation-agreement discipline as every prior phase. The kernel gets the real thing: a new `kernel/timi_runtime.c` implements `timi_rt_resolve/objsize/objtype` as bounded linear scans over the live `object_catalog[]`, gated on `.active`, returning `base_vaddr`/`size_pages*4096`/`SLSObjectType` — the first TIMI runtime function in this project that touches real kernel state rather than translating self-contained bytecode.

**A real bug, caught the same way as Phase 3's ModRM bug and Phase 5's `reg_disp()` bug — by execution, not review.** The RV64 verify harness needs *some* way to invoke a "real" host C function from code executing on a purpose-built, non-hardware RV64 model (`rv64_exec.c` — there's no real RV64 CPU underneath it, see §12). The fix: reserve a sentinel address range, hand out `BASE + idx` as the "function address" for host-callback slot `idx`, and have the executor's `JALR` handling intercept jumps into that range instead of faulting on a fake fetch address. First version spaced slots by 1 (`BASE+0`, `BASE+1`, `BASE+2`). `obj_ops.tmo` segfaulted inside the *host* `mock_rt_resolve` on its **second** hostcall — ASan pointed at a `strcmp` dereferencing `0x2000`, which turned out to be `OBJSIZE`'s own correct argument (the vaddr `RESOLVE` had just returned), but arriving at the **`RESOLVE` callback (slot 0) instead of `OBJSIZE`'s (slot 1)**. Traced with an instrumented build logging every `emit_li64`/literal-pool-patch/hostcall: the literal pool held the right values (`…00`, `…01`, `…02`) and the register holding the target address was correctly `…01` right up to the `JALR` — the bug was in real RV64 semantics, not this project's code: **`jalr` unconditionally clears the target address's least-significant bit** (per spec: "...and then setting the least-significant bit of the result to zero"), silently turning sentinel slot 1's `…01` into `…00` and invoking the wrong callback with the right caller's argument. A one-line-looking encoding choice, invisible to review, caught only because the *right* argument showed up at the *wrong* function — exactly the kind of mismatch execution surfaces and static reading doesn't. Fixed by spacing sentinel slots by 16 instead of 1 (any stride ≥ 2 works; see `rv64_exec.h`'s `RV_EXEC_HOSTFN_STRIDE`).

**Verification.** `tests/obj_ops.timi` passes identically — result `305` — through the Phase 1 interpreter (`make test`), the x86 host JIT harness executing real translated machine code that makes real C-ABI calls out to a mock catalog (`make test-native`), and the RV64 verify harness (`make test-riscv`), alongside the full pre-existing Phase 1–5 suite (all still passing, confirming v0.3's format/opcode additions didn't regress anything). The kernel side — `kernel/timi_x86.h/.c` kept byte-identical to the host copy (diffed to confirm: identical from the `#include` line onward, only the top-of-file comment differs, same convention established in Phase 3), `kernel/loader.h/.c`'s header parsing extended for `num_names`, new `kernel/timi_runtime.c` wired into `kernel/timi_translate.c`'s two translate passes — compiles cleanly under the real kernel build flags (`-ffreestanding -mno-red-zone -mno-sse …`, the same flags `make x86-run` uses). **What's not verified**: an actual booted-kernel spawn of a v0.3 TIMI object with a live `RESOLVE` call against the real `object_catalog[]` — this sandbox has no `x86_64-elf-gcc` cross-compiler (the same hard constraint every prior phase's kernel work hit), so unlike Phase 4's real-QEMU-hardware verification, this phase's kernel wiring has only been compile-checked, not boot-tested. A natural next step on real hardware: upload a `.timi` program that `RESOLVE`s an already-`valloc`'d catalog object by name and confirms the returned `base_vaddr`/size/type match what `sys_sls_obj_stat` reports for the same object.

**What Phase 6 deliberately did not do.** No capability/authority semantics on the `T_OBJREF` values `RESOLVE` returns — they're plain `base_vaddr`s, forgeable/copyable like any other pointer-shaped value, with no unforgeability guarantee. That's explicitly Phase 7's job ("capability/authority model on pointers — System/38 'pointer' semantics — tagged, unforgeable references"), and `T_OBJREF` existing as a distinct type tag now (rather than reusing `T_PTR`) is the deliberate hook Phase 7 hangs off of, not a Phase 6 implementation of it.

## 14. Phase 7 findings — capability tags and authority-checked RESOLVE

**What "System/38 pointer semantics" meant in practice.** System/38's ODT pointers are *tagged* (a hardware/microcode bit marks a word as a valid pointer, invisible and inaccessible to ordinary instructions) and *unforgeable* (no arithmetic or bit-twiddling sequence can synthesize one — only a small set of privileged "resolve" operations can mint a new one, and every mint is authority-checked against the requester). Two independent properties, and TIMI's Phase 6 `T_OBJREF` had neither: it was a plain 64-bit `base_vaddr` with a type label, freely copyable, constructible via `LOADI`, and never checked against the caller's permissions. Phase 7 closes both gaps with the smallest mechanism that actually delivers them, deliberately not modeled as a new opcode or a new memory region — a **shadow tag bit per symbolic register**, enforced identically in every implementation, plus an **authority check at the one place capabilities are minted** (`RESOLVE`, in the kernel binding only — the host/mock bindings have no permission model to check against).

**The tag contract, stated once and implemented three times.** Every symbolic register gets a paired 1-bit (stored as a byte for addressability) tag: 1 means "this register currently holds an unforged `T_OBJREF` capability," 0 means "plain value." Four rules cover every opcode: `RESOLVE` sets the destination's tag **iff** the result is non-zero (a failed resolve is not a capability); `MOV` propagates the source's tag to the destination (copying a capability is legitimate — that's how it gets passed to a callee via `r0`-`r7` or used in a comparison); every other register-writing opcode — arithmetic, `LOADI`/`LOADI64`, `CMP`, `LEA`, `PTRADD`, typed `LOAD` — unconditionally clears the destination's tag; `OBJSIZE`/`OBJTYPE` require the operand register's tag to be set, and reject with the pre-existing "not found" sentinel (`0` / `0xFFFFFFFF`) **without ever calling the runtime catalog function** when it isn't. `PTRADD` clearing the tag is the load-bearing rule for unforgeability: if pointer arithmetic preserved a tag, `RESOLVE`-ing a real object and then `PTRADD`-ing by an attacker-chosen offset would forge a "valid" capability for an arbitrary neighboring address. `CALL`/`RET` do not propagate tags across the call boundary in this v1 — a documented narrowing, not an oversight (see below).

**Interpreter first, as ground truth.** `timi_interp.c` grew a `cap_tag[NREGS]` array per `Frame`, a `set_tag()` helper, and the rule set above applied at every `case` in the main switch. This is where the exact semantics — especially "RESOLVE tags conditionally, not unconditionally" and "PTRADD always untags" — were pinned down before touching either native translator, the same discipline as every prior phase.

**x86: a stack-frame tag region and a new local-branch primitive.** The translator's per-procedure frame already held `TX_MAX_REGS` 8-byte register slots at negative `rbp` displacements; Phase 7 adds a second, `TX_MAX_REGS`-byte region directly below it (`TX_TAG_BASE`, contiguous with the register region, zeroed in the same `rep stosq` pass at `ENTER` by growing the frame from 512 to 576 bytes). New byte-level helpers (`ld_tag_cl`/`st_tag_cl`/`st_tag_imm`) and `st_rax_untag`/`st_rcx_untag` wrappers thread through every existing register-write call site. `RESOLVE`'s conditional tag uses `test rax,rax; setne cl` computed *before* the result is stored (TEST/SETcc don't clobber `rax`). `OBJSIZE`/`OBJTYPE` needed an actual conditional branch — skip the runtime call entirely when untagged — which doesn't fit the existing global fixup table (that's keyed by TIMI `target_pc`, resolved in a pass once every instruction's *TIMI-level* offset is known; this branch's target is a few bytes further into the *same instruction's own* generated code). Solved with a new, deliberately separate **local (intra-instruction) branch mechanism** — `emit_jz_placeholder`/`emit_jmp_placeholder`/`patch_local_rel32` — that patches immediately in the same emit call rather than deferring to the global pass.

**RV64: the same mechanism, ported instruction-for-instruction.** The tag region sits below the register-slot region the same way (`TX_RV_TAG_BASE`, frame grown from 512 to 576 bytes, zeroed via an explicit per-byte `sb x0` loop — RV64 has no `rep stosq` equivalent, consistent with the existing register-zero loop's own style). `RESOLVE`'s conditional tag collapses to a *single instruction* on this target — `sltu t1,x0,a0` computes "a0 != 0" directly, RV64's one-shot analog of x86's test+setne, no branch required. `OBJSIZE`/`OBJTYPE` do need a branch, built with new `emit_beqz_placeholder`/`emit_jal_placeholder`/`patch_local_beqz`/`patch_local_jal` helpers — the direct RV64 port of x86's local-branch primitive, using `beqz`/`jal`'s real B-type/J-type encodings rather than threading through the global `g_fixups` table for the same reason as x86. `rv64_exec.c`, the purpose-built verify-harness executor, needed **no changes at all** — `LBU`/`SB` (the tag byte load/store) and `BEQ`/`BNE` (the local branch) were already implemented for Phase 1–6's typed `LOAD`/`STORE` and `BC` opcodes, so this is the first phase where a new TIMI-level opcode/semantics addition required zero executor changes, purely a translator-side story.

**The regression test: forge a capability, confirm the forgery is rejected, not just that the legitimate path works.** `tests/cap_forge.timi` `RESOLVE`s a real object (`"timi_add_test2"`, the shared mock catalog entry every implementation binds to `base_vaddr=0x2000`), takes its `OBJSIZE`/`OBJTYPE` (expected 303/1), then separately does `LOADI r3, #8192, objref` — the identical numeric value, `0x2000`, but never passed through `RESOLVE` so never tagged — and takes `OBJSIZE`/`OBJTYPE` of *that*, which must come back rejected (`0` / `0xFFFFFFFF`) despite being bit-for-bit the same address a legitimate capability would carry. The expected result (`3000304 = 303 + 1 + 1,000,000 + 2,000,000`) is constructed so the two million-scale terms only appear if both rejections actually fired — a silent forgery-acceptance bug would land nowhere near that value, not off by a rounding error, the same "impossible to mistake for a coincidence" test-design principle used for every prior phase's expected-value constants.

**Verification.** `cap_forge.tmo` passes with the identical result `3000304` through all three implementations: the Phase 1 interpreter (`make test`), the x86 host JIT harness *actually executing* the `test`/`setne`/local-`jz` byte sequence on the host CPU (`make test-native`), and the RV64 verify harness executing the `sltu`/local-`beqz`/`jal` sequence through `rv64_exec.c` (`make test-riscv`) — real execution of the forgery-rejection branch logic in two of the three environments, not just static review of the encoder output, plus independent-implementation agreement across all three. The full pre-existing Phase 1–6 suite still passes unmodified in every environment, confirming the tag region's added stack space and the new local-branch mechanism didn't regress anything. `kernel/timi_x86.c` was re-synced to stay byte-identical to the host copy from the `#include` line onward (diffed to confirm), and compiles cleanly under the real kernel build flags.

**Authority-checked RESOLVE: reusing existing infrastructure, not building new security primitives.** The kernel already had everything the authority check needed — `object_catalog.c`'s `catalog_check_access(uid, obj_name, needed_perm)` (role-based rules for `DB_ADMIN`/`APP_USER`/`GUEST`/owner, kernel `uid=0` always passes) and `user/permissions.h`'s `PERM_READ`. The one missing piece was identifying *who's asking*: the kernel had no existing "currently running process" accessor, only `schedule_ring3()`'s ad hoc inline scan of `proc_table[]` for `active && state==PROC_RUNNING && kernel_rsp!=0`. That scan is now a proper function, `process_find_current()` (`kernel/process.c`/`.h`), and `kernel/timi_runtime.c`'s `timi_rt_resolve()` calls it to find the caller's `owner_uid`, then gates the returned `base_vaddr` on `catalog_check_access(uid, name, PERM_READ)` — denial returns `0`, byte-for-byte the same as "object not found." That indistinguishability is deliberate: a distinct "exists but denied" result would let a process fish for which object names exist in the catalog by `RESOLVE`-ing guesses and comparing "not found" against "denied," even though it could never mint a capability for anything it isn't already allowed to read. No Ring-3 process running (`process_find_current()` returns `NULL`, e.g. a call from pure kernel context) is treated as `uid=0`/`ROLE_SYSTEM_KERNEL`, which `catalog_check_access()` always passes.

**Verification of the kernel-side authority check.** `kernel/process.c`, `kernel/process.h`, and `kernel/timi_runtime.c` all compile cleanly under the real kernel build flags (`-ffreestanding -mno-red-zone -mno-sse …`). **What's not verified**: an actual booted-kernel `RESOLVE` call from a real Ring-3 process against a live `object_catalog[]` with a non-kernel `owner_uid`, exercising the deny path end-to-end — the same hard constraint every prior phase's kernel-side work has hit (no `x86_64-elf-gcc` cross-compiler in this sandbox, see §6's and §13's identical caveats), so this piece is compile-checked and logic-reviewed, not execution-verified, consistent with how Phase 6's real `object_catalog[]` wiring was itself verified. A natural next step on real hardware: spawn two Ring-3 processes under different `owner_uid`s/roles, have one `valloc` an object it owns, and confirm the other's `RESOLVE` of that name returns `0` (denied) while the owner's `RESOLVE` succeeds and can `OBJSIZE`/`OBJTYPE` the result — the boot-tested completion of what `cap_forge.timi` already proves for the tag-forgery half in every environment that *can* be executed here.

**What Phase 7 deliberately did not do.** No capability propagation across `CALL`/`RET` — a capability passed as an argument or returned as a result arrives untagged on the other side of a call boundary in this v1 (`emit_call_site`'s result store always untags, on both native targets; the interpreter's `OP_CALL`/`OP_RET` clear tags the same way). A TIMI procedure that wants to pass a capability to a callee and have the callee treat it as one currently must `RESOLVE` again inside the callee, or the callee must trust an out-of-band convention — neither is built here. This is flagged as a narrowing, not a design dead-end: propagating tags across calls is straightforward (copy the tag byte into the outgoing-argument marshaling, same as `MOV` already does within one frame) but was left out to keep this phase's surface area to "mint and check," matching the same "narrow v1, flag it, move on" discipline used for every prior phase's scope cuts (Phase 3's `TX_MAX_REGS` cap, Phase 5's host-only kernel wiring, Phase 6's 31-character name limit). No revocation — once a `RESOLVE` succeeds, the resulting capability is valid for the lifetime of the register holding it (or any register it's `MOV`'d into), with no mechanism to invalidate it if the underlying object is `vfree`'d mid-flight; `OBJSIZE`/`OBJTYPE` still re-check the live catalog on every call (so a freed object's slot going `inactive` is caught then), but nothing stops a stale capability from being held. And no LPAR/partition-boundary interaction yet — that's explicitly Phase 8's job, and this phase's authority model (single flat `object_catalog[]`, single role table) is the substrate Phase 8 would need to partition, not an implementation of the partitioning itself.
