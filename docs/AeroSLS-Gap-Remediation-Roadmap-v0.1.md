# AeroSLS Gap Remediation Roadmap v0.1 — plan of attack

Source: `docs/AeroSLS-Gap-Analysis-v0.1.md`. That document found one dominant pattern (capabilities that exist in C but that no live caller can reach), plus a handful of system-wide properties (never booted, RAM-only persistence in the newer subsystems, shallow auth) that cut across every roadmap. This document turns those findings into an ordered plan, sequenced by what `/slos-sim` actually needs first and what blocks what.

## Sequencing principle

Everything here is sequenced against one question: **what does `/slos-sim` need to be a real IBM i Navigator-style console, and in what order does the backend have to exist for that to be true?** The gap analysis's own §8 ranking is the starting point; this plan makes it executable — phases, concrete deliverables, and explicit dependency edges rather than a flat priority list.

One phase (A) sits outside that logic entirely — it's an environment/infra task, not a code gap — and is called out as such rather than numbered with the rest.

## Phase A (parallel, not blocking): real boot verification

Not a code fix — this project has never linked and booted a kernel containing the RDBMS or Vector Store work, because this sandbox lacks `x86_64-elf-gcc`/`nasm`/`qemu-system-x86_64`. This should happen on whatever machine will actually run AeroSLS (most likely yours, given `my_sls_kernel.bin`'s history), using `setup-toolchain.sh` already in the repo, followed by one `make x86-run` cycle against current `main`. It costs nothing to run in parallel with everything below, and it's the one action that would upgrade every other finding in the gap analysis from "confirmed in source" to "confirmed running." Recommend doing this first if you have a slot for the 20-40 minute toolchain build, but it shouldn't gate Phase B starting.

## Phase B: RDBMS reachability — the single biggest lever for the frontend

This closes the gap that blocks `/slos-sim` from doing anything with real tables at all.

1. **`rowstore_create_table()` gets a live path**: syscall number, `syscall_dispatch.c` case, shell command (`table create <name> <schema>`), and an HTTP route (`POST /api/tables`). Today there is *no* way to create a row-set table outside a host test — this is the one true blocker, everything else in Phase B is reachable once this exists.
2. **`sql_execute()` gets an HTTP route** (`POST /api/sql`, request body `{query, params?}`, reusing the existing bearer-token auth gate). The shell and syscall paths already exist; this is a straight port of the same JSON-response pattern already used by `/api/objects` etc. in `net/http.c`.
3. **Fix the Phase 22 planner regression**: `sql_exec.c`'s live query path currently full-table-scans regardless of indexes. Since the indexes are still correctly maintained on every write, this is a scoped fix — reintroduce an index-consult step into the existing execution path rather than rebuilding the planner. Do this in Phase B rather than later because a frontend demoing "query a table" on anything but toy data will otherwise look slow for a reason that has nothing to do with the frontend.
4. **`GET /api/tables` / `GET /api/tables/:name/schema`** — new, small, needed so a Navigator-style tree view can enumerate what tables exist and their columns without already knowing the name.

Dependency note: steps 2-4 are independent of each other once step 1 lands; step 1 blocks nothing upstream and should be done first.

## Phase C: Vector Store reachability

Mirrors Phase B's shape, scoped smaller since exact search already has a full syscall/shell/HTTP-adjacent path from Phase 4.

1. **HTTP routes for the existing exact-search syscalls** (`POST /api/vec/collections`, `POST /api/vec/insert`, `POST /api/vec/embed-insert`, `POST /api/vec/search`) — direct ports of the Phase 4 syscall adapters, no new kernel logic.
2. **Syscall + shell + HTTP surface for `vec_join_resolve()`** (Phase 5) and **`vec_index_search()`** (Phase 6's HNSW) — both are currently pure C functions reachable only from their own host tests. Needed if `/slos-sim` is going to demonstrate either the relational join-back or the approximate search, and neither has any live surface today.
3. **A list/stat syscall** for collections and indexes (`vec list`, `GET /api/vec/collections`) — currently a caller must already know a collection's name; no enumeration exists at all. Needed for the same reason as Phase B's table-listing route.

Do Phase C after Phase B: it's smaller and lower-risk, and Phase B's HTTP-route pattern (step 2 above) becomes the template to copy rather than a second design decision.

## Phase D: persistence for the newer subsystems

Indexes, constraints, the row journal, and the entire vector store are RAM-only today, while base row data survives reboot via the existing NVMe paging path. This matters most once Phases B/C make these things visible in a UI — "the table is still there but every index and the vector collection vanished on restart" is a worse-looking bug once there's a console showing it happening than it is as a line in a findings doc.

Two viable approaches, in increasing order of effort:
- **Rebuild-on-boot**: cheapest — re-derive indexes and the vector store from the row data that already persists, at boot time, rather than persisting the structures themselves. Works cleanly for B-tree indexes and HNSW (both are pure functions of the underlying data plus config). Constraints just need their definitions (not their state) persisted, which is far smaller than full row persistence.
- **Direct persistence**: extend `persist.c`'s existing NVMe write path to cover index/constraint/journal/vector-store pages directly, avoiding a rebuild pause on boot. More work, only worth it once rebuild-on-boot's boot-time cost is actually measured and found to matter.

Recommend starting with rebuild-on-boot — it reuses Phase B/C's own logic (an index rebuild is just "call the existing insert path for every persisted row") rather than adding a new persistence format.

## Phase E: auth and transport hardening

Scope this explicitly as **"before `/slos-sim` is reachable from anywhere but localhost/your own machine,"** not as a blocker to starting frontend work — the current posture (unauthenticated GET, static demo tokens, no TLS) is honestly fine for a local dev console talking to a local QEMU instance, which is what this will be at first.

1. Apply the existing bearer-token gate to GET routes, not just POST — cheapest fix, closes the "read anything with no token" gap immediately.
2. Add TTL/expiry checking to `auth_validate_token()` — the check point already exists (`kernel/auth.c` ~line 136), just doesn't compare against a timestamp today.
3. Either implement real encryption in `sys_sls_secure_seal()` or rename/relabel it so it stops claiming a security property it doesn't provide — the second option is one line and should happen regardless of whether real encryption is ever built, since a function that logs "successfully encrypted" while doing nothing is actively misleading to anyone reading the log.
4. TLS is a real cross-compile/cert-story project of its own (no libc crypto exists in this kernel today) — flag as a distinct, larger future item rather than folding it into this pass. Not needed for a localhost-only frontend.

## Phase F: partition reachability + the memory leak

Lower priority than B/C for an MVP frontend (the legacy object/process/program surface already gives `/slos-sim` something to show without this), but blocking for a *complete* IBM i Navigator-style console, since partitions are this OS's closest analog to IBM i's own subsystem/LPAR management views.

1. Shell commands + HTTP routes for partition create/list/destroy/assign (currently zero of either exist, despite every partition syscall being correctly wired).
2. Assign `sys_sls_partition_quota_list()` a syscall number — it's fully implemented and simply never wired into the dispatcher.
3. Implement `free_physical_ram_frame()` and call it from `partition_destroy()` — currently a permanent leak; low-risk, self-contained fix once the function exists (frame_pool.c has no free primitive for *any* subsystem today, so this benefits more than just partitions).
4. Thread real caller identity through the HTTP program-spawn path instead of the hardcoded `owner_uid = 0` — the plumbing pattern already exists for agent/workflow routes in the same file; this is copying it over, not inventing it.

## Phase G: TIMI reachability polish

Smallest and lowest-priority phase — TIMI programs already upload/spawn transparently through the generic program routes, so this is about introspection quality, not a hard blocker.

1. Refactor `loader_timi_info()` (and the equivalent partition-info path) to return structured data to the caller instead of writing to the serial console — same JSON-builder pattern as every other `/api/*` handler.
2. Add the missing TIMI/partition syscall bindings to `user/libsls/sls.h` so a native program can call them without hand-rolling a trap.

## Phase B findings addendum

All four planned deliverables are done and verified: `rowstore_create_table()` has a live syscall (`SYS_SLS_ROWSTORE_CREATE_TABLE=225`) plus shell (`table create <name>`) and HTTP (`POST /api/tables`) surfaces; `POST /api/sql` wraps the existing `sys_sls_sql_execute()`, with SELECT results embedded directly in the response (columns + rows) rather than requiring a second round trip; `GET /api/tables` and `GET /api/tables/<name>/schema` give a Navigator-style tree view something to enumerate; and the Phase 22 full-scan regression is fixed for the one case that can be fixed safely — see below.

**The planner fix ended up needing two new pieces, not one.** The original framing ("reintroduce an index-consult step") undersold what Phase 22's own header comment had already flagged: a B-tree index only stores physical `RowId`s, and under MVCC a column value's physical row changes on every UPDATE, so a raw index hit has no notion of which version is visible to a given transaction's snapshot. Fixing just that required `mvcc_resolve_physical()` (`mvcc.c`) — an O(1)-average hash lookup (a small fixed open-addressed table, `mvcc_versions[]`-slot-indexed, sized 2× `MVCC_MAX_VERSIONS`) that resolves a physical id to "is this the version visible right now," backed by the same `mv_visible()` rule `mvcc_table_scan()` already used.

Implementing that surfaced a second, independent hole: a B-tree leaf's duplicate list caps at `BTREE_MAX_DUPES_PER_KEY=16` entries per distinct value, and under MVCC that list fills with every *version* a value ever had, not just current ones — a hot value can silently exceed the cap with no signal available from the existing `row_index_lookup()` API. Trusting an index result blindly here would have traded "always correct, always slow" for "usually correct, silently wrong under load" — the exact "denial looks like absence" failure class this project has named repeatedly. The fix: a new sticky `key_capped` flag per B-tree leaf key (row_index.h/.c), set the moment the cap is ever hit for that exact key and never cleared, plus `row_index_lookup_checked()`, which reports completeness alongside the result.

Given both pieces, the planner (`sql_exec.c`'s `try_index_assisted_eq()`) only accelerates a single top-level equality comparison against an indexed column, and only trusts the index when `row_index_lookup_checked()` reports the result as complete. Range comparisons (`<`, `>`, `<=`, `>=`) and AND/OR predicates were NOT restored — the completeness problem applies independently to every distinct key a range touches, with no aggregate "was anything dropped" signal available without a larger change to `row_index.c`'s range-scan path. This is narrower than Phase 19's original design, on purpose: a real, named scope cut rather than a rushed unsafe fix. Every index candidate is still rechecked against the full predicate before being trusted, exactly matching this whole roadmap's "index narrows, recheck is the authority" posture since Phase 19 — this is a genuine algorithmic improvement, not a relaxation of the correctness bar.

**Verification.** All pre-existing host tests pass unchanged (mvcc: 55/55, row_index: 39/39, sql_exec: 45 pre-existing scenarios, rowstore, legacy-boundary, vec_join, vecstore-syscall, vec_index, row_constraint+journal). `sql_exec_host_test.c` gained two new scenarios: scenario 13 builds a table with 25 rows sharing one indexed string value (over the 16-cap) and confirms the SELECT still returns all 25 — proving the cap-detection fallback prevents silent under-counting, the single most important correctness property of this fix; scenario 14 runs an UPDATE through the index-assisted path and confirms the OLD value stops matching while the NEW value starts, proving MVCC-version-aware resolution works across a real write. Both pass.

**One pre-existing gap found and fixed in passing, unrelated to this phase's own scope**: `vec_join_host_test.c`'s documented build line never linked `kernel/vec_index.c`, left broken since Phase 6 added `vec_index_notify_insert()/_delete()` calls into `vecstore.c` without anyone updating this particular test's link line. Fixed (one line) and reverified — not a Phase B regression, a Phase 6 regression-sweep gap this pass happened to surface.

Not addressed in this pass, named rather than silently skipped: `schema_set` (needed to define a table's columns) still has no HTTP route — `POST /api/tables` promotes an object that already has one; getting there today still requires the shell or a raw syscall. This is a smaller, separate gap, left for a future pass.

Phase B closes out. Recommend Phase C (Vector Store reachability) next, per the original sequencing — it reuses this phase's HTTP-route pattern directly.

## Phase C findings addendum

All three planned deliverables are done and verified, plus one real bug class found along the way. `POST /api/vec/collections` (+ `GET` for listing), `POST /api/vec/insert`, `POST /api/vec/embed-insert`, and `POST /api/vec/search` give the four Phase 4 syscalls a full HTTP surface, reusing the same route-registration pattern Phase B established rather than inventing a second one; `POST /api/vec/join` (+ the new `SYS_SLS_VEC_JOIN=226` syscall and `vec join` shell command) makes `vec_join_resolve()` reachable outside a host test for the first time since Phase 5 built it; `POST /api/vec/indexes` (+ `GET`) and `POST /api/vec/index/search` (+ the new `SYS_SLS_VEC_INDEX_CREATE=227`/`SYS_SLS_VEC_INDEX_SEARCH=228` syscalls and `vec index create`/`vec index search` shell commands) do the same for Phase 6's HNSW index, which had zero live surface by its own explicit scope cut; and `GET /api/vec/collections`/`GET /api/vec/indexes` (backed by new `SYS_SLS_VEC_LIST=229`/`SYS_SLS_VEC_INDEX_LIST=230` syscalls and `vec list`/`vec index list` shell commands) close the same "must already know the name" enumeration gap Phase B's table-listing route closed for the RDBMS side.

**`POST /api/vec/join`'s request shape is a join primitive, not a search-then-join convenience call.** It takes `matches[]` directly (typically copied from a prior `/api/vec/search` or `/api/vec/index/search` response) rather than re-running a search itself, matching `vec_join_resolve()`'s own real contract — a caller who already has a result set from elsewhere shouldn't pay for a second search just to join it. This did require one new JSON helper not needed by any earlier phase: `json_array_object_at()`, which extracts the Nth `{...}` object out of a JSON array of objects (the `matches` array's own elements), since every prior route only ever needed scalar or flat-array JSON values.

**One pre-existing gap found and fixed in passing, in the same file class as Phase B's own `vec_join_host_test.c` fix**: `vecstore_syscall_host_test.c`'s documented build line also never linked `kernel/vec_index.c`, broken for the identical reason (Phase 6 added `vec_index_notify_insert()/_delete()` calls into `vecstore.c`'s insert/delete paths but never swept every host test's own build-line comment). Found while running this phase's own regression sweep, fixed the same one-line way, reverified — all 8 pre-existing scenarios still pass.

**A second, more consequential bug found while writing this phase's own new code, unrelated to Vector Store specifically**: `kernel_serial_print(const char* s)` takes exactly one argument and does no format processing — `kernel_serial_printf(const char* fmt, ...)` is the variadic one. Three pre-existing files called `kernel_serial_print()` with a format string plus several extra arguments, a call that should have used `kernel_serial_printf()`: `object_catalog.c`'s `sys_sls_obj_list()`, `tier_mgr.c`'s `sys_sls_tier_list()`, and `microkernel.c`'s `sys_sls_svc_list()`, all in each function's own header-row print. This compiled without error only because none of those three files include `kernel_io.h` themselves — with no prototype in scope, the compiler falls back to implicit-declaration behavior, which accepts any argument count. This is a direct, concrete instance of the gap analysis's own top finding (§1): this project has never been compiled end-to-end with a real cross-compiler and every header correctly included, so a call like this — which would fail loudly under a real compile — has sat unnoticed. Caught while writing `vecstore.c`'s new `sys_sls_vec_list()` (which copies the same header-row idiom) before it could become a fourth instance; the original three were then found via a repo-wide grep sweep and fixed the same way (switched to `kernel_serial_printf`). A second grep pass (checking both multi-line and single-line call forms) confirmed no further instances exist — the remaining matches are all genuine single-argument calls where a comma appears inside the string literal's own text (e.g. `"not 64-bit, aborting."`), not extra arguments.

**Verification.** `vec_join_host_test.c` gained three new scenarios, extended in place rather than forked into a new file (same precedent as Phase B's own `sql_exec_host_test.c` extension): scenario 4 creates an HNSW index over the already-populated `employee_embeddings` collection via `sys_sls_vec_index_create()`, confirms `vec_index_create()`'s documented no-backfill behavior by inserting fresh, position-controlled vectors afterward and confirming only those are found via `sys_sls_vec_index_search()`; scenario 5 feeds `sys_sls_vec_join()` real matches from a live `vecstore_search()` call and confirms it resolves identically to the direct `vec_join_resolve()` call already verified in scenario 1; scenario 6 smoke-tests `sys_sls_vec_list()`/`sys_sls_vec_index_list()` against real, populated data, matching this project's established precedent of not asserting on `sys_sls_obj_list()`-style console-dump content. One test-authoring mistake was caught and fixed during this pass, not shipped: scenario 4's first draft placed its post-index probe vectors at the same coordinates as scenario 1's own employee vectors, which (correctly) produced a tied nearest-neighbor result and made scenario 5's own top-3 assertion fail — not a product bug, a test-data collision, fixed by moving the probe vectors to a disjoint coordinate range and adding an explicit assertion on scenario 5's own top-3 identity so a future regression here fails clearly rather than ambiguously. All checks across the file's 6 scenarios (3 pre-existing + 3 new) pass — 39 individual `CHECK()` assertions in this run, up from 23 before this phase's additions. `vecstore_syscall_host_test.c` (all 8 scenarios), `vec_index_host_test.c` (all scenarios, including its own recall verification), and `legacy_rowstore_boundary_host_test.c` (confirming the `object_catalog.c` bug fix didn't disturb the legacy/rowstore boundary guard) all still pass unchanged. `net/http.c`, `user/shell.c`, `kernel/syscall_dispatch.c`, `kernel/tier_mgr.c`, `kernel/microkernel.c`, `kernel/object_catalog.c`, `kernel/vecstore.c`, `kernel/vec_index.c`, and `kernel/vec_join.c` all syntax-check clean (`gcc -fsyntax-only`, exit 0) — remaining warnings on the shell/dispatch/tier_mgr/microkernel/object_catalog files are the same pre-existing implicit-declaration noise the bug above already explains, not new issues from this phase's own changes.

Not addressed in this pass, named rather than silently skipped: `vec_index_create()`'s no-backfill behavior (a Phase 6 scope cut, not a Phase C regression) means an index created over an already-populated collection stays empty until new inserts arrive — a real caller wanting to index existing data still needs a `vecstore_collection_scan()`-driven backfill helper, which doesn't exist yet. HNSW deletion is still tombstone-only (Phase 6's own named limitation); Phase C added no new persistence for either subsystem (that's Phase D's scope, unchanged by this pass).

Phase C closes out. B and C together mean every RDBMS and Vector Store engine function this project has built through Phase 24/Phase 6 now has a real syscall/shell/HTTP path — the two subsystems most of this project's recent effort went into are now actually reachable, not just host-testable. ## Phase D findings addendum

All four RAM-only subsystems named in scope — indexes, constraints, the row journal, and the vector store — now survive a reboot, plus a real, previously-latent bug in the MVCC layer that Phase D's own persistence work would otherwise have made functionally invisible. The original framing offered rebuild-on-boot as the cheaper default across the board; in practice the right call turned out to split by subsystem, not apply uniformly.

**Split strategy: direct persistence where no other copy of the content exists, rebuild-on-boot where the content is a pure derivation of already-persisted data.** `vecstore.c`'s raw vector data, `row_constraint.c`'s constraint definitions, and `row_journal.c`'s audit-trail entries all got direct NVMe persistence — none of that content can be regenerated from anything else (a vector's own float values have no source of truth to rebuild from; a journal entry is, by definition, a historical record). `row_index.c`'s B-tree indexes and `vec_index.c`'s HNSW indexes instead persist only their lightweight definitions (index name, table/column, metric) and get rebuilt at boot by re-running the real `row_index_create()`/`vec_index_create()` functions against the already-restored underlying rows — reusing Phase B/C's own insert-path logic exactly as the original plan intended, just scoped to the two subsystems where it actually applies cleanly. `vecstore.c` follows the same lazy-load/eager-flush page pattern `rowstore.c` established in Phase 16 (`vecstore_load_page()`/`vecstore_flush_page()`, mirroring `rowstore_load_page()`/`rowstore_flush_page()`), reserving a fresh LBA region (`VECSTORE_LBA_BASE=4200000`) clear of `rowstore.c`'s own reserved span. All six new persisted regions (row constraints, row-index definitions, vecstore headers, vec-index definitions, row-journal entries + attachments) get their own magic-tagged header in `persist.h`'s LBA layout, following the exact "denial looks like absence" restore-or-cold-start convention every prior persisted subsystem already uses — struct-size validation and a magic-number check before trusting anything read back.

**A real, previously-latent bug found during this phase's own design review, fixed rather than deferred: MVCC had no persistence or rebuild step at all.** `mvcc.c`'s own header comment had explicitly said persistence was "out of scope, nothing forces it yet" — true when written, but Phase 22 had since routed `sql_exec.c`'s entire live query path through `mvcc_table_scan()` rather than `rowstore_table_scan()` directly, and `row_constraint_check_write()`'s UNIQUE/REFERENCE checks depend on the same table scan. Combined with this phase's new rowstore/vecstore persistence, that meant a restart would leave `mvcc_versions[]` empty — the SQL engine and every constraint check would see a table as empty even though the raw rows and B-tree indexes were both correctly restored underneath it. This is the exact "denial looks like absence" failure class this project has named and fixed on sight in every prior phase, so it was fixed the same way here rather than logged as a known gap: `mvcc_bootstrap_from_rowstore()` (`mvcc.c`), called unconditionally during `persist_restore_all()` right after rowstore's own restore block, synthesizes one `MvccVersion` per restored physical row with `xmin=0`/`xmin_committed=1` (a reserved "committed since before time began" sequence number, safe since real commits start at seq=1) and `xmax=0`/`xmax_committed=0` (never superseded). Every boot-time rebuild call in this phase (`row_index_create(0, ...)`, `vec_index_create(0, ...)`, the MVCC bootstrap's own `rowstore_table_scan(0, ...)`) uses `caller_uid=0`, the existing `ROLE_SYSTEM_KERNEL` convention `object_catalog.c` already established — not a new convention invented for this phase.

**One pre-existing gap found and fixed in passing, a third instance of the same Phase 6 regression-sweep miss named in the Phase B and Phase C addenda above**: `vecstore_host_test.c`'s own documented build line was also still missing `kernel/vec_index.c`. Fixed the same one-line way as the other two instances, reverified.

**Regression sweep.** Adding `persist_*()` calls inside `vecstore.c`/`vec_index.c`/`row_index.c`/`row_constraint.c`/`row_journal.c`, and `persist_restore_all()`'s new references into all five of those plus `mvcc.c`, broke the link step of every pre-existing host test that links one side of that dependency but not the other — an expected, mechanical consequence of the design, not a logic bug. Fixed via the same minimal-stub pattern `persist_partition_host_test.c` already established (stub globals + no-op/safe-failure stub functions, not real dependency linking) across eleven files: `vecstore_host_test.c`, `vecstore_search_host_test.c`, `vec_index_host_test.c`, `vecstore_syscall_host_test.c`, `rowstore_host_test.c`, `sql_exec_host_test.c`, `sql_join_host_test.c`, `sql_tx_host_test.c`, `row_constraint_journal_host_test.c`, `mvcc_host_test.c`, and `row_index_host_test.c`. All eleven, plus the new `persist_rdbms_vecstore_host_test.c` (below) and the unaffected `vec_join_host_test.c`, build clean and pass every pre-existing check unchanged.

**Verification.** New file `persist_rdbms_vecstore_host_test.c` proves the full round-trip in one pass across all five persisted subsystems plus the MVCC bootstrap fix: builds a real employees table (5 rows via real `mvcc_begin()`/`mvcc_row_insert()`/`mvcc_commit()`), two constraints (UNIQUE id, NOT NULL name), a journal attachment plus five journal entries, a B-tree index on `employees.id`, a vecstore collection with 7 embeddings (2 inserted after HNSW index creation, to prove index backfill via `vecstore_collection_scan()` during rebuild), and an HNSW index — then simulates a reboot (`memset()` of every RAM structure, followed by re-running every subsystem's real `_init()` in actual kernel boot order), calls `persist_restore_all()`, and asserts on all five subsystems plus a corrupted-magic "denial looks like absence" check on the vecstore region specifically. All checks pass. `net/http.c`, `user/shell.c`, and `kernel/syscall_dispatch.c` all syntax-check clean (`gcc -fsyntax-only`, exit 0) — Phase D touched none of the HTTP/shell/dispatch layers, so this is a confirmation of no incidental breakage rather than new coverage.

Not addressed in this pass, named rather than silently skipped: the rebuild-on-boot path for large tables has no measured boot-time cost yet — the original plan flagged this as worth measuring before ever moving to direct persistence for indexes, and that measurement still hasn't happened (host-test tables are a handful of rows). If a real boot ever shows this mattering, `row_index.c`/`vec_index.c` could move to direct persistence using the same LBA-region pattern already built for the other four subsystems. Vecstore page persistence is whole-page-granularity (a single point update rewrites the containing page's full 4 KiB rather than a smaller record), matching `rowstore.c`'s own existing precedent rather than inventing a new one, but is worth naming as a real space/write-amplification tradeoff distinct from any of B/C's own findings.

Phase D closes out. Recommend Phase A (real boot verification) if a slot exists — the same standing recommendation as before, now covering four completed phases' worth of "confirmed in source and host-tested, not yet confirmed running." Otherwise E/F/G remain sequenced per the summary table, unchanged by this pass.

## Phase E findings addendum

All three planned deliverables are done, plus one real bug this phase's own TTL work would otherwise have introduced. GET routes now carry the same bearer-token gate POST routes already had; `auth_validate_token()` enforces a real TTL instead of accepting any never-expiring token forever; and `sys_sls_secure_seal()` no longer claims a security property it doesn't provide. TLS remains explicitly out of scope, per the original plan's own framing.

**GET-route gate is scoped to `/api/*` data routes, not applied blanket.** A blanket gate would have broken the thing it's meant to protect: `/auth/token` and `/auth/verify` can't themselves require a valid token (that's how a caller gets one, and how a caller checks whether the one they have still works), and the compiled-in Navigator SPA bundle plus dynamic `WEB_APP` assets (everything not under `/api/`) has to stay reachable so the login screen itself can load before any token exists. `/api/health` is the one explicit exemption inside `/api/*`, kept public as the conventional unauthenticated liveness probe every other system in this project's design vocabulary already assumes; every other `/api/*` GET route — objects, tables, vec collections/indexes, services, WAL, tiers, processes, programs, streams, query, constraints, cursors, locks, MQTs, indexes, journals, agents, workflows — now returns 401 without a valid bearer token, mirroring the POST gate's existing `{"error":"Unauthorized..."}` response shape exactly rather than inventing a second one.

**TTL is expressed in LAPIC timer ticks, not TSC cycles — a deliberate deviation from the obvious-looking approach.** This codebase has no calibrated TSC-to-wallclock ratio anywhere; `dashboard.c`'s own `average_fault_latency_cycles` is deliberately left as raw, uncalibrated cycles, fine for a relative latency metric, useless as an absolute expiry threshold across different CPUs. `timer.c`'s `kernel_tick_counter`, by contrast, increments at a documented ~100 Hz (`init_timer()`'s own comment: "fires roughly every 10 ms"), making it the only monotonic counter in the kernel with even an approximate real-time meaning — `AUTH_TOKEN_TTL_TICKS` (360,000 ticks, ~1 hour) is therefore approximate wall-clock, not exact, the same honesty-over-precision tradeoff `init_timer()`'s own comment already makes, not a new one invented for this phase.

**A real bug found while implementing TTL, fixed in the same pass rather than shipped and found later: naive expiry would have made `POST /auth/token` permanently useless for any email once its token aged out.** `auth_create_token()`'s and `auth_http_issue()`'s own "does this email already have a token" reuse checks originally only tested `active`, never expiry — once a token crossed the TTL, both functions would keep handing back that same permanently-dead token forever, with no way for a caller to get a working one short of an admin-initiated `auth_revoke_by_email()` call. This is the same "denial looks like absence" failure class this project has named and fixed on sight in every prior phase (most recently Phase D's MVCC bootstrap gap), so it was fixed the same way: a shared `au_token_expired()` helper now backs all three decision points (`auth_validate_token()`, and the two reuse checks), and an expired slot triggers a genuine re-issue — a fresh token value in the same slot, not a refreshed timestamp on the stale one, so a leaked old token can't be silently revived by someone else re-requesting a token for that email.

**Standing demo accounts are deliberately exempt from TTL, via an explicit `no_expiry` flag rather than an overloaded sentinel value.** The four `auth_init()` accounts are local-dev credentials, not runtime session tokens — enforcing TTL against them would silently break the simulator's login flow after an hour of kernel uptime, contradicting this very phase's own "still fine for local dev" framing in the plan above. The original code used `created_tsc=0` as a "skip rdtsc during early boot" sentinel; since a real, safe read of `kernel_tick_counter` (a plain global, not the `rdtsc` instruction) turned out to carry no such early-boot risk, that sentinel is gone entirely — `created_tick` now always holds a real, meaningful tick value for every token, demo or runtime-issued, and a dedicated `no_expiry` field carries the "this one never dies" intent explicitly rather than overloading `0` to mean two different things.

**`sys_sls_secure_seal()` relabeled, not reimplemented, per the roadmap's own stated preference.** The function derives a password-based key and stores it in the kernel's key directory — nothing more. It never reads, transforms, or writes back the target object's actual stored data, so calling it left (and still leaves) that object exactly as readable as before. The `SLSSealRequest.encryption_algorithm_flags` field (`"e.g., 1 = ChaCha20-User"`) is accepted but was never consulted by anything — no cipher was ever run, ChaCha20 or otherwise. The runtime log previously claimed the object was "successfully encrypted"; that's now `"Key derived and stored... (data NOT re-encrypted -- key registration only)"`, and both `secure_api.h`'s struct comment and `secure_api.c`'s function header now say the same thing in more detail. The syscall number, function name, and shell command (`seal <name> <password>`) are unchanged — renaming those would touch `syscall_dispatch.c`, `shell.c`, and `docs/SLS-OS.md`'s own architecture description for a wire-format/UX change the underlying finding didn't actually require; the concrete problem was the misleading runtime claim, not the name "seal" itself, which is documented precisely enough now to not overpromise. Real encryption remains a distinct, much larger future project — it would need an actual cipher plus hooks into every object-storage read/write path this kernel has (rowstore, vecstore, legacy KV records) to transform data at rest, not just register a key nobody consumes yet.

**Verification.** New `auth_host_test.c` links the real, unmodified `kernel/auth.c` and defines `kernel_tick_counter` by hand (matching `timer.c`'s own definition — `timer.c` itself isn't linked, since it depends on the LAPIC/IDT arch layer this sandbox doesn't have), advancing it manually to simulate uptime. Seven scenarios, all passing: demo tokens validate correctly at boot; an unknown token fails cleanly with `out_uid=0`/`out_role=GUEST`; a freshly-issued token validates immediately; advancing the tick counter past the TTL expires the runtime token while the demo tokens keep validating unchanged; re-issuing for an expired email produces a genuinely different token value and permanently kills the old one; `auth_http_issue()`'s public HTTP-facing path shows the same issue/reuse-before-expiry behavior end-to-end; and `auth_revoke_by_email()` is confirmed unregressed. `net/http.c`, `user/shell.c`, `kernel/syscall_dispatch.c`, and `kernel/kernel.c` all syntax-check clean (`gcc -fsyntax-only`, exit 0) — the last of these wasn't in Phase D's own sweep list but is checked here since this phase edited `auth.h`, which `kernel.c` includes directly. `kernel/secure_api.c` syntax-checks clean too, modulo the same pre-existing implicit-declaration warnings the Phase C addendum already found and explained (that file still doesn't include `kernel_io.h` itself) — unrelated to this phase's own changes.

Not addressed in this pass, named rather than silently skipped: expired token slots are never reclaimed (bump-allocated, no-reclaim, matching this project's existing precedent for fixed-size registries under pressure, e.g. `row_constraint.h`'s `ROW_CONSTRAINT_MAX`) — a long-running kernel could eventually exhaust `AUTH_MAX_TOKENS=32` on dead slots even though `auth_create_token()`'s reissue path now reuses a matching email's own slot; a future pass could free slots outright on expiry rather than just on `auth_revoke_by_email()`. A related, narrower gap: `auth_http_issue()`'s reissue path always grants `uid=9999`/`ROLE_GUEST` on expiry (matching what it already did for brand-new emails), which is correct for every token this function can itself issue — but if `SYS_SLS_AUTH_CREATE` was used to hand someone an elevated role (DB_ADMIN/APP_USER) directly, and that token then expires, `POST /auth/token` will downgrade them to GUEST rather than restoring their real role; an admin has to re-run the create-token path to restore it. This is a real, if narrow, gap worth naming rather than papering over with a bigger change to `auth_http_issue()`'s request shape. TLS remains a distinct, larger future item, unchanged from the original plan's own framing — not needed for a localhost-only frontend.

## First real cross-compile findings (post-roadmap, Phase A territory)

Immediately after this roadmap's own six phases closed out, `make x86-run` was run against the real `x86_64-elf-gcc` toolchain for the first time in this project's history — confirming the gap analysis's own top finding was not hypothetical. The build failed on `user/shell.c:751: error: SSE register return with SSE disabled`.

**Root cause.** `X86_CFLAGS` (Makefile) has always included `-mno-sse -mno-sse2 -mno-mmx`, and the x86-64 System V ABI has no calling-convention path for returning a `float`/`double` by value without SSE registers (XMM0) — there is no x86-64 equivalent of the x87-based `-mfpmath=387` fallback GCC offers on x86-32. Every function in the codebase that returned `float` by value (`vs_sqrtf`/`vs_dot`/`vs_distance_cosine`/`vs_distance_l2` in `vecstore.c`, `vi_distance`/`vi_node_distance` in `vec_index.c`, `sh_atof` in `shell.c`, `json_float` in `http.c`) was therefore uncompilable under the project's own declared flags — a defect that every previous phase's "syntax-check clean" verification could never have caught, since none of those checks ever passed `-mno-sse` (host `gcc -fsyntax-only` was always run with default flags). Confirmed by direct experiment (host gcc reproduces the identical error with the identical flags) that the breakage is narrowly scoped to by-value float/double *returns* — float/double *parameters* and all-local float arithmetic compile and work fine under `-mno-sse` (GCC silently falls back to stack-passing for parameters and x87 instructions for local math), so no float storage format or numeric behavior needed to change.

**Also found, not yet acted on:** `arch/x86/lazy_fpu.c` + `arch/x86/switch_lazy.asm` implement a CR0.TS-triggered lazy AVX-512 context-switch mechanism (XSAVE/XRSTOR per task) that looks like it was meant to eventually let the kernel enable SSE/AVX properly, but it is unfinished scaffolding, not a working subsystem: no `#NM` (interrupt vector 7) IDT gate is registered anywhere, and `CR4.OSFXSR`/`CR4.OSXSAVE`/`XCR0` (via `xsetbv`) are never configured at boot. `switch_lazy.asm` does set `CR0.TS=1` on every context switch, so if SSE codegen were ever enabled without finishing this wiring, the very first float touch after any context switch would raise an unhandled `#NM` fault. Left entirely alone this pass — named for whoever picks up Phase A's real boot verification next, since finishing it (vs. the fix actually applied) is real new boot-path code that needs QEMU testing, not something a host compile can verify.

**Fix applied — chosen over finishing the SSE/AVX wiring, given to the user as an explicit choice.** Every by-value float-returning function was converted to an out-parameter convention (`void f(float* out, ...)` instead of `float f(...)`) — mechanical, behavior-preserving, no change to any numeric formula, precision, or storage layout. All call sites across `kernel/vecstore.c`, `kernel/vec_index.c`, `user/shell.c`, and `net/http.c` were updated to match. This keeps `-mno-sse` exactly as-is, matching the kernel's actual current (unfinished) FPU-switching state, rather than enabling a compiler codegen path the interrupt/context-switch machinery isn't actually ready to support safely.

**Verification.** Every file in the Makefile's `X86_C_SRC` list (all ~70 files) was compiled with host gcc using the exact `X86_CFLAGS` line from the Makefile (`-ffreestanding -O2 -Wall -Wextra -mcmodel=small -mno-red-zone -mno-sse -mno-sse2 -mno-mmx -fno-pie -fno-pic -fno-tree-vectorize`) — host gcc reproduces the identical SSE-disable behavior since this is an x86-64-architecture effect, not a toolchain-specific one (confirmed: the exact same error message, on the exact same line, from both compilers).
