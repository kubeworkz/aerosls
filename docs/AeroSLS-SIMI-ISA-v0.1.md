# AeroSLS SIMI: Rudimentary Instruction Set — Design Plan v0.2

**Status:** Draft for review — encoding, comparison model, ABI, and numeric semantics now resolved (see §3, §4, §8 changelog); floats confirmed deferred.
**Scope:** Core execution instruction set only (arithmetic, control flow, memory/pointer ops, calls). Object-typing, capability/authority instructions, and template-based translation hooks are explicitly deferred to later phases (see §7).

---

## 1. Where this fits

AeroSLS is building toward a System/38-style stack: Single Level Storage (SLS) as the addressing foundation, SIMI as the technology-independent instruction layer that sits between compiled application objects and the native hardware, and SLIC as the licensed-internal-code layer that implements SIMI and manages the machine underneath it. LPAR-style partitioning is a longer-term goal that depends on both being solid.

This document only covers step one: what a SIMI *instruction* looks like. Nothing here about object types, authorities, or partitioning — those come after there's a working bytecode to build on.

### 1.1 What we're borrowing, and from where

Two IBM lineages matter here, and they pull in different directions:

- **System/38 MI / SIMI** — the *architectural role*: a virtual ISA that is never interpreted. It is generated by compilers, embedded in the program object alongside (or instead of) native code, and translated to the host machine's real instructions before first execution. If the object moves to a machine with a different native ISA, it gets re-translated from the embedded MI/SIMI, not the native code. That's the "technology independence" property we want.
- **PL.8's intermediate language ("W-code" in the prompt's sense)** — the *instruction shape*: PL.8 compiles to "an intermediate machine-independent language with symbolic registers and machine-like operations" before its backend maps it to a real register file and ISA (801, S/370, POWER, etc.). That's a three-address, symbolic-register RISC-like IR — closer to LLVM IR or a load/store RISC ISA than to a stack machine.

These are not the same thing architecturally. Actual System/38 MI instructions are *not* simple fixed-format RISC ops — they're variable-length, opcode-plus-up-to-four-operand instructions where each operand is an index into an Object Definition Table (ODT) that supplies type, size, and location out of band. That's what makes MI opcodes generic (one `ADD NUMERIC` handles every numeric type — the ODT tells the machine how). It's powerful but heavyweight for a first cut.

The plan below takes System/38's *role* (embedded, translated-not-interpreted, technology-independent) but gives SIMI v0.1 PL.8-style *instruction shape* (fixed-format, symbolic-register, three-address, RISC-like) instead of the full ODT/template machinery. That's what "rudimentary" and "resembling W-code" both point at. Full ODT-style generic typing is a §7 phase, not v0.1.

## 2. Design principles for v0.1

1. **Never interpreted, always translated.** SIMI code is a compile-time artifact. SLIC translates it to native machine code before the object runs — at install time, first invocation, or on move to new hardware. There is no SIMI interpreter loop in the execution path (a *reference interpreter* is still useful for testing — see §6 — but it's a development tool, not part of the production architecture).
2. **Symbolic registers, not a stack.** Instructions name an unbounded set of symbolic (virtual) registers, not stack slots. SLIC's translator is responsible for allocating these to whatever real registers the host provides. This matches PL.8's IL and is what makes the format "W-code-like" rather than "p-code/JVM-like."
3. **Fixed-width, three-address encoding.** Every instruction is the same size and shape: opcode, up to three register operands, optional immediate. No variable-length operand streams, no out-of-band descriptor tables in v0.1. This is the single biggest simplification relative to real MI, and it's deliberate — it makes the assembler, translator, and object format all trivial to bring up first.
4. **Minimal but real typing.** Each register operand carries a small type tag (integer width, pointer, float) inline in the instruction rather than in a separate ODT. Enough to make load/store and arithmetic well-defined; not enough to be "generic opcodes" yet. That richer, MI-style out-of-band typing is exactly what §7 phases in.
5. **Embedded in the OS object, alongside a translation record.** Every program object carries its SIMI instruction stream plus enough metadata (entry points, register count, symbol/relocation info) for SLIC to regenerate native code without the original compiler. This is what gives you the System/38 "re-translate on new hardware" property.
6. **Addressing is SLS-native.** SIMI has no notion of physical addresses or segments-and-offsets in the traditional sense — all memory operands are single-level-store pointers. Pointer arithmetic and load/store instructions operate on that pointer model from day one, even before capability/authority checking exists.

## 3. Instruction encoding

Fixed 64-bit instruction word. v0.2 tightens the register fields from v0.1's placeholder 16-bit width down to 10 bits, and spends the reclaimed bits on a much wider inline immediate (see §8 changelog for the reasoning).

```
 63        56 55   52 51        42 41        32 31                          4 3   0
+-----------+-------+-------------+-------------+------------------------------+----+
|  opcode   | type  |    rD       |     rA       |          rB / imm           |flg |
|  (8 bits) |(4 bit)|  (10 bits)  |  (10 bits)   |          (28 bits)          |(4) |
+-----------+-------+-------------+-------------+------------------------------+----+
```

- **opcode (8 bits):** up to 256 instructions; §4 defines the v0.1/v0.2 subset (~34).
- **type (4 bits):** operand type tag — `i8 i16 i32 i64 u8 u16 u32 u64 f32 f64 ptr bool` (12 used, 4 reserved for later phases). Applies to the instruction's operation, not each register independently. Signed vs. unsigned integer variants of an opcode (e.g. `DIV`, `MOD`, arithmetic vs. logical shift eligibility) are selected by this tag, not by separate opcodes — see §4.7.
- **rD, rA (10 bits each):** symbolic virtual register numbers, 0–1023, **scoped per procedure** (renumbered from 0 at each `ENTER`, not a global SSA id). See §8 Q1 for why 1024 is enough and what happens when it isn't.
- **rB / imm (28 bits):** dual-purpose, selected by a flag bit:
  - *Register mode:* low 10 bits are `rB` (0–1023); upper 18 bits reserved, must be zero.
  - *Immediate mode:* full 28 bits, sign-extended, giving a range of roughly ±134 million — enough for the overwhelming majority of constants and load/store displacements without touching the literal pool. Values that don't fit still go through `LOADI64`'s two-word literal-pool form.
- **flags (4 bits):** per-opcode-class modifiers — register/immediate select, relation code on `CMP`, branch-invert bit on `BC`, signed/unsigned or volatile/atomic markers elsewhere. See §4 for per-opcode assignment.

This is intentionally closer to a MIPS/RISC-V-style fixed encoding than to MI's variable-length ODT-indexed stream — that's the "lower-level, W-code-like" trade-off the plan calls for. It is *not* meant to be the final encoding; §7 covers the migration path once generic/typed opcodes are needed.

## 4. Core opcode classes (v0.1)

### 4.1 Arithmetic / logical
`ADD SUB MUL DIV MOD` (int + float variants via type tag), `AND OR XOR NOT`, `SHL SHR SAR`, `NEG`.

### 4.2 Data movement
`MOV` (register-register), `LOADI` (immediate/literal), `LOADI64` (wide literal, two-word form).

### 4.3 Comparison — resolved: separate, register-based, no condition-code register
`CMP rD, rA, rB` — **no hidden flags/condition-code register.** SIMI has no implicit machine state (consistent with design principle #2 — everything is a named symbolic register), so `CMP` writes an ordinary `bool`-typed result into `rD`. The 4-bit flag field selects the relation being tested: `EQ NE LT LE GT GE LTU LEU GTU GEU` (10 codes used, 6 reserved). Signed vs. unsigned is therefore selected by the relation code for `LT/LE/GT/GE` vs. `LTU/LEU/GEU/GTU`, independent of the operand type tag.

Keeping `CMP` and `BC` as two instructions (rather than a fused compare-and-branch) was the deliberate choice: the result register can be reused by multiple downstream branches, and it gives SLIC's translator freedom to schedule the comparison away from the branch, or to lower it onto a host's hardware condition codes, register-compare, or predicate-register model as appropriate for that target — the IR itself stays target-agnostic either way.

### 4.4 Control flow
`BR` (unconditional, PC-relative via imm), `BC rA, imm` (conditional — branches if `rA` is nonzero, or if `rA` is zero when the invert flag bit is set; `rA` is normally a `CMP` result but any nonzero register works, matching common `BEQZ`/`BNEZ`-style RISC idioms), `CALL` (symbolic target — resolved by SLIC at translation time against the object's entry-point table), `RET`.

### 4.5 Memory / pointer
`LOAD`, `STORE` (both take an SLS pointer register + signed displacement via rB/imm — now up to 28 bits, see §3), `LEA` (compute pointer = base + offset without dereferencing), `PTRADD` (pointer + integer, typed by element size — placeholder until §7's real pointer/type model lands).

Pointers are a fixed **64-bit** width (matches the SLS single-level virtual address precedent). Instruction words, the literal pool, and all in-object data are encoded **little-endian** — an arbitrary but now-fixed choice, stated here so the object format and any cross-tool (dis)assembler agree.

### 4.6 Procedure frame
`ENTER` / `LEAVE` — bracket a procedure body, tell SLIC how many symbolic registers (≤1024, see §8 Q1) and how much local storage the translator needs to provision. Not "real" instructions in the ALU sense; more like directives, but keeping them in-stream (rather than in a side table) keeps the "everything is a flat instruction stream" property intact.

### 4.7 Numeric semantics — resolved
- **Overflow:** `ADD/SUB/MUL/NEG` wrap silently in two's-complement, matching plain hardware ALU behavior and most RISC ISAs. No trap. A checked/trapping-arithmetic variant is deferred to a later phase (flag-bit variant or new opcode) if AeroSLS ever needs it.
- **Division:** `DIV`/`MOD` by zero, and signed `MIN_INT / -1`, are **trap conditions**, not wraparound — division can't produce a meaningful wrapped result. The exact trap/exception delivery mechanism depends on SLIC's activation and exception infrastructure, which doesn't exist yet (Phase 4); this section fixes the *contract* now so Phase 1's reference interpreter and Phase 3's translator implement the same behavior from day one instead of drifting apart.
- **Shifts:** `SHL` (left, fills zero) is unambiguous either way. Logical vs. arithmetic right-shift are kept as **separate opcodes** rather than derived from the type tag: `SHR` (logical, zero-fill — valid on any type) and `SAR` (arithmetic, sign-extending — valid only on signed integer types; the assembler should reject `SAR` on a `u*` type tag rather than silently reinterpreting it). Shift amount comes from `rB`/imm, masked to 6 bits (0–63) regardless of operand width.

### 4.8 Calling convention (ABI) — resolved for v0.1/v0.2
Simplest workable option, chosen over adding dedicated argument-marshaling opcodes: **`r0`–`r7` are a reserved, caller-saved argument/return convention by documentation, not by hardware.** Up to 8 integer/pointer arguments go in `r0`–`r7` before `CALL`; the return value comes back in `r0` after `RET`. Anything beyond 8 arguments is passed via frame-local memory (`STORE` into the callee's incoming stack area, mirroring what every real ABI does once it runs out of registers) — exact frame-argument layout is a Phase 1 assembler/ABI-doc detail, not an ISA encoding concern.

This was chosen over first-class `SETARG`/`GETPARAM`/`SETRET`/`GETRET` opcodes to keep the opcode count down for a "rudimentary" v0.1 — it costs SLIC's register allocator a documented convention it must respect around call sites (values live across a call into `r0`–`r7` need to be treated as caller-saved), but adds zero new instructions. If that convention proves awkward once real procedures are being compiled (e.g. it collides badly with the per-procedure renumbering in §8 Q1), promoting to explicit marshaling opcodes is a candidate for Phase 6, not a v0.2 blocker.

### 4.8.1 Aggregate (struct-by-value) ABI convention — Gap Remediation Phase 14 Part A

**Zero new opcodes — a documented calling-convention extension on top of §4.8, exactly as scoped in §16 Phase 14.** SIMI's r0-r7 argument slots are already genuine memory-based marshaling (confirmed by Phase 12's own research into `emit_call_site`), so "does this struct fit in the argument registers" isn't a hardware question here the way it is on a real machine ABI — every slot is just an 8-byte cell regardless of what bit pattern it holds. The convention:

- **Aggregates ≤8 bytes** are passed and returned **by value**, occupying exactly one r0-r7 slot like any scalar. A caller packs multiple sub-8-byte fields into one slot with ordinary `SHL`/`OR` (and a callee unpacks with `SHR`); no new marshaling primitive is needed because this is just integer bit-packing.
- **Aggregates >8 bytes** are passed **by invisible reference**: the caller builds its own on-stack copy of the aggregate (`STORE` into frame-local memory), then `LEA`s a pointer to it into an argument slot. The callee treats that slot as an ordinary pointer, addressing individual fields with `LOAD`/`STORE`/`PTRADD` exactly as it would any other pointer argument.
- **Returns >8 bytes** use the classic **`sret`** pattern: the caller allocates space for the result (again, frame-local memory) and passes a pointer to it as an extra leading argument; the callee `STORE`s each field through that pointer instead of returning through r0. r0's own contents after such a call are unspecified by this convention (a real compiler front end may additionally choose to have the callee echo the sret pointer back through r0 for chaining, but nothing in the ISA or translators requires it).

All three cases compose entirely from opcodes that already existed before Phase 14 (`LEA`, `LOAD`, `STORE`, `PTRADD`, `SHL`/`SHR`/`OR`, `CALL`, `RET`) — there is no ABI-specific opcode and no object-format change. This is purely a frontend/compiler lowering responsibility, matching the original design decision in §16.

**Verified** by `tools/simi/tests/aggregate_abi.simi` — see §16 Phase 14's "Implemented — findings" addendum for the concrete test shape and three-way (interpreter / x86-native / RV64) results.

### 4.9 Fixed-point math (interim substitute for floats)
Since floats are deferred (§8 Q3), fixed-point is available now as a **pure software convention on top of existing integer opcodes — no new instructions required**:
- Represent fractional values as `Q16.16` in an `i32` register or `Q32.32` in an `i64` register (scale factor is a frontend/compiler convention, not something SIMI encodes).
- `ADD`/`SUB` work unmodified — fixed-point addition is just integer addition at a common scale.
- Multiply: `MUL` the two scaled values, then `SAR` by the fractional bit count (16 or 32) to bring the result back to the right scale.
- Divide: `SHL` the dividend by the fractional bit count first, then `DIV`, to avoid losing fractional precision.
No `MULFX`/`DIVFX` opcodes are being added in v0.1 — the two-instruction idiom above is enough to unblock fractional math for now. If profiling later shows that idiom is a hot-path cost, a fused fixed-point multiply/divide pair is a small, self-contained Phase 6 addition.

That's roughly 32–34 opcodes — enough to translate straight-line arithmetic, conditionals, loops, and non-virtual procedure calls with a real (if minimal) ABI. No object creation, no capability/pointer-authority instructions, no floating point, no generic/ODT-typed ops — all explicitly out of scope for v0.1/v0.2 per §7.

## 5. Object embedding format (sketch)

A SIMI-bearing OS object needs, at minimum:

1. **Header** — object type, SIMI version, entry point count, symbolic register count (or "high water mark"), literal pool size.
2. **SIMI instruction stream** — the fixed-width words from §3, back to back.
3. **Literal pool** — wide constants (`LOADI64` targets, float constants).
4. **Entry point table** — symbolic name → instruction offset, for `CALL` resolution both within the object and from other objects.
5. **Relocation/reference table** — external symbols the translator must bind at translate time (calls out to other objects, referenced by name/pointer, not by resolved address — consistent with SLS single-level addressing).
6. **(Optional, post-translation) Native code cache** — the last-translated native instructions for the current host, plus a fingerprint of that host, so SLIC can skip re-translation when nothing has changed. This is what System/38-style hardware migration invalidates and regenerates.

This is deliberately thinner than System/38's Program Template (instructions + ODT + user data) — there's no ODT because v0.1 has no out-of-band operand descriptors. Adding one is the natural extension point in §7.

## 6. Toolchain needed to validate this

A spec is only real once something can run against it:

1. **Assembler** — hand-writable text form of §3/§4, emits the §5 object format. Needed before any compiler backend exists, for hand-testing.
2. **Reference interpreter** — a *development-only* tool that directly executes SIMI for correctness testing. This deliberately violates principle #1 ("never interpreted in production") but is standard practice for bringing up a new ISA — you need a ground truth before you trust a translator.
3. **Disassembler** — for debugging translator output and object dumps.
4. **Native translator (one target first)** — SLIC-side component that walks the instruction stream and emits real machine code for a single initial host architecture. Register allocation here can start as simple as linear-scan; it doesn't need to be good, it needs to exist.
5. **Test suite** — small SIMI programs (arithmetic, loops, calls) with known outputs, run through interpreter and translator, results compared.

## 8. Open questions — resolutions (v0.2)

**Q1. Register count ceiling per procedure — resolved: 1024 (10-bit field), scoped per procedure, not global.**
The 16-bit field in v0.1 was not free — every bit spent on register width is a bit not spent on immediate range, and it bloats every single instruction word regardless of whether that instruction needs a big register number. Bit budget for the 64-bit word: 8 (opcode) + 4 (type) + 4 (flags) = 16 bits fixed overhead, leaving 48 bits for operands. v0.1 spent that as 16/16/16 (three register-sized fields, immediates capped at 16 bits). v0.2 spends it as 10/10/28: two 10-bit register fields plus a 28-bit register-or-immediate field — register ceiling drops from 65,536 to 1024, and the payoff is a ±134M-range inline immediate instead of ±32K, which cuts a lot of literal-pool traffic for everyday constants and displacements.

The reason 1024 is enough, not just "smaller": register numbers are **scoped per procedure** and reset to 0 at each `ENTER` (§4.6) — this was already implicit in v0.1 but is now stated explicitly because it's load-bearing for this answer. That means the ceiling isn't "1024 registers total across the program," it's "1024 distinct symbolic temporaries live within one procedure's numbering," which is a different and much easier bar to clear. The remaining risk is a frontend that does naive SSA numbering (fresh id per assignment, no coalescing) on a very large generated procedure. Rather than widen the encoding for that edge case, the contract is: frontends must renumber/coalesce densely before emission (standard compiler practice — real backends do this regardless of encoding width), and any procedure that still can't fit under 1024 live temporaries spills the excess to frame-local memory via ordinary `LOAD`/`STORE`, exactly like a physical register allocator does under real register pressure. No ISA change needed for that case.

**Q2. CMP/BC — resolved: kept separate, register-based (no hardware condition-code register).**
See §4.3 for the encoding. The deciding factor beyond "SSA-friendliness": SIMI's own design principle #2 already commits to no implicit machine state (symbolic registers, not a stack, not hidden flags), so a condition-code register would have been an inconsistency with the rest of the ISA, not just a stylistic choice. No code-size or hardware constraint is currently known that would push toward fusion — if one shows up once a real target's calling/branch costs are measured in Phase 3, fusing is a localized encoding change, not a redesign.

**Q3. Floats — resolved: deferred; integer/pointer-only for v0.1/v0.2.**
Three items identified as the actual remaining gaps for the integer/pointer core (now resolved above rather than left open):
1. Integer overflow/wraparound and division-trap contract — §4.7.
2. Calling convention for `CALL`/`RET`/`ENTER`/`LEAVE` — §4.8.
3. Pointer width and endianness — §4.5 (64-bit pointers, little-endian).

Fixed-point as an interim substitute for fractional math is specified in §4.9 as a pure software convention on existing opcodes — no new instructions needed for v0.1.

**Q4. Object format reuse vs. divergence from the existing SLS object header — resolved by Phase 2 (see §9).**
Looked at the real `kernel/object_catalog.h` and `kernel/loader.c` in the AeroSLS repo. Short answer: reuse the §5 layout byte-for-byte as the *payload*, don't touch the catalog entry struct itself. Details in §9.

## 9. Phase 2 findings — wiring §5 into the real AeroSLS repo

Connected the actual `aerosls2` codebase and read `kernel/object_catalog.h`, `kernel/loader.c/.h`, `kernel/process.c`, and `kernel/persist.c` rather than guessing. Findings:

**The catalog entry (`struct SLSObjectEntry`) is metadata-only** — object_id, name, type, `base_vaddr`, `size_pages`, owner/tier/role/perm, active flag. It does not carry program bytes. `OBJ_TYPE_PROGRAM` (type 7) already exists, documented as "the SLS-native replacement for a filesystem exec." Good news: no new catalog object type was needed for SIMI — it slots in as a third payload format under the type that already exists.

**Program bytes actually live in a separate staging array**, not in SLS-mapped memory: `struct ServiceBinary service_binaries[MAX_SERVICE_BINARIES]` in `loader.c` — 16 slots, 16 KiB each, uploaded in chunks via `SYS_SLS_UPLOAD_BINARY`, then mapped into a process's page table at spawn time by `loader_load_into_process()`. That function already switches on a detected format (`is_elf` magic-sniffed at upload time) between an ELF64 loader and a flat-binary loader, both of which map the bytes as **directly executable** native pages. This is the exact seam SIMI needed: a third format, detected the same way, that does *not* get mapped executable (design principle #1 — never interpreted, never run as native code, always translated first).

**What Phase 2 actually changed** (in the real repo, not just this doc):
- `kernel/loader.h` — added `SIMI_MAGIC`, `struct SimiObjectHeader` and `struct SimiEntryRec` (packed, byte-identical to the Phase 1 host tool's `.tmo` layout — verified, see below), an `is_simi` flag on `ServiceBinary`, and a `SYS_SLS_SIMI_INFO` (173) syscall number.
- `kernel/loader.c` — `simi_validate()` (magic + bounds-checked header, rejecting truncated/corrupt uploads before anything walks the arrays), SIMI detection alongside the existing ELF magic-sniff in `sys_sls_upload_binary()`, a `loader_simi_info()` introspection function, and a new branch in `loader_load_into_process()` that validates a SIMI object and then **refuses to spawn it** with a clear message — instead of falling through to the flat-binary path, which would have mapped bytecode words in as fake x86-64 instructions and crashed (or worse).
- `kernel/syscall_dispatch.c` — wired `SYS_SLS_SIMI_INFO`.
- `net/http.c` — three sites that reported `is_elf ? "ELF64" : "flat"` (JSON object listing, upload response, binary-download header) now go through a shared `binary_format_name()` helper so a SIMI upload doesn't get silently mislabeled "flat" over the HTTP API.
- `kernel/object_catalog.h` — one-line comment update on `OBJ_TYPE_PROGRAM` noting the SIMI sub-format.

**Verification:** no `x86_64-elf-gcc` cross-compiler is available in this environment, so I couldn't build the actual kernel binary. Two things I could and did check: (1) the modified/touched files (`loader.c`, `loader.h`, `http.c`, `syscall_dispatch.c`, `object_catalog.c`) all compile clean with host `gcc -ffreestanding` — not a perfect stand-in for the real cross-compiler, but it does catch type errors, undeclared identifiers, and struct mismatches; (2) I extracted `simi_validate()`'s exact logic into a standalone host program and ran it against all 6 real `.tmo` files produced by the Phase 1 assembler — all 6 validated with the correct instruction/literal/entry counts and correct entry-point names, and two deliberately corrupted files (truncated, bad magic) were correctly rejected. That's a real, tested proof that a `.tmo` from `simi-asm` is byte-compatible with the kernel's parser, not just an assertion that it should be.

**A pre-existing gap this surfaced, not caused by SIMI:** `OBJ_TYPE_PROGRAM` objects don't actually live in single-level storage today — they're staged in that bounded 16×16 KiB side array and only touch the SLS-mapped `base_vaddr` range at spawn time (as native page-table mappings, not as a persistent object). That's a real divergence from "everything is a persistent SLS object," but it predates SIMI and applies equally to ELF/flat programs — not something to silently paper over inside the SIMI work. Worth a decision later: whether Phase 3 (native translator) should also be the point where PROGRAM objects become properly SLS-resident (bytecode *and* translated-native-cache both persisted as real objects, per §5/§7 phase 4's original intent), or whether that's tracked as a separate, non-SIMI cleanup.

## 10. Phase 3 findings — the native x86-64 translator, and proof it actually works

**Design chosen:** naive load-operate-store codegen — every symbolic register is a memory slot in an rbp-relative stack frame (`TX_MAX_REGS = 64` slots, a documented v1 narrowing from the ISA's 1024 ceiling), not a real allocated host register. Every SIMI instruction becomes "load operand(s) into RAX/RCX, compute, store result back." Correct over fast, deliberately — the goal was proving the translate-and-map-and-run pipeline end to end, not a competitive JIT. Real register allocation is future work, not a blocker.

**Calling convention:** `CALL`/`RET` use real x86-64 `call`/`ret` (so nested/recursive SIMI calls get a real hardware stack for free), with r0–r7 marshalled through a small outgoing-argument area on the stack at each call site, and the callee's `ENTER`-emitted prologue reading them back into its own frame slots. This is the concrete mechanism behind §8 Q1's "register numbers reset per procedure" and §4.8's "r0–r7 argument convention" — those two answers only work together if something physically copies values across the frame boundary, and this is that something.

**Two hard problems, both resolved:**
1. *How does a translated SIMI program get a legitimate memory address to `LOAD`/`STORE` against?* SIMI has no real object/pointer allocation model yet (that's §7 Phase 6/7). Resolved with a stopgap: register **r7 is reserved by Phase 3 trampoline convention** as a pointer to a small scratch page the trampoline provides — a real mapped address, not part of the ISA spec itself. `mem_ops.simi` (Phase 1, uses address 0 — valid only in the interpreter's simulated memory) was left untouched, and `mem_ops_native.simi` was added alongside it using the r7 convention for real hardware.
2. *Where does that scratch page's address come from, when the trampoline needs to bake it in as an immediate before the page is mapped, and the mapping address depends on the translated code's size?* Resolved by translating twice — once with a placeholder to learn the byte length (the scratch pointer's numeric value never changes how many bytes get emitted, only what immediate gets baked in), then once for real with the now-known scratch address. Cheap, deterministic, no patch-after-the-fact fragility.

**Verification — this is the part worth being precise about, because "I wrote a translator" and "I proved a translator correct" are different claims:**
- No `x86_64-elf-gcc` cross-compiler or QEMU is available in this environment, so the actual kernel binary was not built or booted.
- What *was* done: the translator core (`simi_x86.c`) was written as a portable, no-libc file — the same file living at both `outputs/simi_x86.c` (host toolchain) and `kernel/simi_x86.c` (kernel), byte-identical. A host test harness (`simi-jit-test`) translates a real `.tmo` from the Phase 1 assembler, `mmap`s the output `PROT_EXEC`, and **calls it as a real function on the host CPU** — genuine execution, not a simulation or a static read of the bytes.
- First run: 5 of 6 tests passed, one crashed. The crash was real and useful — it caught a genuine ModRM byte-encoding bug (`0x81` used where `0x88` was correct, a reg/rm-field arithmetic mistake affecting every `LOAD`/`STORE` variant) that static review had missed. Fixed, re-verified: **9/9 programs pass under the Phase 1 interpreter, 8/8 applicable programs pass under real native execution** (one skipped by design — see below), covering the full opcode set: arithmetic, `DIV`/`MOD`, shifts, signed and unsigned `CMP` relations, `BR`/`BC`/`CALL`/`RET`, typed `LOAD`/`STORE`, `LEA`/`PTRADD`, `LOADI`/`LOADI64`. Two new test programs (`extra_ops.simi`, `ptr_ops.simi`) were added specifically to exercise opcodes the original six didn't touch.
- The kernel-side files (`kernel/simi_x86.c`, `kernel/simi_translate.c`, and the modified `kernel/loader.c`) all compile clean under host `gcc -ffreestanding` — not the real cross-compiler, but real evidence against type errors, undefined references, and struct mismatches, which is what actually broke last time (a `struct TxEntryRec` name collision between `loader.h` and `simi_x86.h`, caught immediately by this same check and fixed by renaming).

**What's still unverified, stated plainly:** whether this actually boots and runs correctly inside real AeroSLS — page-table mapping, the `SYS_SLS_EXIT` handoff, and the kernel's own scheduler/paging code have not been exercised by anything in this session. That requires the real toolchain and QEMU, neither available here.

**Kernel wiring:** `loader_load_into_process()`'s SIMI branch (Phase 2's "refuse to spawn" stub) now calls `simi_translate_and_map()`, which runs the translator into a static 192 KiB output buffer (no general-purpose allocator exists in this kernel — matches the codebase's existing static-array discipline), maps the resulting code executable page by page via the same `allocate_physical_ram_frame()`/`user_map_page()` calls the ELF64/flat paths already use, maps one additional R+W scratch page for r7, and appends a small kernel-specific stub (kept out of the portable translator core) that turns the translated program's return value into a `SYS_SLS_EXIT` syscall. **v1 limitation:** only a SIMI object's `.entry main` is spawnable — no plumbing exists yet to choose among multiple entry points through `program_spawn()`'s call chain.

**Known, explicitly flagged gap:** `DIV`/`MOD` by zero and signed `MIN_INT / -1` are left to real x86-64 hardware `#DE` (divide error) rather than a software-checked trap. The ISA spec's §4.7 calls these trap conditions, and hardware `#DE` is literally that — but AeroSLS doesn't yet wire an ISR0 handler for user processes, so today an unhandled `#DE` likely hangs or resets rather than delivering a clean fault. Not solved here; it's process-fault-handling infrastructure, not specifically a SIMI problem, similar in kind to the `OBJ_TYPE_PROGRAM` staging gap flagged in §9.

## 11. Phase 4 findings — activation cache (translate-on-first-use)

**What "activation" means here, concretely:** on System/38, `CREATE PROGRAM` compiles source to MI and stores it; the separate *activation* step — first call — translates MI to native machine code and caches that translated form, so a program is translated once and invoked many times, not re-translated per call. Phase 3 didn't have that split: `loader_load_into_process()` called `simi_translate_and_map()` on every single spawn, re-running the full two-pass encoder from scratch each time. Phase 4 closes that gap.

**The key architectural fact that made this simple rather than requiring a redesign:** `process.c`'s `program_spawn()` (the `OBJ_TYPE_PROGRAM` path SIMI objects use) always maps a program's code at the same fixed constant, `USER_PROC_CODE_BASE`, in every process's page table — not a per-spawn or per-process-chosen address. That's true single-level-store semantics: an object's effective address doesn't depend on who's looking at it. It also means the translated code's internal offsets and the r7 scratch-pointer `movabs` immediate baked into the trampoline (§10) are identical across every activation of the same object — nothing about the cached bytes needs to change per process. That's what makes sharing physical code frames across processes safe without any general capability for shared-memory mapping elsewhere in the kernel: this one case happens to satisfy the precondition for free.

**Design:** a bounded `struct SimiActivation g_activations[MAX_SERVICE_BINARIES]` in `kernel/simi_translate.c` (one slot per possible SIMI object, mirroring `ServiceBinary`'s own capacity — consistent with the codebase's static-array-everywhere discipline, no allocator involved). Each slot records the object name, an FNV-1a hash + byte size of the `.tmo` content it was translated from, and the physical frame addresses of the translated code (code + appended exit stub — both immutable at runtime, so both are safely shared).

- **Cache hit** (`simi_translate_and_map()` finds a slot whose name, hash, and size all match the current upload): skips the translator entirely. Loops over the cached frame array and `user_map_page()`s each one — same physical frames — into the new process's page table as R+X. Allocates one **fresh** physical frame for the scratch page (never shared: SIMI code writes through r7, so two concurrent processes sharing one scratch frame would corrupt each other's transient state) and maps it R+W+NX at the same scratch vaddr the cached code already expects.
- **Cache miss** (no slot yet, or the object was re-uploaded with different content — hash/size mismatch): runs the existing Phase 3 two-pass translate-and-stub-append pipeline unchanged, then additionally stashes the resulting frame pointers into the object's activation slot before returning, so the *next* spawn takes the hit path.
- **Invalidation:** keyed on an FNV-1a hash of the uploaded bytes (not just the object name), so re-uploading the same-named object with different bytes is correctly detected as a cache miss rather than serving stale native code. The hash is not cryptographic and doesn't need to be — the only question it answers is "did this object's bytes change since I last translated it," and an object's own owner already controls its own bytes.
- **Known, explicitly flagged gap:** a re-upload that invalidates a cache entry does not reclaim the old translation's physical frames — this kernel has no frame-free path anywhere yet (not for `ServiceBinary` slots, not for the process table, nothing), so the old frames are simply overwritten in the activation struct and orphaned, exactly like every other static-array resource here today. Not a regression Phase 4 introduces so much as the same pre-existing gap now applying to one more resource type. A real fix needs kernel-wide frame refcounting/freeing, out of scope for SIMI specifically.
- **Diagnostics:** `simi_activation_info()` reports cache state (page count, entry offset, content hash, or "not yet translated") and is now called from `loader_simi_info()`, so the existing `simi-info` report shows activation status alongside the header/entry-point dump it already printed.

**Verification:** no cross-compiler is available in this sandbox (same constraint as Phases 2–3), so the kernel binary wasn't rebuilt here. What was checked: (1) the FNV-1a hash function was copy-pasted into a standalone host test and run against all 9 real `.tmo`-source `.simi` files in `tools/simi/tests/` — deterministic (same bytes hash the same twice), zero pairwise collisions across all 9, and a single-byte mutation of one file changes its hash, confirming the invalidation trigger actually fires on content changes rather than being a no-op; (2) both modified files, `kernel/simi_translate.c` and `kernel/loader.c`, compile clean with zero warnings under host `gcc -fsyntax-only -ffreestanding -Wall -Wextra -std=c11` — the same catch-type-errors-and-undeclared-identifiers check used in Phases 2–3, which is what caught real bugs (the `TxEntryRec` collision, the `0x81`/`0x88` ModRM bug) before this session.

**What's still unverified, stated plainly:** whether a cache hit's re-mapping of shared frames into a second process's freshly-cloned page table actually works correctly on real hardware — i.e., whether spawning the same SIMI object twice concurrently produces two working processes that don't corrupt each other — has not been exercised by anything in this session. That requires the real toolchain, a running kernel, and at least two concurrent spawns of the same object, none of which are available here. The architectural argument above (fixed `base_vaddr`, private scratch frame) is sound on paper; it hasn't been run.

**Addendum — this was subsequently verified for real, and it surfaced a genuine pre-existing kernel bug unrelated to SIMI.** With the user's own toolchain and QEMU (this session still has neither), the full pipeline was exercised end to end via the real HTTP API: upload a `.tmo`, spawn it (cache MISS, translate), spawn the same object again (cache HIT, reuse frames). The first spawn worked; the second triple-faulted the VM. The serial log showed a clean exit for PID 100 (`exited (code=5)`) followed by a `program_spawn` line for PID 101 with no matching exit — QEMU's exception trace (`-d int -no-reboot`) pinned it to a General Protection Fault at a kernel address with a garbage `RSP`, cascading through a double fault into a triple fault.

Root cause, and it has nothing to do with the activation cache: `process_exit()` (`kernel/process.c`, reached via `SYS_SLS_EXIT`) restores the kernel's `CR3`/`RSP` and `ret`s straight back into `program_spawn()`'s C code, bypassing `syscall_entry_stub`'s (`arch/x86/syscall.asm`) matching `swapgs` in `.syscall_return`. `syscall_entry_stub` does `swapgs` on entry so `[gs:0]`/`[gs:8]` address this core's `PerCPUData` (`user_rsp`/`kernel_rsp`, `arch/x86/user_paging.h`); skipping the exit-side `swapgs` leaves `GS_BASE`/`KERNEL_GS_BASE` permanently swapped after the *first* process ever exits this way. The *next* process's own syscall entry then swaps them the wrong direction — `GS_BASE` becomes `0` instead of `&per_cpu_data[0]` — so `mov rsp,[gs:8]` reads from physical address `0x8` instead of the real per-CPU struct. That's exactly the garbage `RSP` in the crash dump. This is why it never showed up before: nothing in this project had ever spawned a process, had it exit via a syscall, and then spawned a second one in the same boot session until this activation-cache verification actually exercised that sequence.

Fix: one extra `swapgs` added to `process_exit()`'s inline asm, immediately before the `CR3`/`RSP` restore, undoing the entry-side `swapgs` exactly as `.syscall_return` would have. Verified afterward with two full MISS-then-HIT cycles back to back against two different objects (PIDs 100-103), all four exiting cleanly with the correct result (`code=5`). This closes the "what's still unverified" gap above for real: the activation cache's shared-frame re-mapping across independently-cloned page tables is now confirmed correct on (emulated) real hardware, not just architecturally sound on paper.

## 12. Phase 5 findings — second native target (RV64), and what "prove re-translation-on-migration works" actually means

**Scoping this against what actually exists.** The roadmap's Phase 5 line reads "second native target, to prove re-translation-on-migration actually works." Two things had to be checked before writing any code: whether this sandbox can build/run RV64 code at all, and whether the RISC-V side of the AeroSLS kernel has anywhere to wire a translator into. Neither was available: no `riscv64-unknown-elf-gcc`, no `qemu-riscv64`/`qemu-system-riscv64`, and no root to install either (same hard constraint Phase 2/3 hit for the x86 cross-compiler). More importantly, `kernel/kernel_riscv.c` is a 10-line boot stub — `process.c`, `loader.c`, and `object_catalog.c` are x86-only in the root `Makefile`'s `X86_C_SRC`; the RISC-V kernel has no process/loader/object-catalog subsystem at all yet. So Phase 5's real, honest deliverable is the host toolchain: a second translator that consumes the exact same `.tmo` bytecode format and produces correct results, proving the retargeting claim at the level where it's actually checkable right now. Kernel-side wiring is deferred, blocked on RISC-V kernel infrastructure that's a separate, non-SIMI undertaking — not something to fake by wiring into a subsystem that doesn't exist.

**Design, mirrored deliberately against `simi_x86.c`:** `tools/simi/simi_riscv.c` uses the identical naive load-operate-store model (every symbolic register is a stack slot, not a real allocated register) and the identical real-call-instruction calling convention (`jal ra,target` / `jalr x0,0(ra)` in place of x86's `call`/`ret`, still giving nested/recursive SIMI calls a real call chain), so the two translators are easy to audit side by side. A few things came out differently, forced by real ISA differences rather than by choice:
- **No movabs equivalent.** RV64 has no single instruction that loads an arbitrary 64-bit constant. Every 64-bit value the translator needs — `LOADI64` literals, the trampoline's `scratch_ptr` — goes through a literal pool appended after all code, loaded via the standard `auipc rd,%pcrel_hi; ld rd,%pcrel_lo(rd)` pattern real RISC-V toolchains use for symbol loads, resolved in a second patch pass once the pool's address is known (the same two-pass shape as the branch/call fixups, just a second independent fixup list).
- **12-bit immediates.** RV64 load/store/addi immediates are 12 bits (±2048), far short of SIMI's 28-bit displacement field. Every `LOAD`/`STORE`/`LEA`/`PTRADD` displacement is materialized into a register first (a plain `ADDI` if it fits, the literal-pool path otherwise) rather than relying on the instruction's own immediate field the way x86's disp32 could.
- **`ra` isn't pushed automatically.** x86's `call` pushes a return address for free; RV64's `jal` just clobbers `ra`. Every procedure's prologue/epilogue explicitly saves/restores `ra` around its own body, and the trampoline does the same around its one call into the entry procedure — x86's trampoline needed no equivalent bookkeeping.
- **CMP and DIV/MOD are simpler, not harder.** RV64's M-extension `div`/`divu`/`rem`/`remu` are independent single-result instructions — no `cqo`/paired-register dance like x86's `idiv`. All 10 CMP relations reduce to 1-2 instructions each via `slt`/`sltu`/`xori`/`sub` (see `emit_cmp()`), and RV64 has no flags register at all, so — unlike x86 (§8 Q2) — there was never a "should CMP set implicit machine state" question to weigh; register-based comparison is just how the ISA works.

**A real bug, caught the same way the x86 ModRM bug was — by execution, not review.** The first version of `reg_disp()` placed symbolic register slot 0 at `[s0-8]` and slot 1 at `[s0-16]` — exactly where the prologue also saves the caller's `ra` and `s0`. Static review missed it (it's a one-line arithmetic choice that looks fine in isolation). Running `add.tmo` through the verifier below produced a wrong return value that traced back to `RET`'s `ld ra,8(sp)` reloading `5` (a value the test program had stored into register slot 0) instead of the real return address — the two were the same memory cell. Fixed by shifting every slot down by 16 bytes so the register-slot region sits below, not overlapping, the saved `ra`/`s0` pair (see the comment on `reg_disp()` in `simi_riscv.c`). A second, smaller bug — a sign-extension cast in the verifier's own `sext()` helper landing an intermediate result in `uint32_t` before narrowing to `int32_t`, so negative offsets silently became huge positive ones — was caught the same way, one level down: it's a bug in the verification tool, not the translator, and would never have been visible without single-stepping actual execution and noticing `sp` jump by `2^32`.

**Verification — and an honest comparison to Phase 3's standard.** No riscv64 cross-compiler and no QEMU means the encoder's output was never run on real or emulated RISC-V hardware. What was built instead, in the same spirit as Phase 1's reference interpreter existing to give something independent to check against: `tools/simi/rv64_exec.c`, a small purpose-built RV64 decoder+executor covering exactly the instruction encodings `simi_riscv.c` emits (no CSRs, no FP, no atomics, no compressed instructions, no traps — not a general RISC-V simulator). `tools/simi/simi_riscv_verify.c` translates a real `.tmo`, feeds the bytes into this executor with a sentinel return address standing in for "the call has returned," and checks the result register. This is real execution of the actual encoded bytes — it caught two genuine bugs static review didn't — but it is **weaker evidence than Phase 3's real-CPU execution**: a matching bug in both the encoder and this independently-written decoder wouldn't be caught, whereas x86's `mmap`+`PROT_EXEC`+call ran the translator's output on silicon that has no opinion about what the encoder intended. Stated plainly, not glossed over. All 8 applicable programs from the same test suite Phase 3 uses (`mem_ops.simi` skipped for the same address-0 reason it's skipped in the x86 native suite) pass with results identical to the x86 target — `make test-native` and `make test-riscv` from `tools/simi/` run the same `.tmo` files through both targets and agree on every result, which is the concrete, checkable form of "the same bytecode retargets to a second ISA."

**What's still unverified, stated plainly:** whether this encoder's output runs correctly on real or QEMU-emulated RV64 hardware — the executor above is an independent implementation, not a reference, and independent-implementation agreement is meaningfully weaker evidence than real hardware execution (see above). Also unverified: everything about kernel integration — there is no RISC-V process/loader subsystem to wire into yet, so unlike x86's Phase 3→4 progression, there's no "port into the kernel" step to even attempt here. Both are natural follow-ups once a riscv64 toolchain/QEMU is available and the RISC-V kernel grows a process/loader subsystem, respectively — the second of which is a substantially larger, non-SIMI undertaking.

## 13. Phase 6 findings — object-typed operations (ODT-equivalent), and the first opcodes that make real calls out of translated code

**What "MI-style generic/typed operations" meant in practice.** The roadmap line for Phase 6 pointed at System/38's ODT (Object Definition Table) — the mechanism by which MI instructions reference other system objects by name/handle rather than by raw address. SIMI's answer, kept deliberately small: one new type tag (`T_OBJREF`) and three new opcodes — `RESOLVE rD, "name"` (look up a live SLS object by name, return its `base_vaddr` as a `T_OBJREF`, or 0 if not found), `OBJSIZE rD, rA, TYPE` (byte size of the object a `T_OBJREF` points at), and `OBJTYPE rD, rA, TYPE` (its `SLSObjectType` tag). This is the first point where translated SIMI code needs to ask the *host environment* a question it cannot answer from its own bytecode alone — every prior opcode was self-contained arithmetic/control-flow/memory-access over values already in registers or the object's own bytes.

**Object format v0.2 → v0.3: a fourth pool, a breaking header change.** `RESOLVE`'s name operand is resolved at assemble time to an index into a new name pool appended after the entry table — fixed 32-byte slots (`SimiName`/`TxEntryRec`-shaped), deduplicated by exact string match (`tools/simi/simi_asm.c`'s `find_or_add_name()`), populated either explicitly via an optional `.name "..."` directive or implicitly on `RESOLVE`'s first use of a given string. The `.tmo` header grew a fifth `uint32_t` field (`num_names`), which shifts every fixed-offset calculation after it — the same "clean bump, no back-compat shim" call made for every prior format change in this project (nothing external consumes `.tmo` files; everything is regenerated from source by this same toolchain). `simi_isa.h`, `simi_obj.c`, `simi_asm.c`, `simi_interp.c`, `simi_disasm.c`, `simi_x86.h/.c`, `simi_riscv.h/.c`, and `kernel/loader.h/.c` all needed their header-parsing/size-accounting updated in lockstep; a compile-time or first-test-run mismatch would have caught a missed one immediately (`simi_validate()`'s 64-bit arithmetic size check is exactly the belt-and-suspenders that exists for this).

**The runtime-call ABI: `r6` joins `r7`, and translated code makes real C-ABI calls.** `RESOLVE`/`OBJSIZE`/`OBJTYPE` can't be pure codegen the way `ADD` or `LOAD` are — answering them means querying a live, environment-specific object catalog. The design: every native translator gains three new parameters — `rt_resolve_fn`, `rt_objsize_fn`, `rt_objtype_fn` — addresses of real C functions (`uint64_t simi_rt_resolve(const char*)`, `uint64_t simi_rt_objsize(uint64_t)`, `uint64_t simi_rt_objtype(uint64_t)`), baked into the generated code as translate-time constants exactly the way `scratch_ptr` already was for `r7`. A new reserved register, **`r6` = `namepool_ptr`**, mirrors `r7`'s existing scratch-pointer convention: the translator computes it *itself* from `obj_data` (a raw pointer straight into the SIMI object's own name-pool bytes, never copied) and bakes it into every trampoline, so `RESOLVE` can compute `namepool_ptr + idx*32` and hand that string pointer to `rt_resolve_fn` with zero marshaling. On x86 this is `movabs r11,fn_addr` / `call r11` (real indirect call, wrapped in an explicit `mov r10,rsp` / `and rsp,-16` / … / `mov rsp,r10` realignment — unlike this translator's own intra-object `call`s, the callee here is a normal compiler-generated C function that may assume strict SysV 16-byte stack alignment for its own stack-local spills, an assumption the rest of this naive codegen never had to satisfy before). On RV64 it's the direct analog: the function address goes through the same `auipc+ld` literal-pool path as any other 64-bit constant, called via a real `jalr ra,0(reg)`, with the single argument passed in `a0` (the real RV64 ABI arg0/return register — every other opcode in this translator uses only `t0`–`t2`, so `a0` is reserved exclusively for this).

**Each environment binds those three function pointers to something different — that's the whole point.** The Phase 1 reference interpreter, the x86 host JIT harness, and the RV64 verify harness all bind them to small identical mock catalogs (`"simi_add_test2"` → size 303/type PROGRAM, `"simi_verify3"` → size 100/type DATA), so `tests/obj_ops.simi` — `RESOLVE` a known name, `OBJSIZE`/`OBJTYPE` it, `RESOLVE` an unknown name and confirm it comes back 0 — produces the identical expected result (305) across every implementation, the same cross-implementation-agreement discipline as every prior phase. The kernel gets the real thing: a new `kernel/simi_runtime.c` implements `simi_rt_resolve/objsize/objtype` as bounded linear scans over the live `object_catalog[]`, gated on `.active`, returning `base_vaddr`/`size_pages*4096`/`SLSObjectType` — the first SIMI runtime function in this project that touches real kernel state rather than translating self-contained bytecode.

**A real bug, caught the same way as Phase 3's ModRM bug and Phase 5's `reg_disp()` bug — by execution, not review.** The RV64 verify harness needs *some* way to invoke a "real" host C function from code executing on a purpose-built, non-hardware RV64 model (`rv64_exec.c` — there's no real RV64 CPU underneath it, see §12). The fix: reserve a sentinel address range, hand out `BASE + idx` as the "function address" for host-callback slot `idx`, and have the executor's `JALR` handling intercept jumps into that range instead of faulting on a fake fetch address. First version spaced slots by 1 (`BASE+0`, `BASE+1`, `BASE+2`). `obj_ops.tmo` segfaulted inside the *host* `mock_rt_resolve` on its **second** hostcall — ASan pointed at a `strcmp` dereferencing `0x2000`, which turned out to be `OBJSIZE`'s own correct argument (the vaddr `RESOLVE` had just returned), but arriving at the **`RESOLVE` callback (slot 0) instead of `OBJSIZE`'s (slot 1)**. Traced with an instrumented build logging every `emit_li64`/literal-pool-patch/hostcall: the literal pool held the right values (`…00`, `…01`, `…02`) and the register holding the target address was correctly `…01` right up to the `JALR` — the bug was in real RV64 semantics, not this project's code: **`jalr` unconditionally clears the target address's least-significant bit** (per spec: "...and then setting the least-significant bit of the result to zero"), silently turning sentinel slot 1's `…01` into `…00` and invoking the wrong callback with the right caller's argument. A one-line-looking encoding choice, invisible to review, caught only because the *right* argument showed up at the *wrong* function — exactly the kind of mismatch execution surfaces and static reading doesn't. Fixed by spacing sentinel slots by 16 instead of 1 (any stride ≥ 2 works; see `rv64_exec.h`'s `RV_EXEC_HOSTFN_STRIDE`).

**Verification.** `tests/obj_ops.simi` passes identically — result `305` — through the Phase 1 interpreter (`make test`), the x86 host JIT harness executing real translated machine code that makes real C-ABI calls out to a mock catalog (`make test-native`), and the RV64 verify harness (`make test-riscv`), alongside the full pre-existing Phase 1–5 suite (all still passing, confirming v0.3's format/opcode additions didn't regress anything). The kernel side — `kernel/simi_x86.h/.c` kept byte-identical to the host copy (diffed to confirm: identical from the `#include` line onward, only the top-of-file comment differs, same convention established in Phase 3), `kernel/loader.h/.c`'s header parsing extended for `num_names`, new `kernel/simi_runtime.c` wired into `kernel/simi_translate.c`'s two translate passes — compiles cleanly under the real kernel build flags (`-ffreestanding -mno-red-zone -mno-sse …`, the same flags `make x86-run` uses). **What's not verified**: an actual booted-kernel spawn of a v0.3 SIMI object with a live `RESOLVE` call against the real `object_catalog[]` — this sandbox has no `x86_64-elf-gcc` cross-compiler (the same hard constraint every prior phase's kernel work hit), so unlike Phase 4's real-QEMU-hardware verification, this phase's kernel wiring has only been compile-checked, not boot-tested. A natural next step on real hardware: upload a `.simi` program that `RESOLVE`s an already-`valloc`'d catalog object by name and confirms the returned `base_vaddr`/size/type match what `sys_sls_obj_stat` reports for the same object.

**What Phase 6 deliberately did not do.** No capability/authority semantics on the `T_OBJREF` values `RESOLVE` returns — they're plain `base_vaddr`s, forgeable/copyable like any other pointer-shaped value, with no unforgeability guarantee. That's explicitly Phase 7's job ("capability/authority model on pointers — System/38 'pointer' semantics — tagged, unforgeable references"), and `T_OBJREF` existing as a distinct type tag now (rather than reusing `T_PTR`) is the deliberate hook Phase 7 hangs off of, not a Phase 6 implementation of it.

## 14. Phase 7 findings — capability tags and authority-checked RESOLVE

**What "System/38 pointer semantics" meant in practice.** System/38's ODT pointers are *tagged* (a hardware/microcode bit marks a word as a valid pointer, invisible and inaccessible to ordinary instructions) and *unforgeable* (no arithmetic or bit-twiddling sequence can synthesize one — only a small set of privileged "resolve" operations can mint a new one, and every mint is authority-checked against the requester). Two independent properties, and SIMI's Phase 6 `T_OBJREF` had neither: it was a plain 64-bit `base_vaddr` with a type label, freely copyable, constructible via `LOADI`, and never checked against the caller's permissions. Phase 7 closes both gaps with the smallest mechanism that actually delivers them, deliberately not modeled as a new opcode or a new memory region — a **shadow tag bit per symbolic register**, enforced identically in every implementation, plus an **authority check at the one place capabilities are minted** (`RESOLVE`, in the kernel binding only — the host/mock bindings have no permission model to check against).

**The tag contract, stated once and implemented three times.** Every symbolic register gets a paired 1-bit (stored as a byte for addressability) tag: 1 means "this register currently holds an unforged `T_OBJREF` capability," 0 means "plain value." Four rules cover every opcode: `RESOLVE` sets the destination's tag **iff** the result is non-zero (a failed resolve is not a capability); `MOV` propagates the source's tag to the destination (copying a capability is legitimate — that's how it gets passed to a callee via `r0`-`r7` or used in a comparison); every other register-writing opcode — arithmetic, `LOADI`/`LOADI64`, `CMP`, `LEA`, `PTRADD`, typed `LOAD` — unconditionally clears the destination's tag; `OBJSIZE`/`OBJTYPE` require the operand register's tag to be set, and reject with the pre-existing "not found" sentinel (`0` / `0xFFFFFFFF`) **without ever calling the runtime catalog function** when it isn't. `PTRADD` clearing the tag is the load-bearing rule for unforgeability: if pointer arithmetic preserved a tag, `RESOLVE`-ing a real object and then `PTRADD`-ing by an attacker-chosen offset would forge a "valid" capability for an arbitrary neighboring address. `CALL`/`RET` do not propagate tags across the call boundary in this v1 — a documented narrowing, not an oversight (see below).

**Interpreter first, as ground truth.** `simi_interp.c` grew a `cap_tag[NREGS]` array per `Frame`, a `set_tag()` helper, and the rule set above applied at every `case` in the main switch. This is where the exact semantics — especially "RESOLVE tags conditionally, not unconditionally" and "PTRADD always untags" — were pinned down before touching either native translator, the same discipline as every prior phase.

**x86: a stack-frame tag region and a new local-branch primitive.** The translator's per-procedure frame already held `TX_MAX_REGS` 8-byte register slots at negative `rbp` displacements; Phase 7 adds a second, `TX_MAX_REGS`-byte region directly below it (`TX_TAG_BASE`, contiguous with the register region, zeroed in the same `rep stosq` pass at `ENTER` by growing the frame from 512 to 576 bytes). New byte-level helpers (`ld_tag_cl`/`st_tag_cl`/`st_tag_imm`) and `st_rax_untag`/`st_rcx_untag` wrappers thread through every existing register-write call site. `RESOLVE`'s conditional tag uses `test rax,rax; setne cl` computed *before* the result is stored (TEST/SETcc don't clobber `rax`). `OBJSIZE`/`OBJTYPE` needed an actual conditional branch — skip the runtime call entirely when untagged — which doesn't fit the existing global fixup table (that's keyed by SIMI `target_pc`, resolved in a pass once every instruction's *SIMI-level* offset is known; this branch's target is a few bytes further into the *same instruction's own* generated code). Solved with a new, deliberately separate **local (intra-instruction) branch mechanism** — `emit_jz_placeholder`/`emit_jmp_placeholder`/`patch_local_rel32` — that patches immediately in the same emit call rather than deferring to the global pass.

**RV64: the same mechanism, ported instruction-for-instruction.** The tag region sits below the register-slot region the same way (`TX_RV_TAG_BASE`, frame grown from 512 to 576 bytes, zeroed via an explicit per-byte `sb x0` loop — RV64 has no `rep stosq` equivalent, consistent with the existing register-zero loop's own style). `RESOLVE`'s conditional tag collapses to a *single instruction* on this target — `sltu t1,x0,a0` computes "a0 != 0" directly, RV64's one-shot analog of x86's test+setne, no branch required. `OBJSIZE`/`OBJTYPE` do need a branch, built with new `emit_beqz_placeholder`/`emit_jal_placeholder`/`patch_local_beqz`/`patch_local_jal` helpers — the direct RV64 port of x86's local-branch primitive, using `beqz`/`jal`'s real B-type/J-type encodings rather than threading through the global `g_fixups` table for the same reason as x86. `rv64_exec.c`, the purpose-built verify-harness executor, needed **no changes at all** — `LBU`/`SB` (the tag byte load/store) and `BEQ`/`BNE` (the local branch) were already implemented for Phase 1–6's typed `LOAD`/`STORE` and `BC` opcodes, so this is the first phase where a new SIMI-level opcode/semantics addition required zero executor changes, purely a translator-side story.

**The regression test: forge a capability, confirm the forgery is rejected, not just that the legitimate path works.** `tests/cap_forge.simi` `RESOLVE`s a real object (`"simi_add_test2"`, the shared mock catalog entry every implementation binds to `base_vaddr=0x2000`), takes its `OBJSIZE`/`OBJTYPE` (expected 303/1), then separately does `LOADI r3, #8192, objref` — the identical numeric value, `0x2000`, but never passed through `RESOLVE` so never tagged — and takes `OBJSIZE`/`OBJTYPE` of *that*, which must come back rejected (`0` / `0xFFFFFFFF`) despite being bit-for-bit the same address a legitimate capability would carry. The expected result (`3000304 = 303 + 1 + 1,000,000 + 2,000,000`) is constructed so the two million-scale terms only appear if both rejections actually fired — a silent forgery-acceptance bug would land nowhere near that value, not off by a rounding error, the same "impossible to mistake for a coincidence" test-design principle used for every prior phase's expected-value constants.

**Verification.** `cap_forge.tmo` passes with the identical result `3000304` through all three implementations: the Phase 1 interpreter (`make test`), the x86 host JIT harness *actually executing* the `test`/`setne`/local-`jz` byte sequence on the host CPU (`make test-native`), and the RV64 verify harness executing the `sltu`/local-`beqz`/`jal` sequence through `rv64_exec.c` (`make test-riscv`) — real execution of the forgery-rejection branch logic in two of the three environments, not just static review of the encoder output, plus independent-implementation agreement across all three. The full pre-existing Phase 1–6 suite still passes unmodified in every environment, confirming the tag region's added stack space and the new local-branch mechanism didn't regress anything. `kernel/simi_x86.c` was re-synced to stay byte-identical to the host copy from the `#include` line onward (diffed to confirm), and compiles cleanly under the real kernel build flags.

**Authority-checked RESOLVE: reusing existing infrastructure, not building new security primitives.** The kernel already had everything the authority check needed — `object_catalog.c`'s `catalog_check_access(uid, obj_name, needed_perm)` (role-based rules for `DB_ADMIN`/`APP_USER`/`GUEST`/owner, kernel `uid=0` always passes) and `user/permissions.h`'s `PERM_READ`. The one missing piece was identifying *who's asking*: the kernel had no existing "currently running process" accessor, only `schedule_ring3()`'s ad hoc inline scan of `proc_table[]` for `active && state==PROC_RUNNING && kernel_rsp!=0`. That scan is now a proper function, `process_find_current()` (`kernel/process.c`/`.h`), and `kernel/simi_runtime.c`'s `simi_rt_resolve()` calls it to find the caller's `owner_uid`, then gates the returned `base_vaddr` on `catalog_check_access(uid, name, PERM_READ)` — denial returns `0`, byte-for-byte the same as "object not found." That indistinguishability is deliberate: a distinct "exists but denied" result would let a process fish for which object names exist in the catalog by `RESOLVE`-ing guesses and comparing "not found" against "denied," even though it could never mint a capability for anything it isn't already allowed to read. No Ring-3 process running (`process_find_current()` returns `NULL`, e.g. a call from pure kernel context) is treated as `uid=0`/`ROLE_SYSTEM_KERNEL`, which `catalog_check_access()` always passes.

**Verification of the kernel-side authority check.** `kernel/process.c`, `kernel/process.h`, and `kernel/simi_runtime.c` all compile cleanly under the real kernel build flags (`-ffreestanding -mno-red-zone -mno-sse …`). **What's not verified**: an actual booted-kernel `RESOLVE` call from a real Ring-3 process against a live `object_catalog[]` with a non-kernel `owner_uid`, exercising the deny path end-to-end — the same hard constraint every prior phase's kernel-side work has hit (no `x86_64-elf-gcc` cross-compiler in this sandbox, see §6's and §13's identical caveats), so this piece is compile-checked and logic-reviewed, not execution-verified, consistent with how Phase 6's real `object_catalog[]` wiring was itself verified. A natural next step on real hardware: spawn two Ring-3 processes under different `owner_uid`s/roles, have one `valloc` an object it owns, and confirm the other's `RESOLVE` of that name returns `0` (denied) while the owner's `RESOLVE` succeeds and can `OBJSIZE`/`OBJTYPE` the result — the boot-tested completion of what `cap_forge.simi` already proves for the tag-forgery half in every environment that *can* be executed here.

**What Phase 7 deliberately did not do.** No capability propagation across `CALL`/`RET` — a capability passed as an argument or returned as a result arrives untagged on the other side of a call boundary in this v1 (`emit_call_site`'s result store always untags, on both native targets; the interpreter's `OP_CALL`/`OP_RET` clear tags the same way). A SIMI procedure that wants to pass a capability to a callee and have the callee treat it as one currently must `RESOLVE` again inside the callee, or the callee must trust an out-of-band convention — neither is built here. This is flagged as a narrowing, not a design dead-end: propagating tags across calls is straightforward (copy the tag byte into the outgoing-argument marshaling, same as `MOV` already does within one frame) but was left out to keep this phase's surface area to "mint and check," matching the same "narrow v1, flag it, move on" discipline used for every prior phase's scope cuts (Phase 3's `TX_MAX_REGS` cap, Phase 5's host-only kernel wiring, Phase 6's 31-character name limit). No revocation — once a `RESOLVE` succeeds, the resulting capability is valid for the lifetime of the register holding it (or any register it's `MOV`'d into), with no mechanism to invalidate it if the underlying object is `vfree`'d mid-flight; `OBJSIZE`/`OBJTYPE` still re-check the live catalog on every call (so a freed object's slot going `inactive` is caught then), but nothing stops a stale capability from being held. And no LPAR/partition-boundary interaction yet — that's explicitly Phase 8's job, and this phase's authority model (single flat `object_catalog[]`, single role table) is the substrate Phase 8 would need to partition, not an implementation of the partitioning itself.

## 15. Phase 8 findings — LPAR groundwork

**What "groundwork" means here, stated up front.** The roadmap names Phase 8 "partitioned SLIC instances sharing/isolating single-level store" and flags it "out of scope for near-term planning" — a real LPAR is a hypervisor: separate SLIC instances, separate physical memory carve-outs, a partition-aware scheduler, migration of whole partitions between physical machines. None of that exists after this phase, and building it wasn't the goal. What Phase 8 actually delivers is the *identity and boundary concept* a real implementation would need first: a way to say "this object belongs to partition X" and "this uid belongs to partition Y," and one enforced rule connecting the two. That's a deliberately small, retrofittable addition — not a partitioned kernel, a labeling system plus a gate.

**Design: reuse `role_table`'s shape, reuse `catalog_check_access`'s choke point.** Phase 7 already established the pattern this phase leans on entirely: `role_table[]` maps `uid -> role`, looked up via `catalog_get_role(uid)`, consulted inside `catalog_check_access()` — the one function every authority-checked call path in the kernel (DB select/insert/update/delete, grant/revoke, and as of Phase 7, SIMI `RESOLVE`) already goes through. Phase 8 adds a sibling: `kernel/partition.c`'s `partition_assign_table[]` maps `uid -> partition_id`, looked up via the new `partition_get_for_uid(uid)`, and `catalog_check_access()` gained exactly one new check — `if (e->partition_id != partition_get_for_uid(uid)) return 0;` — placed as the *outermost* gate, below only the "kernel always passes" rule and above even the owner-always-passes rule. That ordering is the load-bearing design decision: partition isolation is meant to model a hypervisor-level boundary, which should be stronger than any in-partition authority rule, including ownership. The practical effect: every existing Phase 6/7 call path gets partition isolation for free, without any of those call sites needing to know partitions exist — the same "add the check once at the choke point, not at every caller" leverage Phase 7 got from adding its authority check inside `simi_rt_resolve()` alone.

**Objects get tagged the same way.** `struct SLSObjectEntry` gained a `partition_id` field, populated in `sys_sls_valloc()`. The interesting design decision here isn't the field, it's the default: when a caller leaves `SLSVallocRequest.partition_id` at `0` (the same "zero means use a sensible default" idiom `perm_mask` already used one line above it), the object doesn't default to a hardcoded `PARTITION_SYSTEM` — it defaults to **the owner's own assigned partition**, via `partition_get_for_uid(req->owner_uid)`. That distinction matters because of the ordering decision above: if a uid assigned to partition 3 vallocs an object that silently landed in partition 0, `catalog_check_access()`'s new outermost check would deny the owner access to their own freshly-created object before the owner-check even runs — a confusing footgun. Defaulting to the owner's partition means "I didn't say" and "I'm in partition 3" resolve the same way a person would expect.

**Backward compatibility, made structural rather than promised.** `PARTITION_SYSTEM` and `PARTITION_DEFAULT` are deliberately the same value (`0`). Every object that existed before Phase 8 (or is created by code that never sets `partition_id`) defaults to `0` via ordinary struct zero-init; every uid that's never been through `partition_assign_uid()` resolves to `0` via `partition_get_for_uid()`'s fallthrough. Since both sides of the new boundary check default to the identical value, the check is a no-op — `0 == 0` — for the entire pre-Phase-8 system, unless someone explicitly calls `partition_create()` and `partition_assign_uid()` to opt something out of it. This is the same "narrow v1 that provably can't regress what came before" discipline as Phase 6's `T_OBJREF` addition (a new type tag, zero effect on existing `T_PTR` code) and Phase 7's tag region (zeroed at `ENTER`, so untagged-by-default matches every pre-Phase-7 register).

**A real bug, caught by review before it shipped — not by execution.** Auditing every call site that constructs a `struct SLSVallocRequest` (ten of them, across `kernel/agent.c` ×3, `user/shell.c` ×2, `net/http.c` ×2, `kernel/mqt.c` ×1, `kernel/stream.c` ×2) found that **none of them zero-initialize the struct** — every existing field (`owner_uid`, `perm_mask`, `type`, …) is explicitly assigned at each call site instead. Adding a new field to the struct without also adding an explicit assignment at all ten call sites would have meant `partition_id` carried whatever garbage happened to be on the stack — and per the previous paragraph's design, a nonzero garbage value would silently place a freshly-`valloc`'d object in a bogus partition, invisible to everyone including its own owner, with the failure mode being "object mysteriously not found" rather than a crash. All ten call sites were updated to explicitly set `.partition_id = 0` (matching this codebase's existing convention of assigning every field, rather than switching those ten sites to a zero-init style that isn't used anywhere else in the project). This is the same category of bug Phase 6's `num_names` header field and Phase 7's tag-region frame growth both could have hit — a new field silently uninitialized somewhere in a multi-call-site codebase — caught here by grepping every construction site before considering the change done, the "compile succeeds" bar being necessary but not sufficient.

**Verification.** `kernel/partition.c` has exactly one external dependency (`kernel_io.h`'s two logging functions), unlike `object_catalog.c` (six subsystem headers — `persist.h`, `journal.h`, `lock_mgr.h`, `index_mgr.h`, `constraint.h`, `mqt.h`) or `process.c` (`loader.h`, `user_paging.h`, physical-frame allocation) — both of which were compile-check-only in Phases 6/7 because stubbing their full dependency graphs wasn't practical. `partition.c` doesn't have that problem: `partition_host_test.c` (repo root) links the real, unmodified `kernel/partition.c` against two one-line stubs and actually **executes** twenty-one checks covering the full new API — default resolution (`uid 0` / unassigned uid / the `SYSTEM == DEFAULT` invariant itself), partition creation and id uniqueness, assignment insert-then-update-in-place, every rejection path (reassigning uid 0, assigning to an undefined partition id, an oversized name), and both tables' capacity edges (including the specific edge case that an *update* to an already-assigned uid must keep succeeding even after the assignment table is completely full of *other* uids — a plausible off-by-one this project's own history suggests would otherwise slip through, see Phase 5's `reg_disp()` bug and Phase 6's RV64 `jalr` bug, both real bugs execution caught that review missed). All twenty-one checks pass against the real subsystem code, real execution rather than a reimplementation-for-testing's-sake. The `catalog_check_access()` boundary check itself, and every touched call site (`kernel/object_catalog.c`, `kernel/syscall_dispatch.c`, `kernel/agent.c`, `kernel/mqt.c`, `kernel/stream.c`, `net/http.c`, `user/shell.c`, `kernel/kernel.c`, `kernel/partition.c`), compiles cleanly under the real kernel build flags. **What's not verified**: an actual booted-kernel scenario — two uids in different partitions, one `valloc`s an object, confirm the other's `SELECT`/`RESOLVE`/`OBJ_STAT` on it is denied while same-partition access still works — the same no-cross-compiler constraint every prior phase's kernel-side work has hit (§6, §13, §14). A natural next step on real hardware: `partition create tenant-a`, `partition create tenant-b`, assign two shell uids to each, `valloc` an object under each, and confirm `ls objects`/`select` cross-partition access is denied while same-partition access and the `role`/`grant` model both still work exactly as before within a partition.

**What Phase 8 deliberately did not do.** No process-level partitioning — `struct ProcessDescriptor` was not touched, and `process_create()`/`program_spawn()` still resolve `SERVICE_PROCESS`/`PROGRAM` objects via a raw linear scan of `object_catalog[]`, not through `catalog_check_access()` at all (true before this phase too — process spawning has never gone through the Phase 6/7 authority model, a pre-existing gap this phase chose not to fix, since closing it is Phase 6/7 catch-up work, not Phase 8 scope). A partitioned process — one that can only see and spawn objects within its own partition — needs that fixed first; right now partition isolation only covers the object-catalog-mediated paths (DB operations, SIMI `RESOLVE`, `grant`/`revoke`), not process creation itself. No persistence — `partition_table`/`partition_assign_table` are plain BSS arrays like `role_table` always has been; a reboot forgets every partition and assignment (`persist_catalog()` isn't called anywhere in `partition.c`, unlike `object_catalog.c`'s role/grant/valloc paths). No nested or hierarchical partitions — the model is a single flat namespace of sibling partitions plus one system partition, no partition-of-partitions. And no resource isolation whatsoever — memory, CPU scheduling, and the physical address space are still fully shared; this phase partitions *visibility* (what `catalog_check_access()` will admit to being findable) and nothing about *capacity or scheduling fairness* between partitions, which is most of what a real LPAR's value proposition actually is. All four are natural, separable follow-ups, not oversights — the roadmap's own "out of scope for near-term planning" framing for Phase 8 was about exactly this: doing the groundwork now that the earlier phases could hang authority-checking off of, without committing to what a full hypervisor-grade implementation looks like before there's a concrete reason to build one.

## 16. Gap Remediation Roadmap (Phases 9–14) — planned, not started

Everything below is a plan, not a findings section — none of Phases 9–14 has been implemented. This roadmap exists because Phases 1–8 each closed out with an honest "what's not addressed" paragraph (§9 through §15 above all have one), and those paragraphs have accumulated into a real, actionable backlog rather than staying as scattered footnotes. This section collects them into one ordered plan, the same way `docs/AeroSLS-Gap-Remediation-Roadmap-v0.1.md` did for the RDBMS/Vector Store/LPAR/auth findings elsewhere in this project. A seventh known gap — no compiler front end exists for SIMI at all — is deliberately *not* a phase here; it already has its own dedicated scoping document (`docs/AeroSLS-SIMI-LLVM-Backend-Scoping-v0.1.md`) because it's a different shape of project (a new tool, not a fix to existing code) and a much larger one than any single phase below.

**Sequencing principle.** Phase 9 (RISC-V kernel wiring) is the highest-leverage item and should come first if only one phase happens: it's the one gap that undermines SIMI's own core claim — "technology-independent, re-translatable to a second real ISA" — by leaving that claim proven only at the host-toolchain level, never inside a running kernel. Phases 10–13 are independent of each other and of Phase 9; do them in any order dictated by what's actually needed next (e.g., Phase 12's capability-tag/CALL boundary matters more once real multi-procedure programs exist, which in turn matters more once Phase 9 or a real compiler front end makes writing such programs easier). Phase 14 is explicitly preparatory — it exists because the LLVM scoping document's own Phase 2 needs an aggregate ABI convention and named its absence as a real gap; doing Phase 14 here, ahead of any LLVM-backend work actually starting, means that future work isn't blocked re-discovering the same gap.

### Phase 9: RISC-V kernel wiring — concrete design, broken into sub-phases

**The gap, re-audited by directly reading the actual coupling, not re-stating the summary.** `kernel/kernel_riscv.c` is confirmed a genuine 10-line stub (a boot banner over `sbi_putchar` plus a `wfi` loop — nothing else). A direct grep of the three files the roadmap named as "x86-only" tells a more precise, more encouraging story than "port everything":
- **`kernel/object_catalog.c` is already effectively arch-neutral** — its only `x86` hit in the whole file is a comment ("avoid mixed-type variadic printf which crashes on freestanding x86"), not a dependency. It very likely compiles under RV64 with zero real changes.
- **`kernel/loader.c`'s real coupling is exactly one include**: `#include "../arch/x86/user_paging.h"`, for process address-space/page-table setup.
- **`kernel/process.c`'s real coupling is two things**: the same `user_paging.h` include, and — more interesting — a documented dependency on x86's `swapgs` instruction and GS-segment-relative per-CPU data (`arch/x86/syscall.asm`'s syscall-entry stub), which RISC-V has no direct equivalent for and needs its own mechanism, not a port of the x86 assembly.

This changes the shape of the work: it's not "port three x86-coupled files," it's "build one missing arch-abstraction layer (`user_paging`) that two already-mostly-portable files depend on, plus one RISC-V-native replacement for a swapgs-based mechanism that has no direct analogue." `arch/riscv/walk_page_tables_riscv.c` already exists and is already built into `RV_OBJECTS` per the Makefile — the low-level page-table-walking primitive this needs already exists; it's just never been wrapped in the loader-facing abstraction `arch/x86/user_paging.c` provides for `loader.c`/`process.c` on the x86 side.

**Sub-phase 9a — confirm and lock in `object_catalog.c`'s arch-neutrality.** Lowest-risk, do first: best-effort structural/grep review for any x86-specific intrinsics or header assumptions (the same fallback methodology already used for the slsos-sim frontend when a real cross-compiler wasn't available), then add it to `RV_C_SRC` and confirm it compiles clean. If true, this sub-phase is nearly free — it validates the "less porting than assumed" finding above rather than taking it on faith.

**Sub-phase 9b — build `arch/riscv/user_paging_riscv.{h,c}` and port `kernel/simi_riscv.c` into `kernel/`.** The address-space abstraction `loader.c`/`process.c` need, built on top of the already-existing, already-RV64-built `walk_page_tables_riscv.c` (mirroring how `arch/x86/user_paging.c` already wraps `walk_page_tables_x86.c` today — this is filling in a known-shape gap, not inventing a new one). In parallel: port `tools/simi/simi_riscv.c` (804 lines, complete, fully tested at the host level via `rv64_exec.c`) into `kernel/simi_riscv.c`, the same "portable, no-libc, byte-identical host/kernel copy" treatment `simi_x86.c` got in Phase 3.

**Sub-phase 9c — port `process.c`, including a real RISC-V per-hart-data mechanism.** Address-space setup reuses 9b's `user_paging_riscv`. The `swapgs`/GS-segment trick has no RISC-V equivalent and needs a genuine RISC-V-native design, not a literal port: the natural analogue is dedicating `sscratch` (swapped with a live register on trap entry/exit, the standard RISC-V pattern for reaching per-hart kernel state from a trap handler) to hold a pointer to this hart's per-CPU data, mirrored through `arch/riscv/context_riscv.S`'s trap entry/exit path the same way `arch/x86/syscall.asm` uses `swapgs` for GS-relative access — same problem, RISC-V's own idiomatic solution, not x86's. `USER_PROC_CODE_BASE`'s canonical-address constant also needs its own RV64 value (Sv39/Sv48's canonical range is shaped differently from x86-64's).

**Sub-phase 9d — scope the first reachable RV64 syscall surface to headless/serial, explicitly not the full HTTP/RDBMS/vector-store stack.** Confirmed via the Makefile: `net/http.c` and everything it pulls in (`object_catalog`, `transaction`, `microkernel`, `tier_mgr`, `webapp`, and transitively the whole RDBMS/vector-store surface) is entirely absent from `RV_C_SRC` today — it is, and always has been, an x86-only build artifact, not something RV64 ever partially supported. Rather than replicating that entire surface, scope RV64 v1 to what `kernel_riscv_main()` already gestures toward: a headless, `sbi_putchar`-console target, wired just far enough to spawn and run one translated `.simi` program through a minimal syscall surface (a process-list-equivalent syscall, matching the existing x86 hello-world program's own reachability test per `loader.c`'s comments) — not the full HTTP/SQL surface. This is the same "host toolchain only" scoping discipline Phase 5 already used for exactly this reason: don't take on more than can actually be verified.

**Sub-phase 9e — real hardware/QEMU verification, honestly scoped around a real, still-unresolved sandbox constraint.** `riscv64-unknown-elf-gcc` and `qemu-system-riscv64` have been unavailable in this sandbox at every point in this project's history — the same constraint Phase 5 hit and worked around via `rv64_exec.c`'s purpose-built host-side decoder+executor. That same technique extends naturally to whatever of 9a-9d can be exercised at the host level (compile-checks, unit tests against stubbed hardware primitives), but genuine boot-time verification on real QEMU stays explicitly unverified until the user has that toolchain available — to be documented honestly as an open gap when this phase ships, not claimed as tested.

**Ordering and dependency**: 9a is independent and should go first (cheapest, and empirically validates the smaller-than-expected scope). 9b and 9c share the `user_paging_riscv` dependency and should be *designed* together even though 9b (loader + translator port) can land before 9c (full process subsystem). 9d depends on both 9b and 9c existing. 9e verifies the assembled whole, but each sub-phase's host-testable pieces should be verified incrementally as they land — matching the phase-by-phase verification discipline used in every prior implemented phase of this project — rather than deferred to one big-bang verification at the very end.

**Why this is still Phase 9 and not folded into a smaller catch-all, even with the smaller-than-expected scope found above**: five real sub-phases with a genuine new-mechanism design problem in the middle (9c's per-hart-data replacement) is still substantially larger and riskier than any single prior SIMI gap-remediation phase — closer in shape to a new subsystem than a gap fix — and should stay scoped and estimated as its own multi-step effort, not assumed to be "just wire it up" scale just because 9a turned out to be nearly free.

### Phase 10: Float opcodes — concrete design

**The gap, precisely.** `T_F32`/`T_F64` are real values in the `SimiType` enum (`tools/simi/simi_isa.h`) and every instruction word already carries a 4-bit type field (`opcode(8)|type(4)|rD(10)|rA(10)|rB/imm(28)|flags(4)`) — the encoding was built for this from v0.1. Direct reading of `tools/simi/simi_interp.c` confirms the actual current state precisely: the interpreter *decodes* `type` into a local on every instruction (line 182) but only two opcodes ever look at it — `OP_DIV`/`OP_MOD` explicitly `die("float arithmetic not implemented in Phase 1 interpreter")` when `type == T_F32 || T_F64`. `OP_ADD`/`OP_SUB`/`OP_MUL`/`OP_NEG`/`OP_CMP` don't check `type` at all — they silently run integer semantics even when an instruction's type field says float. This means the ISA's own type-tag mechanism was already designed to carry this distinction; the gap is that nothing honors it yet, not that new opcode space is needed.

**Design decision 1 — reuse the existing arithmetic opcodes, dispatched on the type field they already carry, rather than adding FADD/FSUB/FMUL/FDIV.** The encoding already has the type tag; adding parallel float opcodes would duplicate opcode space the ISA doesn't need duplicated. `ADD`/`SUB`/`MUL`/`DIV`/`NEG`/`CMP` each grow a `type == T_F32 || type == T_F64` branch (replacing `OP_DIV`/`OP_MOD`'s current defensive `die()`, which becomes real handling for everything except `MOD`, addressed below) instead of `die()`ing.

**Design decision 2 — representation.** `regs[]` is a single unified `uint64_t[NREGS]` file (confirmed by reading `simi_interp.c` line 41) — there is no separate float register bank, matching how RV64's `f` registers are never used and x86 never allocates an XMM-backed symbolic register today. `T_F64` reinterprets the full 64-bit slot as an IEEE-754 double, bit-for-bit (`union { uint64_t u; double d; }` in the interpreter; real `movq`/`movsd` bit-moves in the native translators — no conversion, just a different read of the same bits). `T_F32` reinterprets the low 32 bits as an IEEE-754 single with the high 32 bits zeroed on write, matching this ISA's existing convention for narrower integer types (`T_U32`/`T_I32`) occupying the low bits of the same 64-bit slot.

**Design decision 3 — `DIV` under float types is a real, implementable op; `MOD` under float types is permanently out of scope, not temporarily deferred.** IEEE-754 float division by zero is a *defined* result (signed infinity or NaN, no trap) — unlike integer `DIV`, there's no hazard `die()` was ever protecting against; it can be replaced with real `divsd`/`divss` (x86) or `fdiv.d`/`fdiv.s` (RV64) once the FPU-usability gap below is closed. `MOD`, by contrast, has no single hardware instruction for float remainder on either target — a real `fmod()` would need to be composed from `DIV`+`MUL`+`SUB` at the `.simi` source level, the same way libm's own `fmod()` is implemented. Recommend keeping `OP_MOD`'s `die()` for float types permanently, documented as intentional rather than "not yet."

**Design decision 4 — `CMP` under float types must implement IEEE-754 unordered comparison semantics exactly, including for NaN**, via `ucomisd`/`ucomiss` (x86) or `feq.d`/`flt.d`/`fle.d` (RV64) rather than integer `cmp`: `REL_LT`/`REL_LE`/`REL_GT`/`REL_GE` are all false when either operand is NaN, while `REL_NE` is true and `REL_EQ` is false for NaN compared to anything including itself. This should be implemented as specified, not "corrected" to be more intuitive — any future LLVM front end (per the scoping doc) will assume real IEEE semantics.

**Design decision 5 — literals need an assembler front-end change, not an ISA change.** A float literal's bit pattern is just a 64-/32-bit immediate; `LOADI64`/`LOADI` already exist. `tools/simi/simi_asm.c` needs new literal syntax (e.g. `3.14f64`) that emits the IEEE bit pattern through the existing opcodes — no new encoding.

**The real blocking prerequisite this phase surfaces (found by reading the actual code, not assumed from the gap-remediation roadmap's summary): the x86 kernel cannot safely execute *any* SSE/AVX instruction today, in kernel C or JIT-emitted code, despite `arch/x86/lazy_fpu.c` looking complete.** Tracing it end to end: `arch/x86/switch_lazy.asm` (line 32-36) already sets `CR0.TS = 1` on every context switch, correctly arming the lazy-FPU trap. `arch/x86/lazy_fpu.c`'s `handle_device_not_available_fault()` is a complete, correct-looking `XSAVE`/`XRSTOR`-based handler for exactly that trap (AVX-512 `zmm0`-`zmm31`, which covers XMM/SSE state too). But **`handle_device_not_available_fault()` is never registered as the IDT vector-7 (`#NM`) handler** — a targeted grep of `arch/x86/idt.c` and `kernel/kernel.c` finds zero references to it or to vector 7 outside `lazy_fpu.c` itself. So today, the moment any code (kernel or a SIMI-JIT-emitted `movsd`) executes a float/SSE instruction after a context switch, `CR0.TS=1` correctly traps to `#NM` — and hits an unhandled vector with no installed ISR, which is exactly why `-mno-sse` is load-bearing for kernel C right now: without the trap handler wired up, using SSE at all is a guaranteed crash, not a slow path. **This is a small, well-scoped, one-time fix — install one IDT entry pointing at code that already exists and already looks correct — not new infrastructure**, and it should be treated as Phase 10's true first sub-step (before any float opcode lands), verified by a minimal test that deliberately triggers `#NM` (e.g. an early `movsd` after a forced context switch) and confirms it doesn't crash. Only once this is wired can the x86 native translator safely emit real `movsd`/`addsd`/`divsd`/`ucomisd`.

**RV64 has no equivalent mechanism at all, and it's a separate, real gap, not an oversight in this write-up.** `-march=rv64gcv -mabi=lp64d` already implies hardware float in the kernel's own C-code ABI, but a direct read of `arch/riscv/context_riscv.S` shows zero `fsd`/`fld`/FCSR save-or-restore of any kind — no F/D register context switching exists at all today. The one lazy-trap-and-save mechanism RV64 does have, `arch/riscv/lazy_vector.c`, is scoped specifically to the **V** (vector) extension via `sstatus.VS`, using a real, working pattern (trap on `sstatus.VS == OFF`, flip to `Initial/Clean`, save/restore, resume) — but it does not cover **F**/**D** (scalar float) or `sstatus.FS` at all. Building an `sstatus.FS`-based twin of `lazy_vector.c` for `f0`-`f31` would be the direct analogue of the x86 fix, but unlike the x86 case, this is genuinely new code, not "wire up a handler that already exists."

**Design decision 6 — scope RV64 out of Phase 10's v1, matching the LLVM scoping doc's own precedent for excluding RISC-V.** Land float support against x86 only (interpreter + x86 translator), once the `#NM` IDT fix lands; defer RV64 float codegen to ride alongside Phase 9 (RISC-V kernel wiring), since that phase already has to build substantial RV64 kernel-side infrastructure from nothing and an `sstatus.FS` lazy-save mechanism fits naturally alongside it rather than being solved in isolation here.

**Verification plan**: a new test program exercising `ADD`/`SUB`/`MUL`/`DIV`/`CMP` under `T_F32` and `T_F64`, run first through the interpreter (host-compiled C gets real hardware float for free via the union trick, so this is the cheap, trivially-correct reference) and cross-checked bit-for-bit; then, once the `#NM` handler is wired, the same program run through the x86 JIT test harness (`tools/simi/simi_jit_test.c`) to prove emitted `movsd`/`addsd`/`divsd`/`ucomisd` produce results bit-identical to the interpreter — matching this project's established two/three-way cross-verification discipline, RV64 excluded from this phase per decision 6 above.

**Blocks**: any future LLVM-backend Phase 6 (per the LLVM scoping doc), and any hand-written `.simi` program that needs real (non-fixed-point) fractional math.

**Implemented — findings.** All six design decisions above landed, plus the `#NM` blocking prerequisite (decision-before-decision-1, per the design's own framing) — with real deviations and real bugs found beyond the design's original scope, recorded honestly rather than glossed over.

**The `#NM` prerequisite turned out to gate three genuine, pre-existing bugs, not just a missing IDT entry.** Registering `isr7_stub` at vector 7 in `arch/x86/idt.c`'s `init_idt()` (matching every other vector's existing pattern — no separate per-vector init function needed) was the easy part. Reading `arch/x86/lazy_fpu.c` and its callers closely enough to write a correct custom stub (not built from the `FAULT_STUB_NOEC` macro — `#NM` must resume via `iretq` after the trapping instruction re-executes with FPU state now loaded, not route through `handle_ring3_fault()`'s kill-the-process path, which is for real faults, not this lazy-save mechanism's own expected control flow) surfaced three unrelated, real problems: (1) `kernel_get_current_task_struct()` was a stub unconditionally returning `NULL`, with a comment claiming "`lazy_fpu.c` guards against this" — true only for the very first `#NM` trap ever taken; every trap after that hit the same `NULL == NULL` early-return, meaning no task's AVX-512 state was ever actually saved or restored across a context switch, a silent correctness bug (cross-task register leakage), not a safe no-op. Fixed with a real per-thread-id side table in `kernel/stubs.c`, keyed by `kernel_get_current_thread_id()` (`kernel/scheduler.c`) — the only "current execution identity" this kernel has, honestly scoped as tied to the kernel-thread-id space rather than a distinct user-process identity, since `kernel/process.c` has no current-process accessor today. (2) `fpu_hardware_owner` was declared `static` inside `arch/x86/scheduler_lazy.h` — a file-scope static in a header gives every including translation unit its own private copy rather than one shared global, harmless only by accident since `lazy_fpu.c` was the sole includer; moved to a real `extern` declaration with the definition in `lazy_fpu.c`. (3) `lazy_fpu.c` contained two `lapic_write(LAPIC_REG_EOI, 0)` calls — semantically wrong, since `#NM` is a synchronous CPU exception, never LAPIC-routed, and almost certainly a copy-paste artifact from IRQ-handling code; removed. A fourth, smaller finding: `scheduler_lazy.h` also declared `init_interrupt_7_handler(void)` with zero definitions anywhere in the codebase (confirmed via grep) — removed rather than implemented, since vector 7 registration folding directly into `init_idt()` made the dangling declaration simply unneeded, matching how no other vector has a separate init function either.

**One real implementation obstacle while fixing (1): a header enum collision, found by trying the straightforward fix first and hitting real compiler errors, not anticipated in advance.** `kernel/stubs.c` already `#include`s `process.h`, which transitively includes `scheduler.h` — and `scheduler.h`'s `enum TaskState` shares every enumerator name with `arch/x86/scheduler_extended.h`'s own `enum TaskState` (`TASK_READY`/`TASK_RUNNING`/`TASK_BLOCKED` all collide). A first attempt to `#include scheduler_extended.h` directly and return a real `struct ExtendedTask*` failed to compile with genuine `redeclaration of enumerator` errors under the exact kernel `X86_CFLAGS`. Fixed with a `struct ExtendedTaskShadow` — a byte-for-byte layout-compatible mirror of the real struct built from plain integer types instead of the colliding enum — so `lazy_fpu.c`'s `XSAVE`/`XRSTOR` calls, which only ever touch the `avx512_state_buffer` bytes through their own real-typed view of this same memory, stay correct without either view needing to be the literal same named C type.

**One real, honestly-flagged verification gap**: `nasm` is not installed in this sandbox and cannot be installed (no root access), so `arch/x86/interrupt.asm`'s new `isr7_stub` could not be assembled or syntax-checked in this session — verified only by careful manual comparison against the already-proven-correct `FAULT_STUB_NOEC` macro's register save/restore shape, not by an actual assembler run. Flagged here rather than silently presented as fully verified.

**Decisions 1-5 (interpreter, representation, DIV/MOD split, CMP semantics, literal syntax) landed exactly as designed**, with zero deviations worth recording: `tools/simi/simi_interp.c`'s `OP_ADD`/`OP_SUB`/`OP_MUL`/`OP_DIV`/`OP_NEG`/`OP_CMP` each grow the `type == T_F32 || type == T_F64` branch as specified, `OP_MOD` keeps its `die()` for float permanently (message reworded to say so explicitly rather than implying "not yet"), and `CMP`'s float path uses native C `double`/`float` comparison operators — which the C standard already defines with correct IEEE-754 unordered/NaN semantics, so no hand-written NaN-handling logic was needed there at all. `tools/simi/simi_asm.c` gained `parse_float_imm()`, recognizing `#<number>f64`/`#<number>f32` tokens via `strtod` and a bit-reinterpreting union, feeding the existing `LOADI64` literal-pool path unchanged.

**The x86 translator (decision 1-4's native-code half) landed with two real deviations from the design sketch, both simplifications discovered while implementing, not planned in advance.** First, the design's decision 2 didn't commit to a specific mechanism for getting operand bits into XMM registers; the implementation uses a GP-register bounce (`MOVQ xmm0, rax` / `MOVQ xmm1, rcx`, immediately after the existing `ld_rax`/`ld_rcx` slot loads, then `MOVQ rax, xmm0` before the existing `st_rax_untag` slot store) rather than a second, XMM-direct memory-addressing path — this reuses every existing slot-loading/storing helper unchanged, at the cost of two extra `MOVQ` instructions per float op, a deliberate simplicity-over-speed choice consistent with Phase 3's own "naive, not fast" v1 stance. Second, `NEG`'s float case needs no XMM instruction at all, and was not sketched this way in the original design: negating an IEEE float is purely a sign-bit flip, so a plain GP-register XOR against a sign-mask constant (`0x8000000000000000` via `movabs`+64-bit `xor` for f64; `0x80000000` via a 32-bit `xor eax,ecx` for f32) does the whole job — and the 32-bit form's architectural zero-extension of `rax`'s upper 32 bits conveniently satisfies this translator's existing "zero high 32 bits on f32 write" convention for free, with no extra instruction. `CMP`'s float path uses `UCOMISD`/`UCOMISS` as designed, implemented with one refinement beyond decision 4's text: `GT`/`GE` resolve with a single `seta`/`setae` each, and `LT`/`LE` resolve with the *same* `seta`/`setae` codes by swapping the compared operands — because `UCOMISD`'s flag encoding happens to make the unordered case and the "first-operand-is-less" case both set `CF=1`, `seta`/`setae` (which key off `CF=0`) are false for NaN automatically with no separate parity check. `EQ` still needs an explicit `sete`+`setnp`+`and` (unordered also sets `ZF=1`, so equality has no operand-swap trick), and `NE` needed no comparison of its own: IEEE-754 defines `!=` as the exact logical negation of `==` in every case including NaN, so `NE` is just `EQ`'s result XORed with 1 — narrower and simpler than a from-scratch unordered-not-equal computation. `FLAG_IMM` combined with a float type is rejected via a new `TX_ERR_FLOAT_UNSUPPORTED` error code (also covering float `MOD`), added identically to both `tools/simi/simi_x86.h` and `kernel/simi_x86.h`.

**RV64 (decision 6) landed exactly as scoped — explicit rejection, not silent mishandling.** `tools/simi/simi_riscv.c`'s `emit_instr()` gained one check, at the top alongside the existing register-range validation: any of `ADD`/`SUB`/`MUL`/`DIV`/`MOD`/`NEG`/`CMP` combined with `T_F32`/`T_F64` returns the new `TX_RV_ERR_FLOAT_UNSUPPORTED` immediately, before any codegen runs — confirmed by direct test that this fires cleanly (`FAIL ... translate error: float (T_F32/T_F64) not supported by the RV64 translator (scoped out of Phase 10 v1)`, not a wrong numeric answer) rather than the alternative of silently reading the IEEE bit pattern as a plain integer, which is what every one of those opcodes would have done without this check (`simi_riscv.c` has no type dispatch anywhere else). `tests/run_riscv_tests.sh` gained an explicit `SKIP` case for the new float test program, matching the file's own existing precedent for `mem_ops`' target-specific skip, rather than letting a permanently-expected rejection register as a perpetual regression-suite failure.

**Verification.** A new self-checking test program, `tools/simi/tests/float_ops.simi`, exercises `ADD`/`SUB`/`MUL`/`DIV`/`NEG` under both `T_F64` and `T_F32`, plus all four ordered `CMP` relations and reflexive `EQ`/`NE`, summing seventeen individual pass/fail checks (each a `CMP` result) into one `i32` total via plain integer `ADD` — chosen so a single `Expected result: 15` comment can catch a wrong answer anywhere in the chain without needing one test file per operation, matching this project's existing test-fixture style. Three of the seventeen checks specifically exercise IEEE-754 NaN unordered-compare semantics (`EQ`/`NE`/`LT` against a raw `0x7FF8000000000000` quiet-NaN bit pattern loaded via plain `i64` `LOADI64` and reinterpreted by the `CMP` itself) — `EQ`/`LT` both correctly resolve false and `NE` correctly resolves true, the concrete proof the unordered-compare codegen (interpreter's native C operators; x86's `UCOMISD` operand-swap scheme) is right, not just "doesn't crash." Result: **15 on both the interpreter (`make test`) and the real x86-64 JIT harness (`make test-native`)** — every arithmetic result, relation, and NaN case bit-identical between the two, independently cross-checked against Python's own `struct.pack`/`struct.unpack` IEEE bit patterns during development. `make test-riscv` shows 11 passed, 0 failed, 3 skipped (the pre-existing `mem_ops`/`cap_forge_debug` skips plus the new, expected `float_ops` skip) — no regression. Float `MOD` and `FLAG_IMM`+float were independently confirmed to fail cleanly with the new `TX_ERR_FLOAT_UNSUPPORTED` error, not silently misexecute. A full 76-file kernel compile-check sweep (every file in the Makefile's `X86_C_SRC` list, compiled individually under the exact `X86_CFLAGS` including `-mno-sse -mno-sse2 -mno-mmx`) is clean — zero errors across all 76, and zero warnings in every file touched this phase (`kernel/simi_x86.c`, `arch/x86/idt.c`, `kernel/stubs.c`, `arch/x86/lazy_fpu.c`) — confirming the float translator's hand-emitted SSE2 byte sequences compile correctly into a file that itself builds under strict `-mno-sse`, with zero new C `double`/`float` types introduced anywhere in `kernel/simi_x86.c`, exactly as decision 1's constraint required.

### Phase 11: Real register allocation — concrete design

**The gap, confirmed.** A direct read of `kernel/simi_x86.c` confirms every symbolic register is an rbp-relative stack slot (§ "rbp-relative symbolic register slot helpers"), loaded before and stored after every use, with no exceptions — the file's own reserved-usage registers are `rax`/`rdx` (hardwired by the `MUL`/`DIV` lowering idiom, which needs the x86 `mul`/`div` instructions' fixed operand positions) and an explicit `r10`/`r11` scratch pair (used for the "save value, do a thing, bring it back" idiom the tag-check codegen already relies on). Every other GP register sits unused for the entire duration of translated code.

**Design decision 1 — scope v1 to straight-line live ranges; force any symbolic register live across a backward branch (i.e. any loop) to keep its existing stack slot.** SIMI registers are non-SSA/mutable and the instruction stream has no CFG structure computed today (consistent with the LLVM scoping doc's own §6 note that there's "no PHI/SSA construct"). Full correctness for allocation across loops needs fixed-point liveness over a real control-flow graph — solvable, but a second, harder increment. A v1 that computes each symbolic register's live range with a single forward pass (first def to last use in program order) and conservatively excludes from allocation any register whose live range crosses a backward-branch target is simple to implement, simple to prove correct, and still captures the common case (straight-line arithmetic inside a basic block, which is most of what any real procedure spends its instruction count on) without needing dataflow analysis in the first cut.

**Design decision 2 — allocate from a fixed, conservatively-chosen register pool, explicitly excluding everything the existing codegen idioms already hardwire.** For x86: reserve `rbp`/`rsp` (frame pointer/stack pointer, untouchable), `rax`/`rdx` (the `MUL`/`DIV` idiom), and `r10`/`r11` (the existing scratch pair) exactly as they are today; allocate from the remaining pool (`rbx`, `rcx`, `rsi`, `rdi`, `r8`, `r9`, `r12`-`r15`, a first estimate — needs a real pre-implementation audit of every codegen idiom for narrower reservations, e.g. shift-amount lowering conventionally needs `cl` specifically on x86, which would additionally reserve `rcx` for any symbolic register live across a `SHL`/`SHR`/`SAR`). RV64 needs the equivalent audit against `tools/simi/simi_riscv.c`'s own scratch-register conventions (`t0`-`t6` vs `s0`-`s11` vs `a0`-`a7`) before its own pool can be finalized — not done in this write-up, flagged as a real pre-implementation task rather than guessed at.

**Design decision 3 — the capability-tag region stays keyed by symbolic-register index, unmodified, regardless of where a symbolic register's value physically lives.** This is the one easy-to-get-wrong interaction with Phase 7's tag mechanism: the `TX_MAX_REGS`-byte tag region is one byte per *symbolic* register index today, and every tag set/clear in emitted code already addresses it by symbolic index, not by physical register. Allocating symbolic register 5 into `rbx` for one live range and symbolic register 9 into `rbx` for a later, non-overlapping live range in the same procedure must not let register 9's tag write clobber or be confused with register 5's — since tag storage is indexed by symbolic index (a fixed memory location per procedure activation, same as today), not by physical register, this is naturally correct without any change to the tag mechanism itself; this needs to be called out explicitly in the implementation so nobody "optimizes" tag storage to be physical-register-keyed, which would be wrong.

**Algorithm shape**: classic linear-scan (Poletto & Sarkar) restricted to the pool from decision 2, run once per procedure at translate time, over the live ranges from decision 1 — spill (fall back to the existing stack-slot path) on pool exhaustion exactly the same way a register already excluded by decision 1 does, so "spilled" and "never eligible" are the same code path, not two.

**Verification plan**: run every existing `.simi` test fixture (all files under `tools/simi/tests/`) through both the current zero-allocation path and the new allocator, and diff final register/memory state and program output for byte-for-byte equality — this is pure codegen-quality work with zero intended observable-behavior change, so "identical output, different code path" is the correct and sufficient proof, the same technique already used to verify the earlier SSE/float-ABI fix. A secondary loop-heavy benchmark (e.g. a tight-loop Fibonacci or sum-to-N `.simi` program) run before/after, timed, demonstrates the performance win is real and not just harmless.

**Explicitly not a prerequisite for anything else in this list** — named here because it's a real, confirmed gap (§16 intro), not because it blocks Phase 9/10/12/13.

**Implemented — findings.** Decisions 1 and 3 landed exactly as designed. Decision 2 required a real, substantive correction beyond what the design text anticipated, caught only by the regression suite — recorded honestly below rather than glossed over, since it's the kind of bug that silently corrupts unrelated code if missed.

**The gap, confirmed exactly as described, plus one omission the design text didn't call out: `CALL` needed the same boundary treatment as a loop, and `OBJSIZE`/`OBJTYPE` needed the same treatment as `RESOLVE`.** `emit_prologue()` (`OP_ENTER`) marks the start of a fresh rbp-relative frame — confirmed to be the natural per-procedure register-numbering boundary the implementation needed, since `rbp` itself resets there. Live ranges and the allocation pool are therefore recomputed fresh at every `OP_ENTER`, over that procedure's own `[start_pc, next_ENTER_pc)` slice of the flat instruction array, mirroring how the rbp-relative frame itself is already scoped per procedure. Beyond decision 1's loop exclusion, the implementation adds a second, equally necessary exclusion the design doc's text doesn't discuss: any symbolic register whose live range spans a `CALL` (to another translated SIMI procedure) or a `RESOLVE`/`OBJSIZE`/`OBJTYPE` (a real C runtime-function call) is also forced to stay on the stack. The reason is structural, not incidental: a `CALL`'s callee is separately, independently allocated code with no save/restore convention between caller and callee (the callee's own `ENTER` prologue is free to clobber any pool register for its own symbolic registers, and nothing tells it not to touch whatever the caller happened to be using); a `RESOLVE`/`OBJSIZE`/`OBJTYPE` invokes an arbitrary compiled C function which, under the SysV ABI, is free to clobber every caller-saved register internally, not merely the ones this translator's own pre-call codegen happens to touch explicitly. Building real cross-procedure register preservation is a genuine, much larger feature (this translator has no push/pop preservation anywhere, for any register); v1 instead treats any such instruction as a single-point "boundary," using the exact same overlap-exclusion mechanism decision 1 already established for loops — "spilled and never-eligible are the same code path," now generalized to every kind of external control transfer, not just backward branches. The first implementation pass added this boundary for `CALL` and `RESOLVE` but missed `OBJSIZE`/`OBJTYPE` — a real, caught-by-testing omission, not a design gap; fixed identically to `RESOLVE`'s own treatment once found.

**The real, substantive correction: decision 2's proposed pool (`rbx, rcx, rsi, rdi, r8, r9, r12-r15`) is unsafe as written, and the regression suite caught it immediately and unambiguously.** `rbx`, `r12`, `r13`, `r14`, and `r15` are SysV-ABI *callee-saved* registers — and this translator's trampoline is a real, externally-callable x86-64 function, invoked directly by ordinary compiled C (the host JIT test harness today; the kernel's own loader once wired in). A callee-saved register must be restored to its entry value before such a function returns, or the *caller's* own live values in it are silently corrupted — and this file has no push/pop preservation logic anywhere (`emit_prologue`, `OP_RET`, and `emit_trampoline` none of them save a single register). Putting `rbx`/`r12`-`r15` in the pool broke the test suite in a genuinely confusing way that's worth recording as a caution for future work on this file: `make test-native` printed `PASS` for several tests — the *translated function's own return value* was correct — but the harness process then exited with an unrelated nonzero code (observed values included 1, 20, 96, none of them `simi-jit-test`'s own defined exit codes). The actual bug was in `simi_jit_test.c`'s own `main()`, not the SIMI program under test: the *host test harness's own* compiler-generated code kept some of its local variables live in `rbx`/`r12`-`r15` across the call to `fn()`, exactly as SysV entitles it to given the callee-saved contract — and the translated code silently violated that contract, corrupting those locals in a way that only surfaced later, in `main()`'s own print/exit path, well after the comparison that had already (correctly) printed "PASS." This is precisely the failure mode callee-saved-register rules exist to prevent, and precisely why "prints the right answer" is not sufficient verification for a change like this — the exit code check in `run_native_tests.sh` (`[ "$fail" -eq 0 ]`, driven by the harness's real exit status, not a text match against stdout) is what actually caught it. **Fix**: the pool is caller-saved-only — `rsi`, `rdi`, `r8`, `r9` (4 registers, down from the proposed 9) — registers the SysV ABI already treats as freely clobberable by any call, including the outermost call from real C into the trampoline, meaning zero extra preservation logic is needed for them at all, by construction, rather than as an invariant this file has to maintain by hand. `r10`/`r11` stay excluded exactly as decision 2 specified (the existing runtime-call scratch pair); `rax`/`rcx`/`rdx`/`rbp`/`rsp` stay excluded exactly as decision 2 specified. Confirmed via direct grep that `rsi`/`rdi`/`r8`/`r9` are otherwise completely unused anywhere in this file's own codegen outside the new allocation-aware paths (the one partial exception, `rdi` as `RESOLVE`/`OBJSIZE`/`OBJTYPE`'s own call-argument register and `rep stosq`'s zero-fill destination in `emit_prologue`, is already safe by construction: both uses are transient, sequential, and — for the runtime-call case — already covered by the same boundary-exclusion that protects against the ABI-clobber hazard in the first place).

**Mechanism, as actually implemented**: rather than touching each of this file's roughly twenty opcode cases individually, the four primitives every one of them already funnels through — `ld_rax`, `ld_rcx`, `st_rax`, `st_rcx` — were made allocation-aware once, at the base. Each now checks a `g_physreg_of[symbolic_index]` table (-1 = stack, populated fresh at every `OP_ENTER` by the linear-scan pass); if allocated, the memory load/store redirects to a `mov <physreg>, rax`-shaped register-to-register move instead. This is why zero opcode case bodies needed to change, including the ones Phase 10 added for float support (`MOVQ xmm0, rax` doesn't care whether `rax` was just populated by a real memory load or a cheap register move) — a nice, unplanned confirmation that funneling every opcode through a small set of shared primitives back in Phase 3 paid for itself here.

**Verification.** Live-range/boundary/linear-scan computation, and the four allocation-aware primitives, are the only changes — applied identically to `tools/simi/simi_x86.c` and `kernel/simi_x86.c`, re-diffed afterward to confirm the two files' code bodies are still byte-identical below their differing header comments. The full existing regression suite (interpreter `make test`: 14/14 passing, including the new tests below; real x86-64 hardware `make test-native`: 13/13 passing; RV64 `make test-riscv`: 12/12 passing, decision 2's RV64 half being out of v1 scope per below) shows **zero output changes** across every pre-existing test — exactly the "identical output, different code path" proof the design's own verification plan called for, with the added confirmation that catching a real bug (the callee-saved corruption above) *required* checking real exit status, not just comparing printed values, which this project's existing harnesses already did correctly. A new test, `tools/simi/tests/straight_line_bench.simi`, was added specifically to give the allocator something to do: six independent, simultaneously-live values (deliberately exceeding the 4-register pool, to exercise both real allocation and the spill-when-exhausted path in one program) combined via a small expression tree, no loop, no call. Measured effect, with identical output in both cases (`= 21`): generated code size dropped from 698 to 618 bytes (11.5% smaller) with allocation enabled versus a temporarily pool-disabled build of the same file, confirmed via a small throwaway timing/size harness (`tools/simi/bench_harness.c`, not part of any Makefile target) built against both configurations. Every one of the twelve pre-existing native tests also shrank measurably (2-13% smaller generated code, e.g. `add.tmo` 490→442 bytes, `extra_ops.tmo` 734→638 bytes, `ptr_ops.tmo` 705→625 bytes) with byte-for-byte identical program output — real, reproducible, deterministic evidence of the win. A wall-clock timing comparison was also attempted (calling the JIT'd `straight_line_bench` function 100 million times, allocator-enabled vs. disabled) but produced inconsistent results across repeated runs in this sandboxed, likely-shared/virtualized environment (differences within run-to-run noise, sometimes favoring one build, sometimes the other) — reported honestly as inconclusive rather than cherry-picking a favorable run; the code-size reduction is the reliable, reproducible evidence for this phase, and directly implies fewer memory operations per allocated register access regardless of what a noisy timer shows.

**RV64 pool audit (decision 2's flagged pre-implementation task) — completed as a design-only decision, RV64 allocator implementation itself scoped OUT of v1**, mirroring Phase 10's own precedent for deferring RV64 work to a later, dedicated pass. Direct grep of `tools/simi/simi_riscv.c` confirms its own scratch-register convention: `X_T0`/`X_T1`/`X_T2` (`x5`-`x7`, the RV64 analogue of x86's `rax`/`rcx` ALU scratch), `X_S0` (`x8`, the frame-pointer analogue of `rbp`), and `X_A0` (`x10`, used for real runtime-call argument/return, the analogue of `rdi`/`rax` there) are the only register macros this file defines or uses anywhere — meaning `a1`-`a7` (`x11`-`x17`) and `t3`-`t6` (`x28`-`x31`), eleven registers total, are completely untouched by any existing RV64 codegen. Applying the exact lesson learned from the x86 correction above (this translator has no callee-saved push/pop preservation on RV64 either, and its trampoline is likewise a real external entry point), any future RV64 allocator's pool must be caller-saved-only under the standard RISC-V calling convention: `a1`-`a7` and `t3`-`t6`, excluding every `s`-prefixed (`s0`-`s11`) callee-saved register entirely, exactly mirroring the x86 pool's rsi/rdi/r8/r9-only shape. Implementing the actual RV64 allocator (live-range computation is architecture-independent and could mostly be shared; only the pool constant and the physical-move encoding are RV64-specific) is deferred, matching this project's established practice of landing x86 first and RV64 second for native-target work.

### Phase 13: Activation cache frame reclamation — concrete design

**The gap, confirmed by direct reading of `kernel/simi_translate.c`** (not assumed): `simi_translate_and_map()`'s cache-miss path (lines ~220-344) allocates fresh physical frames into `act->frame[i]` one at a time inside its mapping loop (line 314: `if (act) act->frame[i] = (uint64_t)(uintptr_t)frame;`), silently overwriting whatever address was previously stored there. If this is a re-upload (i.e. `act` already had `act->valid == 1` from an earlier translation), every old frame address that gets overwritten before the loop's index reaches it is lost with no free ever issued; if the new translation is *smaller* than the old one (`total_pages < act->code_pages`), the old frames at the tail indices are never touched by the loop at all and become permanently unreachable — `act->code_pages` gets overwritten to the new, smaller count at line 330, so nothing in the struct even remembers those addresses existed.

**Design.** Insert one new block immediately after the existing `total_pages > SIMI_ACT_MAX_PAGES` capacity check (line ~298) and before the frame-allocation loop begins (line ~300) — i.e. after every failure-prone step of *translating* has already succeeded, but before any *old* state is destroyed:

```c
/* Gap Remediation SIMI Phase 13: free this slot's OLD code-page frames
 * before they're overwritten below. Placed here (after translation has
 * fully succeeded, before any old state is touched) so a translation
 * failure earlier in this function never destroys a still-good cached
 * activation -- only a *successful* re-translation retires the old one. */
if (act && act->valid) {
    for (uint32_t i = 0; i < act->code_pages; i++) {
        free_physical_ram_frame((void*)(uintptr_t)act->frame[i]);
    }
    act->valid = 0;   /* defensive: if anything below still fails (e.g. an
                        * OOM mid-allocation-loop), this slot must report
                        * "not cached" rather than a half-populated
                        * frame[] array claiming to be a valid activation. */
}
```

Three things make this the right placement, not just *a* placement:
1. **Old-frame addresses are still intact when we free them** — nothing overwrites `act->frame[]` before this point, so every address freed here is real and correct, not already-clobbered.
2. **A failed re-translation can't destroy a working cache entry.** Both translation passes (`simi_x86_translate()` ×2) and the capacity check all happen *before* this block. If bytecode translation itself fails, this block never runs, `act` is untouched, and the *old* translation — while now orphaned relative to the object's new (broken) bytecode, since the cache-hit check recomputes the hash from `sb->data` fresh every call and will keep missing — is at least not leaked or double-freed. That orphaning is a separate, pre-existing, much smaller concern (an object re-uploaded with bytecode that fails to translate becomes unspawnable until fixed) unrelated to this phase's own scope.
3. **The `act->valid = 0` defensive reset** means any failure *after* this block (e.g. `allocate_physical_ram_frame()` returning null mid-loop, line ~302) correctly leaves the slot reporting "not cached" via `simi_activation_query()`/`simi_activation_info()`, rather than a slot whose `frame[]` array is a mix of freed-and-reused and not-yet-written addresses masquerading as `valid`.

**Header declaration needed**: `free_physical_ram_frame()` is declared in `kernel/frame_pool.h`, not currently included by `simi_translate.c`. Match the file's own existing convention (line 58: `extern void* allocate_physical_ram_frame(void);` — a bare extern, not a full header include, presumably to avoid pulling in `frame_pool.h`'s larger surface) with a matching `extern int free_physical_ram_frame(void* frame);` declaration near it, rather than adding a new `#include`.

**Verification plan**: extend `tests/simi_info_host_test.c` (the existing host test that already links the real, unmodified `kernel/simi_translate.c` and calls `simi_translate_and_map()` directly, per §16 intro's own established pattern) with a new scenario — translate `add.simi` once, record its `activation.content_hash`/frame addresses via the existing query path, then simulate a re-upload with different content (the test already stubs `nvme_write_sync`-adjacent upload plumbing per its own header comment), re-translate, and confirm: (a) the new activation's `content_hash` differs from the old one, proving a genuine re-translation happened, not a stale cache hit; (b) the fake frame pool this test presumably stubs `allocate_physical_ram_frame()`/`free_physical_ram_frame()` against shows the old frame addresses as freed (available for reallocation) after the second translate call — the concrete, causally-proven form of "this doesn't leak," matching this project's own established "prove it by feeding the freed address back through, not by reading the code" verification discipline (the same technique `frame_quota_host_test.c`'s free-then-reallocate-returns-same-address scenario already uses for Phase F's own `free_physical_ram_frame()` addition).

**Implemented, exactly as designed above — findings.** The free-old-frames block landed verbatim (the `if (act && act->valid) { ... act->valid = 0; }` block shown above, unchanged from the design) at the exact insertion point named, plus the matching `extern int free_physical_ram_frame(void* frame);` declaration alongside the existing `allocate_physical_ram_frame()` extern. `kernel/simi_translate.c` compiles clean (zero warnings) under the exact kernel `X86_CFLAGS`.

`tests/simi_info_host_test.c` gained a real `free_physical_ram_frame()` stub (previously the test had none — `free_physical_ram_frame` wasn't referenced by the file at all before this phase) that validates the freed pointer falls within the fake frame pool and records every freed address, plus a new Scenario 7: translate `addobj` (real assembled `add.simi` bytes), mutate one byte of the uploaded object (deliberately chosen — the *first* attempt flipped the buffer's last byte and broke translation outright, because that byte turned out to be the entry table's `offset` field, not inert padding; corrected to flip a byte inside the entry name field's zero-padding, strictly between the "main" name's null terminator and the `offset` field, confirmed inert by re-running with `kernel_serial_printf` temporarily un-stubbed to see the real translator error message before finding the safe byte), re-upload under the same object name, and re-translate. Confirms, causally: zero frames freed across a miss-then-hit against *unchanged* content (no spurious frees); exactly `old_code_pages` frees on the re-upload-with-different-content path; and every one of the freed addresses matches the exact old frame pointers by value, not just by count. Full suite: 36 checks, all passing.

Regression: the existing `tools/simi/tests/` three-way suite (`make -C tools/simi test`) still passes unchanged, 11/11, including `cap_forge` — confirming this phase's changes to `simi_translate.c` (x86-kernel-side only) have no effect on the interpreter/RV64 paths or on Phase 7's forgery-prevention behavior, as expected since this phase never touches capability tags.

### Phase 12: Capability tag propagation across CALL/RET — concrete design

**Confirmed current behavior, not assumed** (verified by reading all three implementations directly): `CALL` carries zero register operands in the ISA — it is a pure PC-relative jump-with-return-address-push, resolved at translate time against the object's entry-point table. Argument passing is realized entirely through the pre-existing, already-real r0-r7 marshaling step every implementation already has: the interpreter copies `regs[0..7]` value-only into a freshly-pushed `Frame` (`tools/simi/simi_interp.c` `OP_CALL`); the x86/RV64 translators marshal through a caller-allocated stack scratch area written before the real `call`/`jal` and read by the callee's real prologue. All three currently **force the tag byte to 0** at this exact marshaling step — the interpreter via an explicit `memset` on the new frame's `cap_tag[]`; x86/RV64 via prologues that zero their whole tag region before copying only 8-byte *values* in. The same forced-zero happens symmetrically on return: the interpreter's `OP_RET` calls `set_tag(..., 0)` on the caller's r0 after copying the value; x86/RV64's call-site epilogue uses `st_rax_untag()`/`st_slot_untag()` — a helper that unconditionally zeroes the destination tag, regardless of what the callee's r0 tag actually was.

**Design decision: make the tag ride along with the existing value-marshaling copy, unconditionally — the same rule `MOV` already uses, generalized across the call boundary.** No new opcode, no ISA/encoding change (`CALL` stays zero-operand). The justification for why this doesn't reopen Phase 7's forgery-prevention hole: the tag invariant is "this register's current bits are the untampered output of a real `RESOLVE` (or a chain of plain copies from one)." The r0-r7 argument copy and the r0 return-value copy are *already* plain, uncomputed copies — literally what `MOV` does within one frame today, just now crossing a frame boundary via a call/return instead of staying within one. Propagating the tag through a plain copy preserves the invariant exactly as safely as `MOV` already does; it does not let a callee (or caller) synthesize a fake capability from arithmetic, because nothing about this change touches how tags get *set* (`RESOLVE`) or *cleared* (every arithmetic/pointer-computing opcode still clears unconditionally, unchanged).

**Interpreter — small, direct fix.** In `OP_CALL`: replace the blanket `memset(nf->cap_tag, 0, sizeof(nf->cap_tag))` with a zero of registers 8-1023 only, plus an explicit `nf->cap_tag[i] = fr->cap_tag[i]` alongside the existing `nf->regs[i] = fr->regs[i]` loop for `i` in 0..7. In `OP_RET`: replace `set_tag(&frames[frame_top], 0, 0)` with `set_tag(&frames[frame_top], 0, fr->cap_tag[0])`.

**x86/RV64 — the value-marshaling code already exists per-argument; the tag needs a parallel side-channel through the same scratch memory, since it can't steal bits from the 64-bit value.** Concretely:
- *Arguments (CALL)*: extend the call site's existing 64-byte (8 × 8-byte) outgoing-argument scratch push to 72 bytes. After writing each argument's value (unchanged), build one additional 8-bit mask (bit *i* = source register *i*'s current tag, read via the existing `ld_tag_cl`-style helper Phase 7 already introduced for `RESOLVE`'s own tag-check codegen) and store that mask word at `[rsp+64]`. The callee's prologue, immediately after copying each argument's *value* from `[rbp+16+8*i]` into slot *i* (unchanged), additionally reads the mask word from `[rbp+16+64]`, tests bit *i*, and calls the existing `st_tag_imm()`-style helper with the result — a fixed, branch-free `test`/`setne`/store sequence per argument, the same "compute a boolean into a tag byte without branching" idiom Phase 7's own `RESOLVE` codegen already established (`kernel/simi_x86.c`'s `test+setne...before storing` sequence, per the earlier capability-tag research), not a new code shape being invented for this phase.
- *Return value (RET)*: simpler — only one bit needs to cross. The callee's `RET` codegen, immediately before tearing down its frame, additionally loads its own r0 tag byte into a scratch register that survives the real `ret`/`jalr` (a register not part of the return-value or stack-pointer path — e.g. `dl` on x86, a free temporary like `t1` on RV64), the same place it already loads r0's *value* into `rax`/`a0`. Back at the call site, after the real `call`/`jal` returns, that scratch register's value replaces the current unconditional `st_rax_untag()`/`st_slot_untag()` call with a conditional set (`test`/`setne`/store, same branch-free idiom).

**What does not change**: the tag-clearing rule for every arithmetic/pointer opcode (`ADD`, `PTRADD`, `LOAD`, etc.) — those still unconditionally clear the destination tag, exactly as Phase 7 established; this phase touches only the two marshaling points (arguments in, return value out), nothing about how a tag is earned or lost *within* a procedure.

**Verification plan**: a new test program (interpreter + x86 + RV64, matching Phase 7's own `cap_forge.simi` three-way-agreement discipline) that `RESOLVE`s a real object, `CALL`s a procedure passing the capability as an argument, has that callee run `OBJSIZE`/`OBJTYPE` on the received argument and confirm they succeed (proving the tag survived the call), then has the callee itself `RESOLVE` a second object and `RET` it, with the caller running `OBJSIZE`/`OBJTYPE` on the returned value and confirming success too (proving the tag survives the return). Critically, **`cap_forge.simi` itself must still pass unmodified** — its forged-capability-gets-rejected assertion is the one this phase must not weaken, and re-running it after this change (across all three implementations, matching the project's established regression discipline) is the concrete proof this phase didn't quietly loosen Phase 7's unforgeability guarantee while fixing this narrower gap.

**Implemented — findings.** All three implementations landed close to the design above, with two real deviations worth recording honestly rather than glossing over:

1. **The x86/RV64 mask mechanism ended up simpler than "test+setne."** The design sketch proposed extracting each argument's mask bit via a `test`/`setne` comparison; the actual implementation instead does a direct `shr`/`and 1` (x86) or `srli`/`andi 1` (RV64) per bit — since the mask bit is already a boolean value sitting at a known position, shifting it down and masking to 1 bit produces the 0/1 result directly, with no comparison step needed at all. Still branch-free, still one instruction sequence per argument, just one instruction shorter than originally sketched. Return-value propagation landed exactly as designed: `dl` on x86, `t1` on RV64, both loaded by `RET` immediately before the epilogue and both confirmed (by reading the epilogue's own instruction sequence) to be untouched by `mov rsp,rbp`/`pop rbp`/`ret` (x86) or `addi sp`/`ld ra`/`ld s0`/`addi sp`/`jalr` (RV64).
2. **RV64 needed one real, unanticipated fix outside `simi_riscv.c` itself**: the host verification decoder (`tools/simi/rv64_exec.c`) had never needed to decode `SRLI` or `ANDI` before — nothing emitted either opcode prior to this phase — so its OP-IMM dispatch only implemented `ADDI`/`XORI`/`SLTIU`/`SLLI`, and every RV64 test started failing with "unimplemented or malformed instruction" at the exact point the new prologue's mask-consuming loop first ran. Traced to the decoder's `switch (funct3)` inside its `case 0x13` (OP-IMM) block having a `default: return RV_EXEC_BAD_INSTR` for anything not already listed — not a bug in the new codegen, a real gap in the verifier that had simply never been exercised. Fixed by adding both cases, using the same "funct6 assumed 0, not checked" simplification the file's own pre-existing `SLLI` case already documents (safe here because every shamt this translator ever emits is a small argument index, 0-7).

Both host copies of the x86 translator (`tools/simi/simi_x86.c` and `kernel/simi_x86.c`) received the identical code change and were re-diffed afterward to confirm the two files' code bodies are still byte-identical below their differing header comments, per this project's own established convention. `kernel/simi_x86.c` compiles clean (zero warnings) under the exact kernel `X86_CFLAGS`.

A new test, `tools/simi/tests/cap_call_ret.simi`, was added rather than modifying `cap_forge.simi` — it isolates the two propagation directions into two separate `CALL`s (one testing argument-tag survival via `OBJSIZE` on a received argument, one testing return-tag survival via `OBJSIZE`/`OBJTYPE` on a `CALL`'s result) so a failure in either direction produces an unambiguous, un-mistakable wrong answer rather than a coincidental near-miss. Result: **2100303 on all three implementations** (interpreter, x86 JIT, RV64 host-decoder) — the first real, causal proof that a genuine capability now survives both a `CALL` argument and a `RET` return value. `cap_forge.simi` was re-run unmodified across all three and still passes (3000304), confirming Phase 7's forgery-rejection guarantee was not weakened by this change. Full regression: interpreter 12/12, x86 native 11/11, RV64 11/11 (both native suites carry 2 intentional skips, unrelated to this phase, pre-existing).

### Phase 14: Aggregate ABI + multi-way dispatch (LLVM-backend prerequisite) — concrete design

**Two independent gaps, grouped only because both exist to unblock future compiler-front-end work.** Confirmed by reading the ISA header: `FMT_BR` is `BR label` and `FMT_BC` is `BC rA, label [, INV]` — both take a *static* label operand baked in at translate time. There is no indirect/computed-target branch primitive of any kind today — not "no jump-table opcode specifically," but no way to branch to an address held in a register at all. That's a real gap, not just a missing optimization.

**Design decision — aggregates: pass/return by invisible reference above 8 bytes, no new opcodes needed.** SIMI's r0-r7 argument convention is already genuine memory-based marshaling through a stack scratch area, not real register-file aliasing (confirmed in Phase 12's research) — so the usual "does this struct fit in two SysV registers" classification doesn't apply; every argument slot is already just an 8-byte memory cell regardless of what's in it. This makes the convention simple: aggregates ≤8 bytes are passed by value, their raw bytes occupying one argument slot exactly like any scalar — SIMI doesn't care what bit pattern a slot holds. Aggregates >8 bytes are passed by invisible reference: the caller `LEA`s the address of its own on-stack copy into an argument slot, and the callee treats that slot as a pointer, using `LOAD`/`STORE`/`PTRADD` against it — composed entirely from opcodes that already exist. Returns follow the same rule: aggregates ≤8 bytes return through `r0` like any scalar; aggregates >8 bytes return via a caller-supplied "return slot" pointer passed as an implicit leading argument (classic `sret` lowering), also zero new opcodes. This is purely a documented convention (a new subsection alongside §4.8) plus a future compiler front end's lowering responsibility — no ISA or runtime change.

**Design decision — indirect jump: exactly one new opcode, `JMPR rA` (jump to the code-relative address held in rA), not a specialized jump-table opcode.** This matches the ISA's existing composability discipline (`LEA`+`PTRADD` already compose instead of a hypothetical single "index into array" opcode) — a real jump table is then just: bounds-check the switch index against the case count (existing `CMP`/`BC`), `LOAD` the matching entry from a table of code-relative offsets stored in the object's existing literal pool (no new pool type — the literal pool already exists in the `.simi`/`.tmo` format), `PTRADD` it against the code segment's base, then `JMPR` to the result. One opcode, composed with primitives that already exist, exactly like decision 1's aggregate handling.

**Non-negotiable safety requirement, not a nice-to-have**: `JMPR` is a real control-flow-integrity hazard — an indirect jump to a bug- or attacker-controlled register value can land mid-instruction or outside the object's own code region entirely. Every translator's `JMPR` lowering must emit a bounds check (target must fall within the current activation's own translated code range, else trap/kill the process) before performing the actual jump, on every implementation (interpreter + both native translators) — the same "verify before trusting" discipline Phase 7's capability tags already established for object references, applied here to code addresses. This is required for `JMPR` to ship at all, not an optional hardening pass added later.

**Why now, even though nothing currently needs them**: `docs/AeroSLS-SIMI-LLVM-Backend-Scoping-v0.1.md` §6 and §7 both name these as real open design questions its own Phase 2 (minimal integer-only subset) deliberately scopes around rather than solves. Deciding both conventions here, before any LLVM-backend work starts, means that project inherits answers instead of inventing them under its own time pressure — the same "decide the boundary before building against it" discipline this project used for the Phase 24 legacy-KV-vs-rowstore fork.

**Verification plan**: a hand-written `.simi` test program using the aggregate convention (a small struct passed by value, a larger one passed by reference, one returned via `sret`) verified for correct byte layout across a `CALL`/`RET`; a second program building and dispatching through a real jump table via `JMPR`, plus a deliberate out-of-bounds-target test confirming the bounds check actually traps rather than jumping wild — run across the interpreter and both native translators per this project's standard three-way discipline.

**What's explicitly not in scope for Phase 14**: actually building the LLVM backend itself — that remains entirely the separate, much larger, not-yet-started project the scoping document describes.

#### Phase 14 — Implemented, findings

Both halves of the design above shipped: the aggregate ABI convention (§4.8.1, purely documentation + a verification test, zero code changes) and the `JMPR` opcode (a real ISA addition, implemented in all three execution engines — reference interpreter, x86-64 native translator, RV64 native translator).

**Part A — aggregate ABI.** Documented as new §4.8.1, exactly as scoped: no ISA or object-format change. Verified by `tools/simi/tests/aggregate_abi.simi`, a single self-checking program exercising all three conventions in sequence — a ≤8-byte struct packed into one register and passed by value (`sum_packed`, `(7,9)` packed via `SHL`/`OR`, unpacked via `SHR`/`SHR` → 16), a >8-byte struct passed by invisible reference (`sum_struct3`, a 3-element `i64` array built in the caller's r7-scratch memory, a pointer to it passed as the sole argument → 6), and a >8-byte `sret` return (`fill_struct3`, called with a caller-supplied return-slot pointer plus a scalar `x=2`, filling `{x*10,x*20,x*30}` through it → 120). Expected/actual result 142 on all three engines — interpreter, x86-native, and RV64 — first try, no bugs found. This is the expected outcome for Part A specifically: it adds no new machine-code-emission logic anywhere, so there was nothing new for the translators to get wrong.

**Part B/C — `JMPR` ISA + assembler + interpreter.** Added `OP_JMPR`/`FMT_JMPR` to `simi_isa.h` (a single register operand, the abstract SIMI pc to jump to — never a raw machine address, see the opcode's own doc comment), assembler parsing, disassembler printing, and the interpreter case (`fr->regs[ra]` read as the target pc, bounds-checked against `obj.num_instr`, `die()`s on an out-of-range value). Verified with a hand-built jump/no-jump test and a deliberate out-of-range test before any native-translator work started.

**Part D — `JMPR` in the x86 translator.** The runtime jump table is reserved as raw data at `out_buf` offset 0 (`hdr.num_instr * 8` bytes, one absolute-machine-address slot per abstract pc), gated behind a cheap pre-scan (`g_has_jmpr`) so an object with no `JMPR` at all pays zero extra bytes. Table contents are backfilled in a pass after the main translate loop, the trampoline loop, and the existing branch-fixup pass, once `g_instr_off[]` is known for every real pc. Codegen: load the target pc into `rcx`, `cmp` against `g_num_instr`, `jae` to a trap on out-of-range, otherwise index the table (`rax = g_out_buf_addr; lea rax,[rax+rcx*8]; mov rax,[rax]`) and `jmp rax` — an indirect jump that never falls through. The out-of-range path is a bare `ud2`, reusing this codebase's existing "real hardware trap" idiom (the same idea as `#DE` for integer div-by-zero and `#NM` for lazy FPU) rather than inventing new software abort infrastructure. The Phase 11 register allocator's `compute_alloc_for_proc()` needed one addition, caught during implementation rather than left as a latent gap: `OP_JMPR` previously fell through to that switch's `default:` case, meaning a register holding a `JMPR` target would never be touch-tracked or boundary-excluded. Added an explicit case (`tx_touch(ra,pc); tx_boundary(pc,pc);`), treating `JMPR` the same conservative way `CALL`/`RESOLVE` already are. Applied identically to `kernel/simi_x86.c` — re-diffed against the host copy afterward and confirmed byte-identical below the two files' differing header comments, then compiled in isolation under the exact kernel `X86_CFLAGS` (`-ffreestanding -O2 -Wall -Wextra -mcmodel=small -mno-red-zone -mno-sse -mno-sse2 -mno-mmx -fno-pie -fno-pic -fno-tree-vectorize`) with zero warnings.

**Part E — `JMPR` in the RV64 translator, and a real bug caught by actually running the output.** The first implementation pass mechanically ported the x86 design: bake `(uintptr_t)out_buf` as `g_out_buf_addr`, store `g_out_buf_addr + g_instr_off[pc]` in each table slot, `jalr` through the loaded value. This compiled cleanly and *looked* correct, but `make test-riscv` failed it immediately with `execution error: fetch/load/store outside guest memory buffer` — `rv64_exec.c`'s guest CPU doesn't execute this file's output at its true host address; it fetches and executes against a small `cpu->mem` buffer indexed from 0 (see `rv64_exec.c`'s own `fetch32`/`load_mem`/`store_mem`), and *every* control-transfer instruction this translator already emits — `JAL`, `BEQ`, `BNE`, even `AUIPC` — computes its target purely as an offset relative to the guest's own pc, never a real host address. The only place a genuine host pointer legitimately reaches emitted RV64 code is a host-function call (`RESOLVE`/`OBJSIZE`/`OBJTYPE`'s `rt_*_fn` literals), and even those aren't fetched as guest instructions — `rv64_exec.c`'s `JALR` case recognizes a reserved sentinel bit pattern (`RV_EXEC_HOSTFN_MASK`/`RV_EXEC_HOSTFN_BASE`) and redirects to a host callback table instead. `JMPR` is real guest-to-guest control flow, indistinguishable from what `JAL` already does, so its table needed to hold plain byte offsets into `out_buf` — exactly what `g_instr_off[]` already stores — not `out_buf`'s host address plus that offset. Fixed by removing `g_out_buf_addr` from the RV64 file entirely (rather than leaving an unused-but-misleading field) and backfilling the table with `g_instr_off[pc]` directly; the table's own base address is likewise just its offset within `out_buf`, reached the same way every other 64-bit constant in this file already is (a literal-pool slot loaded via the standard `auipc`+`ld` pattern, lazily reserved on the first `JMPR` instruction seen via `g_jmpr_lit_idx`, so an object with no `JMPR` pays nothing extra). The out-of-range trap is `e32(cb, 0)` — the all-zero-bits 32-bit word, permanently reserved by the base RV64 ISA as illegal (opcode field `0000000` matches none of the base opcodes, falls to `rv64_exec.c`'s own `default: return RV_EXEC_BAD_INSTR;`) — the direct RV64 analog of x86's `ud2`. After the fix, `jmpr_basic.tmo` passed on RV64 (222, matching interpreter and x86), and the deliberate out-of-range test correctly produced `execution error: unimplemented or malformed instruction` rather than a memory fault or (worse) silently jumping into garbage. This is exactly the kind of bug this project's own verification discipline exists to catch — wrong by inspection-passing review, caught only because `rv64_exec.c` actually executes the emitted bytes against a real (if simulated) CPU model instead of trusting that "x86's approach ported cleanly."

**Verification.** Full three-way regression, no prior test's output changed: interpreter 16 passed / 0 failed (`make test`), x86-native 15 passed / 0 failed / 3 skipped (`make test-native`), RV64 14 passed / 0 failed / 4 skipped (`make test-riscv`) — the skips are all pre-existing and unrelated to this phase (`cap_forge_debug` has no self-check comment by design, `mem_ops` is an interpreter-only convenience superseded by `mem_ops_native`, `float_ops` is explicitly out of RV64's v1 scope per Phase 10, and the two new `jmpr_oob`/no-`Expected-result` tests are deliberately excluded from auto-grading — see below). Two new tests joined the permanent suite: `jmpr_basic.simi` (a real jump through the runtime table, expected result 222, PASS on all three engines) and `aggregate_abi.simi` (expected result 142, PASS on all three engines). `jmpr_oob.simi` (a deliberate out-of-range `JMPR` target) is intentionally *not* self-grading — there is no valid "expected result," the whole point is that normal execution must not produce one — and was instead verified by hand on all three engines: the interpreter `die()`s with exit code 2 (`"JMPR target out of range (bad indirect jump)"`), the x86-native harness's SIGILL handler reports `"CRASHED during execution (bad codegen)"` (a real `#UD` fault from the emitted `ud2`, correctly caught — the harness's message is generic/pessimistic wording written for genuine codegen bugs, but the crash itself is exactly the intended outcome here), and `simi-riscv-verify` reports `"execution error: unimplemented or malformed instruction"` (the real illegal-instruction decode path). Full 76-file kernel compile sweep (every file in the root Makefile's `X86_C_SRC` list, compiled individually under the exact `X86_CFLAGS`) — zero failures, including `kernel/simi_x86.c` itself, re-diffed against the host copy and confirmed byte-identical below the two files' differing header comments.
