# AeroSLS Relational Layer Roadmap v0.1 — from record store to real RDBMS

## 0. Where this picks up

`AeroSLS-TIMI-ISA-v0.1.md` (Phases 1–8) and `AeroSLS-LPAR-Roadmap-v0.1.md` (Phases 9–15, closed) cover the instruction set, native translators, capability/authority model, and partition isolation. Separately, and largely as a side effect of exercising the object catalog, this project has already built a working transactional record store directly into the kernel: `object_catalog.c`'s `sys_sls_select/insert/update/delete/schema_set`, `journal.c`'s WAL-based commit/rollback/recovery, `lock_mgr.c`'s per-key locking, `index_mgr.c`'s sorted-array indexes, `constraint.c`'s UNIQUE/NOT_NULL/RANGE/REFERENCE checks, `mqt.c`'s materialized aggregates, and `cursor.c`'s DECLARE-CURSOR-shaped iteration API. This is the Db2-for-i model — a database integrated directly into the kernel as native object types, not a userspace process ported in — and it is the right architecture for an SLS OS. It is not, however, a relational database yet.

The hard ceiling: `struct SLSObjectEntry` (a "table") pairs 1:1 with exactly one `struct SLSObjectRecord`, capped at `RECORD_MAX_FIELDS = 32` key-value fields **total** — not per row, total, for the whole table, forever. There is no row concept in the storage layer at all. Multiple rows are simulated today purely by naming convention (packing `"alice_dept"`, `"alice_salary"`, `"bob_dept"`, `"bob_salary"` as four of a table's 32 available fields) — meaning a "table" tops out at something like 8 rows × 4 columns before `sys_sls_insert` starts returning "Record capacity full." `CATALOG_MAX_OBJECTS = 128` then caps the total number of named objects — tables, processes, streams, everything — system-wide. No amount of indexing, query language, or join logic matters until tables can hold real, unbounded row sets. That is what this roadmap is for.

This document follows the same shape as the LPAR roadmap: design principles, a phase table with dependencies, and a per-phase Why/Scope/Verification-plan section. Nothing below is built yet — this is a planning document. As each phase lands, its findings slot in as a "Findings addendum (as built)" under that phase's section, exactly like Phases 9–14 did in the LPAR roadmap.

## 1. Design principles carried forward

1. **Real execution beats compile-check beats logic review**, in that order, and this project is honest about which ceiling each phase actually hit — the same verification hierarchy every phase since Phase 4 has used.
2. **Backward compatible by construction.** Every object created before this roadmap's Phase 16 lands — and every object that never opts into row-set storage after — keeps working exactly as `sys_sls_select/insert/update/delete` behave today. The new row-set storage is an additional capability a table can be created with, not a forced migration.
3. **Groundwork, not a full RDBMS.** This roadmap's job is to make AeroSLS's relational layer real — actual row sets, actual indexes, actual joins, a usable SQL subset — not to match Postgres feature-for-feature. Cost-based optimization, multi-way joins, MVCC snapshot isolation with full serializability, and a complete SQL grammar are the kind of scope this project has consistently deferred until a concrete need names them (see Phase 22, and the LPAR roadmap's own Phase 15 precedent).
4. **Audit every call site before calling a phase done.** The LPAR roadmap's standing lesson (Phase 8's uninitialized `partition_id` bug, Phase 13's frame-pool call-site audit) applies here at least as hard: `RECORD_MAX_FIELDS`, `struct SLSObjectRecord`, and the point-syscall API are referenced from `agent.c`, `agent_tools.c`, `cursor.c`, `mqt.c`, `constraint.c`, `stream.c`'s metadata records, and more — Phase 16 in particular needs a full grep-and-check pass, not just the obvious call sites, before it's called done.
5. **Design pass before implementation**, per phase, same as every LPAR phase — this document sets direction and dependency order, not final per-phase design decisions. Phases 16 and 21 are flagged below as needing the most real design work before code gets written.

## 3. Phase 16 — Row-set storage engine

**Why this is the heaviest lift in this roadmap.** Every existing DB primitive in this kernel — `journal.c`'s WAL, `lock_mgr.c`'s locking, `index_mgr.c`'s indexes, `constraint.c`'s checks, `mqt.c`'s aggregates, `cursor.c`'s iteration — all operate on the same underlying assumption: one catalog object has one `SLSObjectRecord` of up to 32 fields. That assumption was fine for a config-blob-style KV store and is the wall every relational ambition in this codebase hits immediately. `sys_sls_insert` already returns "Record capacity full" at 32 fields today — this isn't a future scaling concern, it's the present behavior. Nothing downstream in this roadmap (real indexes, WHERE clauses over real rows, joins, SQL) is meaningful until a table can hold an actual, growable set of rows.

**Design investigation (this pass).** Three real shapes were considered before picking one:
- **Grow `SLSObjectRecord`'s fixed array.** Rejected — not a fix, a bigger version of the same wall: still a fixed in-RAM cap, still one array per table, still doesn't page to NVMe.
- **Rows as catalog objects.** Each row as its own `SLSObjectEntry`/`SLSObjectRecord` pair with a `parent_table_id`. Rejected — mixes row semantics into `CATALOG_MAX_OBJECTS = 128`'s same namespace as processes/streams/everything else, and `catalog_check_access()`'s linear-scan-by-name pattern doesn't scale to thousands of rows searched by name.
- **A dedicated row heap, paged and NVMe-backed, with a shared page pool.** Selected. Full design below.

Investigation covered `stream.c`'s existing NVMe-paging precedent, `persist.c`'s LBA layout, and a full call-site audit — the concrete facts that shaped the decision:

- `stream.c` is the closest existing precedent (a `frames[]` pointer array of lazily-`nvme_read_sync`-loaded 4 KiB pages, flushed via one `nvme_write_sync` per dirty page), but its *allocation* shape doesn't generalize: each of `STREAM_MAX = 8` named slots gets its own **pre-reserved, fixed 64 MiB LBA region** regardless of use. Applied to tables, that would hard-cap table count at a small fixed number and waste enormous NVMe space on small tables. The row store reuses `stream.c`'s lazy-page-load *mechanism* but not its per-object-fixed-region *allocation policy* — row pages are drawn from one shared, monotonically-growing page pool (a bump allocator, page ids `0..N` shared across every row-set table), addressed as `ROWSTORE_LBA_BASE + page_id * 8`. Simple, first-cut, no reclaim (explicitly deferred, same "fail-cleanly-only, no eviction in the first cut" posture Phase 13 took for frame quotas and Phase 14 took for frame reclamation) — deleting a row marks its slot inactive but does not free the page for reuse yet.
- `persist.c`'s regions run `1024`–`5816` (catalog/records/schemas/programs/partitions), then `STREAM_DIR_LBA = 8192` through stream data ending at LBA `1,114,112`. Only ~2,376 sectors of gap exist *between* the persist regions and stream's start — too tight for a new region. The real disk image, however, is **10 GiB** (`qemu-img create ... sls_storage.img 10G` in the Makefile), not the 1 GiB `stream.h`'s own header comment claims — a stale comment worth fixing when this phase is implemented, noted here rather than fixed now since this pass was investigation-only. That leaves roughly 9.4 GiB genuinely free after stream's region ends. `ROWSTORE_LBA_BASE = 2,000,000` (round number, well clear of stream's end) with a first-cut reservation of `ROWSTORE_MAX_PAGES = 262144` (1 GiB of row data, 2,097,152 sectors) leaves the rest of the disk untouched for whatever comes after. `table_headers[]` (below) is small and fixed-size, so it persists the normal way — a new `PERSIST_MAGIC_ROWSTORE` region following `persist.c`'s existing magic-header + `persist_write_array`/`persist_read_array` pattern exactly, sized like the Phase 10 partition regions were. Individual row *pages* are NOT part of that array snapshot (they're large and sparse, not a small fixed struct array) — they persist via `stream.c`'s per-page lazy-read/eager-write pattern instead, keyed by `page_id`.
- Row width is fixed and schema-derived, decided at table-creation time, not reused wholesale from `SLSObjectRecord`'s per-object shape: `RECORD_MAX_FIELDS = 32` fields × `RECORD_VAL_LEN = 256` bytes was sized for "one record describing one object," and would only pack ~4 rows into a 4 KiB page if reused per-row — too sparse. Row-set tables instead cap column width at a narrower `ROWSTORE_STRING_LEN = 64` bytes (deliberately narrower than the legacy 256, a first-cut density trade-off worth stating explicitly, same spirit as Phase 17's deliberate single-column-only index limit) and compute each row's fixed byte width from its schema at creation time (`UINT64`/`FLOAT` = 8 bytes, `BOOL` = 1 byte, `STRING` = 64 bytes inline, no TOAST/overflow support in this first cut), so `ROWS_PER_PAGE = 4096 / row_width` is table-specific. A `row_id` encodes `(page_id, slot_index)` for O(1) lookup with no directory scan.
- Call-site audit (§1 principle 4) found **13** files touching `object_records`/`SLSObjectRecord`/`RECORD_MAX_FIELDS`/`rec->fields`/`.field_count`, the full blast radius Phase 16 needs to either migrate or deliberately leave on the legacy path: `object_catalog.c` (defines it), `agent.c`, `agent_tools.c`, `aggregate.c` (a real ad hoc query executor — WHERE/GROUP BY/ORDER BY/aggregates over one object's fields, confirmed by direct read, not just grep — the closest existing prior art for Phase 18/19, and a real future migration target), `constraint.c`, `cursor.c`, `index_mgr.c`, `mqt.c`, `persist.c`, `query_engine.c` (a keyword-routed diagnostic/introspection console — "Cognitive Direct Object Space Query... No SQL"; touches `object_records[]` for its dump handlers but is unrelated to relational query work, confirmed by direct read), `net/http.c`, and `persist_partition_host_test.c`. Every row-set table stays reachable through this same set of subsystems eventually, but Phase 16 itself only has to prove the new storage works — migrating each of these 13 call sites onto it is follow-on work for Phases 17–19, tracked here so it isn't lost.
- Boot wiring: `kernel.c`'s init order is `... → process_init → partition_init → loader_init → ... → journal_init → lock_mgr_init → index_mgr_init → constraint_init → cursor_mgr_init → mqt_init → agent_init → microkernel_init → ... → persist_restore_all() → stream_init()`, with `persist_restore_all()`/`stream_init()` both gated inside the "NVMe found" branch. `rowstore_init()` slots in right alongside `stream_init()` — after `persist_restore_all()` restores `table_headers[]`, before general availability — and, like `stream_init()`, still runs on a cold start with no NVMe (empty row store, no restore).

**Scope.**
- A new subsystem, `kernel/rowstore.c`/`.h`: a `table_headers[CATALOG_MAX_OBJECTS]` array (index-aligned with `object_catalog[]`, same idiom as `object_records[]`/`object_schemas[]` today) holding `{object_id, row_count, first_page_id, row_width, next_free_page_id}` per row-set table, plus the shared page pool described above.
- A table object gains an opt-in flag (new `uint8_t uses_rowstore` field on `SLSObjectEntry`, defaulting to 0 — every pre-Phase-16 object keeps its exact current single-record behavior, the same backward-compatible-by-construction discipline as every LPAR phase's new fields). Row-set tables get a schema (reusing `SLSObjectSchema`'s existing field-type model, not a new one) and a row heap instead of a `SLSObjectRecord`.
- Row-level CRUD: insert-row/get-row/update-row/delete-row-by-id, addressable by the `(page_id, slot_index)`-encoded `row_id`, with `partition_id` inherited from the owning table — row operations route through the same table-name lookup + `catalog_check_access()` gate the legacy path already uses, so no new partition-boundary logic is needed at the row layer (Phase 8's "one choke point" principle, reused, not rebuilt).
- Explicitly NOT in scope for this phase: WHERE-clause filtering (Phase 18), indexes over rows (Phase 17), SQL (Phase 19), page reclamation/compaction after delete (named gap, deferred), migrating the 13 audited call sites onto the new store (follow-on work for Phases 17–19). This phase only has to prove rows can be stored, retrieved by id, updated, deleted, and paged to/from NVMe — a full-table scan by row_id is enough to verify it.

**Verification plan.** `rowstore.c`'s dependencies are now known concretely, not hypothetical: `frame_pool.h` (proven shallow and host-testable in Phase 13) for RAM pages, and a `stream.c`-shaped lazy NVMe read/write pattern (proven fakeable behind a small in-memory LBA map in `persist_partition_host_test.c`, Phase 10). Plan for a real host test — page allocation/row insert/get/update/delete-by-id, and a restart-simulating round trip (write rows, wipe RAM state, reload from the fake NVMe, confirm they come back) — executing the real, unmodified row-store code, not a reimplementation, the same tier Phases 10/11/13 reached. Full call-site audit (already done above, 13 files) re-confirmed against whatever the implementation actually touches, before this phase is called done.

**Findings addendum (as built).** Landed close to the investigated design, with one correction to a claim that turned out to be wrong once implementation started, and one deliberate strengthening.

**Correction to the investigation pass:** §3 previously said row operations would route through "the same table-name lookup + `catalog_check_access()` gate the legacy path already uses." That's inaccurate — direct inspection of `object_catalog.c` during implementation found `sys_sls_select/insert/update/delete` never call `catalog_check_access()` at all; that function is only wired into TIMI `RESOLVE` (Phase 7) and process spawn (Phase 9). The legacy KV record path has **no partition or permission boundary at that layer today** — a real, pre-existing gap, not a Phase 16 regression, and out of scope to fix here (parallel to how Phase 9 found and explicitly scoped-out an analogous gap in process spawning before fixing it). Given that, row-set tables' CRUD (`rowstore_row_insert/get/update/delete`, `rowstore_table_scan`) were built to gate on `catalog_check_access()` anyway — `PERM_WRITE` for insert/update/delete, `PERM_READ` for get/scan — a deliberate strengthening over the legacy path rather than a copy of it, justified because this is new functionality with zero existing callers to break. `rowstore.h`'s header comment now states this accurately.

Otherwise the design landed as investigated: `kernel/rowstore.h`/`.c` (new), a `table_headers[128]` array index-aligned with `object_catalog[]`, a shared bump-allocated page pool (`ROWSTORE_LBA_BASE = 2,000,000`, `ROWSTORE_MAX_PAGES = 262144`), schema-derived fixed-width rows (`ROWSTORE_STRING_LEN = 64` inline per STRING column, 8 bytes UINT64/FLOAT, 1 byte BOOL), and a 4-byte `next_page_id` header at the start of every page forming a singly-linked per-table page chain. A `RowValues` struct (`{count, values[16][256]}`) carries text in/out at the API boundary, matching this codebase's existing "everything is a string, the schema coerces" convention — parsing/formatting is hand-rolled (no libc `atoi`/`printf` in a freestanding kernel file): strict unsigned-decimal for UINT64, a minimal signed-decimal-with-one-`.`-no-exponent parser for FLOAT, `"true"/"1"` and `"false"/"0"` for BOOL. A STRING value at or over `ROWSTORE_STRING_LEN` bytes is **rejected**, not silently truncated — verified explicitly, not just asserted in a comment.

`uses_rowstore` landed as a new `uint8_t` on `struct SLSObjectEntry` (`object_catalog.h`), zero-default, exactly as scoped. `rowstore_create_table()` sets it and calls **two** separate persist functions — `persist_rowstore_headers()` for `table_headers[]` and the pre-existing `persist_catalog()` for the flag itself, since the two facts live in two different structs/arrays that already have two different persist paths. Missing the second call would have been exactly the kind of "field added, one persist call updated, the other forgotten" bug Phase 8's own audit lesson exists to prevent — caught by design, then confirmed by the round-trip test actually exercising it (below).

`persist.c` gained `PERSIST_ROWSTORE_HDR_LBA = 5824` / `PERSIST_ROWSTORE_ENT_LBA = 5832` (comfortably inside the 2,376-sector gap the investigation found before `STREAM_DIR_LBA`, using only ~39 of the ~297 available frames), `PERSIST_MAGIC_ROWSTORE`, `persist_rowstore_headers()`, and restore block 6 — following the existing five blocks' exact shape (magic check → size-validated `persist_read_array` → log line). The page-pool bump-allocator cursor (`rowstore_next_free_page_id`) rides along in the header frame's otherwise-unused v1 slot rather than needing its own region, the same "steal a `write_hdr()` field" trick used nowhere else in this file but available since `write_hdr()` always had three general-purpose `uint32_t` slots.

`kernel.c` wiring: `rowstore_init()` (RAM-only: zeroes `table_headers[]`/the page cache, resets the cursor) sits alongside `mqt_init()`/`agent_init()`, unconditionally early in boot — not gated behind NVMe detection, since it has nothing to do with disk yet. The real restore happens inside `persist_restore_all()`'s block 6, at the same point in boot `stream_init()`'s restore already happens, matching that precedent exactly.

**Verification exceeded the plan, same as Phases 10/11/13.** `rowstore_host_test.c` (repo root) links the real, unmodified `kernel/rowstore.c` **and** `kernel/persist.c` together — not just rowstore.c in isolation — against `frame_pool.h`'s `allocate_physical_ram_frame()` stub (a real per-call heap allocation this time, since row pages genuinely need to be distinct, independently-writable buffers, unlike simpler fixed-pointer stubs used elsewhere), a call-tracked `catalog_check_access()` stub (records call count and last requested permission, and can be told to force-deny — letting the test verify rowstore.c calls it correctly, not just that *something* gates access), and the same in-memory fake-NVMe pattern Phase 10's own test established. 39 checks, all passing, all real execution: table creation validates and computes the correct layout (row_width=74, rows_per_page=55 for a 3-column UINT64/STRING/BOOL table, both hand-verified against the formula); 60 inserts correctly span exactly 2 pages with the page boundary landing exactly where the arithmetic predicts; every one of the 60 rows reads back correct across all three column types; update-in-place leaves neighboring rows untouched; delete tombstones without disturbing anything else and a second delete on the same row fails cleanly rather than double-freeing; a full table scan visits exactly the 59 surviving rows and never the tombstoned one; every documented error path fires correctly (value-count mismatch, unparseable UINT64, oversized STRING rejected not truncated, permission denial, unknown table, out-of-range `RowId`); page-pool exhaustion was forced deterministically (setting the cursor to `ROWSTORE_MAX_PAGES` directly, rather than actually allocating a quarter million real pages) and fails cleanly. The strongest check: a full restart-simulating round trip — create a table, insert two real rows, call the real `persist_catalog()`, wipe **every** bit of relevant in-memory state (`object_catalog[]`, `object_schemas[]`, `table_headers[]`, the page cache, the bump cursor) exactly like a fresh boot's BSS-zero would, then call the real `persist_restore_all()` — confirms the table's identity, its `uses_rowstore` flag (proving the two-persist-call design actually works, not just compiles), its layout, its row count, and both rows' actual data (read back through a page lazily reloaded from the fake NVMe, exercising `rowstore_load_page()`'s restore-time branch for real) all survive.

Full compile-check sweep (20 files: `rowstore.c`, `persist.c`, `object_catalog.c`, `kernel.c`, and every one of the 13 audited call-site files plus `process.c`/`loader.c`/`frame_pool.c`/`partition.c`/`ipc.c`) — all compile cleanly under the real kernel build flags, zero new warnings. Re-ran every pre-existing host test (`partition_host_test.c`, `persist_partition_host_test.c`, `ipc_partition_host_test.c`, `scheduler_fairness_host_test.c`, `frame_quota_host_test.c`) to confirm no regression from the new `uses_rowstore` field or `persist.c`'s new restore block — one genuine breakage found and fixed: `persist_partition_host_test.c` didn't define `table_headers[]`/`rowstore_next_free_page_id`, which `persist.c`'s new block 6 now references unconditionally, so linking failed until two dummy declarations (matching the same pattern that test already used for the other four subsystems it doesn't exercise) were added. All five pre-existing suites pass in full after the fix.

## 4. Phase 17 — B-tree indexing

**Why this can't wait for the SQL surface.** `index_mgr.c` today is a single flat array (`INDEX_ENTRIES_MAX = 32`, explicitly sized to match the old `RECORD_MAX_FIELDS` ceiling) maintained by O(n) shifts on every insert/delete, with binary-search lookup. That's adequate for 32 entries; it is not adequate for a row-set table with real row counts once Phase 16 removes the 32-row wall. A query planner (Phase 19) choosing between "scan" and "use an index" only makes sense if the index itself doesn't degrade to another O(n) scan under the hood.

**Scope.**
- A real B-tree or B+-tree, keyed on a single indexed column's value, storing `row_id` (Phase 16) rather than a field-array slot.
- Single-column indexes only for this first cut — composite/multi-column indexes are explicitly deferred, matching this roadmap's Phase 20 precedent of "two tables, not N" and the LPAR roadmap's own repeated "first cut, not the general case" discipline.
- Range scan support (needed by Phase 18's `<`/`>`/`<=`/`>=` operators and by ORDER BY in Phase 19), not just exact-match lookup.
- Every existing `index_mgr.c` caller (`constraint.c`'s REFERENCE checks, `cursor.c`'s `order_index`, `mqt.c`) audited and updated to the new index shape or explicitly left on the old array-index path for non-row-set tables.

**Verification plan.** Real execution is very plausible here — a B-tree's insert/delete/range-scan logic is exactly the kind of self-contained, low-dependency data structure this project has host-tested successfully before (`partition.c` in Phase 8, `frame_pool.c` in Phase 13). Plan for a dedicated host test exercising insert/delete/range-scan against a real, unmodified build of the new index code, not a reimplementation.

**Findings addendum (as built).** Landed as a *new*, parallel capability rather than a migration of `index_mgr.c` — the audit below confirms that split is safe, not just convenient.

**Design decisions made during implementation, not fully pinned down by the scope bullets above:**

- **Duplicate values, and why leaf entries are keyed by distinct value, not one array slot per row.** The scope bullets didn't specify how the tree handles multiple rows sharing one indexed value. The first design tried — one leaf array slot per row, duplicates as plain repeated keys — turned out to have a real correctness hazard: a run of equal keys can straddle a leaf split boundary, and a lookup that lands on only one side of that boundary would silently miss entries on the other side. Rather than add split-time logic to avoid splitting mid-run (itself only a partial fix — an arbitrarily long duplicate run can still force it), the tree instead consolidates each *distinct* key into one leaf slot holding a small fixed-capacity array of row_ids (`BTREE_MAX_DUPES_PER_KEY = 16`). This makes "does this key already live in a leaf" and "which leaf" both unambiguous — a plain descent, never a multi-leaf check — at the cost of a bounded per-key duplicate cap. Hitting the cap fails loud and narrow (the 17th duplicate for one key is dropped from that index, logged, while the row itself is untouched and every *other* index and the table itself are unaffected) rather than the alternative design's silent, hard-to-notice missed-lookup failure mode. This is the same "denial looks like absence" posture named in `row_index.h`'s header comment, applied to a data-structure choice rather than an access-control one. Documented consequence: low-cardinality columns (booleans, small enums) are a poor indexing candidate under this cap once row counts grow past it — true of most real query planners too, which generally skip indexes on low-selectivity columns regardless.
- **Key encoding.** Not specified in the scope bullets at all. Landed on a canonical, fixed-width (`BTREE_MAX_KEY_BYTES = 64`), memcmp-comparable byte encoding per `SLSFieldType`: `UINT64` as 8-byte big-endian (memcmp already gives correct unsigned order); `FLOAT` as the standard IEEE-754-to-memcmp-orderable transform (flip all bits if negative, set the sign bit if non-negative — verified explicitly across the negative/positive boundary in the host test, not just asserted); `BOOL` as 1 byte; `STRING` as up to 64 bytes, zero-padded (zero-padding at the end preserves correct lexicographic order for text with no embedded NULs before the logical end — also verified explicitly, with a deliberately chosen name set where lexicographic and numeric order diverge, `"emp19" < "emp2"`).
- **Automatic maintenance, not an opt-in step.** The scope bullets named indexes and callers but not *when* an index gets updated. Rather than require every future caller to remember to call an update hook after a row mutation (an easy-to-forget step, and exactly the failure mode a "the mutation function itself calls the hook" design avoids), `kernel/rowstore.c`'s `rowstore_row_insert/update/delete` were extended to call `row_index_notify_insert/update/delete` directly after a mutation commits — the same shape the *legacy* `index_mgr.c` hooks already use (`object_catalog.c`'s `sys_sls_insert/update/delete` call `index_on_insert/on_update/on_delete` directly), not a new pattern. `rowstore_row_update`/`_delete` needed a small addition to support this: both now deserialize the row's *pre-mutation* values before overwriting/tombstoning the slot, since the old indexed key is otherwise unrecoverable once the slot is overwritten — `row_index_notify_update` needs both old and new values to remove the stale key and insert the fresh one; `row_index_notify_delete` needs the old values to know which key to remove the row from at all.
- **Persistence: explicitly out of scope, not an oversight.** Indexes are a derived, in-RAM structure over already-persisted row data (`rowstore.c` persists the rows). `row_index_create()` always rebuilds correctly and cheaply from a real table scan, so there's nothing today forcing indexes to survive a restart — a future phase can add it if rebuild cost ever becomes a real problem, not assumed here.

**Audit of existing `index_mgr.c` callers (§4's scope bullet), confirmed by direct search, not assumed:** `object_catalog.c` (the hook call sites themselves), `constraint.c` (REFERENCE checks), `cursor.c` (`order_index`), `mqt.c`/`aggregate.c`, `net/http.c`, and `user/shell.c` all reference `index_mgr.h`/`index_lookup`/`index_range_scan`/`index_on_*`/`order_index` — and none of them reference `rowstore.h` or `row_index.h` at all. They operate exclusively through `object_catalog.c`'s legacy `sys_sls_select/insert/update/delete` path, which is gated on nothing about `uses_rowstore` and never touches row-set tables. This isn't a claim that needed new code to make true — it's the same structural separation Phase 16 already established (row-set tables are opt-in via `uses_rowstore`, and every legacy call site predates that flag and never checks it), confirmed here specifically for Phase 17's new file rather than assumed to still hold. No changes were needed to any of the six files; they stay exactly on the array-index path for every table that doesn't opt into row-set storage, precisely the "explicitly left on the old array-index path for non-row-set tables" outcome the scope called for.

**What actually got built:** `kernel/row_index.h`/`.c` (new). A fixed-fanout B+-tree, `BTREE_ORDER = 4` (deliberately small — a real production fanout would be tuned to a page/cache-line size, meaningless for this first-cut, unpersisted, in-RAM structure; a small order instead makes multi-level splitting easy to force and verify with a modest number of host-test insertions, which it does: 50 distinct keys against a 3-keys-per-leaf cap forces both leaf splits and at least one round of internal-node splits, verified, not assumed). Nodes live in one shared, array-backed, index-addressed pool (`BTREE_MAX_NODES = 16384`, bump allocator, no reclaim — the same "no reclaim in first cut" posture `rowstore.c`'s page pool and `frame_pool.c`'s quota system already established, applied here to a third subsystem rather than invented fresh). `struct RowIndex` (`row_indexes[ROW_INDEX_MAX = 16]`) records `{index_name, table_object_id, column_index, column_type, root_node, entry_count}`. Insert does real recursive-up split propagation via a recorded descent path (no parent pointers stored in nodes — the standard iterative technique); leaf splits copy the promoted separator (classic B+-tree: leaves retain every key), internal-node splits remove and promote it. Delete is a tombstone within a key's duplicate-id array — no rebalancing/merging, consistent with the duplicate-cap design's own "fail narrow, not silently" posture, and with every other "no reclaim in first cut" precedent in this project. Range scan descends to the leaf that would contain the lower bound (or the leftmost leaf if unbounded), then walks the leaf-to-leaf chain (`next_leaf`) collecting matches until the upper bound is exceeded or the chain ends; exact-match lookup is implemented as `range_scan(value, value)` rather than a separate code path, avoiding a second, potentially-inconsistent traversal implementation. `kernel/kernel.c` gained `row_index_init()` (RAM-only, no restore step needed — see the persistence note above), wired immediately after `rowstore_init()`.

**A real, pre-existing gap found and fixed, unrelated to the index logic itself:** while doing the compile-check sweep below, `kernel/rowstore.c` (Phase 16) turned out to have never been added to the `Makefile`'s `X86_C_SRC` list at all — meaning the real cross-compiled kernel binary would have failed to link the moment `kernel.c`'s `rowstore_init()` call was reached, despite every host test passing throughout Phase 16. Host tests link `rowstore.c` directly by filename and so never would have caught this; only a check against the actual build system would. Fixed by adding both `kernel/rowstore.c` (retroactively, Phase 16's own omission) and `kernel/row_index.c` (this phase) to `X86_C_SRC`. Worth naming explicitly: host-test-only verification, however thorough, does not substitute for checking the real build's source list — a gap in this project's own verification discipline up to this point, now closed for these two files and worth remembering for future phases.

**Verification, matching the plan's real-execution target:** `row_index_host_test.c` (repo root) links the real, unmodified `kernel/row_index.c`, `kernel/rowstore.c`, and `kernel/persist.c` together — not row_index.c in isolation — against the same fake-NVMe/call-tracked-`catalog_check_access()` harness `rowstore_host_test.c` established. 40 checks, all passing, all real execution, across a 4-column (`UINT64`/`STRING`/`BOOL`/`FLOAT`) fixture: index creation validated against every documented error path (unknown table, permission denial, unknown column, duplicate index name, `ROW_INDEX_MAX` exhaustion — filled to exactly 16 and confirmed the 17th fails cleanly); 50 distinct ids inserted through the real `rowstore_row_insert → row_index_notify_insert` path, individually looked up afterward (all 50, each returning exactly its own `row_id` — the strongest per-key routing check, since a wrong split anywhere would misroute at least one); a full unbounded range scan confirmed strictly ascending order with no gaps or duplicates across however many splits 50 inserts triggered (the strongest whole-tree check — any broken leaf-chain link or mis-promoted separator would show up here as an ordering violation, not just a wrong count); ## 5. Phase 18 — Predicate evaluation (WHERE engine)

**Why this is its own phase, not folded into Phase 19.** Filtering today is exactly one shape: `cursor.c`'s `where_field`/`where_value`, single-field string equality. A real WHERE clause needs comparison operators (`=`, `!=`, `<`, `>`, `<=`, `>=`), boolean combinators (AND/OR, at minimum), and type-aware comparison against `SLSFieldType`-typed values (comparing `"100"` and `"9"` as strings gives the wrong answer for UINT64/FLOAT columns — a real bug class this phase exists to prevent). Keeping this separate from Phase 19's parser means the predicate evaluator is independently testable against Phase 16's row store before any SQL syntax exists to drive it.

**Scope.**
- A small expression evaluator: parses/represents a predicate as a simple tree (comparison nodes, AND/OR nodes, literal nodes), evaluates it against one row given its table's schema.
- Explicitly out of scope for this phase: arithmetic expressions, string functions, subqueries, `LIKE`/pattern matching, `IN` lists — deliberately narrow, matching this roadmap's stated non-goal of a full SQL expression language.
- Used by: a table-scan-with-predicate function (walks Phase 16's row heap, evaluates the predicate per row) and, once available, an index-assisted path when the predicate's operator and column match an existing Phase 17 index.

**Verification plan.** Real execution — this is a pure function (row + schema + predicate tree → bool) with no I/O and no kernel dependencies beyond the row/schema structs themselves, an even shallower dependency graph than `frame_pool.c`. A host test table-driven over every operator, every `SLSFieldType`, and AND/OR combinations should be cheap and thorough.

**Findings addendum (as built).** Landed close to scope, with the representation and integration shape both decided during implementation (the scope bullets named *what* the evaluator needed to do, not *how* the tree is stored or *how* scanning plugs in).

**Representation decision.** Not specified by the scope bullets. Landed on a small, fixed-capacity, array-backed tree (`PREDICATE_MAX_NODES = 32` — a deliberate echo of legacy `index_mgr.c`'s own `INDEX_ENTRIES_MAX = 32`, not a coincidence, same "small first cut" sizing instinct) rather than a pointer-based AST with heap allocation — consistent with every other kernel subsystem this roadmap has built (`row_index.c`'s node pool, `rowstore.c`'s page pool), not a new pattern invented for this phase. Two node kinds: a comparison leaf (`column_name`, operator, literal text) and an AND/OR combinator (two child node indices). Since Phase 19's parser doesn't exist yet, callers assemble a `Predicate` directly via `predicate_add_comparison()`/`_add_and()`/`_add_or()`, threading the returned indices together and setting `Predicate.root` themselves — the same "caller assembles it from small primitives, no invented DSL yet" shape `row_index.c`'s own builder functions already established for indexes with no SQL to drive them either.

**Type-aware comparison, not string comparison.** The whole reason this phase exists (per its own "why" — comparing `"100"` and `"9"` as raw text gives the wrong answer for a UINT64 column) was verified explicitly, not just implemented and assumed correct: the host test's UINT64 checks include `25 > 9` evaluating true (numeric), which would evaluate false under a naive string compare (`"25" > "9"` lexically is false, since `'2' < '9'`) — the exact bug class this phase is named to prevent, caught in the test itself rather than left as a comment claiming it's handled.

**Fail-closed, not fail-open.** A comparison against a column name absent from the table's layout, or a literal that fails to parse against that column's `SLSFieldType`, evaluates to `false` for that comparison — never a crash, and never treated as "unknown, so let it through." A malformed tree (a `root` index pointing past the node pool's populated range — reachable today only by a caller manually corrupting `Predicate.root`, since every node-adding function validates its own indices, but worth guarding since `Predicate` is a plain public struct, not an opaque handle) also fails closed rather than reading out of bounds. Both are exercised directly in the host test, not just asserted in the header comment.

**`predicate_table_scan()` — a thin wrapper, not a new gate.** Built as a small trampoline over the existing `rowstore_table_scan()` (Phase 16): it looks up the table's layout, then re-invokes `rowstore_table_scan()` with an internal callback that evaluates the predicate per row and only forwards matches to the caller's own callback. This reuses `rowstore_table_scan()`'s own `catalog_check_access()` gate completely unmodified — no second permission check was added, and the host test confirms both that access denial still propagates through the wrapper and that a `NULL` predicate visits every row exactly like calling `rowstore_table_scan()` directly would (the wrapper adds filtering, not a new code path with new failure modes). `predicate.c` has no dependency on `row_index.c` at all and needed no changes to `rowstore.c` — unlike Phase 17, this phase's integration point was a pure addition, not a modification of Phase 16's own functions.

**Deliberately deferred, per scope:** the index-assisted scan path (using a Phase 17 B-tree index when a predicate's column/operator match one) was not built this phase. There is no query planner yet to decide table-scan vs. index-scan — that choice is Phase 19's job once a planner exists to make it — so building the integration now would be speculative plumbing with nothing to drive it, the same posture this project took toward Phase 15's nested-LPAR question and Phase 22's cost-based optimizer.

**Verification, matching the plan's real-execution target and then some:** `predicate_host_test.c` (repo root) links the real, unmodified `kernel/predicate.c` for a pure-function half (zero rowstore/persist dependency, exactly as light as the plan predicted) and additionally the real `kernel/rowstore.c` + `kernel/persist.c` for a second half exercising `predicate_table_scan()` against an actual rowstore-backed table — a heavier bar than the plan asked for, matching this roadmap's running pattern of verification meeting or exceeding its own stated ceiling. 51 checks, all passing: all six operators against all four `SLSFieldType`s (24 base combinations, plus the numeric-vs-lexical UINT64 check above and a signed-FLOAT-comparison check); AND/OR including a nested `AND(OR(...))` tree; every fail-closed edge case named above, plus builder-level bounds checks (`predicate_add_and` rejecting an out-of-range child index, the node pool filling to exactly `PREDICATE_MAX_NODES` and failing cleanly on the next call); and the `predicate_table_scan()` half against a real 20-row table (`active=true` matching exactly the 7 rows it should, unknown-table and permission-denial paths returning 0 cleanly, and direct confirmation via the call-tracked `catalog_check_access()` stub that no duplicate gate was added).

Full compile-check sweep: `predicate.c` syntax-checks cleanly in isolation. Added `kernel/predicate.c` to the `Makefile`'s `X86_C_SRC` proactively (not retroactively, this time — the Phase 17 findings named host-test-only verification as insufficient without a build-system check, and this phase closes that same gap before it could recur, even though nothing in the real kernel calls into `predicate.c` yet). Re-ran all seven pre-existing host test suites (`partition_host_test.c`, `persist_partition_host_test.c`, `ipc_partition_host_test.c`, `scheduler_fairness_host_test.c`, `frame_quota_host_test.c`, `rowstore_host_test.c`, `row_index_host_test.c`) — all pass, zero regressions, and none needed a stub fix this time (unlike Phase 17), since `predicate.c` adds no new calls into `rowstore.c` or `row_index.c` at all. No `kernel.c` boot wiring was needed either — `predicate.c` carries no global state to initialize, unlike every prior phase's subsystem.

## 6. Phase 19 — Minimal SQL surface

**Why the scaffolding is already halfway there.** `cursor.c`'s API (`cursor_open`/`cursor_fetch`/`cursor_close`, explicitly modeled on `DECLARE CURSOR`/`OPEN`/`FETCH`/`CLOSE`) is already SQL-cursor-shaped at the interface level — it just sits on top of the old single-field-equality filter and the old single-record table model. This phase's job is to give that shape a real query language and real row sets underneath it.

**Scope.**
- A hand-written recursive-descent parser (matching this project's established style — the TIMI assembler is hand-rolled, not generated) for a genuinely minimal SQL subset: `SELECT <cols> FROM <table> [WHERE <predicate>] [ORDER BY <col>] [LIMIT <n>]`, and `INSERT`/`UPDATE`/`DELETE ... [WHERE <predicate>]`.
- A trivial planner: table scan vs. single-index scan (first applicable index wins — no cost model, that's Phase 22). Executes via Phase 16's row store, Phase 17's indexes, Phase 18's predicate evaluator.
- `cursor.c` becomes (or gains a mode that is) the execution/iteration layer under `SELECT`, reusing its existing pagination model rather than inventing a second one.
- Explicitly out of scope: joins (Phase 20), subqueries, views, aggregate functions beyond what `mqt.c` already does, `GROUP BY` in the general SQL sense (MQTs remain the aggregate story for now).

**Verification plan.** Parser correctness is real-execution-testable in isolation (feed strings, check the parse tree) with zero kernel dependencies, the same way the TIMI assembler was verified in Phase 1. End-to-end execution against Phase 16's row store is a heavier dependency graph — likely compile-check plus logic review for the full pipeline, real execution for the parser alone, an honest split matching this roadmap's stated verification discipline.

**Findings addendum (as built).** Landed as four new files (`sql_parser.h/.c`, `sql_exec.h/.c`) plus a real, if modest, extension to `cursor.c` — and verification *beat* its own stated plan: both halves got real execution, not the compile-check-plus-logic-review ceiling the plan allowed for the executor half.

**Grammar and representation.** A hand-written lexer + recursive-descent parser (`sql_parser.c`), matching this project's TIMI-assembler-style "hand-rolled, not generated" convention. WHERE clauses parse directly into a `struct Predicate` (Phase 18) via its own `predicate_add_comparison()`/`_add_and()`/`_add_or()` builders — no second expression-tree type invented. AND binds tighter than OR for free from the grammar's own nesting (`predicate := and_expr (OR and_expr)*`), giving correct standard SQL precedence without needing user-supplied parentheses; explicit parenthesized grouping beyond that natural precedence was the one deliberate grammar simplification made during implementation, narrower than originally scoped but consistent with the roadmap's own "no arithmetic expressions" non-goal. String literals support the standard `''`-escaped-quote convention. Keywords are case-insensitive, matching standard SQL and verified explicitly (a lowercase `select ... from ... where` statement is one of the host test's own scenarios).

**Planner.** `sql_find_matching_rows()` (private to `sql_exec.c`) implements the scoped "first applicable index wins" rule: only a WHERE clause whose ROOT is a single, non-`!=` comparison against an indexed column uses `row_index_lookup()`/`row_index_range_scan()` (Phase 17); everything else — AND/OR-wrapped predicates, comparisons against unindexed columns, `!=`, or no WHERE at all — falls back to `predicate_table_scan()` (Phase 18). One design decision not pinned down by the scope bullets: the index path is never trusted as the sole correctness gate. Every candidate row_id the index returns is re-fetched and re-checked against the FULL predicate via `predicate_eval()` before being accepted into the result. This was chosen specifically because it sidesteps a subtle correctness question — whether `row_index_range_scan()`'s inclusive-bounds contract exactly matches a strict `<`/`>` operator's semantics — by making the answer never matter: the recheck enforces the real semantics regardless of what the index scan over-shoots. A new one-function addition to Phase 17, `row_index_find_for_column()`, gave the planner a clean way to ask "does an index exist for this table+column" without reaching into `row_indexes[]` directly from a different file.

**`cursor.c`'s row-set mode — a real extension, not a rewrite.** Per the roadmap's own "cursor.c becomes (or gains a mode that is)" framing: `struct SLSCursor` gained four zero-default fields (`is_rowset`, `result_rows[CURSOR_MAX_ROWSET_ROWS]`, `result_count`, `result_pos`) and two new functions, `cursor_open_rowset()`/`cursor_fetch_rows()`, reusing the exact opaque-id/position/done pagination shape `cursor_open()`/`cursor_fetch()` already established — but touching none of the legacy path's fields, JSON output, or field-iteration logic. A real bug was caught and fixed during this extension, not just anticipated: `cursor_open()` stamps every field it cares about onto a freshly-claimed slot except the ones it doesn't touch at all, and a slot previously used in row-set mode would leave a stale `is_rowset = 1` behind for the next (legacy) claimant of that slot — the same "slot reuse leaves stale fields" class of bug Phase 8's `partition_id` initialization gap and Phase 14's frame-usage reset both taught this project to watch for. Fixed by having `cursor_open()` explicitly zero `is_rowset` when claiming a slot, and `cursor_open_rowset()` explicitly clear the legacy-path fields (`where_field`/`where_value`/`order_index`/`position`/`index_pos`) it doesn't use, so `cursors_to_json()`'s introspection view never shows a prior occupant's leftover state either way.

**A first-cut scope narrowing worth stating plainly:** column projection in SELECT is metadata-only. `SqlResult.columns[]`/`column_count` correctly report what was asked for (`*` vs. an explicit list), but the cursor's materialized rows always carry the table's FULL row — genuine column-level narrowing at the storage/serialization layer was not built. This was a deliberate scope cut made during implementation (not in the original scope bullets) to keep result-set materialization simple and correctly ordered/limited without also having to re-index every row into a second, differently-shaped `RowValues`; a future phase can add real projection if it ever matters for something beyond metadata.

**Static scratch buffers, not stack allocation — a real embedded-systems constraint, not a style choice.** `sql_execute()`'s `struct SqlStatement` (which embeds a `struct Predicate` — 32 nodes × a 256-byte literal each, several KB on its own) and `exec_select()`'s `struct RowValues[CURSOR_MAX_ROWSET_ROWS]` materialization buffer (~4 KiB per row × up to 256 rows, approaching 1 MiB) are both function-static globals, not stack locals — a freestanding kernel's stack is nowhere near large enough for either as a repeated local variable. This is an explicit, documented consequence of Phase 21 (real concurrency control) not existing yet: `sql_execute()` is not reentrant today, and nothing in this project's row-set execution path claims to be safe under concurrent invocation — the same gap this whole roadmap has named Phase 21 to eventually close, encountered here directly rather than abstractly.

**A correctness gap found and closed during test-writing, not left implicit:** the original implementation validated SELECT's column list and ORDER BY column against the table's schema up front, but did NOT validate WHERE-clause column names the same way for any of SELECT/UPDATE/DELETE — a WHERE clause with a typo'd column name would silently match zero rows via `predicate_eval()`'s own fail-closed-per-row behavior, rather than surfacing as an error. That's a real usability gap masquerading as "correctly failing closed": a malformed QUERY and a legitimately-empty RESULT look identical to the caller. Fixed by adding `predicate_columns_valid()`, a small recursive walk over a predicate tree confirming every referenced column exists, applied uniformly to SELECT/UPDATE/DELETE's WHERE clauses before any scan runs — now all four statement kinds are consistent about rejecting an unknown column immediately (`SQL_ERR_COLUMN_NOT_FOUND`) rather than three of them silently matching nothing.

**Verification, exceeding the plan's own stated ceiling for the executor half:** `sql_parser_host_test.c` links the real, unmodified `sql_parser.c` + `predicate.c` (the parser's only real dependency, since it calls `Predicate`'s builder functions directly) — genuinely zero rowstore/row_index/cursor dependency, the "total isolation" the plan asked for. 44 checks, all passing: the full grammar in one statement, WHERE tree SHAPE verified node-by-node (not just parse success) including AND/OR precedence (`OR(a, AND(b,c))` confirmed as the actual tree produced by `a OR b AND c`), all seven comparison-operator spellings including `<>` as a `!=` synonym, escaped-quote string literals, INSERT/UPDATE/DELETE, and seven distinct syntax-error scenarios each producing a clean rejection with a message. `sql_exec_host_test.c` links the full real stack — `sql_exec.c`, `sql_parser.c`, `predicate.c`, `row_index.c`, `rowstore.c`, `persist.c`, `cursor.c` together — real execution for the "heavier dependency graph" half the plan explicitly said might only reach compile-check-plus-logic-review. 45 checks, all passing: 20 rows inserted via real SQL INSERT statements; index-assisted exact-match AND range-scan SELECTs verified against a hand-computed expected row set; full-scan-fallback paths (AND-wrapped WHERE, an unindexed column, and `!=`, which never uses the index) all independently verified correct; ORDER BY ASC and DESC checked against the actual returned column values, not just a row count; LIMIT verified to keep the first N rows post-sort, not an arbitrary N; a real multi-call pagination test (fetch 7 at a time across 20 rows, confirming the exact 7/7/6/0 sequence and that every row arrives exactly once, in order); UPDATE and DELETE verified via a real re-SELECT afterward, not just an affected-row count; permission denial and unknown-table/column error paths for every statement kind. One genuine bug was caught by this test during authoring, not anticipated in the design: the host test initially left every SELECT's cursor open across scenarios, and by scenario 8 had silently exhausted `CURSOR_MAX`'s 8 slots — `cursor_open_rowset()` was returning 0 (correctly, per its own documented contract) but the test's `SqlResult.row_count` field (set from the executor's own row count, independent of whether the cursor actually opened) masked the failure until the subsequent `cursor_fetch_rows(0, ...)` calls returned nothing. Fixed by adding `cursor_close()` calls matching realistic cursor lifecycle usage — a real "real execution catches real bugs" instance, this time in the TEST rather than the implementation, and left in the addendum because it's exactly the kind of thing this project's whole verification discipline exists to surface.

Full compile-check sweep: `sql_parser.c`, `sql_exec.c`, `predicate.c`, `row_index.c`, `rowstore.c`, and `cursor.c` all syntax-check cleanly (one pre-existing, unrelated warning in `cursor.c`'s legacy `cursor_fetch()` — present before this phase, untouched by it). `kernel.c` grepped specifically for `sql_`/`cursor_open_rowset`/`cursor_fetch_rows`/`row_index_find_for_column` — no related errors (same targeted-grep technique Phase 17's findings established for whole-`kernel.c` checks without the freestanding cross-toolchain available). `net/http.c` and `user/shell.c` — the two files outside `cursor.c` itself that reference the cursor API — both grepped clean for cursor/sql/row_index/rowstore-related errors, confirming the new `struct SLSCursor` fields didn't disturb either legacy caller. Proactively added both `kernel/sql_parser.c` and `kernel/sql_exec.c` to the `Makefile`'s `X86_C_SRC` (continuing the practice Phase 18 started after Phase 17 found `rowstore.c` had been retroactively missing). Re-ran all nine pre-existing host test suites (`partition_host_test.c`, `persist_partition_host_test.c`, `ipc_partition_host_test.c`, `scheduler_fairness_host_test.c`, `frame_quota_host_test.c`, `rowstore_host_test.c`, `row_index_host_test.c`, `predicate_host_test.c`) — all pass, zero regressions; none needed a stub fix this time, since `cursor.c`'s extension added no new calls into any of them and `sql_parser.c`/`sql_exec.c` are purely new call sites, not modifications to existing ones' contracts.

## 7. Phase 20 — Two-table JOIN

**Why nested-loop first, and why only two tables.** The LPAR roadmap's Phase 12 precedent is directly on point: "a naive version can slot in without restructuring the core loop... a real guarantee wants a bigger change this phase should not attempt in its first cut." The same logic applies here — nested-loop join (for each row in table A, scan table B for matches) is the simplest correct join algorithm, and restricting this phase to exactly two tables with `INNER JOIN ... ON <col> = <col>` keeps the planner trivial (no join-order search problem yet, because there's only one order).

**Scope.**
- `SELECT ... FROM A JOIN B ON A.col = B.col [WHERE ...]` added to Phase 19's parser and planner.
- Plain nested-loop join as the baseline; when Phase 17 has an index on B's join column, use it (indexed nested-loop join) — a natural, small addition once both pieces exist, not deferred to Phase 22.
- Explicitly out of scope: three-or-more-table joins, OUTER JOIN (LEFT/RIGHT/FULL), join reordering, hash join, merge join. "Groundwork, not a query optimizer" — matching this roadmap's own stated non-goals.

**Verification plan.** Compile-check plus logic review for the full pipeline (same heavy dependency graph as Phase 19's execution path), with a hand-traced worked example (two small tables, known expected output) as the concrete check, mirroring Phase 12's own "hand-traced scenario" verification ceiling before its host test proved out.

**Findings addendum (as built).** Landed exactly at the scoped grammar and algorithm, and verification beat its own stated ceiling again — real execution against the full stack, not the compile-check-plus-hand-trace bar the plan allowed for.

**The one genuinely new idea this phase adds: a qualified name is just a string.** Every other design question this phase raised had an obvious answer once this one was picked. `exec_select_join()` (`sql_exec.c`) builds a synthetic `struct RowTableLayout` at execution time whose `column_names[]` holds `"tablename.column"` for every column of both joined tables, and materializes each combined row as a plain `RowValues` with A's columns first, then B's. Because Phase 18's `predicate_eval()`/`predicate_columns_valid()` and Phase 19's `find_column_index()`/`compare_rows_by_column()` only ever do string comparison against a layout's `column_names[]`, every one of those functions works against a joined result with **zero code changes** — no join-aware branch was added to any of them. This is the same "one choke point" reuse discipline named in this roadmap's design principles, just applied to a data shape instead of a permission gate.

**"Indexed nested-loop join when available" fell out for free, not from new index-selection logic.** The scope bullet called for using a Phase 17 index on B's join column "when available." Rather than write a new decision (index-vs-scan) specific to joins, the join probe step constructs a single-comparison `Predicate` (`B.join_col = <this A-row's value>`) per outer row and calls the exact same `sql_find_matching_rows()` Phase 19's single-table planner already uses. Phase 19's planner already knows how to use an index when one exists on the compared column and fall back to a full scan otherwise — the join probe inherited that behavior directly, and this phase added no index-selection code of its own. Verified concretely, not just claimed: the host test runs the same JOIN shape once against a joined column with a Phase 17 index (`departments.id`) and once against a joined column with no index at all (`notes.emp_id`, a table created with zero indexes), and both return the correct, hand-computed row set.

**Qualifier resolution: parsing doesn't decide which side is which.** The ON clause's two `qualifier.column` halves are parsed and split (`split_qualified()`) but deliberately NOT matched against `table_name`/`join_table` at parse time — the parser has no table-existence knowledge at all (by design, per `sql_parser.h`'s own "zero dependency on rowstore.h/row_index.h" discipline). `exec_select_join()` resolves `a_is_left` by checking both orderings (`join_left_qualifier` matches `table_name` and `join_right_qualifier` matches `join_table`, OR the reverse) once real table names are in hand — `SQL_ERR_JOIN_INVALID` if neither ordering matches. Verified in both directions: the host test runs the identical JOIN once with the ON clause written `employees.dept_id = departments.id` and once reversed (`departments.id = employees.dept_id`), confirming qualifier resolution isn't silently assuming a fixed position.

**No predicate pushdown — a named non-goal, not an oversight.** WHERE filtering happens entirely after the join, evaluated against the combined row. Decomposing a predicate by which table's columns it references, to filter each side's scan before joining, is real query-optimizer territory — explicitly out of scope per this roadmap's own Phase 22 framing, and not attempted here even though disallowing `!=`-style early filtering costs nothing extra to add now. The nested-loop join's outer table (`FROM`'s table) always drives the loop; there is no cost-based or size-based side selection (also Phase 22 territory).

**Combined-row width cap: reusing `RowValues`'s existing capacity, not inventing a wider type.** A joined row is still a plain `RowValues` (`ROWSTORE_MAX_COLUMNS = 16` total). `exec_select_join()` checks `layout_a->column_count + layout_b->column_count` up front and returns the new `SQL_ERR_JOIN_TOO_WIDE` before touching any row data if the combined width would exceed that cap — verified directly with two real 9-column tables (18 combined), not just a boundary-value comment.

**A minor, accepted trade-off, not a bug:** rewriting `parse_comparison()` to route through the new generic `parse_column_ref()` helper (needed so a `column_ref` can optionally be qualified everywhere one appears — SELECT list, WHERE, ORDER BY, not just the ON clause) lost the more specific original parse-error message "expected a column name in WHERE clause" in favor of `parse_column_ref()`'s generic "expected a column name." Accepted as a non-blocking cost of code reuse rather than maintaining two near-identical column-parsing functions.

**Verification, exceeding the plan's own stated ceiling:** `sql_join_host_test.c` (repo root) links the full real stack — `sql_exec.c`, `sql_parser.c`, `predicate.c`, `row_index.c`, `rowstore.c`, `persist.c`, `cursor.c` — real execution for the "same heavy dependency graph as Phase 19's execution path" the plan predicted might only reach compile-check-plus-hand-trace. 25 checks, all passing, against a fixture deliberately including a dangling foreign key (`employees` row "dave," `dept_id = 99`, matching no department): basic indexed-nested-loop JOIN with a qualified SELECT list and `ORDER BY` on a qualified column, confirming dave is correctly excluded (real INNER JOIN semantics, not an accidental row count match) and that each returned row's two halves genuinely correspond to each other, not just to the right counts; the ON clause reversed (`B.col = A.col`), proving qualifier-resolution isn't order-dependent; JOIN against a column with no index at all, proving the full-scan probe fallback path; post-join `WHERE` filtering on a qualified column from the non-driving table; `ORDER BY ... DESC` combined with `LIMIT`, confirming LIMIT applies after sorting, not before (a distinction the test makes concrete: the two rows kept under `DESC LIMIT 2` are specifically the *last* two in ascending order, not an arbitrary two); five distinct error paths (unknown joined table, ON clause qualifier naming neither joined table, ON clause column missing from its own table, an unknown qualified column in WHERE, an unknown qualified column in the SELECT list); the `SQL_ERR_JOIN_TOO_WIDE` boundary with two real 9-column tables; and permission denial propagating cleanly through the reused `catalog_check_access()` gate inside the join probe (0 rows back, no crash), confirming no second, join-specific permission gate was accidentally introduced. One real bug was caught during test-writing, not anticipated: the test's row-collection helper initially assumed a joined row's materialized column order matched the SELECT list (e.g., `employees.name, departments.title` landing at indices 0,1) — but per Phase 19's own "column projection is metadata-only" decision (inherited unchanged by JOIN), a materialized row always carries the FULL combined row in table-definition order regardless of what was SELECTed, so `employees.name` actually lands at index 1 (after `employees.id`) and `departments.title` at index 4. Three assertions initially failed against correct executor output because of this wrong assumption in the TEST; fixed by having the collector take explicit combined-row indices per scenario instead of assuming column 0/1 — the same "real execution catches real bugs, sometimes in the test" pattern Phase 19's own findings named.

Full compile-check sweep: `sql_parser.c` and `sql_exec.c` both re-verified clean under `gcc -fsyntax-only -Wall -Wextra -std=c11` after the JOIN changes. `kernel.c`, `net/http.c`, and `user/shell.c` grepped for `sql_exec`/`sql_parser`/`has_join`/`exec_select_join` — no references at all in any of the three, confirming (as Phase 19's findings already established) the SQL layer isn't wired into kernel dispatch yet, so this phase's parser/executor changes have zero blast radius outside the two files themselves. No Makefile change was needed — `sql_parser.c`/`sql_exec.c` were already added to `X86_C_SRC` in Phase 19. Re-ran all ten pre-existing host test suites (`partition_host_test.c`, `persist_partition_host_test.c`, `ipc_partition_host_test.c`, `scheduler_fairness_host_test.c`, `frame_quota_host_test.c`, `rowstore_host_test.c`, `row_index_host_test.c`, `predicate_host_test.c`, `sql_parser_host_test.c`, `sql_exec_host_test.c`) — all pass, zero regressions; none needed a stub fix, since this phase only modified `sql_parser.c`/`sql_exec.c` (both already-isolated call sites from Phase 19) and touched no other file.

## 8. Phase 21 — Real concurrency control

**Why this can't be folded into Phase 16 or deferred past Phase 20.** `lock_mgr.c` today is fail-fast and single-shot: acquiring a lock another transaction holds fails immediately — no blocking, no wait queue, no deadlock detection, and `LOCK_SHARED` is defined but unused ("reserved for future cursor support," per the existing code comment). That's workable for today's point-operation workload. It will not hold up once Phase 20 lets one transaction touch rows across two tables via a join, or once any real concurrent workload does more than trivial single-key writes — every added lock held across a longer operation multiplies the chance of a fail-fast collision into something closer to "nothing commits under contention." Locks are also keyed on `(object_id, key)` today, which doesn't correspond to anything in Phase 16's row model at all — this phase has to decide what a "row lock" even means before it can build one.

**This phase needs its own design fork, decided at the start, not in this document** — the same treatment Phase 13 gave the accounted/unaccounted frame-allocator split:
- **Option A: upgrade to blocking 2PL with deadlock detection.** Closer to what exists today (still `lock_mgr.c`, still explicit locks), needs a wait-for graph and either timeout-based or graph-cycle-based deadlock detection plus victim selection (abort one transaction to break the cycle). Readers still block writers and vice versa.
- **Option B: MVCC (row versioning + snapshot reads).** Bigger structural change — every row needs a version chain or at least a "as-of" visibility check — but removes reader/writer blocking entirely, which is arguably the more natural fit for an SLS OS's "atomic durability without a WAL bottleneck" framing the original design discussion raised. Would also let Phase 19/20's read-only `SELECT`s never block on concurrent writers, a real usability win.

This roadmap does not pick between them — that's the dedicated design pass's job, informed by whichever concrete workload motivates building this phase at all (matching §1 principle 5).

**Scope (either option).**
- Real per-row lock/version granularity, replacing today's `(object_id, key)` keying with something that maps onto Phase 16's `row_id`.
- Whichever mechanism is chosen, it must not silently break the existing WAL/journal commit-and-recovery path (`journal.c`) that Phases 1–15 already depend on and have verified — this is an extension of the transaction model, not a replacement of it.

**Verification plan.** Whichever option is chosen, deadlock scenarios (Option A) or version-visibility scenarios (Option B) are exactly the kind of concurrent-logic bug this project's own history warns about (Phase 12's starvation bug was caught by execution, not review) — plan for a dedicated host test simulating concurrent transactions against the real lock/version code before considering this phase done, not compile-check-only.

**Findings addendum (as built).** The design fork resolved to MVCC, decided by investigation before any code was written, not preference — and real execution caught a genuine, subtle correctness bug in the visibility logic that a review alone would have been unlikely to catch.

**Why MVCC, not blocking 2PL — two concrete facts, not a preference.** Investigation turned up two decisive findings, both from direct inspection rather than assumption: (1) `lock_mgr.h`'s own existing header comment already states "no blocking / waiting in a bare-metal kernel" as a deliberate, pre-existing architectural stance — blocking 2PL would need a transaction to genuinely suspend on a wait queue and be woken later, real scheduler integration nothing else in this codebase has ever needed, plus a wait-for graph and victim selection for deadlock detection, a substantially larger and riskier lift than this roadmap's own "groundwork, not a full RDBMS" framing invites. (2) Direct inspection of `kernel/rowstore.c` (not assumed) found it calls neither `lock_mgr.c` nor `transaction.c`/WAL staging at all — the only existing concurrency protection in this codebase (`lock_mgr.c`'s fail-fast locking, `transaction.c`'s WAL) belongs entirely to the legacy `sys_sls_select/insert/update/delete` KV-record path, a separate track Phase 17 already established stays untouched by the row-set/SQL work. This phase is therefore not "replacing `(object_id, key)` locking" — it's building the FIRST concurrency protection the row-set/SQL path (Phases 16–20) has ever had, free to be a genuinely new, independent subsystem rather than needing to reconcile with `lock_mgr.c`/`transaction.c` at all. The roadmap's own "must not silently break the existing WAL/journal path" scope bullet is satisfied trivially, by construction: `mvcc.c` calls neither file.

**Layered on top of unmodified `rowstore.c` — the one genuinely new idea this phase adds.** An UPDATE never calls `rowstore_row_update()` (which writes in place, destroying bytes a concurrent older-snapshot reader might still need); it calls `rowstore_row_insert()` to create a brand-new physical row for the new version, and marks the old version's metadata as superseded. `rowstore.c` needed zero modification — every version's bytes live in an ordinary Phase 16 physical row; `mvcc.c` is a pure metadata layer (`struct MvccVersion`: `xmin`/`xmax` commit-sequence numbers, `xmin_txn`/`xmax_txn`, and a `struct RowId` pointing at the real bytes) tracking which physical row is correct for a given transaction's snapshot. This is the same "new parallel capability, not a migration" shape Phase 17's `row_index.c` took relative to legacy `index_mgr.c`, applied here to concurrency control instead of indexing.

**First-updater-wins, detected eagerly — not classic first-committer-wins.** Rather than deferring conflict validation to commit time (a second pass over everything the transaction touched), `mvcc_row_update()`/`mvcc_row_delete()` check the visible version's own `xmax_txn` field directly: if another transaction has already started superseding this exact version (whether that transaction has committed yet or not), the attempt is rejected immediately with `MVCC_ERR_WRITE_CONFLICT` — one field read, no second pass. A deliberate, named trade-off: this can reject a transaction slightly more eagerly than strict first-committer-wins would (a transaction blocked by a sibling that later rolls back could, under the classic deferred rule, have been allowed through) — accepted because it is simpler to implement correctly and, concretely, easier to verify by host test, which mattered more for a first cut. Verified in every direction the design implies: a conflict fires against an in-flight blocker, clears once that blocker rolls back, and — the least obvious case — still fires against a transaction's now-stale snapshot even after the blocker has already *committed*, proving the check is real MVCC lost-update prevention and not merely "is anyone currently active on this row."

**A real bug caught by execution, not review: an off-by-one in snapshot sequencing.** The first implementation set a new transaction's `snapshot_seq` directly to the current value of the global commit-sequence counter — "everything committed as of right now." That's wrong by exactly one: the counter holds the NEXT sequence number to be handed out, not the last one actually used. A transaction beginning immediately before a concurrent commit would be assigned the same number that commit was about to consume, making the newly-committed version incorrectly visible to a snapshot that had, in real time, started before it — a genuine repeatable-read violation. Three of the host test's snapshot-isolation checks failed the first time the test ran (a transaction that should NOT have seen a concurrent commit did), pointing directly at the bug; fixed by taking `snapshot_seq = mvcc_commit_seq - 1` at `mvcc_begin()` instead. This is exactly the kind of concurrent-logic bug the roadmap's own verification plan predicted ("exactly the kind of concurrent-logic bug this project's own history warns about... caught by execution, not review") — found precisely because the plan's own bar (a dedicated host test simulating concurrent/interleaved transactions, not compile-check-only) was met rather than skipped.

**One deliberate second permission gate, named rather than hidden.** Every other mutating call in `mvcc.c` reuses `rowstore.c`'s own `catalog_check_access()` gate by making a real `rowstore_row_insert/get` call underneath — the same "one choke point" reuse this whole roadmap has practiced since Phase 18. `mvcc_row_delete()` is the one exception: because a delete is a pure metadata tombstone with no `rowstore.c` call to piggyback on (calling `rowstore_row_delete()` at delete-time would physically destroy bytes a concurrent older-snapshot reader may still legitimately need), it calls `catalog_check_access(caller_uid, table_name, PERM_WRITE)` directly — a second, genuine gate, not a copy-paste duplicate, and verified explicitly as its own case in the host test rather than assumed to be covered by the insert/update paths' checks.

**Rollback's own permission question, caught during implementation, not after.** `mvcc_rollback()` needs to physically delete a rolled-back INSERT's now-garbage row via `rowstore_row_delete()` — but the first draft of its signature took no `caller_uid` at all, meaning that internal cleanup call had no legitimate identity to gate itself with. Fixed before ever compiling by adding `caller_uid` to `mvcc_rollback()`'s signature; documented explicitly that if the supplied uid lacks `PERM_WRITE` on the affected table, the transaction's version metadata still unwinds correctly (the logical rollback is complete either way) but that one physical row is leaked rather than reclaimed — a storage leak, not a correctness gap, and consistent with this whole roadmap's standing "no reclaim in first cut" posture elsewhere (`rowstore.c`'s page pool, `row_index.c`'s node pool).

**Asymmetric garbage handling: committed history is kept, aborted attempts are not.** Matching the "no reclaim in first cut" posture named above, a version superseded by a COMMITTED transaction keeps its physical row forever (genuine, if currently unexploited, point-in-time history — no time-travel query API was built to read it). A version created by a transaction that ROLLS BACK is real garbage with zero value and is reclaimed immediately via `rowstore_row_delete()` — verified directly in the host test by an independent `rowstore_table_scan()` call (bypassing `mvcc.c` entirely) confirming the physical row count drops by exactly one after a rollback, not just that the logical row becomes invisible.

**Explicitly out of scope, named rather than silently skipped:** wiring `sql_exec.c`'s SELECT/INSERT/UPDATE/DELETE/JOIN (Phases 19–20) to run through `mvcc.c` instead of calling `rowstore.c` directly. That's real, valuable follow-on work — giving every SQL statement a snapshot, routing cursor iteration through a stable view — but a separate, large lift (touching every statement kind's execution path) deserving its own scope, matching this roadmap's own precedent of naming integration debt explicitly (Phase 18 deferred index-assisted scan integration for the identical reason: no consumer yet exists to drive it, so building the integration now would be speculative plumbing). Persistence (surviving a restart) is also out of scope, for the same reason Phase 17 gave for not persisting B-tree indexes — nothing forces it yet, and there's no in-flight-transaction-recovery story to build alongside it regardless.

**Verification, meeting the plan's own stated bar exactly — real execution of simulated concurrent transactions, not compile-check-only.** `mvcc_host_test.c` (repo root) links the real, unmodified `kernel/mvcc.c`, `kernel/rowstore.c`, and `kernel/persist.c` together. 45 checks, all passing, across fourteen scenarios: read-your-own-uncommitted-writes; snapshot isolation proven in both directions (an older transaction never sees a sibling's later commit; a fresh transaction begun after a commit does); repeatable read across a concurrent UPDATE (a transaction keeps seeing the pre-update value throughout its own lifetime even after another transaction commits a change); write-write conflict detection against both an in-flight blocker and a since-committed one; conflict recovery after the blocker rolls back; DELETE participating in the same conflict rule as UPDATE; read-your-own-delete; two transactions touching disjoint rows never conflicting; snapshot-consistent `mvcc_table_scan()` (a sibling's uncommitted insert invisible to others, visible to itself); the "exactly one version ever visible" invariant checked directly via a full scan after multiple updates, not just asserted from the design; independently-verified physical cleanup after rolling back an INSERT; correct reversion and continued usability after rolling back an UPDATE; and a full sweep of error paths (transaction-table exhaustion, unknown table, bogus logical ids, an already-committed/rolled-back txn_id reused, and permission denial through both the shared rowstore.c gate and delete's own direct gate). The off-by-one bug above was found during this test's first run, not anticipated in the design — the strongest possible confirmation that this phase met the verification plan's own bar rather than settling for less.

Full compile-check sweep: `mvcc.c` syntax-checks cleanly in isolation under `gcc -fsyntax-only -Wall -Wextra -std=c11`. `kernel.c` grepped specifically for `mvcc`-related errors — none found (same targeted-grep technique established since Phase 17 for whole-`kernel.c` checks without the freestanding cross-toolchain available). `net/http.c` and `user/shell.c` grepped for any `mvcc` reference — none, confirming (as named above) this phase isn't wired into SQL execution or kernel dispatch yet, so it has zero blast radius outside its own new files. `kernel/mvcc.c` added to the `Makefile`'s `X86_C_SRC`, continuing the practice every phase since 18 has followed. `kernel.c` boot wiring: `mvcc_init()` (RAM-only, no restore step) added alongside `rowstore_init()`/`row_index_init()`, matching this codebase's established "relational-layer subsystems init together" grouping. Re-ran all eleven pre-existing host test suites (`partition_host_test.c`, `persist_partition_host_test.c`, `ipc_partition_host_test.c`, `scheduler_fairness_host_test.c`, `frame_quota_host_test.c`, `rowstore_host_test.c`, `row_index_host_test.c`, `predicate_host_test.c`, `sql_parser_host_test.c`, `sql_exec_host_test.c`, `sql_join_host_test.c`) — all pass, zero regressions; none needed a stub fix, since this phase only added new files and touched no existing one besides `kernel.c`'s boot sequence and the `Makefile`.

## 9. Phase 22 — Make SQL live

**Why this is really two gaps that have to close together, not one.** A direct audit (grep, not assumption) found `sql_execute()` — the whole of Phases 19–21's parser/planner/executor — is called nowhere outside its own host tests: not `kernel/syscall_dispatch.c`, not `kernel.c`, not `net/http.c`, not `user/shell.c`. There is no `SYS_SLS_SQL_*` syscall number and no dispatch case. Three phases of real, tested SQL sit completely inert from the running kernel's point of view. Separately, `sql_exec.c` has zero reference to `mvcc.c` (also grep-confirmed) — every SELECT/INSERT/UPDATE/DELETE/JOIN executes as an implicit, unprotected, single-statement operation straight against `rowstore.c`, with none of Phase 21's snapshot isolation or conflict detection actually in effect. These two gaps have to close in the same phase: exposing a live syscall surface with zero concurrency protection behind it would be worse than leaving it unreachable, since real concurrent callers could then genuinely corrupt each other's reads with no safety net — precisely the failure mode Phase 21 exists to prevent.

**Scope.**
- Rework `sql_exec.c`'s `exec_select/insert/update/delete` and `exec_select_join` to route through `mvcc_row_insert/get/update/delete`/`mvcc_table_scan` instead of calling `rowstore_row_*`/`rowstore_table_scan`/`predicate_table_scan` directly. `predicate_eval()`/`compare_rows_by_column()` etc. stay exactly as they are (Phase 18/19 machinery is storage-layer-agnostic already, per Phase 20's own "qualified name is just a string" finding) — only the row-fetching/row-writing plumbing changes.
- A transaction model for SQL callers: the default should be autocommit (each `sql_execute()` call implicitly wraps its one statement in its own `mvcc_begin()`/`mvcc_commit()`), matching how most SQL engines default and keeping every existing single-statement host test scenario's behavior unchanged. Explicit multi-statement transactions (an SQL-level `BEGIN`/`COMMIT`/`ROLLBACK`, or a syscall-level equivalent mirroring `transaction.h`'s existing `SYS_SLS_TX_BEGIN/COMMIT/ROLLBACK` shape for the legacy path) need a design decision on where the boundary lives — per-session state didn't exist anywhere in the SQL layer before this phase, so this is genuinely new territory, not a lookup of an existing pattern.
- A syscall + dispatch case (numbers in the 220+ range are free — the highest currently assigned is `SYS_SLS_PARTITION_RESUME = 216`) wired to `sql_execute()`, matching the existing `SYS_SLS_SELECT/INSERT` dispatch shape in `syscall_dispatch.c`.
- A minimal reachability path worth having even before any real userspace SQL client exists — likely a `user/shell.c` command, mirroring however the legacy KV path is already exposed there, so this phase's own verification can demonstrate reachability concretely rather than only asserting it via a syscall number existing.
- Explicitly out of scope: prepared/parameterized statements, connection/session pooling, anything resembling a wire protocol — this is exposing the existing engine, not redesigning its interface.

**Verification plan.** Real execution for the `mvcc.c` wiring half — a host test running actual SQL statements against the real, unmodified `sql_exec.c`+`mvcc.c`+`rowstore.c` stack, proving a SQL-level write-write conflict is now detected (two interleaved SQL UPDATEs on the same row, one correctly rejected) and that autocommit boundaries are correct (a SELECT never sees another statement's partial/uncommitted effect). The syscall/dispatch half is a heavier-dependency-graph integration point without an easy host-test harness (the same ceiling every prior phase's kernel-wiring work has hit) — compile-check plus a targeted grep confirming the dispatch case routes correctly, matching this roadmap's own established fallback for whole-`kernel.c` verification.

**Findings addendum (as built).** Landed close to scope on all four bullets, plus a real API decision (an explicit multi-statement transaction surface) and two genuine bugs caught by execution rather than review — one in the implementation, one that would have been invisible without a dedicated SQL-level transaction test.

**The MVCC rewiring: one-for-one replacement, not a redesign.** `exec_select/insert/update/delete` and `exec_select_join` now call `mvcc_row_insert/get/update/delete`/`mvcc_table_scan` exactly where they used to call `rowstore_row_*`/`rowstore_table_scan`/`predicate_table_scan`, with `txn_id` threaded through every function's signature. `predicate_eval()`, `compare_rows_by_column()`, `find_column_index()`, and every other piece of Phase 18/19/20 machinery needed zero changes, confirming those functions really were storage-layer-agnostic (they only ever operate on an already-materialized `RowValues`/`RowTableLayout` pair, never on how a row got fetched) — the same property Phase 20's JOIN work already leaned on.

**A real, load-bearing scope cut: index-assisted planning does not survive MVCC routing.** `sql_find_matching_rows()` (Phase 19's index-vs-scan planner) is gone, replaced by `mvcc_find_matching_rows()`, which always performs a full `mvcc_table_scan()` plus a client-side `predicate_eval()` recheck — no code path that consults a Phase 17 B-tree index remains anywhere in the MVCC-routed executor. This was decided during design (see sql_exec.h's own header comment) once it became clear the B-tree stores physical `RowId`s with no notion of which of a row's many versions is visible to a given snapshot; teaching the index to be MVCC-aware is real, separate work, not attempted here. Verified directly, not just asserted: `sql_exec_host_test.c`'s and `sql_join_host_test.c`'s existing "indexed vs. non-indexed" scenarios both still pass unmodified — proving correctness survived even though the specific code path each scenario originally meant to exercise (index lookup vs. full scan) no longer differs at all under this phase's routing. The "indexed nested-loop join for free" property Phase 20 built and verified is therefore no longer real once a query runs through a transaction — a genuine, named performance regression, not a silent one, and a natural candidate for Phase 25 or a small dedicated follow-up.

**The transaction-boundary design decision the scope bullet flagged as genuinely new territory.** Landed on a two-layer API: `sql_execute()` keeps its exact pre-Phase-22 signature and becomes a thin autocommit wrapper (`mvcc_begin()` → run the one parsed statement → `mvcc_commit()` on success or `mvcc_rollback()` on any error) built on top of a new `sql_execute_tx(txn_id, caller_uid, sql_text, out)`, which runs one statement against an already-open, caller-managed transaction without touching its lifecycle. `sql_tx_begin()`/`sql_tx_commit()`/`sql_tx_rollback()` expose `mvcc.c`'s own transaction primitives under SQL-facing names, so a caller never needs to know `mvcc.h` exists. This gives real multi-statement transactions (proven in the host test: two statements against one open `sql_tx_begin()`-issued transaction, invisible to every other caller until an explicit `sql_tx_commit()`) without touching `sql_parser.c`'s grammar at all — no SQL-level `BEGIN`/`COMMIT`/`ROLLBACK` keywords were added, a deliberate scope narrowing (grammar changes are a bigger, separate decision) noted here rather than silently assumed away.

**Statement-level atomicity for multi-row UPDATE/DELETE — a design decision the scope bullets didn't spell out.** A write-write conflict on any ONE row a multi-row UPDATE/DELETE would touch now aborts the WHOLE statement (`affected_rows` reported as 0), rather than applying a partial update and reporting a partial count. Verified concretely, not just claimed: the host test's scenario 7 sets up three rows sharing a tag, has one transaction quietly lock one of them, then confirms a second transaction's tag-wide UPDATE not only fails on the locked row but leaves the OTHER two, non-conflicting rows completely untouched too — true statement atomicity, not just "the row that conflicted didn't change."

**A real bug found by this phase's own test, not anticipated in the design: "denial looks like absence," recurring in a new place.** `mvcc_table_scan()`'s documented contract treats "transaction not active" and "zero rows matched" identically — both return a plain `0`. Nothing in the initial `sql_exec.c` rewiring compensated for that, so a SELECT run against a bad, expired, or already-committed `txn_id` silently came back as an empty (but technically successful) result instead of a real error — two of the host test's scenario 8 checks caught this directly (`sql_execute_tx()` against a bogus `txn_id`, and reusing an already-committed one, both initially returned "0 rows" instead of `SQL_ERR_TXN_NOT_ACTIVE`). This is the exact same failure shape Phase 19's `predicate_columns_valid()` was built to close for unknown WHERE columns (a malformed QUERY silently looking like a legitimately-empty RESULT) — recognized here, fixed by adding `mvcc_txn_is_active()` (mvcc.h/.c, a one-line lookup) and checking it once, centrally, in `sql_exec.c`'s new `dispatch_stmt()` before any statement-specific logic runs, rather than scattering the check across all four `exec_*` functions.

**A second bug, caught in the test itself, not the implementation.** The host test's own row-value-fetching helper initially assumed a SELECT's materialized row put the requested column first — but per Phase 19's own "column projection is metadata-only" decision (inherited unchanged through every phase since, including this one), a materialized row always carries the FULL row in table-definition order regardless of the SELECT list. Three assertions failed against otherwise-correct executor output before this was caught and the test's column index fixed — the same class of test-authoring mistake this project's own findings have now named twice (Phase 20's findings recorded the identical trap), worth calling out again since it recurred despite being documented, a reminder that "documented" and "internalized" aren't the same thing.

**Verification, meeting the plan's own bar on both halves.** `sql_tx_host_test.c` (repo root, new) links the full real stack — `sql_exec.c`, `sql_parser.c`, `predicate.c`, `row_index.c`, `rowstore.c`, `persist.c`, `cursor.c`, `mvcc.c` — 34 checks, all passing, all real execution: autocommit correctness (a separate `sql_execute()` call sees a prior one's committed INSERT; a rejected INSERT leaves no partial row); real multi-statement transactions (two statements against one explicit transaction, invisible to others until commit, fully discarded on rollback); a genuine SQL-level write-write conflict between two concurrent explicit transactions, its recovery once the blocker rolls back, and confirmation that the losing statement's write never silently took effect; statement-level atomicity for a multi-row UPDATE under a partial conflict; and a full error-path sweep (bad/reused/exhausted transaction ids, both via the explicit API and via `sql_execute()`'s own autocommit wrapper). `sql_exec_host_test.c` and `sql_join_host_test.c` (Phases 19/20's own suites) were updated to link `kernel/mvcc.c` and call `mvcc_init()` and re-verified passing with **zero scenario changes** beyond that — the strongest available evidence that MVCC routing is a transparent, correctness-preserving substitution for a single-statement autocommit caller, not a behavior change in disguise. `mvcc_host_test.c` (Phase 21's own suite) was re-run unmodified and still passes, confirming `mvcc_txn_is_active()`'s addition didn't disturb anything Phase 21 already proved.

Full compile-check sweep: `mvcc.c`, `sql_exec.c`, and `sql_parser.c` all syntax-check cleanly in isolation. `kernel.c` grepped for `sql`/`mvcc`-related errors — none found. `net/http.c` grepped for `mvcc`/`sql_exec`/`SYS_SLS_SQL` references — none (HTTP reachability was not in this phase's scope, named explicitly rather than silently skipped). `syscall_dispatch.c` gained a `#include "sql_exec.h"` and one new dispatch case (`SYS_SLS_SQL_EXECUTE = 220`, chosen since the highest previously-assigned number was `SYS_SLS_PARTITION_RESUME = 216`), compile-checked with the same targeted-grep technique — no SQL-related errors, only pre-existing, unrelated implicit-declaration warnings from checking a freestanding file with a hosted toolchain (the same expected noise every prior phase's kernel.c-adjacent checks have produced). `user/shell.c` gained a `sql <statement>` command that calls `do_syscall(SYS_SLS_SQL_EXECUTE, ...)` — genuinely exercising the new dispatch path, not just the engine directly — plus a row-printing callback for SELECT results; same clean-except-pre-existing-noise compile-check result. ## 10. Phase 23 — Row-set integrity: constraints and journaling

**Why this is a real, confirmed gap, not a hypothetical one.** A direct audit (grep, not assumption) found `constraint.c` and `journal.c` — the legacy KV path's UNIQUE/NOT_NULL/RANGE/REFERENCE enforcement and audit-trail/CDC machinery — have zero reference to `rowstore.h`/`uses_rowstore`/`row_index.h` anywhere in either file. A row-set/SQL table today accepts duplicate values in a column meant to be unique, accepts missing values wherever the type parser alone doesn't reject them, enforces no foreign-key relationships, and leaves no audit trail of any mutation. Phase 16's own type/format validation (rejecting unparseable UINT64s, oversized strings) is necessary but not sufficient — it has never claimed to be a constraint system, and isn't one.

**Scope.**
- A row-set-aware constraint checker, most likely a new, parallel set of functions (matching this whole roadmap's established "new parallel capability, not a migration" precedent from Phase 17's `row_index.c` and Phase 21's `mvcc.c`, both built rather than retrofitting an existing file) covering at minimum UNIQUE and NOT NULL, hooked into the insert/update path (whichever of `rowstore_row_insert/update` or, once Phase 22 lands, `mvcc_row_insert/update` turns out to be the right integration point — a real design question Phase 22's own outcome will help answer, which is part of why this phase is sequenced after it).
- FOREIGN KEY / REFERENCE enforcement across two row-set tables, echoing Phase 20's JOIN precedent for what "two tables, related by a column" looks like in this codebase, but for a write-time check rather than a read-time join.
- Extending `journal.c` (or a parallel row-set-aware equivalent) so row-set table mutations can be attached to a journal the same way legacy KV tables already can — real before/after images, real commit/rollback marking.
- Explicitly out of scope: triggers (a materially bigger feature, not named anywhere in this roadmap so far and deserving its own scoping pass if ever pursued), CHECK constraints beyond RANGE's existing legacy shape, cascading FOREIGN KEY actions (ON DELETE CASCADE and friends) — plain rejection of a violating operation is the first-cut bar, matching every other "first cut, not the general case" precedent in this roadmap.

**Verification plan.** Real execution — constraint checking and journal-entry writing are both pure, low-dependency logic once the row layer is fetched, the same shape Phase 18's predicate evaluator proved out. A host test should cover each constraint type's violation and non-violation cases directly, plus confirming a journaled row-set table's mutations show up correctly via `journal_dump`/`journal_to_json`, matching the legacy path's own already-proven journal correctness.

**Findings addendum (as built).** Landed on scope exactly as written — two new, parallel subsystems (`kernel/row_constraint.h`/`.c`, `kernel/row_journal.h`/`.c`), both wired automatically into `mvcc.c` rather than `sql_exec.c` — plus one confirmed real memory-safety bug caught by the new host test's own execution, not by review.

**Both new subsystems confirm this roadmap's "new parallel capability, not a migration" precedent, for the same reason each prior instance did.** A direct audit (grep, not assumption) found `constraint.c` and `journal.c` have zero reference to `rowstore.h`/`uses_rowstore`/`row_index.h` anywhere — they operate exclusively on `object_records[]`'s one-record-per-object model. `journal.c` specifically has a structural wall of its own: `struct JournalEntry`'s `before_image[64]`/`after_image[64]` fields are 64-byte strings, sized for the legacy KV model's one-field-at-a-time shape (`journal_write(object_name, field_key, before_value, after_value, ...)`), while a full `struct RowValues` (up to 16 columns × 256 bytes) is over 60x too large to fit — the same "hard wall, not a tuning problem" discovery Phase 16 made about `RECORD_MAX_FIELDS` in the first place, now confirmed for `journal.c`'s entry shape specifically. `row_journal.c` therefore uses a deliberately smaller ring (`ROW_JOURNAL_MAX_ENTRIES = 16`, vs. `journal.c`'s own 256) since each entry carries two full `RowValues` — roughly 30x larger per entry, an honest capacity trade-off for the same BSS budget, documented rather than silent (matching `journal.h`'s own precedent of stating its BSS cost explicitly).

**Enforcement lives in `mvcc.c`, not `sql_exec.c` — the design question the scope bullets left open, now answered.** `row_constraint_check_write()`/`_check_delete()` and `row_journal_notify_insert/update/delete()` are called automatically from `mvcc_row_insert()`/`_update()`/`_delete()`, not from any of `sql_exec.c`'s `exec_*` functions. Phase 22 confirmed `mvcc.c` is the one real choke point every row-set mutation now passes through (`rowstore.c` itself has no gate of its own beyond `catalog_check_access()`), so putting the checks there means they can never be bypassed by a future caller that talks to `mvcc.c` directly instead of going through SQL — the same "automatic maintenance, not an opt-in step" reasoning Phase 17 used to justify `rowstore.c` calling `row_index_notify_insert/update/delete()` automatically rather than requiring every caller to remember to. This is a two-way *call* dependency (`mvcc.c` calls into `row_constraint.c`/`row_journal.c`, whose own implementations call back into `mvcc_table_scan()`) but not a circular *header* dependency: `row_constraint.h`/`row_journal.h` declare only primitive types and `struct RowValues` (`rowstore.h`), never anything from `mvcc.h` — only the `.c` implementation files include `mvcc.h`.

**Registration is a direct API call, not SQL DDL — a real, named, accumulating gap, not a new one.** Matching Phase 17's own `row_index_create()` precedent exactly, constraints are registered via `row_constraint_add_unique/not_null/range/reference()` called directly, not through a `CREATE TABLE ... UNIQUE` or `ALTER TABLE ADD CONSTRAINT` statement — `sql_parser.c`'s grammar has no DDL at all today, so tables (and now constraints) are set up via direct API calls, not SQL text. This repeats the exact "not syscall/DDL-reachable yet" gap Phase 22 found and fixed for the SQL engine itself, except here for index AND constraint creation both (`row_index_create()` was never made reachable either) — a real, accumulating gap worth naming plainly rather than treating as new. A future "make DDL live" phase is the natural place to close it for both at once.

**First-cut scope held exactly as written.** UNIQUE, NOT NULL, RANGE (reusing `constraint.c`'s own plain min/max shape), and REFERENCE (foreign key) are all implemented; REFERENCE's delete-side check (`row_constraint_check_delete()`) is plain RESTRICT — block the delete if any other table's REFERENCE constraint still points at the row, never delete-then-fix-up. No triggers, no CHECK constraints beyond RANGE, no cascading FOREIGN KEY actions, no composite (multi-column) constraints (matching Phase 17's own single-column-index precedent) — none of these were attempted.

**Statement-level atomicity, extended from write conflicts to constraint violations — a real bug caught during this phase's own implementation, not anticipated in the design.** Phase 22's `exec_update()`/`exec_delete()` special-cased only `MVCC_ERR_WRITE_CONFLICT` to abort a multi-row statement; any OTHER non-`MVCC_OK` return (constraint violations included, once this phase added them) fell through a bare `if (rc == MVCC_OK) affected++;` with no `else` — silently dropping the row from the affected count and reporting `SQL_ERR_NONE` with a possibly-wrong `affected_rows`, never surfacing the violation to the caller at all. This is the exact "denial looks like absence" failure class Phase 22 already named twice (`predicate_eval()`'s WHERE-column validation, `mvcc_table_scan()`'s bad-`txn_id` ambiguity) — recurring a third time, caught before it shipped by writing this phase's own host test (`row_constraint_journal_host_test.c`'s scenario 9, a batch UPDATE that should abort on a UNIQUE violation) rather than by code review. Fixed by generalizing both loops: any non-`MVCC_OK` return now aborts the whole statement via `map_mvcc_err()`, matching the write-conflict path's existing contract, with a new `SQL_ERR_CONSTRAINT_VIOLATION` `SqlErrorCode` (mapped from all five `MVCC_ERR_CONSTRAINT_*` variants) surfaced to the caller.

**A second, more serious bug — a real memory-safety issue, caught only by execution.** `row_journal.c`'s `rj_format_row()` (formats a `RowValues` as `"v0|v1|v2"` for `journal_dump()`/`journal_to_json()`) looped `for (i = 0; i < rv->count; i++)`, calling `rj_append()` (which null-terminates as it writes) on each iteration — but when `count == 0` (the common, EXPECTED case: an INSERT's before-image, a DELETE's after-image, an UPDATE's UB entry's after-image, and so on are all legitimately empty), the loop body never ran, so the caller-owned output buffer was never touched at all — not even null-terminated. Every call site then fed that buffer into `rj_append_json_str()`/`kernel_serial_printf("%s", ...)` as if it were a valid C string, reading uninitialized stack memory until it happened to hit a stray zero byte. This surfaced concretely, not hypothetically: `row_constraint_journal_host_test.c`'s scenario 12 (confirming a rolled-back transaction's journal entry disappears) found a *different, unrelated, already-committed* entry's JSON output contaminated with the rolled-back row's data — traced by bisecting with checkpoint instrumentation to confirm the underlying `RowJournalEntry` struct data was always correct (`count == 0` throughout) and the corruption was purely in the string formatting, not the data model. Fixed with a one-line change: `rj_format_row()` now writes `buf[*pos] = '\0'` unconditionally before the loop, guaranteeing a real empty string for the `count == 0` case. Worth stating plainly: this is the kind of bug that a compile-check or a logic review would not have caught — only real execution against a scenario that actually produced an empty `RowValues` and then read it back surfaced it, the same lesson every "real execution over review" verification-ceiling decision in this roadmap has been justified by.

**Verification, meeting the plan's own bar.** `row_constraint_journal_host_test.c` (repo root, new) links the full real stack — `sql_exec.c`, `sql_parser.c`, `predicate.c`, `row_index.c`, `rowstore.c`, `persist.c`, `cursor.c`, `mvcc.c`, `row_constraint.c`, `row_journal.c` — 34 checks, all passing, all real execution against two related tables (`accounts`, `orders`, with `orders.account_id REFERENCES accounts.id`): constraint registration (success and every documented failure mode — unknown table, unknown column, REFERENCE type mismatch, unparseable RANGE bounds); UNIQUE/NOT NULL/RANGE/REFERENCE violations on INSERT, each confirmed to leave no partial row; REFERENCE-on-DELETE RESTRICT behavior, both blocked (still referenced) and allowed (unreferenced) cases; UNIQUE-on-UPDATE with correct exclude-self semantics (a row updated to its own existing value does not self-conflict); statement-level atomicity for both write conflicts and constraint violations; journal recording for INSERT (a single PT entry), UPDATE (a UB+UP pair, per `journal.c`'s own before/after-image-as-two-entries precedent), and DELETE, including correct `committed` flag transitions; a rolled-back transaction's journal entries fully purged, never surfacing; and a detached table producing zero journal entries, matching `journal_write()`'s own "silently ignored if not attached" contract. All four host tests that link `mvcc.c` (`mvcc_host_test.c`, `sql_exec_host_test.c`, `sql_join_host_test.c`, `sql_tx_host_test.c`) needed their documented build lines updated to also link `kernel/row_constraint.c`/`kernel/row_journal.c`, since `mvcc.c` now calls into both at link time — none of the four call `row_constraint_init()`/`row_journal_init()` themselves, so every call resolves as a guaranteed no-op (zero registered constraints/attachments), and all four were re-run and still pass with **zero scenario changes**, the same "transparent, correctness-preserving substitution" evidence Phase 22 established for its own `mvcc.c` rewiring.

Full compile-check sweep: `row_constraint.c`, `row_journal.c`, `mvcc.c`, and `sql_exec.c` all syntax-check cleanly in isolation (`gcc -fsyntax-only -Wall -Wextra -std=c11`). `kernel.c` gained `#include`s for both new headers and `row_constraint_init()`/`row_journal_init()` calls in the boot sequence, placed immediately after `mvcc_init()` (grouped with the other relational-layer subsystems, though nothing here actually depends on `mvcc_init()` having run first) — grepped, not fully built (no cross-compiler available in this environment, matching every prior phase's kernel.c-adjacent verification ceiling). `Makefile`'s `X86_C_SRC` gained both new files. Re-ran all thirteen pre-existing host test suites (the twelve from Phase 22 plus `sql_tx_host_test.c`) plus the new `row_constraint_journal_host_test.c` — fourteen suites, all pass, zero regressions.

## 11. Phase 24 — Unify the two database paths

**Why this is its own phase, decided at the start, not discovered along the way.** Every phase since 16 has deliberately kept the row-set/SQL path and the legacy KV path (`object_catalog.c`'s `sys_sls_select/insert/update/delete`, `lock_mgr.c`, `transaction.c`, and — until Phase 23 — `constraint.c`/`journal.c`) as two parallel, non-interoperating implementations, each time as a genuine, named scope simplification rather than an oversight. That was the right call phase-by-phase — it kept each phase small and low-risk — but the accumulated result is two separate databases living in one kernel, which is the opposite of DB2 for i's own defining trait: one integrated database, with SQL and native record-level access as two doors into the same room. This phase is where that accumulated debt gets a real decision, not another deferral — matching the same "this phase needs its own design fork, decided at the start" treatment Phase 21 gave the 2PL-vs-MVCC question.

**The fork, not resolved here.**
- **Option A: migrate the legacy KV path onto row-set storage.** Every `object_catalog.c` table becomes a row-set table under the hood (even ones with exactly one conceptual "row" today); `sys_sls_select/insert/update/delete` become thin compatibility wrappers over `rowstore.c`/`mvcc.c`. Real unification, but touches the single most call-site-heavy subsystem in the codebase (Phase 16's own audit named 13 files referencing `SLSObjectRecord`/`RECORD_MAX_FIELDS` directly) — a large, genuinely risky migration.
- **Option B: deprecate the legacy KV path.** Freeze `object_catalog.c`'s record API where it stands (config-blob-style objects have a real, narrow use case it already serves adequately), and treat the row-set/SQL path as the only path for anything resembling "a table" going forward. Smaller and safer, but doesn't unify anything already built on the legacy path — existing legacy callers stay on a permanently separate track.
- **Option C: a bridging view.** Let a legacy KV object be addressable through the SQL surface (e.g., `sql_execute()` transparently routing a statement at a legacy table name through `sys_sls_select/insert/update/delete` instead of `rowstore.c`), without migrating storage at all. Cheapest, but risks being exactly the "cheap flexibility that gets tangled the first time it's slightly wrong" pattern this project has named and avoided at every prior fork (Phase 6, Phase 22/25) — a real concern to weigh honestly against the other two options, not dismiss reflexively.

**Scope (whichever option).** Must not silently break `journal.c`/`transaction.c`'s existing, already-verified WAL commit-and-recovery path for whatever of the legacy KV path survives the decision — the same non-negotiable constraint Phase 21 held itself to. A concrete call-site audit (matching Phase 16's own 13-file audit) of every legacy KV caller (`agent.c`, `agent_tools.c`, `aggregate.c`, `mqt.c`, `net/http.c`, `user/shell.c`, and whatever else direct inspection turns up) before any migration/deprecation decision is treated as final.

**Verification plan.** Depends entirely on which option is chosen — real execution should remain achievable regardless (a migration is testable the same way Phase 16 proved rowstore.c correct: real round-trip host tests against the real, unmodified code), but the specific shape of "what gets tested" can't be pinned down until the fork is resolved.

**Findings addendum (as built).** The scope's own required call-site audit produced a finding that reframes the fork itself: **Option B was chosen, not as the "smaller and safer" fallback, but because the audit showed Option A rests on a false premise.**

**The audit (real execution and direct reading, not assumption): `object_records[]` is not a competing table implementation at all.** `sys_sls_insert()`'s own rejection text is the tell: `"[DB] INSERT: Key '%s' already exists. Use 'update'."` — INSERT adds a new named FIELD to the object's one `SLSObjectRecord`, not a new row. There is exactly one record per catalog object (`object_records[]` is index-aligned with `object_catalog[]`, confirmed by direct read, matching Phase 16's own original framing of the wall it built `rowstore.c` to get around). Reading the six real call sites the scope bullet named (`agent.c`, `agent_tools.c`, `aggregate.c`, `mqt.c`, `query_engine.c`, `net/http.c`) confirmed every one of them iterates `rec->fields[fi]` treating each FIELD as the unit of data — `aggregate.c` most tellingly: its GROUP BY/WHERE/aggregate query engine runs over one object's own fields as if they were rows, not over multiple table rows at all. This is a per-object attribute/property bag — closer to IBM i's own separate `*DTAARA` (data area) concept, or a document's flat property list, than to a database table. `object_schemas[]`, notably, is NOT forked the same way: `rowstore_create_table()` reuses the exact same schema array `sys_sls_schema_set()` populates (confirmed in `rowstore.h`'s own header comment and `rowstore_create_table()`'s implementation) — the schema layer was already unified before this phase; only the record/row STORAGE layer forks, gated by `uses_rowstore`.

**Given that, Option A would be a category error, not a "large but doable" migration.** Migrating a flat per-object key-value bag onto a paged, columnar, fixed-row-width multi-row table engine doesn't have a natural mapping — there is no "row" in the legacy model to migrate, only fields on one object. Each of the six real callers would need a bespoke, non-generalizable adapter, for a benefit (storage unification) that was never real to begin with: this was never "two databases," it was "one attribute store plus one relational engine," a normal, coherent split (real DB2 for i itself keeps object attributes and database-file rows as genuinely separate concepts too). Option C's bridging view was rejected for the reason already named in the scope bullet — routing SQL text at a flat attribute bag with no row concept would need its own bespoke translation layer for the exact same reason Option A doesn't generalize, the "cheap flexibility that gets tangled" pattern named up front, now confirmed rather than just anticipated.

**Option B, made concrete rather than left as a deferral.** "Freeze the legacy path" isn't a no-op: the audit surfaced a real, currently-live, previously-unguarded gap that Option B's own execution had to close. `object_catalog.c` had **zero** reference to `uses_rowstore` anywhere (confirmed by grep) — meaning nothing stopped a caller from writing through `sys_sls_select/insert/update/delete` into `object_records[obj_idx]` for an object that had ALREADY been promoted to a real row-set table via `rowstore_create_table()`, silently diverging from what SQL/`rowstore.c` actually sees (`rowstore_create_table()` never touches or clears `object_records[idx]`, confirmed by direct read). That's a genuine instance of "two separate databases silently colliding on one object" — the exact failure mode this whole phase exists to prevent, just narrower and more concrete than the original "two accumulated implementations" framing suggested. Fixed with a guard added to the top of `sys_sls_select/insert/update/delete` AND `sys_sls_schema_set` (schema changes after row-set promotion would desync `table_headers[idx].layout`, computed once at promotion time and never recomputed — the same risk, one function later): each now checks `object_catalog[obj_idx].uses_rowstore` before touching any legacy state and returns a new, distinct code (`4`) with a clear log message pointing at `sql_execute()` instead, rather than silently operating on stale/irrelevant memory — the same "denial looks like absence" fix pattern this roadmap has now applied four times (Phase 19's WHERE-column validation, Phase 22's bad-`txn_id` scan ambiguity, Phase 23's swallowed constraint-violation return codes, and now this).

**Verification, real execution proving both halves of the promise.** `legacy_rowstore_boundary_host_test.c` (repo root, new) links the REAL, unmodified `kernel/object_catalog.c` AND `kernel/rowstore.c` — the first host test in this roadmap to link `object_catalog.c` itself rather than stub around it, since it's the file actually under test this phase. Every other host test that needed `object_catalog.c`'s data model instead stubbed `catalog_check_access()` and supplied its own catalog globals (Phase 16's own "as heavy as Phase 9 found process.c's to be" reasoning for why); this test inverts that, stubbing `object_catalog.c`'s own dependencies instead (`tx_get_active`/`wal_stage`/`kernel_get_current_thread_id` — its own forward-declared externs — plus `lock_acquire`, `journal_write`, `index_on_insert/update/delete`, `mqt_refresh_for_table`, `constraint_check_insert/update`, `tier_notify_access`, `partition_get_for_uid`, and `persist_catalog/records/schemas/rowstore_headers`) so the real guard logic runs unmodified. 15 checks, all passing: the legacy record API fully unchanged for a normal, never-promoted object (INSERT/SELECT/UPDATE/SCHEMA-SET/DELETE all still succeed exactly as before); `rowstore_create_table()` really does flip `uses_rowstore`; all five legacy entry points cleanly rejected (return code `4`) once an object is promoted, each confirmed to leave the underlying state completely untouched (`object_records[]`'s `field_count`, `object_schemas[]`'s `field_count`, both checked directly, not just the return code); and a second, never-promoted object confirmed completely unaffected by the first one's promotion — the guard is per-object, not global.

Full compile-check sweep: `object_catalog.c` syntax-checks cleanly against this test's real link line, with only pre-existing, unrelated implicit-declaration warnings from `sys_sls_valloc` (checking a freestanding file with a hosted toolchain, the same expected noise every prior phase's kernel.c-adjacent check has produced — confirmed pre-existing by diffing against the unmodified file, not new). No `Makefile`/`kernel.c` changes were needed — `object_catalog.c` was already built into the kernel image; this phase only added logic inside functions already wired in. Re-ran all fourteen pre-existing host test suites (the thirteen from Phase 22 plus `sql_tx_host_test.c`, plus Phase 23's `row_constraint_journal_host_test.c`) plus the new `legacy_rowstore_boundary_host_test.c` — fifteen suites, all pass, zero regressions (expected: this phase's only code change is new guard logic inside `object_catalog.c`'s own DML functions, which none of the other fourteen suites link).

## 12. Phase 25 (stretch, not scoped) — cost-based query optimization

Deliberately left unscoped, same treatment the LPAR roadmap gave its own Phase 15. Phase 19–20's planner is "first applicable index wins, one fixed join order" — genuinely naive, and genuinely fine until a real workload proves it isn't. A real optimizer needs table/index statistics (row counts, value-distribution estimates), a cost function, and — once more than two tables can be joined — a join-order search. That's a meaningfully bigger scope than any single phase above, and building it before there's a concrete slow query to point at risks the same "cheap flexibility that gets tangled the first time it's slightly wrong" this project has avoided at every prior fork (Phase 6's ODT-avoidance precedent, cited again here deliberately). Revisit only when Phases 16–21 are built and something real is slow.

## 13. Suggested execution order

Phase 16 is the hard dependency for everything — nothing else in this document can start first. Phases 17 and 18 depend only on 16 and have no dependency on each other, so they can run in either order or interleaved, the same "two independent, either-order phases" shape the LPAR roadmap found between its own Phases 9 and 10. Phase 19 needs both 17 and 18. Phase 20 needs 19. Phase 21 depends only on 16, not on 17–20 — it could in principle run any time after 16, interleaved with the SQL-surface work, since concurrency control and query surface are largely orthogonal concerns that happen to share Phase 16's row store as a common foundation. Phase 22 needs both 20 (the full SQL surface it's exposing) and 21 (the concurrency control it's wiring in). Phase 23 depends on 16 directly for row-set access and on 22 for which insert/update entry point (`rowstore.c` or `mvcc.c`) turns out to be the right integration point. Phase 24 depends on all of 16–23 being in place before its own unification design fork can be decided with full information. A reasonable sequence: **16, then (17, 18, 21) in any order or interleaved, then 19, then 20, then 22, then 23, then 24**, treating 25 as unscoped until a concrete slow query demands it. Each phase gets its own "Design Phase N" pass before implementation, exactly like every LPAR phase did — this document sets direction and dependency order, not final per-phase design decisions.
