# AeroSLS Vector Store Roadmap v0.1 — a second database, in concert with the first

## 0. Where this picks up

The RDBMS roadmap (`docs/AeroSLS-RDBMS-Roadmap-v0.1.md`) closed out Phases 16-24 with a real relational engine (row-set storage, B-tree indexing, WHERE/JOIN, MVCC, SQL, constraints, journaling) and a settled boundary against the legacy per-object attribute store. That roadmap's own Phase 24 finding is the starting premise here: AeroSLS's storage subsystems specialize by data shape rather than converge on one engine, and that has been the right call every time it came up (`row_index.c` beside `index_mgr.c`, `mvcc.c` with no legacy analog, `row_constraint.c`/`row_journal.c` beside the legacy path's own versions).

A vector collection — fixed-dimension float arrays searched by approximate or exact nearest-neighbor distance, not by typed-column equality/range predicates — is the starkest case of "wrong shape for the existing engine" this project has hit yet. This roadmap treats it as a genuinely separate subsystem from day one, not a future extension of `rowstore.c`, and defines how it talks to the relational engine rather than absorbing it.

**The integration decision, made up front.** Vector collections are their own object-catalog entries (own capability tag, own partition scoping — reusing Phase 7's authority model exactly the way `uses_rowstore` already reuses the shared schema layer), with their own storage and search engine underneath. A similarity search returns `external_id`s, not full rows; correlating a match back to relational metadata is a second, ordinary query against whatever table the caller associated those ids with. This mirrors how vector search is actually deployed in practice more than it mirrors "vectors as a native SQL column type" — and it means Phase 20's JOIN machinery doesn't need to change at all to make "nearest neighbors, with metadata attached" possible; the caller does two queries and joins them together at the SQL/application boundary. Native SQL-level vector search syntax is an explicit non-goal of this roadmap's first pass (see Phase 5).

## 1. Design principles carried forward

1. **New parallel subsystem, not an extension.** `kernel/vecstore.c`/`.h` is new. Nothing in `rowstore.c`, `mvcc.c`, or `sql_exec.c` changes to accommodate it.
2. **Audit before assuming.** Before Phase 3 was scoped, a direct investigation (not assumption) of `net/http.c`, `net/tcp.c`, and `net/inference.c` found AeroSLS already has a *working outbound HTTP client* aimed at an OpenAI-compatible completions endpoint (`net/inference.c`'s `infer_call()`, header comment explicitly naming Ollama compatibility) built on a real `tcp_connect()`/`tcp_send()`/`tcp_recv()` client (`net/tcp.c`). Phase 3 is scoped as adapting that proven pattern to a different endpoint and response shape, not building an HTTP client from nothing.
3. **Real execution over review**, same verification ceiling every RDBMS phase held itself to — host tests wherever the dependency graph allows it, compile-check-plus-grep only where a heavier kernel-wiring dependency graph forces it (matching the RDBMS roadmap's own repeated, named fallback).
4. **First cut proves correctness, not performance.** Brute-force exact search before any approximate index — the same order Phase 16-19 proved row-set storage and querying correct before Phase 17 added indexing on top.
5. **Explicitly out of scope, named rather than silently dropped:** approximate nearest-neighbor indexing (HNSW/IVF — Phase 6, stretch, unscoped, same treatment the RDBMS roadmap gave cost-based query optimization), native SQL-level vector search syntax, multi-vector/multi-model collections, vector updates-in-place (first cut is insert + delete, no update), any notion of vector-level MVCC (a single global "last committed state" per collection is the first-cut concurrency model — see Phase 1's own scope note).

## 3. Phase 1 — Vector storage engine

**Why this is smaller than Phase 16's row-set engine, not a copy of it.** `rowstore.c` earns its complexity from per-table, schema-derived, multi-column row layouts (`RowTableLayout`, `ROWSTORE_MAX_COLUMNS`, mixed `SLSFieldType`s). A vector collection has exactly one column, of one fixed type (`float`), whose width is set once at collection-creation time (the embedding dimension — 384/768/1024/1536 depending on the model, per Phase 3's own findings once real Ollama models are tested against). This is a much narrower problem: fixed-width entries in a page pool, addressed by `(page_id, slot_index)`, exactly like `rowstore.c`'s own `RowId` — the same mechanism, far less schema machinery around it.

**Scope.**
- A `vector_collections[CATALOG_MAX_OBJECTS]` header array (index-aligned with `object_catalog[]`, same idiom as `table_headers[]`), each holding `{object_id, dimension, entry_count, first_page_id, last_page_id}`.
- A new catalog flag (`uses_vectorstore`, mirroring `uses_rowstore`) or a dedicated `OBJ_TYPE_VECTOR_COLLECTION` object type — a real, small design decision to make at implementation time by checking how `SLSObjectType`'s existing switch statements would need to grow either way, not assumed here.
- Each entry: `{external_id: uint64, active: uint8, values: float[dimension]}`. `external_id` is caller-supplied and opaque to this subsystem — the correlation key a caller uses to join back to a relational row (see Phase 5), not interpreted or validated against any table.
- CRUD: `vecstore_create_collection(name, dimension)`, `vecstore_insert(collection_name, external_id, float* values) -> VecId`, `vecstore_get(collection_name, VecId) -> values`, `vecstore_delete(collection_name, VecId)`. No update-in-place in this first cut (delete + re-insert instead) — same "first cut, not the general case" posture as every RDBMS phase's own scope cuts.
- Concurrency: a single global read/write posture per collection (no MVCC) — real concurrent-write correctness is deferred exactly the way it was for `rowstore.c` before Phase 21 existed; this subsystem starts at that same pre-Phase-21 point deliberately, not by oversight.
- Persistence: mirrors `rowstore.c`'s own NVMe paging story (a new `PERSIST_MAGIC_VECSTORE` region, a new `ROWSTORE`-sibling `VECSTORE_LBA_BASE`), or explicitly deferred to a later phase if the first cut should stay RAM-only for speed of delivery — a real scope call to make explicitly rather than silently assume either way when this phase starts.

**Verification plan.** A host test in the same shape as `rowstore_host_test.c` — real round-trip insert/get/delete against the real, unmodified `vecstore.c`, confirming values survive intact and `external_id` correlation is preserved.

**Findings addendum (as built).** Landed smaller than even this phase's own scope bullets anticipated, on both of the two design questions this section flagged as "real, to decide at implementation time" — plus one deliberate divergence from `rowstore.c`'s own precedent that turned out to be the right call once the actual complexity was visible.

**Catalog integration: neither a flag nor a new object type.** The scope bullet framed this as "`uses_vectorstore` flag vs. `OBJ_TYPE_VECTOR_COLLECTION`" — as built, it's neither. `vector_collections[idx].active` is the ONLY place "is this a vector collection" is recorded; `object_catalog.h`'s `SLSObjectEntry` is untouched. The reasoning, worked out once the actual callers were considered rather than assumed: `uses_rowstore` exists because MULTIPLE subsystems outside `rowstore.c` need to ask that question from the outside (`mvcc.c`, `sql_exec.c`, and Phase 24's own legacy-path guard all consult it) — no such external caller exists yet for vector collections, so there's nothing forcing the flag onto the shared catalog struct today, matching `row_index.c`'s own precedent of keeping its own independent index-existence bookkeeping rather than touching `object_catalog[]` at all. This also sidesteps a real, concrete hazard a persisted flag would have created: `persist_catalog()` snapshots the whole `object_catalog[]` array as raw bytes (confirmed by reading `persist.c` directly, not assumed), so a persisted `uses_vectorstore` flag would survive an NVMe restore while this phase's RAM-only collection data would NOT — a "flag says yes, data says no" inconsistency this design avoids by construction rather than needing a restore-time reconciliation step to fix later.

**Persistence: RAM-only, decided rather than left ambiguous.** The scope bullet listed NVMe persistence as an open call between "mirror `rowstore.c`'s paging story" and "defer for speed of delivery." Decided: RAM-only, following `row_index.c`'s own already-established precedent for a first-cut storage-adjacent subsystem in this codebase. This is the single largest simplification versus `rowstore.c`: no `PERSIST_MAGIC_VECSTORE` region, no `VECSTORE_LBA_BASE` reservation, no lazy-load-from-NVMe path, no `nvme_read_sync`/`nvme_write_sync` calls anywhere in `vecstore.c` at all — pages allocated via `allocate_physical_ram_frame()` simply stay resident for the process/kernel's lifetime. A real, named scope cut for a future phase to close if vector data durability across reboots turns out to matter, not a silent gap.

**A real, deliberate divergence from `rowstore.c`'s own convention: numeric API boundary, not text.** `rowstore.c`'s `RowValues` is text in, text out at the API boundary (matching the legacy KV path's own string-based convention, with `rowstore.c` parsing/formatting internally per column type). `vecstore.c`'s `VecValues` is `float` in, `float` out directly — no text round-trip at all. This wasn't in the original scope bullets and was decided during implementation once it became clear a text round-trip would be pure overhead here: there is no legacy string-based precedent to match for vector data (unlike row values), and Phase 2's distance functions need raw floats directly, so serializing to decimal text and re-parsing on every read would cost real precision and CPU for zero benefit. Storage itself is still fixed-width raw bytes (1 tombstone + 8-byte `external_id` + `dimension` × 4-byte floats per entry, mirroring `rowstore.c`'s own tombstone-byte-plus-fixed-columns page layout exactly) — only the CRUD function signatures differ from `rowstore.c`'s text-based ones.

**Verification, real execution proving both correctness and the page-chain mechanism specifically.** `vecstore_host_test.c` (repo root, new) links the real, unmodified `kernel/vecstore.c` — 39 checks, all passing. Beyond ordinary CRUD round-tripping (bit-exact float storage confirmed component-by-component, not just "looks right"), scenario 12 specifically exercises the multi-page path: a second collection created with `dimension=800` (chosen so `entries_per_page` computes to exactly 1, confirmed by direct assertion), forcing every single insert to allocate a brand-new page and chain it to the previous one — proving the page-linking mechanism actually works across a real 3-page chain, not just within a single page, without needing hundreds of ordinary inserts to reach that state. Scenario 13 confirms two independently-created collections never interfere with each other's `entry_count` or storage. Permission-gating (scenario 10) and unknown-collection handling (scenario 11) are both covered the same way `rowstore_host_test.c` already established the pattern for `rowstore.c`.

Full compile-check: `vecstore.c` syntax-checks cleanly in isolation (`gcc -fsyntax-only -Wall -Wextra -std=c11`) with zero warnings — no implicit-declaration noise at all, unlike every kernel.c-adjacent file checked outside the freestanding toolchain, since `vecstore.c`'s dependency surface (`object_catalog.h`, `frame_pool.h`, `kernel_io.h`, `../user/permissions.h`) is fully self-contained and hosted-toolchain-clean. `kernel.c` gained a `#include` and a `vecstore_init()` call in the boot sequence (grouped with the other RAM-only relational-layer subsystems, right after `row_journal_init()`), grepped rather than fully built (no cross-compiler available in this environment, matching every RDBMS-phase precedent). `Makefile`'s `X86_C_SRC` gained `kernel/vecstore.c`. A sample of the existing RDBMS-roadmap host suites (`rowstore_host_test.c`, `legacy_rowstore_boundary_host_test.c`) was re-run to confirm zero interference — expected, since `vecstore.c` shares no code, globals, or catalog flags with any existing subsystem; a full sweep wasn't necessary given that isolation, unlike every RDBMS-phase change that touched shared machinery (`mvcc.c`, `sql_exec.c`, `object_catalog.c`) and therefore needed the full sweep each time.

## 4. Phase 2 — Brute-force similarity search

**Scope.**
- `vecstore_search(collection_name, float* query_vec, uint32_t k, VecMatch* out) -> uint32_t count`, where `VecMatch = {external_id, VecId, distance}`.
- Cosine similarity as the primary metric (the standard choice for embedding models, including Ollama's own), with L2 (Euclidean) as a second, selectable metric — both hand-rolled (no libc math beyond what's already used elsewhere in this freestanding kernel; a `sqrt` implementation will be needed and doesn't exist yet anywhere in the tree, confirmed by the same audit discipline Phase 3 already applied).
- Top-K selection via a bounded linear scan (a small fixed-size max-heap or insertion-sorted array of size `k`), not a full sort — O(n·d + n·log k), a real, honest complexity bound to state rather than leave implicit.
- Explicitly out of scope: any index structure, approximate results, filtering by metadata during the scan (a `WHERE`-style predicate combined with vector search is real, useful, future work, not attempted here).

**Verification plan.** Real execution against a small synthetic collection with known nearest neighbors (e.g., vectors placed at controlled distances), confirming both the returned ordering and the distance values themselves are correct — matching Phase 18's own predicate-evaluator verification shape.

**Findings addendum (as built).** Landed on scope with one API decision the scope bullets didn't spell out (how a "higher is better" metric and a "lower is better" metric share one top-K selection rule) and one deliberate, named complexity trade-off (insertion-sorted array, not a heap).

**A real design decision: both metrics report "smaller distance = closer," even though cosine's native sense runs the other way.** Cosine SIMILARITY is naturally "higher is more similar" (range [-1, 1]); L2 is naturally "lower is more similar" (0 = identical). Rather than making top-K selection metric-aware, `VEC_METRIC_COSINE` reports `1.0f - cosine_similarity` as its `distance` (0 = identical direction, 2 = exactly opposite) so one selection rule ("smallest distance wins") works for both metrics without a branch in the hot path. A zero-magnitude vector (undefined direction) reports a neutral `1.0f` rather than dividing by zero or arbitrarily ranking it best/worst.

**Reuse, not a parallel scan.** `vecstore_search()` is built directly on top of `vecstore_collection_scan()` — same `catalog_check_access()` gate, same page-chain traversal, zero duplicated logic. This mirrors the RDBMS roadmap's own `predicate_table_scan()`-on-`rowstore_table_scan()` precedent exactly, confirming the "one real choke point, reused" doctrine holds for this new subsystem too, not just the relational one.

**A stated, deliberate complexity trade-off: `O(n·k)`, not `O(n·log k)`.** The scope bullet named a heap as one option; a bounded insertion-sorted array of size `k` was built instead — each candidate costs up to `k` element shifts rather than `log k` heap operations, worse asymptotically but simpler to get correct (no heap-invariant bugs to introduce) and, for the `k` values this phase's own use case realistically needs (tens, not thousands, per Phase 2's own scope framing), the difference is not real in practice. Named explicitly here rather than silently claiming the better bound the original scope bullet suggested.

**The `sqrt` gap the Phase 3 audit already predicted was real, not hypothetical.** A fresh grep before writing any code confirmed zero existing `sqrt`/Newton/`isqrt` anywhere in this tree. `vs_sqrtf()` is a from-scratch Newton-Raphson implementation (bounded at 20 iterations with an early convergence exit, globally convergent for any positive input), matching this codebase's established "every file keeps its own small hand-rolled numeric helpers" convention rather than inventing a shared math utility file for one function.

**Verification, with hand-computed ground truth, not just "looks plausible."** `vecstore_search_host_test.c` (repo root, new) links the real, unmodified `kernel/vecstore.c` — 24 checks. Five 2D points were chosen by hand specifically so L2 and cosine distance from one query point produce two DIFFERENT orderings with NO ties in either metric (worked out algebraically before writing any assertions, then cross-checked against the program's actual float output to 3 decimal places of tolerance) — proof the two metrics genuinely diverge rather than coincidentally agreeing, and a test that never has to reason about tie-breaking behavior. Covers: exact rank-by-rank ordering AND exact distance values for both metrics at `k=5` (all points); `k=3` truncation returning exactly the correct prefix of the full ordering, not an arbitrary subset; `k` larger than the collection's real entry count returning only what exists; a search result's `VecId` round-tripping through `vecstore_get()` to confirm it's a real, usable id and not just an opaque token; and the full error-path sweep (dimension mismatch, unknown collection, permission denial, and a real-but-empty collection all returning `0` cleanly).

Full compile-check: `vecstore.c` (now including the Phase 2 additions) syntax-checks cleanly (`gcc -fsyntax-only -Wall -Wextra -std=c11`), zero warnings. `vecstore_host_test.c` (Phase 1's own suite) was re-run unmodified and still passes all 39 checks — confirming Phase 2's additions are purely additive with no changes to any Phase 1 code path.

## 5. Phase 3 — Ollama embedding client

**Why this is adaptation, not new-HTTP-client work — confirmed by direct investigation, not assumed.** `net/inference.c`'s `infer_call()` already does the complete pattern this phase needs against a different endpoint: `parse_ipv4_str()` → `tcp_connect()` → build a JSON request body with headers → `tcp_send()` → loop `tcp_recv()` until the connection closes → `parse_http_status()` → `find_http_body()` → extract fields from the JSON response. That whole control flow is directly reusable as a template. What does NOT exist anywhere in this codebase, confirmed by the same audit: a JSON parser capable of walking a numeric array (`"embedding":[0.1,0.2,...]`) — only string-field extraction exists (`json_str()`/`json_int()` in `net/http.c`, `infer_json_str()` in `net/inference.c`) — and a string-to-float parser (only integer parsing exists anywhere in this tree).

**Scope.**
- `kernel/ollama_client.c`/`.h`: `ollama_embed(const char* model, const char* text, float* out, uint32_t max_dim) -> int` (0 on success), following `infer_call()`'s connect/send/recv/parse shape against Ollama's `/api/embeddings` endpoint (`{"model": "...", "prompt": "..."}` request, `{"embedding": [...]}` response — a real shape to confirm against a running Ollama instance during implementation, not assumed from documentation alone).
- A new JSON number-array parser (`json_float_array()` or similar) — genuinely new code, not adapted from anywhere in this tree.
- A new string→float parser (no `atof` exists in this freestanding kernel) — small, but real, new code, following the same hand-rolled-per-file convention every other numeric parser in this codebase already uses (`rc_parse_f64()` in `row_constraint.c` is the closest existing precedent to adapt from, though it parses a single float from a fixed-format string, not a JSON array element).
- Explicitly out of scope: any Ollama endpoint other than embeddings (chat/generate are already `net/inference.c`'s concern), batching multiple texts into one request, retry/timeout policy beyond whatever minimal behavior `tcp_recv()`'s existing loop already has.

**Verification plan.** This is the one phase in this roadmap that cannot be fully host-tested without a running Ollama instance reachable over the network — real execution against a live Ollama server is the target verification (matching this project's "real execution wherever the dependency graph allows it" posture), with a fallback of compile-check plus a hand-constructed fake TCP response (mirroring `sql_tx_host_test.c`'s own fake-NVMe-frame pattern, applied here to a fake TCP stream instead) if a live Ollama endpoint isn't reachable from wherever this phase gets implemented and verified.

**Findings addendum (as built).** Landed on the fallback verification path the scope section already anticipated, plus one file-location correction and one confirmed, named, still-unverified assumption.

**Location correction: `net/ollama_client.c`/`.h`, not `kernel/`.** This section's own scope bullet said `kernel/ollama_client.c`/`.h`, written before this phase's own implementation actually looked at where its direct template lives. `net/inference.c` — the proven connect/send/recv/parse pattern this file adapts — lives in `net/`, grouped with `net/tcp.c`/`net/http.c`/`net/dhcp.c`, because it's fundamentally a network client, not a kernel subsystem. The real files are `net/ollama_client.c` and `net/ollama_client.h`; this doc is corrected to match the real layout rather than the other way around.

**No live Ollama reachable — confirmed by direct attempt, not assumed.** Before any implementation code was written, three direct `curl` attempts (`localhost:11434`, `host.docker.internal:11434`, and the Docker gateway `172.17.0.1:11434`, each against `/api/version`) all returned "Connection blocked by network allowlist" — this development sandbox has no path to a live Ollama instance. This confirms the fallback verification path named in this section's own scope is the one that had to be used, not a shortcut taken without checking.

**What got built.** `ollama_embed()` follows `infer_call()`'s exact control flow: `parse_ipv4_str()` → `tcp_connect()` → build a JSON request body (`{"model":"...","prompt":"..."}`) with HTTP headers (including a `Content-Length` computed from the real, just-built body, not a guess) → `tcp_send()` → loop `tcp_recv()` accumulating into a response buffer until the connection closes → `parse_http_status()` (rejecting anything outside 2xx) → `find_http_body()` → hand off to the new parsing machinery below. Per this codebase's own established convention (`net/inference.c` and `net/http.c` each keep their own small string/buffer helpers rather than sharing a library), `ollama_client.c` keeps its own copies too (`oc_*` prefix) rather than extracting a shared net-client helper module.

**The genuinely new machinery, confirmed necessary by the same audit this section's own scope already named.** Two functions needed to be built from nothing, because — confirmed by direct grep before writing either — nothing in this tree parses a full JSON number grammar or any JSON array from external input: `json_str()`/`json_int()` (`net/http.c`) and `infer_json_str()` (`net/inference.c`) only extract string fields; `rowstore.c`'s/`row_constraint.c`'s own float parsers handle a narrower grammar (`[-]digits[.digits]`, no exponent) sized for this codebase's own internally-generated text, not genuinely external JSON. `oc_parse_json_number()` implements the real grammar (optional sign, integer part, optional fraction, optional signed exponent with `e`/`E` and optional `+`/`-`) — verified directly against `4.5e-2` and `-1.25e2` inputs in the host test, not just plain decimals. `oc_json_float_array()` walks to a `"key":` in the JSON text, expects `[`, and parses comma-separated numbers via `oc_parse_json_number()` into a caller-supplied array — stopping cleanly (returning whatever was already parsed, not failing the whole call) at the first unparseable element or at the caller's `max_count`, matching this codebase's own "return what's valid, don't discard partial real data" posture seen elsewhere (`row_journal.c`'s ring buffer).

**A real, named, still-unverified assumption.** This implementation targets Ollama's legacy `/api/embeddings` endpoint shape (`{"model":"...","prompt":"..."}` request, flat `{"embedding":[...]}` response) rather than the newer `/api/embed` endpoint (batch-capable, nested `{"embeddings":[[...]]}` response) — the legacy shape is what this roadmap already committed to before this phase started, and it's simpler (one flat array, not an array of arrays). Because live Ollama is unreachable from this environment (see above), this shape has NOT been confirmed against a real running server — it's built from Ollama's documented API shape, not from an observed real response. This is a real, load-bearing assumption to confirm before depending on this code in anger, named here explicitly so it's never mistaken for a verified fact.

**Verification, with the fallback path this section's own scope plan anticipated.** `net/ollama_client.c` syntax-checks cleanly in isolation (`gcc -fsyntax-only -Wall -Wextra -std=c11 -I kernel -I drivers -I net`), zero warnings. `ollama_client_host_test.c` (repo root, new) links the real, unmodified `net/ollama_client.c` and stubs `tcp_connect`/`tcp_send`/`tcp_recv`/`tcp_close` directly at their own real API boundary (matching this project's established "stub the heavy dependency at its own real API boundary" pattern, e.g. `catalog_check_access()` in `rowstore_host_test.c`) rather than linking the real, much heavier `net/tcp.c` (which depends on ARP resolution and the e1000 driver) — 28 checks, all passing. Beyond the request/response happy path (confirming both a correctly-built HTTP request — method, path, `Content-Length` matching the real sent body, model/prompt present in the JSON body — and correct response parsing, including negative numbers and both positive- and negative-exponent scientific notation), the suite specifically covers: a response delivered across many small (7-byte) `tcp_recv()` chunks rather than one read, proving the accumulate-until-close loop reassembles a fragmented response correctly; `tcp_connect()` failure; a non-2xx (404) response; a 200 response with the wrong JSON shape (a stand-in for accidentally hitting the newer `/api/embed` endpoint's nested-array response); a completely empty response; and a realistic 768-dimension embedding (a real `nomic-embed-text` model dimension) to confirm the response-buffer sizing and array-parsing loop hold up at real scale, not just on 5-element toy arrays. `Makefile`'s `X86_C_SRC` gained `net/ollama_client.c`, placed beside `net/inference.c`. `vecstore_host_test.c` (Phase 1) and `vecstore_search_host_test.c` (Phase 2) were both re-run and still pass all 39 and 24 checks respectively — expected, since `ollama_client.c` shares no code, globals, or files with `vecstore.c` at all; Phase 4 is what will actually connect these two subsystems together.

## 6. Phase 4 — Make it live

**Scope.**
- Syscall + dispatch case(s) for vector insert (raw vector) and embed-insert (raw text, routed through `ollama_client.c` first) and search, matching Phase 22's own `SYS_SLS_SQL_EXECUTE` precedent (next free syscall number after whatever Phase 22/23 last claimed).
- A shell command (`user/shell.c`) for reachability demonstration, matching Phase 22's own `sql <statement>` command.
- Explicitly out of scope: any notion of a query language for vector search (no "VQL") — the syscall/API surface is the only interface in this pass.

**Verification plan.** Same ceiling as Phase 22's own syscall/dispatch work — compile-check plus targeted grep for the heavier kernel-wiring half, real execution for anything host-testable in isolation.

**Findings addendum (as built).** Landed on four syscalls, not the three this section's own scope bullet named, plus a real design decision about where cross-subsystem glue code is allowed to live, and — unusually for this phase's own "compile-check ceiling" plan — real host-test coverage for the parts of it that turned out to be genuinely testable after all.

**A fourth syscall this section didn't name: `SYS_SLS_VEC_CREATE`.** The scope bullet above named only insert, embed-insert, and search. Implementation surfaced a real reachability gap before writing any dispatch code: `vecstore_create_collection()` (Phase 1) had no syscall wrapper, and per `vecstore.h`'s own "Registration" comment, every insert requires a collection to already exist first. Without exposing collection creation too, the three named syscalls would be dispatch-reachable but never actually usable — the same "denial looks like absence" failure class named and fixed four times across the RDBMS roadmap (Phase 19 WHERE-column validation, Phase 22 mvcc bad-txn-id ambiguity, Phase 23 swallowed constraint violations, Phase 24 design notes), here applied to reachability rather than error handling. `SYS_SLS_VEC_CREATE = 221` was added, shifting insert/embed-insert/search to 222-224. Worth naming precisely because it exposed a real, still-open gap in the RDBMS roadmap this project has never fixed: `rowstore_create_table()` has no syscall wrapper anywhere in this codebase either (confirmed by grep — it's reachable only from host tests), and `sql_exec.c`'s grammar has no `CREATE TABLE` statement. Fixing that is out of scope here (it belongs to the RDBMS roadmap, not this one) but it's the reason this phase didn't just quietly repeat the same gap without comment.

**Where cross-subsystem glue code is allowed to live — a real, bounded exception to "no shared code."** Every prior phase's own design principle (§1: "New parallel subsystem, not an extension") held that `vecstore.c` and `net/ollama_client.c` share no code, no globals, no files. `sys_sls_vec_embed_insert()` — the one syscall that must call both `ollama_embed()` and `vecstore_insert()` in sequence — is the deliberate, named exception: it lives in `vecstore.c`, which gained a new `#include "../net/ollama_client.h"` (via `vecstore.h`). The alternative (putting it in `ollama_client.c` instead, with that file including `vecstore.h`) was considered and rejected: `net/ollama_client.c` is meant to stay a generic embedding client, independently reusable without any notion of vector storage, whereas `vecstore.c` is already the "owning" subsystem for everything vector-shaped. This is Phase 4's whole reason to exist — joining Phases 1-3 together — not a violation of the separation principle those phases established; the header comments in both `vecstore.h` and `vecstore.c` name this explicitly at the point it happens, rather than leaving a future reader to wonder why the "no shared code" rule stopped applying.

**Request struct fields exactly mirror the SQL precedent, uid included.** `SLSVecCreateRequest`/`SLSVecInsertRequest`/`SLSVecEmbedInsertRequest`/`SLSVecSearchRequest` all carry `caller_uid` as their first field, filled in by the shell from `current_session_uid` before `do_syscall()` — the identical convention `SLSSqlRequest.caller_uid` established, itself modeled on `SLSVallocRequest.owner_uid`. `SLSVecSearchRequest` mirrors `SqlResult`'s own bounded-result-plus-truncated-flag shape: a fixed `matches[VEC_SEARCH_MAX_K]` array (`VEC_SEARCH_MAX_K = 64`, generous for this roadmap's own "tens, not thousands" k-value framing from Phase 2) plus a `truncated` flag set when the caller's requested `k` had to be capped.

**The adapters are not all one-liners, and that mattered for verification.** `sys_sls_sql_execute()` (Phase 22's precedent) is a genuine one-line pass-through. `sys_sls_vec_search()` has real logic of its own (capping `k`, setting `truncated`), and `sys_sls_vec_embed_insert()` has more: bounding a reported embedding dimension defensively, copying floats into a `VecValues`, and — the part most worth getting right — reporting two independent failure points (`ollama_status` vs. `insert_status`) separately rather than collapsing them into one code, so a caller can tell "Ollama never answered" apart from "Ollama answered fine but the collection/dimension was wrong." That's real, new marshaling logic, not just wiring, and it turned out to be host-testable after all (see below) despite this phase's own scope plan assuming a compile-check-only ceiling for the whole syscall surface.

**Verification: dispatch/shell hit the expected ceiling; the adapters themselves did not.** `kernel/syscall_dispatch.c` (gained `#include "vecstore.h"` plus the four new `case` blocks) and `user/shell.c` (gained `vec create`/`vec insert`/`vec embed-insert`/`vec search` commands plus two new parsing helpers, `sh_token()` and `sh_atof()` — freestanding, since no `atof`/token-splitter existed in this shell before, matching this codebase's per-file hand-rolled-helper convention) both syntax-check cleanly (`gcc -fsyntax-only -Wall -Wextra -std=c11`) with zero new warnings — only the same pre-existing, unrelated implicit-declaration noise every prior phase's kernel.c-adjacent check has produced (confirmed by grep for anything mentioning "vec"/"ollama"/"error": zero matches beyond the include-comment line itself). This is the expected ceiling this section's own scope plan named, and matches Phase 22's identical precedent exactly. But `sys_sls_vec_create/insert/embed_insert/search()` themselves are ordinary functions in `vecstore.c`, not inside `do_syscall()`'s own heavy switch statement — so unlike Phase 22 (which stopped at compile-check for its whole syscall surface), this phase went further: `vecstore_syscall_host_test.c` (repo root, new) links the real, unmodified `kernel/vecstore.c` and stubs `ollama_embed()` directly and controllably (mirroring `catalog_check_access()`'s own stub in `vecstore_host_test.c`) — 22 checks, all passing, covering the marshaling logic specifically: `sys_sls_vec_create` success/failure against a real vs. nonexistent object; `sys_sls_vec_insert` round-tripping exact values and reporting dimension-mismatch cleanly; `sys_sls_vec_search`'s k-capping and `truncated` flag under an intentionally oversized `k`; and all three `sys_sls_vec_embed_insert` paths — Ollama success, Ollama failure (confirming `insert_status` stays a sentinel `-1`, distinguishable from any real `vecstore_insert()` code), and Ollama success with a subsequent insert failure (confirming `ollama_status`/`insert_status` genuinely disagree rather than collapsing) — plus a defensive-cap check with a maximum-sized (2048-dimension) reported embedding to confirm no overflow of `VecValues.values[]`.

**Regression: existing suites needed one link-time fix, not a logic change.** `vecstore.c` gaining a real reference to `ollama_embed()` (via the new embed-insert adapter) broke `vecstore_host_test.c` and `vecstore_search_host_test.c`'s linking, since neither previously linked anything Ollama-related — both gained a two-line stub `ollama_embed()` returning failure, exactly mirroring their own established `catalog_check_access()` stub pattern, to satisfy the linker without pulling in TCP machinery neither suite's actual scope needs. After that fix, both were re-run and still pass all 39 and 24 checks respectively, and `ollama_client_host_test.c` (unaffected — it doesn't link `vecstore.c` at all) still passes all 28. No `Makefile` change was needed this phase — `kernel/vecstore.c`, `net/ollama_client.c`, `kernel/syscall_dispatch.c`, and `user/shell.c` were all already build-listed by prior phases.

## 7. Phase 5 — ID-based join back to relational tables

**Scope.**
- A documented pattern (and likely a small helper function, not new SQL grammar) for taking a `vecstore_search()` result's `external_id` list and turning it into a relational lookup — most simply, a caller-side loop issuing `SELECT * FROM <table> WHERE id = <external_id>` per match (or a single `WHERE id IN (...)`-shaped query, if `sql_parser.c`'s grammar already supports an `IN` clause — a real thing to check against the actual parser rather than assume).
- Explicitly out of scope, named rather than deferred silently: extending `sql_parser.c`'s grammar with vector-search syntax (e.g. a `NEAREST TO` clause) — real, useful, future work, but new grammar work this pass doesn't attempt, matching the RDBMS roadmap's own repeated "not syscall/DDL-reachable yet" gaps being named rather than quietly worked around.

**Verification plan.** Real execution — a host test creating both a vector collection and a relational table, inserting related data into both, running a search, and confirming the returned `external_id`s correctly resolve to the right relational rows via a real SQL query against the real `sql_exec.c`.

**Findings addendum (as built).** Landed on a new, third leaf file rather than extending either existing subsystem, confirmed `IN (...)` really is unsupported (an `OR`-chain is the real mechanism), found the actual binding batch-size limit is a predicate-node budget rather than the SQL text-length cap the scope bullet suggested checking, and produced this project's first host test to link the vector store and the full relational stack together in one binary.

**A new file, `kernel/vec_join.c`/`.h` — not an addition to `vecstore.c`, unlike Phase 4's precedent.** Phase 4 put its one cross-subsystem adapter directly inside `vecstore.c`, because `net/ollama_client.h` is a genuinely light dependency. `sql_exec.h` is not: pulling it into `vecstore.c` would have dragged `predicate.h`, `cursor.h`, `mvcc.h`, `rowstore.h`, `row_index.h`, and `persist.h` — and therefore a fake-NVMe backing store — into `vecstore_host_test.c`, `vecstore_search_host_test.c`, and `vecstore_syscall_host_test.c`, none of which need any of that to test `vecstore.c`'s own storage/search logic. So this phase's join-back logic lives in its own new leaf file instead, depending on both `vecstore.h` and `sql_exec.h` while neither of those depends on it — confirmed to work exactly as intended: all three existing `vecstore.c` suites (39, 24, 22 checks) and `sql_exec_host_test.c` (already-passing count unchanged) were re-run after this phase and needed zero changes, not even a new stub, unlike Phase 4's `ollama_embed()` link-time fix.

**`IN (...)` really is unsupported, confirmed by reading the grammar directly — `OR` is the real mechanism, with zero new parser code.** `sql_parser.h`'s own grammar comment and `predicate.h`'s own scope-cut comment both independently name `IN` lists as explicitly out of scope; no `IN` token exists in the lexer's keyword table. `parse_predicate()`/`parse_and_expr()` already support arbitrary `OR`-chains, so `vec_join_resolve()` builds `SELECT * FROM <table> WHERE <id_column> = <id1> OR <id_column> = <id2> OR ...` as ordinary text and calls the real, unmodified `sql_execute()` — the relational engine has no idea a vector search was ever involved, matching this whole roadmap's "vectors joined at the caller boundary, not the SQL grammar" thesis from §0. Zero grammar changes, exactly as scoped.

**The real, binding limit is `PREDICATE_MAX_NODES` (32), not `SQL_MAX_TEXT_LEN` (512) — found by reading the actual node-consumption code, not assumed from the more obvious-looking text limit.** A flat `OR`-chain of *k* equality comparisons consumes `2k-1` predicate nodes (one comparison for the first term, then one comparison plus one `OR`-combinator per remaining term), so `k <= 16` is the real structural ceiling — `predicate_add_comparison`/`_add_and`/`_add_or` return an exhausted-pool error well before 512 characters would ever be reached for realistic table/column name lengths. `VEC_JOIN_MAX_IDS_PER_QUERY = 16` is set from this, and `vec_join_resolve()` independently also checks the actual SQL text buffer has room for one more term before appending it — whichever limit is hit first ends the current batch, with resolution continuing via an additional query for the remainder, rather than assuming one limit always dominates the other.

**Every real, honest edge case named in the header comment was exercised, not just the happy path.** `vec_join_resolve()`'s contract explicitly allows fewer resolved rows than matches (an `external_id` with no corresponding row — not an error) and more (duplicate `external_id`s, either in `matches[]` itself or in `id_column`'s stored values — both cases `vecstore.h` and ordinary relational data already permit, not vecstore's or this file's job to prevent). `id_col_idx` is looked up by name against each query's own `SqlResult.columns[]` (populated for `SELECT *` in real table-column order, confirmed by reading `sql_exec.c`'s `select_all` handling directly rather than assumed) — no positional assumption about which schema field is "the id column" appears anywhere in `vec_join.c` itself.

**Verification: this project's first host test linking `vecstore.c` and the full relational stack together — real execution, exceeding the scope plan's own bar in exactly the way Phase 19's own `sql_exec_host_test.c` did.** `vec_join_host_test.c` (repo root, new) links `kernel/vec_join.c`, `kernel/vecstore.c`, and the complete `sql_exec.c`/`sql_parser.c`/`predicate.c`/`row_index.c`/`rowstore.c`/`persist.c`/`cursor.c`/`mvcc.c`/`row_constraint.c`/`row_journal.c` stack — confirmed by grep before writing this file that no prior host test combines both subsystems — 22 checks, all passing. Creates 5 real employee rows and 5 real embeddings whose `external_id`s equal the employee's own `id` column, placed along one axis at hand-verifiable distances; runs a real `vecstore_search()`; resolves the results via `vec_join_resolve()`; and confirms both the relational data (employee names) AND the original vector-search metadata (distance, in the correct relative order) travel through the resolved pairs correctly together. A second scenario confirms an orphan embedding (`external_id` with no matching employee) resolves to zero delivered pairs without crashing. A third scenario is the one this phase's own batching logic specifically needed proven, not just plausible: 20 more employees + embeddings, searched with `k=20` (exceeding `VEC_JOIN_MAX_IDS_PER_QUERY=16`), forcing `vec_join_resolve()` to issue multiple internal `sql_execute()` calls — all 20 resolved correctly, confirming the multi-batch path works, not just a single query's worth. One real setup-ordering bug surfaced and was fixed during test-writing: `object_catalog_count` must be incremented before `rowstore_create_table()` is called (it locates its own object by scanning `object_catalog[0..object_catalog_count)`), matching `sql_exec_host_test.c`'s own existing ordering exactly — an easy mistake to make once, now named here for the next combined test that needs it.

`kernel/vec_join.c` syntax-checks cleanly in isolation (`gcc -fsyntax-only -Wall -Wextra -std=c11`), zero warnings. `Makefile`'s `X86_C_SRC` gained `kernel/vec_join.c`, placed beside `kernel/vecstore.c`. No syscall/dispatch/shell surface was added this phase — a deliberate scope decision, not an oversight: unlike Phase 4's `SYS_SLS_VEC_CREATE` (a hard blocking gap where three named syscalls were reachable but structurally unusable without it), `vec_join_resolve()` has no equivalent reachability blocker — it's a plain, fully host-testable kernel function, and this section's own scope bullet asked for exactly that ("a documented pattern ... a small helper function"), not a live syscall surface. If a real caller eventually needs `vec_join_resolve()` from the shell or a syscall, that's a real, separate, future decision to make when a real caller needs it — the same "decide the real design question when a real caller needs it" posture this roadmap has applied before (Phase 1's catalog-flag decision, citing the RDBMS roadmap's own `mvcc_txn_is_active()` precedent).

## 8. Phase 6 (stretch, not scoped) — approximate nearest-neighbor indexing

Not scoped in detail here, matching the RDBMS roadmap's identical treatment of cost-based query optimization. Revisit only once brute-force search (Phase 2) is demonstrably too slow for a real workload this project actually runs — not before. HNSW is the most likely candidate given it's the closest fit to AeroSLS's own pointer/object-graph-friendly SLS design philosophy (the original motivating observation for this whole roadmap), but that's a design fork for whenever this phase actually starts, not a decision to pre-make here.

**Findings addendum (as built).** Started on an explicit, direct user request ("let's go for phase 6"), ahead of this section's own stated gate — recorded honestly, not silently: no real workload has ever shown brute-force search to be too slow anywhere in this project. Framed as the exploratory/architectural phase this section's own text already anticipated ("closest fit to AeroSLS's own pointer/object-graph-friendly SLS design philosophy"), not a performance fix for a measured problem.

**A new, third parallel structure — `kernel/vec_index.h`/`.c` — with the same relationship to `vecstore.c` that `row_index.c` has to `rowstore.c`.** A real HNSW graph: multi-layer, random layer assignment, greedy upper-layer descent, `ef`-bounded beam search at layer 0. It stores no vector values of its own — every distance computation re-fetches through `vecstore_get()` (a direct `(page_id, slot_index)` read, not a scan) rather than caching a second copy, a deliberate reuse-over-raw-speed choice matching Phase 2's own O(n·k) top-K precedent. Distance math and top-K insertion (`vs_sqrtf`/`vs_dot`/`vs_distance_cosine`/`vs_distance_l2`/`vs_topk_insert`) were promoted from `static` to exported functions in `vecstore.c`/`.h` specifically so the approximate and exact search paths share bit-identical math — correctness-critical for this phase's own recall comparison, and a deliberate, narrow exception to this codebase's usual "each file keeps its own hand-rolled copies" convention, named as such in `vecstore.h`'s own comment at the point it happens.

**A single shared, bump-allocated node pool, not a naive per-index array — corrected mid-design once row_index.h's own precedent was reread carefully.** The first draft gave every `VecIndex` its own `VEC_INDEX_MAX_ELEMENTS`-sized node array, eagerly reserved whether or not the index slot was ever used — tens of megabytes of BSS for mostly-empty slots. Rereading `row_index.h`'s own `btree_nodes[BTREE_MAX_NODES]` (one flat pool shared across every B-tree index, not one array per index) caught this before it was built, not after — `vec_index_nodes[VEC_INDEX_MAX_NODES=4096]` is the corrected, shared design, with each node tagged `index_id` only so `vec_index_notify_delete()` can find "the node for VecId X belonging to index Y" via a pool scan; ordinary graph traversal never checks `index_id` at all, since edges only ever connect nodes of the same index by construction.

**Auto-maintenance, mirroring `row_index.c`'s real precedent exactly — and naming a failure class this project hadn't hit before.** `vecstore_insert()`/`vecstore_delete()` now call `vec_index_notify_insert()`/`_delete()` unconditionally after a successful mutation (self-guarding no-ops if no index is defined on that collection), the same "base storage layer calls the optional index layer; the index layer decides whether there's anything to do" shape `rowstore.c` already established for `row_index_notify_insert()`. Named explicitly in `vec_index.h`'s own header comment: an approximate index that silently falls out of sync with its source data is worse than no index — it would return confidently WRONG results while looking healthy, unlike every prior "denial looks like absence" case in this project, which at least failed by omission rather than quiet fabrication. Auto-maintenance from day one is how this phase avoids ever creating that failure mode.

**No `ln()`, no PRNG existed anywhere in this kernel before this phase — both confirmed absent by grep before writing either replacement.** The paper's layer-assignment formula (`floor(-ln(U) * mL)`) needs a natural log this freestanding kernel doesn't have. `vi_random_layer()` uses the equivalent discrete construction instead — flip an M-sided coin via a hand-rolled xorshift32 PRNG (also genuinely new machinery, confirmed via the same audit) and climb one layer per success, capped at `VEC_INDEX_MAX_LAYERS-1` — producing the identical geometric `P(layer >= k) = (1/M)^k` decay via integer arithmetic only.

**Real, named simplifications versus the textbook algorithm, not silent shortcuts.** (1) Neighbor selection keeps the simple M/M0-closest, not the paper's diversification heuristic. (2) Deletion is a tombstone, not a real removal with neighbor re-linking — but named as a MORE severe limitation than `row_index.c`'s own identical-sounding tombstone precedent, not assumed equivalent: a B-tree's routing keys are independent of its leaves' duplicate lists, so deleting a row never affects the tree's ability to route to other rows, while an HNSW node's edges ARE the navigation structure — a node whose underlying vecstore entry was deleted becomes an effectively infinite-distance dead end (`vecstore_get()` fails for a deleted `VecId`), which can fragment the graph or degrade recall for unrelated queries routing through it, not just "can't find the deleted thing." (3) `vec_index_create()` does not backfill an already-populated collection — only vectors inserted afterward, via the auto-maintenance hook, are indexed. (4) No syscall/shell surface this phase, the same deliberate call Phase 5 made and for the identical reason: no reachability blocker forces one.

**Verification: real execution against a genuinely clustered dataset, measuring actual recall — not a plausibility check.** `vec_index_host_test.c` (repo root, new) links the real, unmodified `kernel/vec_index.c` and `kernel/vecstore.c` — crucially, `vecstore_insert()`/`vecstore_delete()` are the REAL versions with the auto-maintenance hooks live, not stubbed, so this test exercises the hook wiring end to end, not just `vec_index.c` in isolation. 24 checks, all passing. 300 points across 6 well-separated clusters (8 dimensions) were inserted via ordinary `vecstore_insert()` calls — never a direct `vec_index` API — into an index created empty beforehand, then compared against `vecstore_search()`'s exact results across 12 queries: **87.5% average recall@10**, individual queries ranging from 10/10 (most clusters) down to 2/10 and 3/10 (both on cluster 5 specifically, the numerically outermost cluster). That spread was investigated, not averaged away: it's the real, predictable consequence of simplification (1) above — the diversification heuristic this phase deliberately skipped exists specifically to preserve long-range "bridge" edges between distant regions of the graph, and cluster 5's lower recall is exactly the kind of local-optimum entrapment that heuristic is designed to prevent. Structural invariants were checked directly (no node's neighbor list exceeds its layer's M/M0 cap; every node has at least one layer-0 edge; the graph genuinely grew past layer 0 with 300 points at M=16). Deletion was verified end to end: a `vecstore_delete()` call is picked up by the auto-maintenance hook, the deleted point never reappears in subsequent approximate search results, and search for an unrelated cluster still returns full results afterward. A second index (cosine metric) was created on the same collection to confirm the shared node pool keeps two graphs genuinely independent via `index_id` tagging — one insert correctly grew both indexes' own, separately-tracked `active_count`s.

**Regression: zero stubs needed, unlike Phase 4's `ollama_embed()` situation.** `vec_index.c`'s only dependencies are `vecstore.h`/`object_catalog.h`/`permissions.h` — genuinely light. All four existing `vecstore.c`-family suites (`vecstore_host_test.c` 39 checks, `vecstore_search_host_test.c` 24, `vecstore_syscall_host_test.c` 22, `vec_join_host_test.c` 22) were rebuilt linking the real `kernel/vec_index.c` directly rather than stubbing the new notify calls, and all passed unmodified — no new stub function needed anywhere, confirming the "light dependency" design choice held in practice, not just in theory. `ollama_client_host_test.c` (28 checks) and `sql_exec_host_test.c` (unrelated regression sanity) were also re-run clean. `kernel.c` gained a `#include` and a `vec_index_init()` call (grouped with `vecstore_init()`, same RAM-only/no-restore-step reasoning), `syscall_dispatch.c` and `user/shell.c` were grep-checked for any stray reference (none) and compile clean with only the same pre-existing, unrelated implicit-declaration noise every prior phase's kernel.c-adjacent check has produced. `Makefile`'s `X86_C_SRC` gained `kernel/vec_index.c`, placed beside `kernel/vecstore.c`.

This closes out every phase this roadmap defined (1 through 6, including the stretch phase). A note for whoever revisits this doc: Phase 6 was built without its own stated trigger condition (brute-force demonstrably too slow) ever being met — if a real workload eventually does hit that wall, this phase's own recall numbers and the cluster-5 finding above are the place to start, not a blank slate.

## 9. Suggested execution order

Phases 1 and 3 have no dependency on each other and could be built in either order or in parallel by design (vector storage doesn't need Ollama to exist; the Ollama client doesn't need vector storage to exist). Phase 2 needs Phase 1. Phase 4 needs 1, 2, and 3 together. Phase 5 needs Phase 4 plus the RDBMS roadmap's own Phase 20. Phase 6 needs 1 and 2 (✅ done) — built ahead of its own "revisit only once too slow" gate on explicit request; see §8's findings addendum.
