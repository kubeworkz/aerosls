# AeroSLS Storage Isolation Roadmap v0.1

## 0. Why this doc exists

The Multitenant Isolation Gap Analysis's §3 table named the gap plainly: *"No partitioning at all. `kernel/persist.c` writes every subsystem's data into one flat, shared LBA space; `kernel/frame_pool.c`'s bitmap is one shared free-list. There is no per-tenant NVMe region, quota, or byte accounting."* That doc's own §7 priority list (item 5, now item 2 after RBAC closed) explicitly deferred a real fix here, calling it "large, foundational for any data-isolation claim; likely deserves its own dedicated roadmap doc the way LPAR and Multi-Node each got one, given its size." This is that doc — scoping only, matching the precedent every other subsystem roadmap here set (LPAR, Multi-Node, VectorStore Interface, SQL Feature Parity): name the real architecture, name the real gap, propose a phased, incrementally shippable plan, and do not write implementation code until a phase is actually being executed.

## 1. What's actually there today (confirmed by direct investigation, not recalled)

Storage in AeroSLS is not one subsystem — it's three, each persisting to NVMe by a different mechanism, and a roadmap that only names `persist.c` (as the gap analysis's one-line summary did) misses two of them:

1. **`persist.c`'s small-struct-array regions.** A hand-curated sequence of fixed LBA regions, one per kernel subsystem (catalog, partitions, databases, tenants, grants, and so on), each documented as a `#define` pair in `kernel/persist.h` (e.g. `PERSIST_TENANT_HDR_LBA 6784`/`PERSIST_TENANT_ENT_LBA 6792`). Every region is a single shared array (`object_catalog[]`, `partition_table[]`, `tenants[]`, ...) written and restored whole, gated only by a magic number and a `sizeof()` check. No region carries a partition_id/tenant_id dimension at all — the whole array belongs to the whole system.
2. **Row-store and vector-store bulk page pools.** `rowstore.h`'s `ROWSTORE_LBA_BASE = 2000000` reserves `ROWSTORE_MAX_PAGES = 262144` pages (~1 GiB) as one flat `page_id -> LBA` space; `vecstore.h`'s `VECSTORE_LBA_BASE = 4200000` reserves `VECSTORE_MAX_PAGES = 65536` pages (~256 MiB) the same way. Neither has any partition awareness in its page-allocation arithmetic — any table or collection, from any tenant, draws from the same shared page pool.
3. **Stream/blob slots.** `stream.h`'s `STREAM_DATA_LBA_BASE = 65536`, `STREAM_MAX = 8` fixed 64 MiB slots. As of the Multitenant Isolation Gap Analysis §10 closure, streams now carry `owner_uid`/`partition_id` tags and their *RAM frame* allocation is quota-checked — but the on-disk LBA slot itself is still just "whichever of the 8 fixed slots is free," no per-tenant reservation.

None of these three carries a compile-time or runtime concept of "how many bytes has partition P consumed on disk" or "which LBA range belongs to partition P." `kernel/frame_pool.c`'s physical-RAM bitmap (`physical_memory_bitmap[TOTAL_FRAMES/64]`, `TOTAL_FRAMES = 1048576`) is the same story one layer down: one flat, shared free-list, scanned linearly with no partition subdivision. `allocate_physical_ram_frame_for_partition()` (LPAR Phase 13) does not change this — it adds a quota *check* (`partition_frame_usage[]`/`partition_frame_quota[]`, two flat arrays local to `frame_pool.c`) before calling the same shared-pool `alloc_raw_frame()`. A partition's frames are therefore scattered arbitrarily across physical RAM, never contiguous or reserved — accounting, not physical isolation. `kernel/tier_mgr.c`'s hot/cold promotion (`SLSObjectEntry.storage_tier`, ticked by `tier_mgr_tick()`) is confirmed to have zero partition/tenant awareness: it iterates `object_catalog[]` globally and reports system-wide byte totals per tier, so one tenant's access pattern can promote or demote another tenant's data with no per-tenant SLA possible.

The disk itself has no in-kernel size constant — the 10 GiB image (`qemu-img create -f raw sls_storage.img 10G`, `Makefile`/`README.md`) is a build-script convention, validated only at runtime against the drive's real NSZE via `nvme_get_capacity_bytes()` (`drivers/nvme_admin.c`). Headroom is real and large: `persist.c`'s own regions end around LBA 6,800; row/vec-store run from 2,000,000 to roughly 4,724,288; against a 20,971,520-sector (10 GiB) image that leaves well over 16 million sectors (~8 GiB) entirely unclaimed above the highest reserved region today.

## 2. The reusable precedent already in the codebase

LPAR Phase 13's frame quota is exactly the shape this roadmap should copy for storage, not reinvent: two flat, `PARTITION_MAX`-sized arrays (`partition_frame_usage[]`, `partition_frame_quota[]`), a setter (`partition_set_frame_quota()`), two getters (`partition_get_frame_usage()`/`_quota()`), quota 0 meaning unlimited (BSS zero-init, no special-case code needed), denial happening cleanly before any allocation side effect, and two syscalls (`SYS_SLS_PARTITION_QUOTA_SET`/`_LIST`) for reachability. This is directly reusable: a parallel `partition_lba_usage[]`/`partition_lba_quota[]` pair, with its own setter/getter/syscalls, can sit beside `persist.c`/`rowstore.c`/`vecstore.c`/`stream.c`'s existing writers without moving or re-laying-out a single existing LBA region — the same "accounting layer bolted onto an unchanged shared pool" posture the frame quota already proved out and that this codebase already trusts (frame_quota_host_test.c has been green since LPAR Phase 13).

One honest caveat inherited from that precedent: frame quotas are accounting-only and do not reclaim physical frames on a plain usage reset (`partition_reset_frame_usage()`) — only the more expensive `partition_reclaim_all_frames()` does real reclaim, called from `partition_destroy()`. An LBA quota built the same way will need the same two-tier design: a cheap accounting reset for normal bookkeeping, and a real "walk this partition's known page/slot allocations and mark them free" reclaim path for `partition_destroy()`, mirroring rather than skipping that lesson.

## 3. What real, physical isolation would require (and why it's out of scope for a first phase)

A tenant genuinely being able to say "my bytes are on a reserved slice of this disk, not interleaved with anyone else's" requires rewriting `rowstore.c`'s and `vecstore.c`'s flat `page_id -> LBA` arithmetic into partition-scoped page pools — each partition getting its own reserved LBA sub-range and its own free-page bitmap within it, the same conceptual jump `frame_pool.c`'s bitmap would need to make to go from "accounted" to "physically isolated." That is a substantially larger redesign than the quota layer above, touches the hottest read/write paths in the SQL and vector engines, and has no existing precedent in this codebase to copy from (unlike the frame quota, which Phase 1 below can lift almost verbatim). It is named here explicitly so it isn't silently lost, but scoped out of Phase 1 for the same reason Multi-Node's own roadmap correctly deferred real cross-node data movement until a dedicated-node tier is actually being sold: don't build the expensive version of a feature before a concrete need for it exists. If AeroSLS ever sells a "dedicated storage" tier as a distinct, physically-true product (as opposed to today's "dedicated partition, shared disk" reality), that is a Phase 2+ undertaking deserving its own design pass once Phase 1's accounting layer has proven the tagging is threaded correctly everywhere it needs to be.

## 4. Proposed phases

1. **Phase 1 — Per-partition LBA quota accounting.** Add `partition_lba_usage[]`/`partition_lba_quota[]` (mirroring `frame_pool.c`'s frame quota exactly), with setter/getter functions and two syscalls. Wire the check-and-increment call into every real disk-page-allocating call site across `rowstore.c`, `vecstore.c`, and `stream.c` (the three subsystems identified in §1), converting "unbounded, unquota'd disk consumption" into "bounded, denied-cleanly-at-quota disk consumption" — the exact same shape of fix Multitenant Isolation Gap Analysis §10 already delivered for stream *frames*, just extended to stream (and rowstore/vecstore) *disk bytes*. This is the concrete, scoped, immediately shippable first step; everything else in this document is context for why it's the right first step, not a prerequisite for it.
2. **Phase 2 — Byte-level usage export.** Once Phase 1's counters exist and are real, expose them via the existing `sys_sls_disk_status()`/tier-totals reporting path (`tier_mgr.c`) broken out per-partition, closing part of the "no usage metering" gap the Multitenant Isolation Gap Analysis's §3 table also named as its own separate, not-yet-scoped item.
3. **Phase 3 (deferred, design-only until a concrete need exists) — real per-tenant LBA sub-ranges.** The rowstore/vecstore page-pool redesign described in §3. Not scoped further in this document; revisit only once a dedicated-storage tier is an actual product decision, per the same discipline the Multi-Node roadmap already modeled for real data movement.

## 5. Open questions for whoever picks up Phase 1

- Should the LBA quota unit be pages, sectors, or raw bytes? Frame quota counts frames (fixed 4 KiB unit); rowstore/vecstore both already have their own fixed page sizes, so a per-subsystem page-count quota (mirroring frame quota's granularity exactly) is likely simpler than a raw-byte quota that has to convert across three different page sizes.
- Does stream storage's existing 8-fixed-64-MiB-slot design even need a quota, or is "one slot per stream, `STREAM_MAX` slots total" already an adequate ceiling until real per-tenant sub-ranges (Phase 3) exist? Worth deciding explicitly rather than defaulting to "add a quota because the other two subsystems got one."
- `PARTITION_MAX = 16` (flagged as a capacity ceiling in the Multitenant Isolation Gap Analysis §4) sizes every one of these flat per-partition arrays. Any Phase 1 implementation inherits that same ceiling by construction — worth resolving the capacity-sizing question (already flagged, not yet actioned) before or alongside Phase 1, since raising `PARTITION_MAX` later means resizing every one of these arrays again.
