# AeroSLS Storage Isolation Roadmap v0.1

## 0. Why this doc exists

The Multitenant Isolation Gap Analysis's §3 table named the gap plainly: *"No partitioning at all. `kernel/persist.c` writes every subsystem's data into one flat, shared LBA space; `kernel/frame_pool.c`'s bitmap is one shared free-list. There is no per-tenant NVMe region, quota, or byte accounting."* That doc's own §7 priority list (item 5, now item 2 after RBAC closed) explicitly deferred a real fix here, calling it "large, foundational for any data-isolation claim; likely deserves its own dedicated roadmap doc the way LPAR and Multi-Node each got one, given its size." This is that doc — scoping only, matching the precedent every other subsystem roadmap here set (LPAR, Multi-Node, VectorStore Interface, SQL Feature Parity): name the real architecture, name the real gap, propose a phased, incrementally shippable plan, and do not write implementation code until a phase is actually being executed.

## 1. What's actually there today (confirmed by direct investigation, not recalled)

Storage in AeroSLS is not one subsystem — it's three, each persisting to NVMe by a different mechanism, and a roadmap that only names `persist.c` (as the gap analysis's one-line summary did) misses two of them:

1. **`persist.c`'s small-struct-array regions.** A hand-curated sequence of fixed LBA regions, one per kernel subsystem (catalog, partitions, databases, tenants, grants, and so on), each documented as a `#define` pair in `kernel/persist.h` (e.g. `PERSIST_TENANT_HDR_LBA 6784`/`PERSIST_TENANT_ENT_LBA 6792`). Every region is a single shared array (`object_catalog[]`, `partition_table[]`, `tenants[]`, ...) written and restored whole, gated only by a magic number and a `sizeof()` check. No region carries a partition_id/tenant_id dimension at all — the whole array belongs to the whole system.
2. **Row-store and vector-store bulk page pools.** `rowstore.h`'s `ROWSTORE_LBA_BASE = 2000000` reserves `ROWSTORE_MAX_PAGES = 262144` pages (~1 GiB) as one flat `page_id -> LBA` space; `vecstore.h`'s `VECSTORE_LBA_BASE = 4200000` reserves `VECSTORE_MAX_PAGES = 65536` pages (~256 MiB) the same way. Neither has any partition awareness in its page-allocation arithmetic — any table or collection, from any tenant, draws from the same shared page pool.
3. **Stream/blob slots.** `stream.h`'s `STREAM_DATA_LBA_BASE = 65536`, `STREAM_MAX = 8` fixed 64 MiB slots. As of the Multitenant Isolation Gap Analysis §10 closure, streams now carry `owner_uid`/`partition_id` tags and their *RAM frame* allocation is quota-checked — but the on-disk LBA slot itself is still just "whichever of the 8 fixed slots is free," no per-tenant reservation.

None of these three carries a compile-time or runtime concept of "how many bytes has partition P consumed on disk" or "which LBA range belongs to partition P." `kernel/frame_pool.c`'s physical-RAM bitmap (`physical_memory_bitmap[TOTAL_FRAMES/64]`, `TOTAL_FRAMES = 1048576`) is the same story one layer down: one flat, shared free-list, scanned linearly with no partition subdivision. `allocate_physical_ram_frame_for_partition()` (LPAR Phase 13) does not change this — it adds a quota *check* (`partition_frame_usage[]`/`partition_frame_quota[]`, two flat arrays local to `frame_pool.c`) before calling the same shared-pool `alloc_raw_frame()`. A partition's frames are therefore scattered arbitrarily across physical RAM, never contiguous or reserved — accounting, not physical isolation. `kernel/tier_mgr.c`'s hot/cold promotion (`SLSObjectEntry.storage_tier`, ticked by `tier_mgr_tick()`) is confirmed to have zero partition/tenant awareness: it iterates `object_catalog[]` globally and reports system-wide byte totals per tier, so one tenant's access pattern can promote or demote another tenant's data with no per-tenant SLA possible.

The disk itself has no in-kernel size constant — the 10 GiB image (`qemu-img create -f raw sls_storage.img 10G`, `Makefile`/`README.md`) is a build-script convention, validated only at runtime against the drive's real NSZE via `nvme_get_capacity_bytes()` (`drivers/nvme_admin.c`). Headroom is real and large: `persist.c`'s own regions end around LBA 6,800; row/vec-store run from 2,000,000 to roughly 4,724,288; against a 20,971,520-sector (10 GiB) image that leaves well over 16 million sectors (~8 GiB) entirely unclaimed above the highest reserved region today.

## 2. The reusable precedent already in the codebase

LPAR Phase 13's frame quota is exactly the shape this roadmap should copy for storage, not reinvent: two flat, `PARTITION_MAX`-sized arrays (`partition_frame_usage[]`, `partition_frame_quota[]`), a setter (`partition_set_frame_quota()`), two getters (`partition_get_frame_usage()`/`_quota()`), quota 0 meaning unlimited (BSS zero-init, no special-case code needed), denial happening cleanly before any allocation side effect, and two syscalls (`SYS_SLS_PARTITION_QUOTA_SET`/`_LIST`) for reachability. This is directly reusable: a parallel `partition_lba_usage[]`/`partition_lba_quota[]` pair, with its own setter/getter/syscalls, can sit beside `persist.c`/`rowstore.c`/`vecstore.c`/`stream.c`'s existing writers without moving or re-laying-out a single existing LBA region — the same "accounting layer bolted onto an unchanged shared pool" posture the frame quota already proved out and that this codebase already trusts (frame_quota_host_test.c has been green since LPAR Phase 13).

One honest caveat inherited from that precedent: frame quotas are accounting-only and do not reclaim physical frames on a plain usage reset (`partition_reset_frame_usage()`) — only the more expensive `partition_reclaim_all_frames()` does real reclaim, called from `partition_destroy()`. An LBA quota built the same way will need the same two-tier design: a cheap accounting reset for normal bookkeeping, and a real "walk this partition's known page/slot allocations and mark them free" reclaim path for `partition_destroy()`, mirroring rather than skipping that lesson.

## 3. What real, physical isolation would require (and why it's out of scope for a first phase)

A tenant genuinely being able to say "my bytes are on a reserved slice of this disk, not interleaved with anyone else's" requires rewriting `rowstore.c`'s and `vecstore.c`'s flat `page_id -> LBA` arithmetic into partition-scoped page pools — each partition getting its own reserved LBA sub-range and its own free-page bitmap within it, the same conceptual jump `frame_pool.c`'s bitmap would need to make to go from "accounted" to "physically isolated." That is a substantially larger redesign than the quota layer above, touches the hottest read/write paths in the SQL and vector engines, and has no existing precedent in this codebase to copy from (unlike the frame quota, which Phase 1 below can lift almost verbatim). It is named here explicitly so it isn't silently lost, but scoped out of Phase 1 for the same reason Multi-Node's own roadmap correctly deferred real cross-node data movement until a dedicated-node tier is actually being sold: don't build the expensive version of a feature before a concrete need for it exists. If AeroSLS ever sells a "dedicated storage" tier as a distinct, physically-true product (as opposed to today's "dedicated partition, shared disk" reality), that is a Phase 2+ undertaking deserving its own design pass once Phase 1's accounting layer has proven the tagging is threaded correctly everywhere it needs to be.

## 4. Proposed phases

1. **[Closed for rowstore+vecstore — see §6] Phase 1 — Per-partition LBA quota accounting.** Add `partition_lba_usage[]`/`partition_lba_quota[]` (mirroring `frame_pool.c`'s frame quota exactly), with setter/getter functions and two syscalls. Wire the check-and-increment call into every real disk-page-allocating call site across `rowstore.c`, `vecstore.c`, and `stream.c` (the three subsystems identified in §1), converting "unbounded, unquota'd disk consumption" into "bounded, denied-cleanly-at-quota disk consumption" — the exact same shape of fix Multitenant Isolation Gap Analysis §10 already delivered for stream *frames*, just extended to stream (and rowstore/vecstore) *disk bytes*. This is the concrete, scoped, immediately shippable first step; everything else in this document is context for why it's the right first step, not a prerequisite for it.
2. **[Closed — see §7] Phase 2 — Byte-level usage export.** Once Phase 1's counters exist and are real, expose them via the existing `sys_sls_disk_status()`/tier-totals reporting path (`tier_mgr.c`) broken out per-partition, closing part of the "no usage metering" gap the Multitenant Isolation Gap Analysis's §3 table also named as its own separate, not-yet-scoped item.
3. **[Closed — see §8] Phase 3 (was: deferred, design-only until a concrete need exists) — real per-tenant LBA sub-ranges.** The rowstore/vecstore page-pool redesign described in §3. A dedicated-storage tier became an actual product decision, so this phase was implemented rather than left design-only — see §8 for what was actually built.

## 5. Open questions for whoever picks up Phase 1

- Should the LBA quota unit be pages, sectors, or raw bytes? Frame quota counts frames (fixed 4 KiB unit); rowstore/vecstore both already have their own fixed page sizes, so a per-subsystem page-count quota (mirroring frame quota's granularity exactly) is likely simpler than a raw-byte quota that has to convert across three different page sizes.
- Does stream storage's existing 8-fixed-64-MiB-slot design even need a quota, or is "one slot per stream, `STREAM_MAX` slots total" already an adequate ceiling until real per-tenant sub-ranges (Phase 3) exist? Worth deciding explicitly rather than defaulting to "add a quota because the other two subsystems got one."
- `PARTITION_MAX = 16` (flagged as a capacity ceiling in the Multitenant Isolation Gap Analysis §4) sizes every one of these flat per-partition arrays. Any Phase 1 implementation inherits that same ceiling by construction — worth resolving the capacity-sizing question (already flagged, not yet actioned) before or alongside Phase 1, since raising `PARTITION_MAX` later means resizing every one of these arrays again.

## 6. Findings addendum: Phase 1 — per-partition on-disk page quota (rowstore + vecstore) — closed; stream deliberately excluded

Implemented: `kernel/storage_quota.h`/`.c`, a new, genuinely standalone subsystem mirroring `frame_pool.c`'s LPAR Phase 13 frame-quota shape exactly — `partition_page_usage[PARTITION_MAX]`/`partition_page_quota[PARTITION_MAX]`, `storage_page_reserve(partition_id)` (quota-checked, denies before any side effect), `storage_page_release(partition_id, count)` (accounting-only, floors at 0), `storage_set_page_quota()`/`storage_get_page_usage()`/`storage_get_page_quota()` (0 = unlimited, sentinel `0xFFFFFFFFFFFFFFFF` for out-of-range), and syscalls `SYS_SLS_PARTITION_STORAGE_QUOTA_SET`/`_LIST` (275–276, confirmed as the next free numbers via a fresh grep before picking them).

**§5's first open question, resolved: ONE combined budget, not two.** `rowstore.c` and `vecstore.c` share a single per-partition page counter rather than two separate quota pools. Both subsystems use an identical 4 KiB page size (`ROWSTORE_PAGE_SIZE == VECSTORE_PAGE_SIZE == 4096`, confirmed by direct read of both headers), so a single "disk pages consumed" counter per partition is simpler than two parallel tables and mirrors how `frame_pool.c` already tracks one flat "frames consumed" number regardless of whether they went to process code, stack, or ELF segments — the resource being protected is "disk space," not "rowstore space" and "vecstore space" as two things a customer would think about separately. This composability isn't a separate integration test's job to prove: `storage_page_reserve()`/`_release()` are keyed purely by `partition_id`, with no knowledge of which subsystem is calling, so any two callers passing the same `partition_id` are, by construction, sharing one counter.

**§5's second open question, resolved: stream storage gets NO quota in this phase.** `STREAM_MAX` (8) fixed 64 MiB on-disk slots is already a hard, small, system-wide ceiling on its own, and stream RAM frame allocation is already quota-checked separately (Multitenant Isolation Gap Analysis §10). A per-partition quota on top of an 8-slot total would be redundant bookkeeping for no real additional protection today — revisit only if `STREAM_MAX` is ever raised enough that "one tenant claims every slot" becomes a real risk again.

**The mechanism.** `rowstore_alloc_page()` and `vecstore_alloc_page()` (previously parameterless bump allocators) now each take `partition_id` and call `storage_page_reserve(partition_id)` as the very first step — before the shared cursor advances or any RAM frame is touched. Both call sites resolve `partition_id` from the real `object_catalog[idx].partition_id` of the table/collection being written to (2 call sites in `rowstore.c`: `rowstore_add_column()`'s migration path and `rowstore_row_insert()`; 1 in `vecstore.c`: `vecstore_insert()`). On the rare case a reservation succeeds but the subsequent `allocate_physical_ram_frame()` call fails, both allocators call `storage_page_release(partition_id, 1)` to roll back the reservation before returning the existing invalid-page sentinel — the counter never drifts out of sync with what was actually consumed.

**Reachability.** `user/shell.c` gained `partition storagequota set <id> <pages>` / `partition storagequotas` commands, mirroring the existing `partition quota`/`partition quotas` pair's shape exactly. `net/http.c` gained `GET /api/partition/storagequotas` (lists every partition with nonzero usage or quota) and `POST /api/partition/storagequota` (body `{"partition_id": N, "page_quota": N}`), mirroring the existing frame-quota HTTP pair's handler structure.

**Verification.** New `tests/storage_quota_host_test.c` — 9 scenarios covering `storage_quota.c`'s own contract in isolation (unlimited-by-default, real quota enforcement with denial-before-side-effect, independent per-partition counters, out-of-range fail-closed everywhere, accounting-only release floored at 0, the reserve-then-rollback pattern, the syscall wrapper, explicit re-uncap via quota=0, and the debug listing), 45 checks, all real execution against the unmodified `kernel/storage_quota.c`. Extended `tests/rowstore_host_test.c` (new scenario 7b) and `tests/vecstore_host_test.c` (new scenario s15) with real end-to-end wiring proof: a quota-denied insert leaves the shared bump-allocator cursor/`page_count` completely untouched and the partition's usage counter unchanged, while a within-quota insert consumes exactly one page and is visible through `storage_get_page_usage()` — proving each subsystem's real call site threads the right `partition_id` through, not just that `storage_quota.c` itself works.

31 pre-existing host test files that link the real `kernel/rowstore.c` or `kernel/vecstore.c` (but not `kernel/storage_quota.c`) needed their own "Build and run:" link commands updated to add `kernel/storage_quota.c` — a new, real, load-bearing dependency now, not a stub, since `storage_quota.c`'s own dependency surface (`kernel_io.h`'s two logging functions) is exactly as shallow as `frame_pool.c`'s own, and every one of those 31 files already stubs that surface. `Makefile`'s `X86_C_SRC` list updated with `kernel/storage_quota.c`.

Full regression: `tests/run_all.sh`, 56/56 host tests passing (55 pre-existing + this phase's new `storage_quota_host_test.c`), zero regressions. All touched/new files (`kernel/storage_quota.h`, `kernel/storage_quota.c`, `kernel/rowstore.c`, `kernel/vecstore.c`, `kernel/syscall_dispatch.c`, `user/shell.c`, `net/http.c`) compile clean under the real kernel cross-compile flags (`gcc -ffreestanding -fno-stack-protector -fno-pic -m32 -I . -I kernel -I drivers -I user -fsyntax-only`), zero new warnings beyond this codebase's pre-existing freestanding-environment implicit-declaration set.

**What this does and does not close.** "Unbounded, unquota'd disk consumption" (§1/§3's original finding) is now bounded and cleanly denied for rowstore and vecstore — the two subsystems that account for essentially all real growable table/collection data. Physical isolation (§3's "my bytes are on a reserved slice of this disk" claim) is unchanged: a partition's pages are still scattered arbitrarily within the shared LBA pools, never contiguous or reserved — this phase is accounting, not physical separation, exactly as scoped. Phase 2 (byte-level usage export via `tier_mgr.c`/`sys_sls_disk_status()`) and Phase 3 (real per-tenant LBA sub-ranges) remain open, Phase 3 deliberately deferred until a dedicated-storage tier is an actual product decision.

**Status:** Phase 1 closed for rowstore + vecstore (the combined budget); stream storage explicitly, deliberately excluded per the resolved open question above. Phase 2 now closed too — see §7. Phase 3 remains open.

## 7. Findings addendum: Phase 2 — byte-level usage export via the existing disk-status reporting path — closed

**The design fork, resolved.** `kernel/usage_metering.h`'s own header comment invites a future disk-byte accounting phase to "extend `usage_table[]` with the new field the same way, not create a second, parallel metering mechanism." That guidance was written before Phase 1 existed. As of Phase 1, `kernel/storage_quota.c`'s `partition_page_usage[]`/`partition_page_quota[]` are *already* the single, real source of truth for on-disk page consumption — copying that data into a second array (`struct SLSUsageEntry`, `usage_table[]`) would itself be exactly the "second, parallel metering mechanism" the comment warns against, just one layer further removed from the truth. `net/http.c`'s `api_usage_report()` already established the alternative pattern this phase follows instead: `frames_now` is a live gauge, computed at request time via `partition_get_frame_usage()` and merged into the `/api/usage` JSON, without ever being stored as a field on `struct SLSUsageEntry`. Phase 2 does the byte-level export the same way — a live read of `storage_quota.c`'s existing counters, converted to bytes, with no new persisted state and no new syscall (`storage_get_page_usage()`/`storage_get_page_quota()` were already directly callable).

**Where it landed.** The roadmap's own wording ("expose them via the existing `sys_sls_disk_status()`/tier-totals reporting path, broken out per-partition") named the destination precisely, so Phase 2 landed there rather than on `/api/usage`: `kernel/tier_mgr.c`'s `sys_sls_disk_status()` gained a new "Per-Partition Storage Usage" section, printed after the existing system-wide tier table, listing every partition with nonzero disk usage or a configured quota (bytes = pages × 4096, same skip-if-both-zero rule `sys_sls_partition_storage_quota_list()` already uses). `net/http.c`'s `api_disk_json()` (the JSON twin behind `GET /api/disk`) gained a matching `"partitions"` array with `partition_id`/`disk_bytes_used`/`disk_bytes_quota` per entry.

**No new syscall, no struct changes.** `SYS_SLS_DISK_STATUS` (253) is reused as-is — `sys_sls_disk_status()`'s existing call site now just does more. `struct SLSUsageEntry`/`usage_table[]` (`kernel/usage_metering.h`/`.c`) are completely untouched by this phase.

**Verification.** `tests/tier_capacity_phase5b_host_test.c` (the existing host test for `tier_capacity_totals()`) picked up `kernel/storage_quota.c` as a new real link-time dependency, since `sys_sls_disk_status()` now calls into it from the same translation unit. New scenario 5 configures a quota on one partition and unquota'd usage on another, then calls the real `sys_sls_disk_status()` as a smoke test proving the new code path links and runs to completion — the same "no serial-capture mechanism exists, so this is a smoke test" posture `storage_quota_host_test.c`'s own scenario 9 already established for `sys_sls_partition_storage_quota_list()`. `net/http.c` is not host-testable directly (too heavy a dependency graph, per `tests/http_rate_limit_host_test.c`'s own header comment), so `api_disk_json()`'s new `"partitions"` array was instead verified by a clean `gcc -fsyntax-only` pass with zero new warnings around the touched code.

Full regression: `tests/run_all.sh`, 56/56 host tests passing, zero regressions.

**Status:** Phase 2 closed. Phase 3 (real per-tenant LBA sub-ranges) remains open, deliberately deferred until a dedicated-storage tier is an actual product decision.

## 8. Findings addendum: Phase 3 — real per-partition page sub-ranges (rowstore + vecstore) — closed

A dedicated-storage tier became an actual product decision, so Phase 3's design-only deferral (§3, §4 item 3) ended and this phase was implemented for real.

**The mechanism.** The page_id space each subsystem already reserves (`ROWSTORE_MAX_PAGES=262144`, `VECSTORE_MAX_PAGES=65536`) is split into `PARTITION_MAX` (16) equal, contiguous, compile-time-fixed sub-ranges: partition p's rowstore pages live in `[p × ROWSTORE_PAGES_PER_PARTITION, (p+1) × ROWSTORE_PAGES_PER_PARTITION)` — 16384 pages (64 MiB) per partition — and the identical shape for vecstore at 4096 pages (16 MiB) per partition. Both constants divide `PARTITION_MAX` exactly today, so nothing is wasted. `rowstore_alloc_page()`/`vecstore_alloc_page()` now bump a NEW per-partition cursor (`rowstore_partition_cursor[]`/`vecstore_partition_cursor[]`, `kernel/rowstore.c`/`kernel/vecstore.c`) confined to the caller's own sub-range, instead of one flat shared cursor — a page physically CANNOT be allocated outside its owning partition's slice anymore, closing §3's own "my bytes are on a reserved slice of this disk" gap for real, not just as accounting.

**What happened to the old flat cursor.** `rowstore_next_free_page_id`/`vecstore_next_free_page_id` were NOT removed or retyped — every existing caller (persist.c's restore block, `rowstore_load_page()`/`vecstore_load_page()`, the row/vector get-update-delete boundary checks, and ~25 host tests' own stub declarations) only ever needed them as a defensive "has this page_id been allocated by anyone" upper bound, never partition attribution. They're kept as a running high-water mark across all partitions' own cursors, preserving every one of those call sites' contracts completely unchanged — a deliberate choice to keep this phase's blast radius to "add a new array and a new bounds check," not "retype a widely-referenced global."

**Quota interaction, an intentional new fact.** Storage Isolation Phase 1's `storage_quota.c` page_quota is a soft, admin-configurable ceiling; a partition's physical sub-range is now a second, hard, structural ceiling underneath it. A Phase 1 quota set above a partition's physical sub-range capacity is not an error — it simply never binds, because the sub-range boundary is always the tighter of the two in that case. Documented in `rowstore.h`'s own header comment, not left implicit.

**Persistence.** `rowstore_partition_cursor[]`/`vecstore_partition_cursor[]` are the one genuinely new piece of state this phase needed to survive a reboot correctly — without it, a restored boot would forget how far each partition's own sub-range had been consumed and could re-issue an already-live page_id. Two new small array regions (`PERSIST_ROWSTORE_PARTCURSOR_LBA`/`PERSIST_VECSTORE_PARTCURSOR_LBA`, `kernel/persist.h`) were added after the last existing region (LBA 6816/6824) rather than squeezed into any existing region's approximate frame-count gap, for the same reason every prior addition to that file picked the list's end: no risk of miscalculating an existing region's exact size. A one-way format-version marker (`PERSIST_STORAGE_ISOLATION_PHASE3_MARK`) rides in the existing rowstore/vecstore header frame's previously-always-0 v2 field, so a restore can tell whether the snapshot being read was written by Phase-3-aware code; a pre-Phase-3 snapshot's page ownership cannot be reattributed after this change (no migration path), so its absence correctly falls back to every partition's cold-start sub-range start instead of trusting raw bytes nothing ever wrote. Same accepted one-way-format-change trade-off this codebase's own prior phases documented explicitly (e.g. SQL Feature-Parity Roadmap Phase 4's row-width change).

**Verification.** New scenario 9 in `tests/rowstore_host_test.c` (13 new checks) and new scenario 16 in `tests/vecstore_host_test.c` (9 new checks) prove, against the REAL unmodified allocators: two different partitions' pages land in disjoint, non-overlapping sub-ranges; one partition's own sub-range can be fully exhausted while a completely different partition allocates normally and unaffected (the actual physical-isolation guarantee, not just two independent quota counters); and the new per-partition cursor arrays survive a real `persist_rowstore_headers()`/`persist_restore_all()` round trip via the new PARTCURSOR LBA regions and format-version gate. ~24 pre-existing host test files that stub `rowstore_next_free_page_id`/`vecstore_next_free_page_id` (because they link one subsystem's real `.c` file and persist.c, but not the other subsystem's `.c` file) needed a matching new stub line added for the new array, the same "new load-bearing symbol, not a stub-avoidable one" posture Phase 1's own 31-file fixup established. `tests/rowstore_host_test.c`'s pre-existing scenario 7 (page-pool exhaustion) was updated from directly manipulating the now-mostly-symbolic flat cursor to manipulating the correct per-partition cursor, since the exhaustion check itself moved.

Full regression: `tests/run_all.sh`, 56/56 host tests passing, zero regressions. All touched/new files (`kernel/rowstore.h`, `kernel/rowstore.c`, `kernel/vecstore.h`, `kernel/vecstore.c`, `kernel/persist.h`, `kernel/persist.c`) compile clean under a freestanding syntax-only check, zero new warnings beyond this codebase's pre-existing freestanding-environment implicit-declaration set.

**What this does and does not close.** §3's "my bytes are on a reserved slice of this disk, not interleaved with anyone else's" claim is now true for rowstore and vecstore: a partition's pages are confined to its own reserved sub-range by construction, not merely accounted for. What remains unchanged: sub-range sizes are fixed and equal per partition today (no per-tenant differently-sized reservations), and `PARTITION_MAX=16` still sizes every array here exactly as it does everywhere else in this codebase (the Multitenant Isolation Gap Analysis §4 capacity-sizing question, still open, still a business input not actioned here). Stream storage remains deliberately excluded, unchanged from Phase 1's own resolved open question.

**Status:** All three phases of this roadmap are now closed.
